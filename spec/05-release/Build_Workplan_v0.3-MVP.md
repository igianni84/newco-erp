## §1 Workplan posture — the three sequencing positions (carried from v1.1, re-baselined)

**Position 1 — Architecture is Phase 1, before any module coding.** Unchanged. Every subsequent phase inherits the architectural decisions made here; the cost of revisiting them after Phase 2 ships is high. Phase 1 ends with a signed-off Architecture Decision Record, a working dev environment, the frontend platform shell deployed, and a "hello world" service exercising the chosen stack end-to-end through CI/CD. **(The stack itself is a dev-team Phase-1 decision — DEC-073 — not pre-decided here.)**

**Position 2 — Backend modules are sequenced in five phases by data dependency.** Unchanged order: foundations (Catalog + Parties) → commercial primitives (Allocation + Procurement) → commerce surface → inventory + fulfilment → financials. Each phase produces primitives the next consumes. Within a phase, modules with no inter-dependency build in parallel; **the only intra-phase ordering constraint is in Phase 5, where Inventory (Module B) must come before Fulfilment (Module C)** because Fulfilment reads physical state from Inventory at every pick. **This Phase-5 ordering is the locus of the launch's most delicate floor-sequencing constraint — the build-sequencing flag (below).**

**Position 3 — Integrations and frontend run in parallel with the backend phase that consumes them, not after.** Unchanged. Each third-party integration is wired inside the phase that owns the consuming module (sanctions + catalog-enrichment in Phase 2; warehouse + WMS + blockchain + NFC in Phase 5; payment + accounting in Phase 6). Frontend builds progressively from Phase 2, lighting up surfaces as their backend dependencies land — so by the end of Phase 6 the customer journey is functionally complete, not stub. **Re-baseline note: the Admin Panel is a first-class frontend build target at every phase** (the manual-first MVP made it the load-bearing operational surface — D24 / Architecture §2.3 — not the implicit DEC-083 parity mirror v1.1 assumed).

**Four consequences worth naming up front** (the first two carried from v1.1; the last two are the MVP carries):

- **Payment is mocked in Phase 4, real in Phase 6.** The Commerce phase (Phase 4) mocks the payment provider; the customer journey is testable end-to-end with a mock through Phase 4–5; it goes "real money" at Phase 6. Unchanged.
- **NFT-related code is decoupled AND feature-flagged in production through the entire build (D12).** Re-baselined from v1.1's "feature-flagged until the blockchain expert review closes": at launch the **serialization workflow is launch-ready** (NFC tag + serial + SerializedBottle ledger + Logilize), while the **NFT mint/burn + on-chain layer is decoupled** — `nft_reference` is nullable + back-fillable, and the **non-serialized (NS) path is the universal fallback** (every downstream degrades gracefully). The code is built in Phase 5; the production **unflag waits for the EXT-1 blockchain-expert review to close** (a Phase-9 gated external workstream — schedule the review early, §4). DECOUPLE ≠ DEFER — the value proposition is preserved.
- **⚠️ The build-sequencing flag (Phase C item G/Q5 — the load-bearing MVP carry).** The no-overselling floor is whole **at the integrated launch** *provided this workplan sequences **Module B's floor artefacts — the Layer-2 push pipeline, InboundBatch, StockPosition, and per-sub-pool ATP — to be integration-ready by the integrated launch, not as a post-launch follow-on.*** Module A's Layer 1 (Phase 3), Module S's lesser-of storefront read (Phase 4), and Module C's no-oversell-at-pick (Phase 5) all depend on Module B's side (Phase 5) being live. The build *phases* (A/D=3, S=4, B/C=5) are an implementation sequence **within ONE coherent launch** (no piecemeal launch-staging), so the phase-3/4 consumers of B's ATP build against scaffolds (Phase 3 ATP-cache scaffold; Phase 4 physical-layer scaffold) that are **concretised when B lands in Phase 5**. This is a **sequencing confirmation, not a cut** — and it is the single most delicate floor item the dev-team sizing exercise must protect (Phase 5; §4).
- **Operations are manual-first at launch (GOV-1).** The back-office automation defers; the **manual-first operator consoles are an in-scope launch build** (the finance-ops console, the shared Logilize discrepancy queue, the white-glove quote flow, the returns/recall consoles, the stocktake/quarantine/adjustment surfaces, the procurement/discrepancy surfaces). The recording is whole at launch; the engines are the post-launch backlog (Phase 9 / roadmap #12).

What follows is **nine phases**, unchanged in shape from v1.1. Phases 1–6 are the build (re-baselined to the launch scope). Phase 7 hardens, builds the launch-minimum customer-support floor, and clears the pre-launch QA gates. Phase 8 is go-live. Phase 9 is the post-launch operating posture **+ the post-launch build backlog** (the deferred set → roadmap #12).

## §2 Phase-by-phase workplan (the nine phases, re-baselined to the launch scope)

### Phase 1 — Architecture and engineering foundations

**What this phase produces**: the locked architectural blueprint, a working dev environment, CI/CD, the observability + audit substrate, the frontend platform shell, and a "hello world" service running through the chosen stack end-to-end. No production functionality. The foundation every later phase inherits. **Unchanged from v1.1** except the explicit MVP carries below.

**Key decisions to make** *(carried as dev-team Phase-1 decisions — DEC-073; not pre-decided here)*: backend stack (language, framework, repo structure, service decomposition); persistence (engine, migration discipline, transaction patterns); event substrate (bus/queue, ordering + idempotency, audit storage); auth (identity provider, sessions, the admin-configurable role-and-permission model); secrets management; observability; CI/CD; hosting (EU data-residency constraint); the **multi-currency dual-record schema pattern** (every financial event records customer-facing amount, EUR equivalent, FX rate, source, fetch timestamp — D18 FLOOR, full pattern); the **i18n schema pattern** (per-locale content + fallback — D2, six-locale-ready); the **audit pattern** (every state transition + financial event captured for 10-year retention — floor); the PCI-scope boundary; the frontend platform; the module-build template. **MVP carries:** **(i)** the **feature-flag infrastructure** that will gate the NFT/on-chain surfaces through the build (D12) is a Phase-1 platform concern; **(ii)** the **naming cascade** (Product spine — `Bottle Reference → Product Reference`; `Wine*/BottleReference* → Product*`; Module 0 §18 is source-of-truth) is the canonical naming the module-build template adopts; **(iii)** the **`actor_role` audit envelope** (every operator action carries actor_role + identity + timestamp + action + entity reference — the Admin-Panel arm of the audit floor) is a Phase-1 audit-pattern concern.

**Subtasks**: architecture working sessions landing each decision as an ADR; stand up source control, CI/CD, test harness, observability, secrets; deploy a "hello world" service exercising DB + event bus + audit trail + pipeline; write the module-build template + integration-test pattern; stand up the **frontend platform shell** (with the Admin Panel as a first-class app, not an afterthought — §1 Position 3) with authentication wired; light-touch security review of the architecture (deep review is Phase 7).

**Frontend**: platform shell only — authentication wiring, design-system foundation, build chain, deployment; **the Admin-Panel app shell stood up as a first-class surface** (it will carry the manual-first operator load — Architecture §2.3).

**Tests**: the CI/CD pipeline exercised by the "hello world" service; its test suite demonstrates the standard patterns.

**Signoff**: every architectural decision in a signed-off ADR; dev environment functional (every team member can build/test/deploy); "hello world" running in a production-like environment with full observability; module-build template + test patterns documented; frontend platform shell (incl. the Admin-Panel app shell) deployed with authentication.

**Output / handoff**: a locked architectural foundation. Phase 2 begins with clear conventions for how every module is built, tested, deployed.
