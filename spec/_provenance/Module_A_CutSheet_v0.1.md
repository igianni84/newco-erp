## §1 Verdict & counts

**Module A is KEPT on its full supply-side primitive — the single Allocation entity, its 4-state lifecycle, the Club/Discovery visibility partition, the unified commercial-terms structure, the sub-pool partition, and (load-bearing) its half of the two-layer no-overselling guard.** Module A sits *inside* the scope floor: it is named in the core-loop floor ("allocation published") and it owns Layer 1 of the no-overselling guard plus the committed-inventory-protection release primitive. Three facts converge to make this — like Module 0 and Module K — a near-full KEEP, with the headline cut **forwarded downstream**, not taken here:

1. **D7 (the kickoff's nominated "primary cut" for Module A) is a Module S cut, not a Module A cut.** The multi-producer atomic-bind logic — "ONE composite Offer reading N constituent Allocations atomically" — lives entirely in **Module S** (DEC-097; rollback-on-issuance per DEC-179; both Wave 3 Module S decisions). Per DEC-061, each constituent of a multi-producer Discovery composite settles to its own producer through its **own single-producer Allocation** at that allocation's per-unit cost `C_i`. Module A's job is therefore *only* to be the per-constituent supply primitive — which it already is, via the single-Allocation entity (DEC-075) + the two-FK `producer_id`/`supplier_id` pattern (DEC-082). **Deferring multi-producer atomic composites removes nothing from Module A**; single-producer offers (club single-producer mixed-case per DEC-019; single-Allocation Discovery offers) ship at launch unchanged. Module A's only D7 obligation is to *preserve the seam* — keep the single-Allocation entity referenceable as one of N constituents and keep per-constituent commercial-terms — which is intrinsic to its design. **The substantive D7 cut lands in the Module S cut-sheet** (exactly as the substantive D8 cut landed in Module S from Module K).

2. **D10 (Allocation) is locked KEEP and is already heavily trimmed in v1.1.** The single Allocation entity (subtypes/split-tables rejected, DEC-075), the 2-value visibility enum (`BOTH` dropped, DEC-076), the sub-pool partition (DEC-080), the harmonized operator-publish activation (DEC-183, which collapsed the Direct-Purchase-specific FSM trigger into a uniform one), and the ATP cache (DEC-187) are all pre-trimmed. Nothing in the Allocation core is heavier than the supply primitive the core loop requires.

3. **Module A carries a piece of the scope floor.** Layer 1 of the two-layer no-overselling guard (`qty − issued ≥ 0`, DEC-099), the per-sub-pool ATP composition with Module B's Layer 2 (DEC-185/187/196), and the `VoucherCancelled` committed-inventory-protection release primitive (DEC-099 + Q-CL-6) are all named-floor in MVP-plan §3. None can be deferred or simplified at the product-spec layer.

The genuine launch-scope reductions that *touch* Module A are thin and **carry zero backend/spec cut**: **(a)** Direct Purchase (`sourcing_model = direct_purchase`) is **deferred at launch** per D11 — but DEC-183 already harmonized activation so deferring it removes essentially no Module A machinery; the enum value + uniform FSM stay as the seam, and the substantive Direct-Purchase deferral is a **Module D** matter (Q2). **(b)** Under **L-PP**, every Module A allocation operation is **operator-driven via the Admin Panel** at launch; the producer-facing *write* UIs are deferred — but DEC-083 admin-parity already makes the Admin Panel functionally complete, so **no backend capability is cut** (Q3). Module A has **zero** retained producer writes at launch (the one retained producer write platform-wide — membership approve/decline — is a Module K surface, not Module A).

## §3 Module-specific changes

Four threads carry the Module A MVP beyond a plain KEEP: the D7 forwarding finding, the D11 Direct-Purchase treatment, the L-PP producer-write treatment, and the naming cascade.

### §3.1 D7 forwarding — the headline cut is taken in Module S, not Module A (the kickoff §9 mandate)

The kickoff nominated D7 (defer multi-producer atomic composites) as Module A's "primary cut" and directed an active hunt to "confirm the multi-producer atomic-bind logic can cleanly DEFER while single-producer composite offers ship, and name the seam precisely." Findings:

1. **The atomic-bind logic is not in Module A.** Multi-producer Discovery composite publication is "**ONE composite Offer reading N Allocations atomically**" — a **Module S** entity-and-binding decision (DEC-097, Wave 3 Module S Q-CL-12), with atomic transactional rollback at voucher issuance also in Module S (DEC-179, Stage 4 Q-AD-12). Module A carries **no** composite-binding attribute, event, or operation.
2. **Per-constituent settlement is per-Allocation.** Per DEC-061, each constituent of a multi-producer composite settles to its own producer through its **own single-producer Allocation** at that allocation's per-unit cost `C_i`; NewCo's margin = `P_d − Σ C_i`. The composite is an N:M Offer-over-Allocations construct (DEC-095/099) — Module A just holds the N constituent rows.
3. **The seam is intrinsic to Module A's design** (P1): the single-Allocation entity (A.1) is referenceable as one of N constituents; the two-FK pattern (A.2) already supports a Discovery Supplier spanning N producers; `commercial_terms` (A.5) already carries per-constituent `C_i`. **Nothing must be added or preserved specially — the seam is the supply primitive itself.**
4. **Single-producer offers ship at launch unchanged.** Club mixed-cases are single-producer by definition (DEC-019); single-Allocation Discovery offers use the same two-FK substrate. The D7 defer touches only Module S's composite-Offer construct.

**Verdict:** Module A needs **no D7 cut**. The substantive D7 decision (defer multi-producer atomic composite Offers; ship single-producer) is **forwarded to the Module S cut-sheet**, where the composite-Offer entity, the atomic N-allocation bind (DEC-097), and the issuance rollback (DEC-179) live. This is the exact mirror of Module K → Module S forwarding for D8. Confirm framing in Q1.

### §3.2 D11 Direct-Purchase treatment — deferred at launch, free for Module A

D11 locked **KEEP both passive-consignment variants (V1 + V2); DEFER Direct Purchase.** Module-A-side treatment:

- **At launch, no `direct_purchase` allocations are created.** Direct Purchase is the fallback sourcing model (used when passive isn't on offer); whether it is needed at launch is **deal-dependent** (D11 is a Paolo/market call — Q2).
- **The deferral is essentially free for Module A** because DEC-183 already harmonized activation: V1/V2/Direct Purchase all transition DRAFT → ACTIVE via the same operator-publish trigger.
