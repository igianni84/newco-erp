## §1 Verdict & counts

**Module B is the inventory-authority + digital-provenance layer — KEPT WHOLE on its integrity floor, with the two heaviest critical-path levers of the whole exercise landing in-module: D12 (DECOUPLE the digital-provenance pipeline off the launch critical path) and D16 (SIMPLIFY the Stage-8 workflow sophistication, keeping the integrity core).** The no-overselling / committed-inventory / InboundBatch / StockPosition / four-way-reconciliation floor stays non-negotiably whole; the cuts land on the *critical-path dimension* (D12) and the *workflow-automation dimension* (D16), not on the integrity core. Six facts shape the verdict:

1. **D12 (NFT/blockchain + NFC) is a DECOUPLE, not a defer — and at ratification Paolo sharpened the boundary: the per-bottle *serialization workflow* stays on the launch critical path; only the *NFT/on-chain layer* decouples.** The physical reality governs — **once bottles arrive, warehouse operators apply the NFC tag + serial to each bottle, and the ERP (Module B) + WMS (Logilize) must be ready to record that at launch**, even if minting is not. So the boundary is: **KEEP / launch-ready** — the physical NFC tag application + serial capture + the `SerializedBottle` inventory-ledger record + the Logilize tagging/receiving integration (B.10/B.11/B.23). **DECOUPLE off the critical path (feature-flagged behind EXT-1)** — the NFT mint/burn on Avalanche + custodial wallet (B.12), the on-chain recovery chains (B.13), and the Bottle-Page chain-link content (B.34). At launch each `SerializedBottle` carries `nft_reference = NULL` (the spec already admits this); the NFT is **back-filled** when the on-chain workstream lands — no rebuild. The launch critical path (inbound → stock → ATP → no-oversell → ship) therefore runs on **two launch-ready paths** — serialized-minus-NFT (per-bottle tag + serial + ledger; NFT deferred) and the non-serialized path (B.14, zero provenance, the residual fallback for NFC-opt-out allocations) — while the two-layer no-oversell guard (FLOOR) composes per sub-pool (§10.5) regardless. **Seam (P1):** (a) `nft_reference` nullable + back-fillable; (b) the Module B Acceptance doc's **EXT-1 feature-flag gate**, but **re-scoped at ratification** so it gates only the **NFT/on-chain** criteria — **not** the physical-tagging/serial/`SerializedBottle`-ledger criteria, which are launch-ready (§5); (c) the NS path. **Two critical dependencies:** the **blockchain-expert review (EXT-1) must be scheduled NOW** (Paolo track) or it becomes the launch critical path; and the **NFC tag-content design (DEC-124) must let tags apply at launch with serial + Bottle Page URL, on-chain reference back-fillable** (the offline-verification feature rides the decoupled workstream) — the B2 feasibility item for the tech team / blockchain review (Q1). **Do not re-cut the already-deferred NFT cluster** (EXT-1/3 smart-contract audit/governance; DEC-120/121/122/124/131 working-hypothesis cluster; hypothesis validation) — carry verbatim (Q1 ratified).

2. **D16 (Stocktake / Quarantine / Inventory-Adjustment — the Stage-8 "single largest v1.1 increment") is a SIMPLIFY of the *workflow automation*, KEEPING the integrity core — and the honest finding is that the cut is thinner than its billing because the Stage-8 workflows are already substantially operator-driven by spec.** KEEP the integrity core whole (the two-layer no-oversell guard Layer 2; committed-inventory protection + `InventoryShortfallDetected`; cost-basis correctness; the four-way reconciliation discipline; the quarantine-before-trust gate — all FLOOR). The Stage-8 workflows (Stocktake §12, QuarantineRecord §14, Inventory-adjustment proposal→approve §13, NS-adjustment §5.5) are **already single-supervisor-approval / operator-initiated** per `feedback_prd_rr_approval`, so the genuine SIMPLIFY is **deferring the *automated reciprocal round-trips* and *reconciliation automation*** (the auto-reopen of Module D's InboundEvent on `InboundBatchDiscrepancy` §11/D.17; the auto cost-basis reconciliation on `BottleQuarantineResolved` §14.4/D.18; the Stocktake tolerance-driven auto-reconciliation engine + cadence automation §12.4) → **manual-first operator handling via the Admin Panel**, decided **in lockstep with Module D's ratified KEEP-pending-B-review** (D.17/D.18) + Module A's forwarded Q4. **Seam (P1):** keep every entity + event + the DEC-194 split + the DISCREPANCY state + the 6-path resolution enum (Module D side) + the StockPosition contract, so the automation is additive when it lands. **Honest calibration:** the integrity core is FLOOR — not a candidate; the simplification is the automation *around* it (Q2).

3. **Module B owns the load-bearing inventory-integrity half of the scope floor — verified whole.** The **two-layer no-overselling guard Layer 2** (per-sub-pool physical-inventory ATP §10.1/10.5; DEC-185/196), the **B→A ATP push (DEC-187, §10.2)**, the **storefront lesser-of read Module S consumes** (§10.4; DEC-185/187), **committed-inventory protection + `InventoryShortfallDetected`** (§13.4/13.5; Q-CL-6), the **InboundBatch** that realises the core-loop "bottle received into inventory" step on the ledger side (DEC-195, §3/§11.3), the **StockPosition 5-dimension aggregated view** (§8; DEC-196), and the **four-way reconciliation discipline** (§2.4) are all named-floor in MVP-plan §3 / kickoff §4. **None is cut.** These compose with Module A's Layer 1 + Module S's shared-pool decrement.

4. **The non-serialized inventory ledger (§5) is doubly load-bearing — it is both Layer 2 for NS stock AND the D12 decouple seam.** DEC-186 (supersedes DEC-133) gave NS stock a full inventory-ledger home (four counters, NS ATP, commitment/reservation lifecycle, conversion path, adjustment workflow) while preserving the digital-provenance no-op. KEEP whole. This is what lets the launch critical path run without the on-chain workstream.

5. **Module B is back-office / warehouse-ops — L-PP / P2 is straightforward (zero producer writes, zero consumer self-serve writes).** Every Module B workflow (stocktake, quarantine, adjustment, recall recording, destruction recording, NFC re-tag authorisation) is operator-driven via the Admin Panel + Logilize/Vinlock execution (DEC-123/188; the personas §20.1 are all ops/back-office). Module B's **one** customer-facing surface is the **Bottle Page render (read-only, DEC-024, zero customer identifiers)** — part of the storefront-exempt consumer experience (kickoff §3). **No backend capability is cut** (Q6).

6. **One scope annotation + Phase-C cascades — no genuine RECONCILE needed.** Unlike Module D (DEC-183) and Module S (DEC-119), Module B carries no stale-framing contract-consistency fix. The Direct-Purchase `ownership_flag = CRURATED`-at-issuance derivation (§2.2) is **not-exercised-at-launch** (consistent with Module A/D's ratified Direct-Purchase deferral — A keeps the enum, D defers the use); a scope annotation, not a cut. ## §3 Module-specific changes

### §3.1 D12 — DECOUPLE only the on-chain layer; the serialization workflow stays launch-ready (the kickoff §9 headline; Paolo brand call, refined at ratification)

The locked dial KEEPS the NFC/NFT value-proposition but DECOUPLES it off the launch critical path — **DECOUPLE ≠ DEFER.** At ratification Paolo sharpened the boundary on a load-bearing operational fact: **once bottles arrive in the warehouse, WH operators still apply the NFC tag + serial to each bottle, and the ERP + WMS must be ready to record that at launch — even if minting/the NFT are not ready.** That governs the cut. Findings:

1. **The decouple boundary is at the *mint*, not the *tag*.** The per-bottle **serialization workflow** — physical NFC tag application + serial capture + the `SerializedBottle` inventory-ledger record + the Logilize tagging/receiving integration (B.10/B.11/B.23) — is **on the launch critical path / ready at launch.** What decouples is the **on-chain layer only:** the NFT mint/burn on Avalanche + the custodial wallet (B.12), the on-chain recovery chains (B.13), and the Bottle-Page chain-link content (B.34). At launch each `SerializedBottle` carries `nft_reference = NULL` (the spec already admits NULL); the NFT is **back-filled** when the on-chain workstream lands — no rebuild. (My initial draft framed the whole serialized cluster as decoupled with the NS path as the only seam; Paolo's correction is materially more accurate — the tag/serial/ledger is a warehouse-floor operation that cannot be deferred.)

2. **Two launch-ready paths, not one.** The launch critical path (inbound → stock → ATP → no-oversell → ship) runs on **(i) serialized-minus-NFT** (per-bottle tag + serial + ledger; NFT deferred/back-filled) and **(ii) the non-serialized path** (B.14 — zero provenance, the residual fallback for NFC-opt-out allocations; the opt-out `non_serialized_offer_admitted` is Module A's lever, ratified A.9). The two-layer no-oversell guard (FLOOR) composes per sub-pool (B.21), so it is intact across both.

3. **The seam.** (a) `nft_reference` is nullable + back-fillable on the `SerializedBottle` record; (b) **the Module B Acceptance doc's EXT-1 feature-flag gate — but RE-SCOPED at ratification** so it gates only the **NFT/on-chain** criteria, **not** the physical-tagging/serial/`SerializedBottle`-ledger criteria (which are launch-ready). The gate currently also covers `NFCTagApplied` (AC-B-EVT-1) + NFC application (AC-B-XM-47); those move to the launch-floor (un-gated) set, leaving the gated set = NFT mint/burn/wallet/on-chain/recovery + the on-chain-reference tag-content. This **resolves AMB-B-5** (Stage-8 vs preserved EXT-1 gate scope) + the Packet's count reconciliation (§5). (c) the NS path.

4. **Two critical dependencies — both must be owned now.** (a) The **blockchain-expert review (EXT-1) must be scheduled NOW** (Paolo track) or it becomes the launch critical path. (b) The **NFC tag-content design (DEC-124) must let tags apply at launch with serial + Bottle Page URL, with the on-chain reference back-fillable** — so the spec's "bake the on-chain reference onto the chip" design is **not a hard prerequisite for applying tags.** The online verification path (scan tag → Bottle Page) works at launch; the offline-verification feature (on-chain ref on the chip) rides the decoupled workstream. Whether tags are rewriteable / two-pass / URL-only-at-launch is tech + procurement (DEC-073) — **the B2 feasibility item for the tech team / blockchain review.** Pre-launch NFC tag-stock procurement lead-time makes this time-sensitive.

5. **Honest framing — the single biggest critical-path removal in the exercise, but the VP is preserved, and tighter than first drafted.** D12 takes Avalanche + the custodial wallet + the mint/burn lifecycle + the on-chain recovery scenarios + the EXT-1 gate off the launch critical path (dials weight: XL). But it is a **decouple, not a delete** — and *only the on-chain layer* decouples; the per-bottle tagging + serial + ledger is launch-ready. If the on-chain workstream lands by launch, the full provenance VP ships; if it slips, serialized stock still ships with per-bottle tags + serial + ledger tracking (NFT back-filled later), NFC-opt-out allocations ship non-serialized, and the on-chain layer lights up additively post-launch. **Do not re-cut** the already-deferred NFT cluster (EXT-1/3 smart-contract audit/governance; DEC-120/121/122/124/131 working-hypothesis; hypothesis validation) — carry verbatim (B.40).

**Verdict (ratified):** DECOUPLE the **NFT/on-chain layer** off the launch critical path; **KEEP the per-bottle serialization workflow (tag + serial + `SerializedBottle` ledger + WMS) launch-ready** with the NFT mint feature-flagged off + back-fillable; KEEP the VP. Seam = `nft_reference` nullable + the re-scoped EXT-1 gate + the NS path. **Schedule EXT-1 now + confirm the tag-content back-fill design (B2 feasibility).** Default fallback = B1 (NFC-opt-out allocations ship non-serialized); B2 (serialized-minus-NFT + back-fill) is the expected posture for serialized stock, feasibility-confirmed in the dev-team sizing exercise.
