## §1 Verdict & counts

**Module C is the fulfilment module — KEPT WHOLE on its load-bearing "ship → cellar" half of the core loop, with FOUR scope dials landing in-module (D3 geography/customs/excise · D13 late-binding pick · D14 returns/replacements · D17 cellar render) plus D15 recall reverse-logistics — making it the broadest-dial-footprint module of the triage, but with the honest finding that three of the four cuts are thinner at the Module-C spec layer than their billing, because v1.1 already pre-factored the hybrid / manual-first posture.** The ship→cellar floor — the shipment gate, the late-binding voucher→bottle bind, the dispatch event that triggers INV2 + the burn chain, the no-oversell-at-pick StockPosition read, the OFAC/eligibility surface, and the in-transit redemption-block — stays non-negotiably whole; the cuts land on the *sophistication around* that floor, never the floor itself. Seven facts shape the verdict:

1. **D3 (Geography + customs/excise/DDP-DAP — THE heaviest Module C lever, #2 lever overall) is a SIMPLIFY (hybrid) — and the hybrid is already v1.1's design.** Paolo's locked D3 decision (SIMPLIFY hybrid — automate low-friction destinations; route complex ones via a manual operator flow) maps **exactly** onto the v1.1 two-tier eligibility model (§7; DEC-147): **Tier 1** automated pre-cleared destinations + **Tier 2** white-glove Customer Care fallback (operator quote per DEC-145 → customer pays → ship). The auto/manual `quote_origin` discriminator (DEC-145), the simple-at-launch US-state (DEC-148) + DDP/DAP (DEC-149) dispositions, and the three automation engines being deferred (US-state matrix DEC-148, DDP/DAP country-by-country DEC-149, excise-rate automation DEC-150) are **all already in v1.1**. So the genuine Module-C-layer cut is thin: **narrow the Tier-1 automated pre-cleared destination list to low-friction (EU/UK/CH) at launch [configuration], and lean on the already-built white-glove manual operator flow for complex destinations (US, high-excise, state-alcohol)** — the three automation engines stay already-deferred (carry verbatim). **OFAC RETAINED — FLOOR** (§7.2: "OFAC screening applies at all destinations regardless of tier"; the eligibility surface composes with Module K's sanctions/Hold gate). **INV2 tax-correctness RETAINED — FLOOR** (the shipping-fee + destination VAT + excise + storage-roll-in composition on INV2; DEC-146). **Honest calibration (the kickoff's explicit ask):** even the "simple" baseline is heavy compliance — the white-glove manual flow still computes excise + customs + DDP/DAP + a correct quote per complex shipment for INV2; the residual heaviness is the **compliance/tax FLOOR itself, not a candidate.** Like Module D's Direct Purchase and Module B's D16, D3 is thinner at the spec layer than its reputation because v1.1 pre-factored the hybrid. *Market call (Q1): the launch destination footprint.*

2. **D13 (Late-binding pick) is a SIMPLIFY → FIFO + manual tiebreak — and the deferred surface is largely Logilize-internal.** v1.1's two-surface algorithm (§3.2; DEC-137) is **Surface 1** voucher-side FIFO-by-expiry (+ earliest-issuance-timestamp tiebreak + `manual_review` tertiary) × **Surface 2** bottle-side Logilize warehouse-efficiency. The D13 SIMPLIFY defers **Surface 2's warehouse-efficiency optimisation** — but that optimisation is **Logilize's responsibility** by spec ("Logilize selects the most efficient physical pick"); Module C only sends the pick instruction naming the allocation-pool boundary + FIFO voucher-side + effective-unbreakable rule. So at the Module-C spec layer the cut is: Logilize picks *simply* (any available bottle from the correct pool) at launch instead of route-optimised — **Module C's instruction contract is unchanged.** KEEP the two-surface contract structure + voucher-side FIFO + the `manual_review` tiebreak + the 7-step event chain (DEC-142) as the seam. **The late-binding *moment* + the voucher→bottle bind is FLOOR** (the core-loop ship step, BMD §5.5) — not a candidate. **The no-oversell-at-pick StockPosition read is FLOOR** (§3.4; reads B's StockPosition at the `(BR, warehouse, case_config, allocation, ownership)` cell, DEC-196; composes with B's Layer 2). Producer-override (Interpretation C, DEC-137/Q-CL-14) already Phase-2+ + explainer-pending — carry verbatim. *Honest: thin (Q2).*

3. **D14 (Returns + Replacement) is a SIMPLIFY → manual-first (Admin Panel) — and the FSM is already operator-driven by spec.** v1.1's Module-C-owned end-to-end Returns/Replacement workflow (§10; DEC-138 + the DEC-184 FSM `REPORTED → INVESTIGATED → APPROVED → REPLACEMENT_ISSUED → CLOSED` + REJECTED/WITHDRAWN off-ramps) is **already a sequence of operator actions** (operator confirms intake, investigates, approves, issues replacement — `feedback_prd_rr_approval` governs the supervisor-override approval workflow).
