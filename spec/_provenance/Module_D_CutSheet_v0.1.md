## §1 Verdict & counts

**Module D is KEPT on its core-loop inbound floor, its uniform procurement flow, its PO-issuance discipline, and its financial-event recording (the D19 settlement seam) — with ONE genuine in-module deferral, D11 Direct Purchase, that lands here but is *thin* at the spec/backend layer.** Module D is the first module in the triage where a headline defer is taken in-module rather than forwarded downstream (Module 0/K/A all forwarded their headline levers). But the honest finding is that even *this* in-module cut is small — for the same structural reason Module A's Direct-Purchase defer was "free": **v1.1 already pre-factored Direct Purchase into a parameterization, not a separate arm** (DEC-093 uniform flow + DEC-183 harmonized activation). Module D owns the core-loop "bottle inbound" step and is load-bearing on the supply side, so it stays near-whole. Four facts converge:

1. **D11 (the headline, forwarded from Module A) lands here — but the deferred surface reduces to one operator action.** Per DEC-093 the procurement flow `PI → PO → InboundEvent → optional ConsignmentReceipt (V2)` is **uniform across all sourcing models** ("No separate Direct Purchase flow or PI subtype in the data model"). Per DEC-183 the DRAFT→ACTIVE activation trigger is **harmonized** to operator-publish-post-PO-commit across V1/V2/Direct Purchase, and `SupplierPaymentCompleted` is **financial-event-only** (no FSM role). The entities (PI, PO, InboundEvent), the `ownership` enum, the timing parameterization, and the 13 events are **all shared with V1/V2 and KEPT**. The *only* genuinely Direct-Purchase-exclusive surface is the **operator-initiated PI-creation path at allocation-creation time + its at-PO-creation timing branch** (and the unexercised `direct_purchase → NEWCO`-at-issuance ownership derivation — note `NEWCO` itself is still reached by V1/V2 at sell-through). So "defer Direct Purchase" at the Module-D layer means: *don't build the operator-initiated PI-creation surface + its timing branch at launch.* Genuine, but thin. **Seam (P1):** the uniform flow stays parameterized (DEC-093); the `trigger_source = operator_initiated` enum value, the `ownership = NEWCO` value, and the at-PO-creation timing rule are retained-but-unexercised; re-enable is purely additive. **Deal-dependent** (mirrors A's ratified Q2): if a launch deal needs Direct Purchase, Module A is unchanged and Module D re-enables the arm (Q1).

2. **Module D carries a piece of the scope floor — the core-loop "bottle inbound" step.** The two-phase InboundEvent (`PHYSICALLY_ACCEPTED` + `COST_FINALIZED`, 5-WD SLA) is the event that brings physical stock into NewCo custody: it creates Module B's InboundBatch (DEC-195 — feeds the inventory ledger / no-overselling floor), gates Module C shipment (DEC-081), and carries the landed cost Module E settles. Named-floor in MVP-plan §3 ("bottle received into inventory"). KEEP — none of it is heavier than the floor needs.

3. **The D16 receiving-workflow sophistication lives in Module B, not Module D.** Module D's receiving surface — two-phase InboundEvent, the documents-side 3-gate QC (DEC-194 split: D = documents-in-order; B = physical-match), the 6-path DiscrepancyResolution **enum** (an event vocabulary per DEC-072, not six automated workflows), and the Stage-8 reciprocal cascades (DISCREPANCY-state re-open; QuarantineRecord cost-basis reconciliation) — is floor or cheap-interlock. The genuine *workflow-depth* SIMPLIFY (physical-match check, QuarantineRecord workflow, Stocktake) is **Module-B-owned** and is forwarded to the joint D16 review at Phase C. Module D's reciprocal side is thin; the integrity interlocks are not unilaterally cut (Q3).

4. **D19 settlement: recording KEPT, automation deferred to Module E.** Module D **records** all procurement/inbound financial events (`SupplierPaymentCompleted` atomic-per-PO, `InboundEventCostFinalized` + landed-cost categories, `DiscrepancyResolutionRecorded`, …). The settlement *engine* (quarterly statement composition, payment execution, Xero GL) is **Module E's, deferred to operator-run** (D19). Module D keeps the recording — that *is* the seam. Net Module-D cut: zero (the automation defer is Module E's).

The L-PP treatment is identical to Module A's: every Module D procurement operation is **operator-driven via the Admin Panel** at launch; producer-facing write UIs are deferred; DEC-083 admin-parity already makes the Admin Panel functionally complete, so **no backend capability is cut**. Module D retains **zero** producer writes (the one platform-wide retained producer write — membership approve/decline — is a Module K surface) (Q4).

**Calibration takeaway (refines the running 0/K/A finding).** The method predicted "the real cutting starts at A/D." Module D sharpens that prediction the same way Module A did: **a supply-side module that sits inside the core-loop floor does not get gutted, even when it is the one that finally takes a headline defer in-module.** Direct Purchase was supposed to be "the substantive procurement-arm deferral"; on close reading the arm was already collapsed into a one-operator-action parameterization by v1.1's own design (DEC-093 uniform flow + DEC-183 harmonized activation), so the in-module cut is *genuine but thin*, and its seam is intrinsic. Module D is therefore **KEEP-heavy on the inbound floor + 1 thin, real in-module defer (Direct Purchase use) + the D16 depth forwarded to Module B + a DEC-183 contract reconciliation**. The genuinely heavy cutting still lands in **S** (D7/D8 commerce machinery), **B** (D16 Stage-8 inventory authority — the single largest v1.1 increment), **C** (D13/D14 fulfilment), and **E** (D19 settlement engine). The supply-side quartet (0/K/A/D) is now complete and near-whole; each forwarded or pre-factored its headline lever.
