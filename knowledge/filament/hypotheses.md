### Hypothesis: Nest a low-level console inside its parent via a RelationManager + create-through-domain-action
A child console that clutters the flat sidebar (Product Variant under Master; Club / Producer Agreement under Producer; Profile under Customer) is better surfaced as a RelationManager on the parent's view page: hide the child with `shouldRegisterNavigation(): false` (routes stay registered, so the RM's row `ViewAction->url(<ChildResource>::getUrl('view', ...))` still resolves), reuse `<ChildResource>::table($table)` for columns (keeps the kit lifecycle badge + i18n), and route the header `CreateAction->using($this->createX(...))` through the owning module's domain action with the owner id injected (`$owner->id`, no parent picker) — never an Eloquent write, so the no-Eloquent-write PHPStan rule stays green. Two gotchas: (1) the RM's `canCreate()` defaults to a policy check that denies when no per-model policy exists → override `canCreate(): bool { return true; }` or the header create button is hidden; (2) Filament's isolated-RM test helpers (`Livewire::test(RM, ['ownerRecord'=>…,'pageClass'=>…])`) don't resolve header-action VISIBILITY (`assertTableActionExists` passes but the button can still be hidden) — assert existence in tests, verify create rendering live; (3) **DECISIVE** — a RM is READ-ONLY on a `ViewRecord` page (`RelationManager::isReadOnly()` = `is_subclass_of(pageClass, ViewRecord::class)`), and a `CreateAction` is DENIED when read-only BEFORE `canCreate()` is consulted (RelationManager.php:360), so the operator console's view-page parents suppress the create button even with `canCreate()→true`. The fix that actually renders it: override `isReadOnly(): bool { return false; }` on the create-bearing RM (safe iff it defines no Edit/Delete actions); pin it with `expect((new RM)->isReadOnly())->toBeFalse()`. See `lessons.md` 2026-06-24. A RM that should stay view-only (e.g. Customer Memberships) keeps the default read-only.
- Confirmations: 2/3
  - 2026-06-24: operator-console UI pass — VariantsRelationManager (Catalog), Clubs/ProducerAgreements (Parties), read-only Memberships on Customer; all green at PHPStan max + 1753 tests.
  - 2026-06-24: operator-console PREMIUM pass — found the create buttons NEVER rendered (gotcha 3, isReadOnly); overriding `isReadOnly()→false` surfaced "New variant"/"New club"/"New agreement" (confirmed live via Playwright); Memberships left read-only; 1753/1753 green.

### Hypothesis: Group "settings"-type reference consoles under a Filament Cluster, not a third nav group
Reference/building-block resources (Format · Case Configuration · Product Reference) read as a "Settings" entry whose sub-navigation is a tab strip — a `Filament\Clusters\Cluster` (discovered via `->discoverClusters(...)`). The cluster carries the sidebar group placement (`getNavigationGroup()` → the module enum) so "Settings" sits INSIDE its module group, not as a peer; each clustered resource sets `protected static ?string $cluster = …` and overrides `getNavigationGroup(): null` so the in-cluster sub-nav renders as flat tabs rather than re-nesting the module sub-heading. Clustered routes gain the cluster slug prefix automatically; `getUrl()` keeps resolving, so existing tests/links survive.
- Confirmations: 1/3

### Hypothesis: `callAction()` / `fillForm()` can GROW or overwrite a list-valued field, but cannot SHRINK it — mount, `set()` the whole list, then `callMountedAction()`
Filament's `setActionData()` → `fillForm()` → `InteractsWithSchemas::fillFormDataForTesting()` writes state through `Arr::dot($state)` + `data_set()`, i.e. **one leaf at a time** (`…data.constituents.0`, `…data.constituents.1`, …). Its `unsetMissingNumericArrayKeys()` prune — the pass that should drop indices the submitted list no longer has — accumulates `$currentStatePath .= ".{$key}"` **inside** the `foreach`, so by the time it reaches the mounted action's `data` key the path already carries a sibling key (`mountedActions.0.name.data.…`). That never `startsWith($schemaStatePath)` (`mountedActions.0.data`), so nothing is pruned. Consequence: `callAction('x', ['ids' => [1]])` on a modal prefilled with `[1, 2]` submits **`[1, 2]`** — the tail survives. The test passes for the wrong reason (here: a 1-element `N ≥ 2` floor test that never reached the floor, and instead performed a real no-op edit that bumped `version`). Shrinking to `[]` DOES work (`Arr::dot` keeps an empty array as a leaf, so `data_set` replaces the array wholesale). The reliable shrink is `->mountAction('x')->set('mountedActions.0.data.<field>', [1])->callMountedAction()` — one `data_set` on the array itself. Corollary: a list-field test whose submitted list is **shorter** than the prefill must never use `callAction()`; equal-length or longer is safe.
- Confirmations: 1/3
  - 2026-07-08: `catalog-module-0-completeness-sweep` task 6.3 — the `editComposition` modal's `N ≥ 2` floor test submitted `[first]` against a `[first, second]` prefill, got zero validation errors and a `version` 1→2 no-op edit; traced to `unsetMissingNumericArrayKeys()` (`vendor/filament/schemas/src/Concerns/InteractsWithSchemas.php:474`); `mountAction → set → callMountedAction` reproduced the operator's real deselection and tripped the floor.

### Hypothesis: `assertHasActionErrors(['field' => $message])` silently truncates the expected message at the first `:`
Livewire's `assertErrorMatchesRuleOrMessage()` opens with `if (Str::contains($ruleOrMessage, ':')) { $ruleOrMessage = Str::before($ruleOrMessage, ':'); }` — so `min:3` can be written as a rule name. It applies to **messages too**, so a localized domain message containing a colon (`'Cannot change the composition of this CompositeSku: it is active, …'`) is compared as its prefix alone, `assertContains()` finds no exact match, and the failure reads as *"Component has no matching failed rule or error message"* — indistinguishable from a missing error. Every colon-free message (`catalog.edit.cannot_edit`, `catalog.composite_sku.insufficient_constituents`) matches fine, which is why the trap only bites some copy. Use Livewire's **Closure overload** — `['field' => fn (array $rules, array $messages) => …]`, which `PHPUnit::assertTrue`s the closure's return and hands it the raw `$messages` list — and assert with `expect($messages)->toContain($message)` inside for a readable diff. The trap is a property of the COPY, not the code: a colon can be introduced into any `lang/*/catalog.php` string later, silently reddening a passing test.
