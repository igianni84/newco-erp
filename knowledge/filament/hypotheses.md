### Hypothesis: Nest a low-level console inside its parent via a RelationManager + create-through-domain-action
A child console that clutters the flat sidebar (Product Variant under Master; Club / Producer Agreement under Producer; Profile under Customer) is better surfaced as a RelationManager on the parent's view page: hide the child with `shouldRegisterNavigation(): false` (routes stay registered, so the RM's row `ViewAction->url(<ChildResource>::getUrl('view', ...))` still resolves), reuse `<ChildResource>::table($table)` for columns (keeps the kit lifecycle badge + i18n), and route the header `CreateAction->using($this->createX(...))` through the owning module's domain action with the owner id injected (`$owner->id`, no parent picker) — never an Eloquent write, so the no-Eloquent-write PHPStan rule stays green. Two gotchas: (1) the RM's `canCreate()` defaults to a policy check that denies when no per-model policy exists → override `canCreate(): bool { return true; }` or the header create button is hidden; (2) Filament's isolated-RM test helpers (`Livewire::test(RM, ['ownerRecord'=>…,'pageClass'=>…])`) don't resolve header-action VISIBILITY (`assertTableActionExists` passes but `assertTableActionVisible` / `callAction` report "not visible") — assert existence in tests, verify create execution live.
- Confirmations: 1/3
  - 2026-06-24: operator-console UI pass — VariantsRelationManager (Catalog), Clubs/ProducerAgreements (Parties), read-only Memberships on Customer; all green at PHPStan max + 1753 tests.

### Hypothesis: Group "settings"-type reference consoles under a Filament Cluster, not a third nav group
Reference/building-block resources (Format · Case Configuration · Product Reference) read as a "Settings" entry whose sub-navigation is a tab strip — a `Filament\Clusters\Cluster` (discovered via `->discoverClusters(...)`). The cluster carries the sidebar group placement (`getNavigationGroup()` → the module enum) so "Settings" sits INSIDE its module group, not as a peer; each clustered resource sets `protected static ?string $cluster = …` and overrides `getNavigationGroup(): null` so the in-cluster sub-nav renders as flat tabs rather than re-nesting the module sub-heading. Clustered routes gain the cluster slug prefix automatically; `getUrl()` keeps resolving, so existing tests/links survive.
