## Audit-only writers for events another module owns (the §11.4 Module-K / Module-E boundary)

**Observation.** When the spec assigns an entity's *lifecycle events* to a **different** module than the one that holds the entity *state*, the state-holding module ships its writer Actions **audit-only**: they `update()` the entity and record **no** domain event, and fabricate **no** event class for the other module's contract. Club Credit is the sharp case — §11.4 makes `ClubCreditIssued` / `ClubCreditApplied` / `ClubCreditRestored` / `ClubCreditForfeited` **and** the upstream `MembershipFeePaid` **Module E's** events; Module K's four writers (`Issue` / `Apply` / `Forfeit` / `RestoreClubCredit`) write `state`/`remaining` only. This is a **stronger** form of the "no event because the spec names none" audit-only precedent (`RecordKycVerified`, the Account family): there the events simply don't exist; here they *do* exist, just owned downstream. When Module E lands, its `MembershipFeePaid` listener + `ClubCredit*` consumers invoke these **same** within-module writers — the entity-state authority (Module K) and the financial-event authority (Module E) compose without rework (the R-reconciliation discipline). Pinned by a §11.4 ownership guard test (`ClubCreditEventOwnershipTest`): no `app/Modules/Parties/Events` class is named for any of the five events, **and** running all four writers records zero `domain_events` rows. (The class-absence half uses a filesystem `glob`, never `class_exists` on a literal-absent FQN — that reds PHPStan max `function.impossibleType`; see `lessons.md`.)

Seen: 2026-06-23 `club-credit` (the four writers + the ownership guard test).

## Restoring into a one-active partial-index slot: guard on no-other-active explicitly, don't lean on the index to throw

**Observation.** A "one active X per parent" invariant enforced by a partial unique index `(parent_id) WHERE state = 'active'` interacts with a *restore* transition (`terminal-ish → active`). Before the write the restored row is **outside** the index's scope (it is `redeemed`), so flipping it back to `active` while the parent already holds another `active` row *would* hit the index — but only as a raw `QueryException`, after attempting the write. `RestoreClubCredit` instead guards **explicitly** first: re-read the parent `lockForUpdate()`, and reject with a localized precondition exception if `parent.activeRelation()->exists()`. Two payoffs — a clean operator-facing rejection rather than a leaked DB exception, and the rejection lands **before** any write (transaction stays clean). The parent lock also serializes a concurrent restore/issue race on the same parent (`RestoreClubCredit` and `IssueClubCredit` both lock the Profile). Contrast issuance, which has no prior row to name a precondition over and therefore **does** lean on the index (a re-issue while `active` is a structural `QueryException`, rolled back). Rule of thumb: when a transition *can* name a clean precondition for the conflict, guard it; when it can't, let the structural index reject.

Seen: 2026-06-23 `club-credit` (`RestoreClubCredit` one-active guard; design L7).
