# Testing — Hypotheses (test when possible)

> A hypothesis carries `Confirmations: N/3`. Confirmed 3× (dated) → promote to `rules.md`. Contradicted → note the contradiction and demote.

## Prove a class is ABSENT by listing the directory at runtime — never `class_exists('<literal>')` (PHPStan max flags it impossible)

**Hypothesis.** A scope guard that asserts a class / Action / Event does **not** exist must not use `class_exists('App\…\Absent')` with a string literal — PHPStan max evaluates the literal and flags the call `function.impossibleType` ("always false", dead code). Instead list the directory at runtime — `glob(app_path('Modules/<M>/<Dir>/*.php')) ?: []` (the `?: []` is required: `glob` returns `list<string>|false` and `array_map` rejects `false`), or a `RecursiveIteratorIterator` — map to basenames, and assert the **set** (`expect($names)->not->toContain('Absent')`, `toEqualCanonicalizing([...])`). A `class_exists($var)` on a variable derived from the listing is fine (not a resolvable literal). Bonus: a directory scan also catches a renamed-but-still-present class that a fixed `class_exists` list would miss.

**Confirmations: 2/3** (need 1 more distinct change).
- 2026-06-15 `catalog-product-spine` — `CatalogNamingCascadeTest` proves absence of a forbidden naming pattern by a recursive walk of the Catalog subtree (`getBasename('.php')` → `array_filter` → `toBe([])`).
- 2026-06-16 `parties-producer-lifecycle` — `SupplyLifecycleChainTest` states the rule + the PHPStan reason explicitly; `glob(...) ?: []` scope-guards both `Actions/` (only the six supply transitions) and `Events/` (no demand-side event exists).
- *(Related ancestor, NOT counted: 2026-06-12 `foundations-modules-skeleton` `ModuleConformanceTest` scans the modules dir for set-equality — same directory-scan idiom, but its driver is "no stray entries", not the `class_exists`-is-impossible mechanism.)*
- *(Related mechanism, NOT counted: `catalog-module-0-completeness-sweep` task 5.2 — `is_a(A::class, B::class, allow_string: true)` asserting two exception classes are unrelated drew the SAME `function.impossibleType`. Same root — PHPStan max resolves class-name literals, so any runtime check of a **static** class relation is dead code — but the prescription differs: there is no runtime substitute to reach for, because the type system has already proven it. **Delete the assertion**; where the relation carries real meaning, let `toThrow(Class::class, 'message fragment')` carry it and prove it bites by mutation. Generalisation to watch: the hypothesis is really "PHPStan max proving your test assertion means the assertion is not a test" — the directory-scan is one prescription of several.)*

**Applies to.** Any test that proves a class / Action / Event is absent (a category-neutrality or scope guard).

## A red-green FSM state-shape flip must invert EVERY observer in the SAME commit — enumerate by grep, sort into four kinds; the isolated writer's own contract + the source-scan guards stay diff-free

**Hypothesis.** When a change alters a state-machine's *shape* (a durable resting state becomes a transient pass-through, two steps merge into one atomic transition, a state is removed), every test that observed the OLD shape must be inverted in the **same commit** so the suite never goes red mid-change. Enumerate them exhaustively by grepping the changed state token + the driving Action's class name across `tests/` **before** editing — the design's enumeration is a checklist, the grep is the proof it is complete. Sort the hits into four kinds, each with a distinct action:
- **(a) outcome observers** — assert state/events AFTER the real Action → **flip** the expected terminal state + the event multiset.
- **(b) precondition helpers** — double-drove the old two-step only to *set up* a later state → **delete** the now-illegal second call (no try/catch); the collapsed Action reaches the terminal state alone.
- **(c) factory-forced-state datasets** — force the state via the factory and never call the flipped Action → **leave untouched** (they exercise the isolated writer, not the flow).
- **(d) source-scan guards** (exact-set Action allow-list, OC/side-write count, creation-chain event set) → **stay diff-free** *iff* no Action/Event class is added/renamed and the literal side-write count is unchanged; a pure shape-collapse rides existing guards without amending them.

The isolated writer's own contract test (factory-forced `From → To` + the illegal-from-state cases) also **stands unedited** — it never observed the flow, only the writer. Verify the whole set green on BOTH engines in one commit.

**Confirmations: 1/3** (need 2 more distinct changes).
- `parties-membership-charge-on-approval` (RM-03 / MVP-DEC-016) — the atomic *approve = charge = activation* collapse: `Approved` durable→transient. 8 outcome observers flipped + 4 precondition helpers' `ActivateProfile` line deleted; `ProfileActivationTest` (isolated writer) + `SupplyLifecycleChainTest` / `ComplianceIndependenceTest` / `SpineCreationChainTest` (source-scan guards) all git-confirmed diff-free; full suite green on SQLite AND PG17. *(Confirmation date = this change's archive-dir date once archived — not yet archived at authoring; do not stamp the authoring date.)*

**Applies to.** Any change that alters a status-FSM's shape — future Module S voucher-FSM transitions, Module A allocation-FSM activation, and further demand-side Profile evolutions.

## A scenario's coverage is an ORDERING claim — verify the order the test builds the world in, not just the facts it asserts

**Hypothesis.** When a spec scenario's **GIVEN** establishes entity state and its **WHEN** is *something landing on that state* (an event delivered to a consumer, a gate re-evaluated, an upstream projection flipping), a test that establishes the state **after** the trigger can assert every individual fact in the **THEN** and still not cover the scenario. The clause it silently drops is always the same one: *"…and X was left unchanged as a side effect"* — unobservable when no X was in flight. A green suite cannot see this; neither can a per-fact grep, nor line coverage (the production branch runs either way).

Two corollaries worth grepping for:
- **Sibling scenarios of one requirement diverge silently in test SHAPE.** They read alike, sit adjacent in the delta, and get written by different iterations. One lands its trigger on a live entity, the other pre-arms the world and never does.
- **The tell is a helper call order**, not an assertion: `projectProducer(...)` before `createMaster(...)` versus after. Grep the trigger helper and the entity-building helper together and compare line numbers per test body.

**Prescription.** Do the delta→test traceability pass by reading each scenario's clauses *in order* against the test's statements *in order*. Where a THEN clause says "unchanged"/"preserved"/"no side effect", require that the test built the thing before the trigger fired. Then mutation-prove the trigger is load-bearing (delete the trigger line; the test must red).

**Confirmations: 1/3** (need 2 more distinct changes).
- `catalog-module-0-completeness-sweep` task 7.2 — the delta's *Producer-State Projection* scenario "ProducerActivated enables a previously-blocked Master" (…"but no Master state changed as a side effect of consuming the event"). **Ten** tests delivered `ProducerActivated`; in all ten it landed before any Master existed, so the side-effect clause had never been observed. Its sibling — "ProducerRetired blocks new activations and preserves existing actives" — had always been tested the right way round, in the same file. Closed by `ProductMasterLifecycleTest::it('unblocks a reviewed Master when ProducerActivated lands, without touching the Master itself')`; mutation-proven. *(Confirmation date = this change's archive-dir date once archived — not yet archived at authoring; do not stamp the authoring date.)*

**Applies to.** Every event-consumer/projection scenario (Module A allocation FSM reading Catalog state, Module B's `SupplierPaymentCompleted` consumers, Module E's financial-event recorder), and any gate whose upstream flips asynchronously. Also the general shape: *"consuming event E must not transition entity X"*.

## An "untouched" snapshot assertion passes for free when the snapshot is EMPTY

**Hypothesis.** The `$before = ids(); …disturbance…; expect(ids())->toBe($before);` idiom — the recommended way to catch a row *added* by a disturbance — is **vacuous when the entity has no rows at snapshot time**: `[] === []`. Whether the trail is empty is a fact about the writers upstream, not about the test (e.g. `CreateProductMaster` records a domain event but **no** audit row, so a Master's audit trail before its first submit is `[]`). Prefer pinning the **literal ordered action list** (`expect(auditActions($e))->toBe(['catalog.product_master.submitted'])`): it catches an appended row exactly as well, and an empty or wrong trail reds at the snapshot instead of sailing through to a green comparison. Generalises: any "compare a collection to its own earlier value" assertion should first assert the collection's *shape*, so it proves its own non-vacuity. Same family as "assert `not->toBe('')` before the string assertions" (`DB::scalar()` narrowing).

**Confirmations: 1/3** (need 2 more distinct changes).
- `catalog-module-0-completeness-sweep` task 7.2 — the new `ProducerActivated`-lands-on-a-reviewed-Master test. First draft snapshotted audit-record **ids**; the Master's only pre-disturbance audit row is its `submitted`, and had the test been written one statement earlier (before the submit) the assertion would have compared `[]` to `[]` and proven nothing. Rewritten to pin the ordered action list. *(Confirmation date = archive-dir date once archived.)*

**Applies to.** Every non-retroactivity / preserve-actives / no-side-effect assertion — the R10 whitelist-reduction proof (3.2) already uses the ids form and is non-vacuous only because its SKU has activation rows; re-check it if that fixture ever moves.

## A test fake of a nullable contract must exercise BOTH arms — PHPStan max audits your fakes, and `return.unusedType` means the fake is too weak

**Hypothesis.** `phpstan.neon` analyses `tests/`, so an in-test (usually anonymous-class) fake implementing a `?T`-returning port is type-checked like production code. A fake that only ever returns a `T` draws `return.unusedType` — *"never returns null so it can be removed from the return type"*. **Never narrow the fake's signature to silence it, and never `@phpstan-ignore` it.** The diagnostic is not a typing nit: it proves the fake cannot drive the consumer's null branch, so any test using it silently leaves the null arm — usually the *safe default / disabled-feature* arm, i.e. the one that governs production — unproven. The fix is to make the fake answer both arms on different inputs (`fn ($id) => $id === 1 ? null : $id * 10`). Same family as the first hypothesis's generalisation — *"PHPStan max proving your test assertion means the assertion is not a test"* — but the prescription inverts: there, delete the assertion; here, **strengthen the fake**.

**Confirmations: 1/3** (need 2 more distinct changes).
- `parties-hero-package` task 1.1 — the swappability fake for `HeroPackageCapacityReader::forClub(): ?int` returned `$clubId * 10` unconditionally. PHPStan max flagged it; the null arm is `null ⇒ uncapped`, which is the **shipped production posture** (dark launch), so the too-weak fake would have left exactly the arm that matters untested. *(Confirmation date = archive-dir date once archived.)*

**Applies to.** Every deferred-module read-port fake (`Parties\Contracts\*Reader` and its Module-A/S/E successors), and any fake of a contract whose `null`/absent return encodes a feature-disabled or unconfigured default.

## A lock-BEFORE-read ordering is provable on SQLite only as SQL statement ORDER — assert it with `DB::listen`, driver-gate the lock clause

**Hypothesis.** When a fix consists of *taking a row lock before reading a derived aggregate* (the classic check-then-act race), no behavioural test on SQLite can observe it: SQLite's grammar compiles **no lock clause at all** (`Grammar::compileLock()` returns `''` and the SQLite grammar does not override it), so the locked and unlocked implementations are byte-identical in behaviour on the dev/test engine, and the sequential acceptance test passes green against the racy code. The property that *does* survive on both engines is the **order of the emitted statements**. Capture it with `DB::listen`, filter to the tables involved, and assert the lock target precedes the aggregate read. Then driver-gate the lock clause itself — `pgsql` ⇒ `expect($sql[0])->toContain('for update')`, else ⇒ `->not->toContain('for update')` — which documents the asymmetry instead of hiding it. `DB::transaction()` issues its `SAVEPOINT` through the raw PDO handle, so it never pollutes the captured list.

**Prescription.** A structural order-pin in the SQLite lane, a behavioural two-connection concurrency proof in the PG lane. Neither replaces the other: the order-pin catches a refactor that moves the lock after the count (silent on both engines otherwise); the PG proof catches a lock that was never a lock. Never `markTestSkipped` the SQLite half — assert its positive content (the house driver-gate idiom, `tests/Feature/Platform/ActorRoleConstraintTest.php:110`; no skip exists anywhere in `tests/`).

**Confirmations: 1/3** (need 2 more distinct changes).
- `parties-hero-package` task 1.2 — `Support\ClubSeatOccupancy::lockAndCountOccupiedSeats()` locks the `parties_clubs` row before counting `Active`+`Suspended` Profiles. The ordering *is* the oversell fix (design D3), and `AC-K-J-13`'s sequential 51st approve passes green **against the racy implementation**, so nothing else in the suite would have caught its removal. Pinned by SQL statement order; the two-connection proof is task 3.2, PG17 only. *(Confirmation date = archive-dir date once archived.)*

**Applies to.** Every `qty − issued ≥ 0` (Module A, invariant 1) and `physical_in_storage − reserved − … ≥ 0` (Module B, invariant 2) enforcement site — both are check-then-act over a derived aggregate and will need the same lock, the same asymmetry, and the same two-part proof. Also the `domain_events` per-consumer delivery ledger's `lockForUpdate` winner-election (`InlineDeliveryExecutor:154`).

## A Pest global test helper is declared once per PROCESS — a duplicate name is a fatal redeclare that kills the whole run

**Hypothesis.** Helper `function`s declared at the top level of a Pest test file land in the **global namespace of the single PHP process** that runs the suite. PHPUnit's `TestSuiteLoader` `include`s every selected test file while *building* the suite — before any test executes — so two files declaring the same helper name abort the entire run with `Cannot redeclare …`, not just the offending file, and not as a shadow. This is invisible while a helper is unique, and surfaces the moment a second test file wants the same fixture (the common case: every file testing one capability wants the same "arrange the aggregate into state X" helper).

**Prescription.** Give each test file's fixture helper a **file-distinct name** (`clubAtCapacity`, `seatClubTo`, `seatLedgerQueries`, …) rather than the most natural one. When several files genuinely need the same arrangement, promote it to a factory state, a `tests/Support/` class, or a Pest `uses()`-bound trait method — never a second global `function`. Do not "reuse" another test file's helper by calling it: the coupling is invisible to `grep`, survives only because both files happen to be loaded, and inverts the file's own arrange-act-assert locality.

**Confirmations: 1/3** (need 2 more distinct changes).
- `parties-hero-package` task 2.2 — `ProfileApprovalCapacityGateTest` needed the same at-parity arrangement `ProfileBirthStateRoutingTest` had already named `clubAtCapacity()`. Verified empirically with a throwaway probe file: the duplicate declaration fataled inside `TestSuiteLoader::load()`, before a single test ran. Renamed to `seatClubTo()` / `freeOneSeat()`. *(Confirmation date = archive-dir date once archived.)*

**Applies to.** Every capability whose tests span more than one file — which is most of them. Tasks 3.1 / 3.2 / 6.1 of this change each want a "Club at exact parity" fixture and each need their own name. Also a live hazard for the Module A / B oversell suites, which will want `allocationAtCapacity()` / `stockAtFloor()` in several files apiece.

## A guard-ORDER claim is provable only NEGATIVELY — assert the skipped gate left no trace, never the reason it reported

**Hypothesis.** When a transition evaluates several rejection gates in a mandated order (from-state → grace → capacity; or KYC → sanctions → Hold), the natural test — *"a past-grace call reports the grace reason"* — **does not discriminate the orderings**. Whenever the later gate would also have passed, both orders report the same reason, so the test is green against the wrong implementation; and when the later gate would have failed, the assertion merely restates the guard it is meant to be checking. The property that actually distinguishes them is what the skipped gate **did not do**: it emitted no query, took no lock, made no port call. Assert that absence — `DB::listen` filtered to the later gate's table, `expect($statements)->toBeEmpty()`; or a counting fake of the port that must record zero calls.

**Prescription.** Pin the order by its side-effect footprint, not by its verdict. Then drive the dataset across the *later* gate's outcomes ({would-fail, would-pass, disabled}) so the earlier gate is shown to be genuinely independent of it rather than merely agreeing with it. The absence is usually also the operationally interesting claim in its own right: a doomed call that takes a row lock serialises unrelated healthy transactions on PostgreSQL for no reason.

**Corollary — TWO mutants, and the second one is the necessity proof.** A mutant that reds *both* the new negative pin and the pre-existing positive tests shows only that the pin **fires**; delete the pin and the suite still catches that mutant, so it does not show the pin is **needed**. Necessity is proved by the *weaker, more realistic* mutant that the existing suite passes and only the new pin fails. For an ordering claim the extreme mutant moves the guard below the whole gate block (the later gate then swallows the call outright, and every positive test screams); the realistic drift moves it just past the lock/count and **before** the gate's `if` — the verdict is unchanged, only the footprint moves. Run that one. Until it has run, "the positive assertion discriminates nothing" is an argument, not a measurement — and it is the argument on which the whole negative-pin technique rests.

**Confirmations: 2/3** (need 1 more distinct change).
- `parties-hero-package` task 2.4 — `RenewProfile` orders from-state → 30-day grace → `parties_clubs` lock → seat count → capacity gate. Asserting the past-grace *reason* proves nothing: at a free seat, capacity-first also reports grace. The discriminator is that a past-grace call emits **no `parties_clubs` statement at all**, asserted across a past-grace × {at parity, free seat, explicitly-uncapped} dataset. *(Confirmation date = archive-dir date once archived.)*
- `parties-hero-package-residuals` task 1.1 — the same shape on `ApproveProfile` (from-state → Club lock → count → capacity gate), and the **source of the two-mutant corollary**. Mutant A (guard below the entire gate block) killed 13 tests, 7 of them in the *pre-existing* positive from-state test — impressive and uninformative. Mutant B (guard between `lockAndCountOccupiedSeats()` and the gate `if`) left all 13 pre-existing tests **green** and reddened exactly the 6 new dataset rows, every one on `expect($statements)->toBeEmpty()`. A guard silently drifting below the Club-row lock was invisible to the entire repo before this pin. *(Confirmation date = archive-dir date once archived.)*

**Applies to.** Every multi-gate transition-initiation surface: the compliance gates of invariant 7 (KYC / sanctions / Hold, which must run before any charge attempt — a Hold-blocked customer must never reach the payment provider), Module A's `qty − issued` gate behind its own from-state guard, and Module S checkout, where an out-of-state order must not lock inventory rows.

## A deliberate NON-gate is invisible to a diff — its test proves nothing until the gate is injected and the test goes red

**Hypothesis.** When a design decides that some guard/gate/side-effect must **not** exist at a given seam (`ReactivateProfile` is never capacity-gated; no listener auto-promotes a waitlisted Profile; an out-of-state call takes no lock), the resulting test asserts a *success* — "this transition reaches `active`". Such a test passes identically against the correct implementation **and** against one carrying the forbidden gate, unless the fixture is arranged so the gate would certainly have fired. Nothing in a diff shows an absent `if`, no reviewer sees the omission, and the acceptance bullet reads as satisfied. The failure mode is real and expensive: a later iteration "completes" the gate for symmetry, the suite stays green, and a temporary restriction silently becomes a permanent one.

**Prescription.** Falsify the test before trusting it. `cp` the target aside, inject the forbidden gate, run **the new test file alone**, confirm it goes red, restore, re-run. Record in `progress.md` how many assertions the mutant killed — an omission-test that survives its own mutant is a placebo. Arrange the fixture at the exact boundary the gate would refuse (for a capacity gate, `0` is at parity for every from-state), and pin the absence structurally as well as behaviourally: constructor **parameter type list** (a `grep` passes on a renamed promoted property), `not->toUse([...])` for the import route, and a `DB::listen`-filtered `toBeEmpty()` for the route that imports nothing at all.

**Corollary — write ONE PIN PER ROUTE IN, and pin by TYPE, not only by NAME.** A structural absence ("no listener/scheduler/job/observer does X") is a claim about a *set of mechanisms*, and each needs its own assertion: the directory name, the **implemented interface**, the container/dispatcher **registration**, and — for a scheduler, which no behavioural test can ever observe, since it does not tick inside the suite — the `Schedule::events()` list. A name-shaped pin (*"this module declares no `Listeners/` directory"*) reads like the whole claim and is not: it merely tells a future promoter where not to stand. Design at least one mutant to **hide** from the pin you are most confident in; if it succeeds, that pin was decorative.

**Corollary — a refactor of an absence test invalidates its earlier falsification.** When an assertion body is rewritten after the mutation run (to satisfy the type-checker, say), re-run the relevant mutant. The mutation result attaches to the assertion as written, not to the test's name.

The Club is at its Hero-Package capacity — 2 of 2 seats are occupied"* — is verbatim the `AC-K-FSM-2a` eviction the non-gate exists to prevent.
  - **4.1** — *no listener, scheduler, job or observer promotes a waitlisted Profile* (D5). Seven mutants, all 15 tests falsified. ## Drive a two-transaction race from ONE process: nest the holder, swap the default connection, and make the block observable with `lock_timeout`

**Hypothesis.** A row-lock serialisation fix (`lock; then count; then act`) is only falsifiable when two transactions are open on the same row at once — and PHP's test process is single-threaded, so the holder can never commit while the waiter blocks. The three-part harness that resolves this, without forking and without touching the Action under test:

1. **Hold the lock by nesting.** Open transaction A **by hand** on the default connection (`DB::connection($default)->beginTransaction()`), then invoke the real Action. Its own `DB::transaction` nests as a SAVEPOINT, so when it returns, A is still open and still holds every row lock it took — exactly where a real in-flight call sits at the gate. Releasing a savepoint does not release locks.
2. **Give B its own session without editing the Action.** Register a second connection whose config is a copy of the default's (`config()->set('database.connections.X', config('database.connections.'.DB::getDefaultConnection()))`), then `DB::setDefaultConnection('X')` for the duration of B's call. The Action's `DB::transaction`, its models and the `DomainEventRecorder` all resolve `DB::connection(null)` → `config('database.default')`, so the **unmodified** Action runs on an independent backend.
3. **Make the wait observable, not eternal.** `SET lock_timeout = '750ms'` on B. B blocks on A's row, the statement is cancelled, and `QueryException` carries SQLSTATE **`55P03`** (`lock_not_available`) — locale-independent, and the right needle when no constraint name exists. *The timeout **is** the serialisation observation:* the racy implementation never blocks at all, it reads a committed occupancy and proceeds. Then commit A and re-invoke B: that is precisely what a real blocked backend executes once the lock is granted (re-run the cancelled statement, then count afresh), and it yields the criterion's outcome.

**`pcntl_fork` is available and is the WRONG tool here.** A forked race is more "real" but nondeterministic *in the red direction*: against the racy implementation the child may finish before the parent starts, so the test passes by luck. Determinism in **both** directions is the criterion for a race test — a proof that only sometimes catches the bug is worse than useless, because a green run then licenses the bug.

**Prescription.** `DatabaseMigrations`, never `RefreshDatabase` — a wrapper transaction leaves the fixture uncommitted and therefore invisible to session B, and demotes the hand-opened transaction to a savepoint holding no top-level lock. Roll the holder back in a `finally` (`if ($holder->transactionLevel() > 0)`) or the trait's teardown `migrate:rollback` blocks on its locks and **hangs the suite**. Set `lock_timeout` well above an uncontended `SELECT … FOR UPDATE` on a one-row table, or the companion *"different rows stay parallel"* assertion turns flaky. Finally, **mutation-prove both lanes**: the PG lane must go red against the racy implementation, and the SQLite lane must stay **green** under the same mutant — that green is the *evidence* for the documented engine asymmetry, not merely a claim about it.

**Confirmations: 1/3** (need 2 more distinct changes).
- `parties-hero-package` task 3.2 — `ApproveProfile`'s `parties_clubs` row lock (design D3). Two sessions, one free seat, two `Applied` Profiles ⇒ exactly one `Active`, one `WaitingList`. Mutating `lockAndCountOccupiedSeats()` → `countOccupiedSeats()` reds the PG lane at the `55P03` assertion (B sails through and oversells 2-of-1) and leaves the SQLite lane green — the asymmetry, demonstrated. *(Confirmation date = archive-dir date once archived.)*

**Applies to.** Every enforcement site of invariants 1 and 2 — Module A's `qty − issued ≥ 0` per sub-pool and Module B's `physical_in_storage − reserved − quarantined − under_adjustment ≥ 0` — both check-then-act over a derived aggregate, both needing this harness for their PG lane. Also `InlineDeliveryExecutor`'s `lockForUpdate` winner-election, where the claim is that two concurrent sweeps deliver an event exactly once.

## `.env.example` is a test-environment file: document a new env var COMMENTED OUT, and pin the comment

**Hypothesis.** An env var added to `.env.example` with an active value silently reconfigures the whole test suite, because three facts compose:

1. `LoadEnvironmentVariables` prefers `.env.{APP_ENV}` **only if that file exists**. This repo has **no `.env.testing`**, so `APP_ENV=testing` falls back to plain **`.env`**.
2. `phpunit.xml` overrides only the ~13 keys it lists (`DB_CONNECTION`, `CACHE_STORE`, …). Every other key in `.env` reaches `env()` and therefore `config()` during tests.
3. `docs/development.md` instructs every developer and CI job to `cp .env.example .env`.

So `.env.example` is transitively a test-environment file. A var whose **absence** is the suite's baseline posture (an uncapped gate, a disabled feature flag, a null driver) must therefore ship **commented out** — the `EVENTS_SWEEP_*` block is the house precedent. Documenting it by activating it is the failure mode, and it fails *quietly*: only the config-sensitive tests move, and they move consistently, so the run looks like a real regression somewhere else.

**Prescription.** Ship the var commented out with a comment saying *why*, then pin the ban mechanically in the change's own test file:
- `.env.example` names the var (the doc exists), and **every line naming it starts with `#`** after `ltrim` (the doc cannot be armed by accident);
- `config/<file>.php` contains `env('<THE_SAME_KEY>')` (a rename cannot orphan the doc).

Read both files with the house idiom, `(string) file_get_contents(base_path(...))`. Mutation-prove it two ways: uncommenting the line and renaming the config key must each red exactly this test and nothing else.

**Corollary — the baseline posture has no other guardian.** The uncapped/disabled default that hundreds of existing tests are written against is protected by the variable's *absence*, not by `phpunit.xml`. Never add such a key to `phpunit.xml` "for clarity": that pins the value, and the next task that needs the other value has nowhere to stand. Bind per-test with `config()->set(...)` instead — and bind it as a **string**, since `env()` yields a string and the config sources the default straight from it (empirically: a `.env` entry arrives as `"2"`, not `2`). Testing with an `int` skips the adapter's cast, which is exactly the boundary worth exercising.

## A chained `expect()->and()` SHORT-CIRCUITS — a conjunction pin needs one mutant PER CONJUNCT, and the assertion COUNT is the proof the later ones ran

**Hypothesis.** A pin written as one chained expectation — `expect($e->causation_id)->toBeNull()->and($e->correlation_id)->toBe($e->event_id)` — reads like a single claim and is **two independent claims**. Pest evaluates the chain left to right and throws on the first failure, so a mutant that breaks conjunct 1 **never evaluates conjunct 2**. The task's prescribed single mutant therefore falsifies only the first half; the rest of the chain is unfalsified, and an unfalsified assertion is decoration that a later iteration will "clean up". The trap is worst where the conjuncts look like restatements of one property (both halves of *root-ness*; `entity_type` + `entity_id`; a status **and** its message) — the reviewer reads one idea, the suite pins one and a half.

**Prescription.** Count the *independent ways the claim can break*, not the assertions, and run one mutant per way. Design each mutant to leave the other conjuncts **green**, so each proves its own half load-bearing on its own. For an event's root-ness the two are asymmetric and both realistic: a `causationId:` re-parents the event under its predecessor, while a shared `correlationId:` — the tempting *"one correlation per transaction"* — leaves `causation_id` null and still collapses two sibling roots into one chain. Neither mutant implies the other.

**The assertion count is the tell, and it is free.** Pest reports `assertions:` per run. A mutant that reds conjunct 1 shows **one fewer** assertion than the green baseline (the chain aborted); a mutant that reds conjunct 2 shows the baseline count (conjunct 1 ran and passed). Read those two numbers and the short-circuit is no longer a hypothesis about your test — it is a measurement. If both mutants report the same count, one of them never reached the assertion you thought it was falsifying.

**Corollary — a conjunct may have NO mutant, and then you must enumerate its CAUSES rather than hunt for one.** "One mutant per conjunct" assumes every conjunct is falsifiable by production drift. Some are not: the property is held up by schema, by a trigger, and by an earlier assertion, and no reachable edit to the Action can break it. Do not conclude the conjunct is decoration and delete it, and do not fabricate a mutant the production code could never contain — enumerate the ways the property could go false, show each is already loud, and write that enumeration into the test comment. An assertion that *cannot* fail today is worth keeping only if the next reader can see why it was kept; otherwise the "clean it up" instinct is correct and it will be deleted. The tell that you are in this case: every candidate mutant reds an *integrity error* or an *earlier assertion*, never the line you aimed at.

**Corollary — an envelope pinned on event E in Action X says NOTHING about event E in Action Y.** The conjuncts multiply by `record()` **call site**, not by event class, and `grep`ping the event name will report the property covered when it is not. The mechanism is the two-entry-point rule (design R3) applied to the *actor* envelope rather than to causality. Locate the call site before concluding a pin is redundant.
