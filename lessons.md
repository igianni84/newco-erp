## 2026-06-12 — pest-plugin-arch: `not` over an ARRAY *source* inverts the aggregate (silently masks a violation)
- **Mistake:** Wrote the platform-direction arch test (task 2.3) as `expect(['App\Providers','App\Models','App\Http'])->not->toUse('App\Modules')`, assuming an array source means "NONE of these may use the target". The suite stayed GREEN even with a deliberate violating fixture (`app/Providers/` class importing a module) present — a vacuously-passing test.
- **Correction:** Empirically probed the API (per design D4): `expect([sources])->not->toUse(X)` negates the *aggregate* positive `toUse` ("ALL sources use X"), so it passes the moment ONE source is clean — `App\Models` not using `App\Modules` satisfied the `not`, masking the dirty `App\Providers`. The fix is to LOOP and assert one single-source `expect($ns)->not->toUse(X)` per source (mirrors the 2.2 per-module loop): reads "this source uses none of X", fails on any one dirty source.
- **Rule:** In pest-plugin-arch, put arrays on the **dependency** side only (`not->toUse([d1,d2,…])` = "uses ANY of these" — safe). NEVER put an array on the **source** side under `not`; loop instead, one `expect($source)->not->toUse(…)` per source. The mandatory red-proof is what caught this — a boundary test never seen red does not count as done (design Risk "vacuously green arch tests").

## 2026-06-13 — `$this->get()` in a Pest test reds PHPStan max — use the `Pest\Laravel\*` typed globals
- **Mistake:** Wrote the first HTTP Feature test (`WelcomePageTest`, task 2.4) with `$this->get('/')->assertOk()->assertSee(...)`. The full suite was green, but PHPStan max threw 11 errors: `Call to an undefined method Pest\PendingCalls\TestCall::get()` (`method.notFound`) + a cascade of `Cannot call method assertOk()/assertSee()/assertDontSee() on mixed` (`method.nonObject`). Inside a Pest `it()`/`test()` closure, `$this` is **statically** `Pest\PendingCalls\TestCall`; the runtime rebinding to `Tests\TestCase` (where `MakesHttpRequests::get()` lives) is invisible to static analysis, and no Pest→PHPStan binding plugin is installed (`pestphp/pest-plugin-laravel` ships runtime globals, not PHPStan rules). Every earlier Feature test passed only because it never called a TestCase *instance* method — facades (`App::`, `Lang::`, `Schema::`), `expect()`, `trans()`/`__()` are all global/static.
- **Correction:** `use function Pest\Laravel\get;` and call the **global** `get('/')` (likewise `post`, `getJson`, `actingAs`, `from`, … in `vendor/pestphp/pest-plugin-laravel/src/*.php`). These are documented `@return TestResponse`, so the whole `get('/')->assertOk()->assertSee($v)->assertDontSee($v)` chain resolves clean (the assertion methods take an untyped `$value` and `return $this`). No new dependency (plugin already installed); did NOT add a Pest PHPStan plugin (this change is dep-bounded to Pennant).
- **Rule:** In Pest tests, reach for a TestCase instance method (HTTP `get/post/getJson`, `actingAs`, `withSession`, `travelTo`, …) **only** via the `Pest\Laravel\*` global function of the same name (`use function Pest\Laravel\<fn>;`) — never `$this-><fn>()`, which is unresolvable at PHPStan max. Facades/`expect()`/helpers stay as-is. This is the standing convention for ALL future route/controller/Livewire/Filament Feature tests (Module S, Admin Panel, …). (Captured as the "HTTP/instance-method Feature test under PHPStan max" Codebase Pattern in the change's progress.md.)

## 2026-06-18 — Two chained `->not->…` assertions red PHPStan max — re-anchor with `->and($x)`

## 2026-06-18 — Spreading a bare-`array` Pest dataset param into a string-keyed literal reds PHPStan max

## 2026-06-20 — A multi-method Pest expectation chain reds PHPStan; `toContain` is clean only as a single modifier
- **Mistake:** In the task-1.3 boundary-test guard I asserted an allow-list with one chained expectation: `expect($list)->toContain($a)->toContain($b)->not->toContain($c)`. Suite green, but `type_check` threw 6 errors — `Access to an undefined property Pest\Mixins\Expectation<list<string>|null>::$not` and `Cannot call method toContain() on mixed`. The larastan Pest extension loses the `Expectation<TValue>` generic across a long mixed chain (multiple `->toContain()` then `->not`), degrading it to `mixed`/`…|null` so the next `->toContain`/`->not` is unresolved. Existing clean usages all use `toContain` as a **single** modifier (`->and($s)->toContain('x')`, `->not->toContain($v)`).
- **Correction:** Replaced the chain with one assertion per `expect()` over an explicit boolean: `expect(in_array($a, $list, true))->toBeTrue();` / `expect(in_array($c, $list, true))->toBeFalse();`. `expect(bool)->toBeTrue()/->toBeFalse()` is the most basic Pest assertion and is unconditionally phpstan-clean. No inline `@var`, no ignore, no baseline.
- **Rule:** Don't chain multiple expectation methods on one Pest `expect()` when `type_check` must stay green — especially `->toContain()->toContain()->not->…`. One assertion per `expect()`; for array membership prefer `expect(in_array($x, $list, true))->toBeTrue()/->toBeFalse()` over `->toContain()`/`->not->toContain()` chains. (Same family as the bare-`array`-spread `argument.type`, the arch-array-under-`not`, and the two-chained-`not` lessons: the runtime is fine, the larastan static analysis is not — verify any non-trivial expectation shape against `phpstan analyse`, not just a green suite.)
