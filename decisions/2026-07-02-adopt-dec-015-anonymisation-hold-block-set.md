## Decision: adopt canon **DEC-015** locally — the GDPR-anonymisation Hold-precedence block-set is **`compliance`-only** over the whole `HoldType` domain; plus three companion scopings this change records (sanctions-retention via a `compliance` Hold, minimal-synchronous J-9b export, a PII-free `CustomerAnonymised` event)

The `AnonymiseCustomer` action's regulatory-retention gate **blocks anonymisation iff the Customer is covered by an active `compliance` Hold, and no other Hold type blocks**. Coverage is read through the within-module `PartyComplianceStatusReader` read-contract (never the `Hold` Eloquent model directly — the no-model-leak boundary law). Precise scope, four recorded points:

- **(a) Block-set = `compliance`-only — and this resolves a contradiction inside our own frozen `spec/`, it does not merely pick a side.** Our frozen spec states the anonymisation × active-Hold precedence in **three** places that **disagree**:
  - `spec/04-decisions/decisions.md` **DEC-027** Stage-6.5 (line 198): regulatory-retention Holds = *"sanctions Holds carrying OFAC / UIF obligations"* block; the non-blocking list is `{payment, fraud, kyc, compliance, admin}` — i.e. **`compliance` does NOT block**.
  - `spec/02-prd/Module_K_PRD_v0.3-MVP.md` **§8.2** (line 454): regulatory = *"sanctions Holds … **and `compliance` Holds**"* block; non-blocking `{payment, fraud, kyc, admin, credit}` — i.e. **`compliance` DOES block** (and `credit` is now the non-blocker instead).
  - `spec/03-acceptance/Module_K_Acceptance_v0.3-MVP.md` **AC-K-J-9a** (line 87): block `{sanctions-OFAC, compliance}`; don't block `{payment, kyc, admin, fraud, credit}` — sides with §8.2.

  So DEC-027 (which even **cites §8.2 as its anchor**) says `compliance` is non-blocking while §8.2 / AC-K-J-9a say it blocks; the two non-blocking lists also disagree on `credit`. **Canon MVP-DEC-015 collapses all of it to `compliance`-only**, adopting the §8.2 / AC-K-J-9a reading (`compliance` blocks) and discarding the phantom sanctions-Hold blocker (point b). This is **count-independent** — the gate keys only on `compliance`, so it is agnostic to whether the domain is the frozen spec's six types or the shipped code's **eight** (DEC-008 / RM-04): the RM-04 `Hold Registry` 6→8 truth-spec debt does **not** gate this change.

- **(b) The "sanctions Hold" all three frozen statements name does not exist — sanctions-retention is enforced via a `compliance` Hold.** The `HoldType` enumeration (§4.8) is **six** in the frozen spec (`admin | kyc | payment | fraud | compliance | credit`) and **eight** in shipped code (`app/Modules/Parties/Enums/HoldType.php` — the six plus `chargeback_review` + `storage_payment_failed`, DEC-008). **None is `sanctions`.** Sanctions is a **separate `sanctions_status` FSM** on the Customer (`app/Modules/Parties/Enums/SanctionsStatus.php`: `pending | passed | failed | under_review`; `spec/…/Module_K_PRD…md` §9.2), exposed *alongside* Holds in the read-API tuple `(sanctions_status, active-Hold-list)`, never *as* a Hold type. The precedence prose conflated an FSM state with a Hold type. Canon's reconciliation, adopted here: **a sanctioned Customer whose identifiable data must be retained is gated by Compliance placing a `compliance` Hold** — the type that actually exists. The `sanctions_status` FSM is deliberately **not** wired into this gate.

- **(c) The J-9b data-export is scoped minimal / synchronous / in-memory** (design D5). Canon `J-9b` is a **NEW** criterion with **no committed definition anywhere** — only the validation canon-overlay parenthetical (`Module_K_Verdict_v0.3-MVP.md:152`) and the frozen PRD §12's statement that right-of-access is *"Operationally executed by Compliance + Customer Care; **not modelled as a state machine**"* (line 599). `ExportCustomerData` therefore assembles an **in-memory** structured payload (Customer PII + a by-id manifest of retained transactional history) and returns it — **no file persistence, no domain event, no mutation**. This satisfies the compliance narrative and the tracker's Done-when without inventing an async/document pipeline and, crucially, **without tripping the still-undecided object-storage-for-documents ADR** (its gate is INV1 issuance, a future change).

- **(d) A PII-free `CustomerAnonymised` domain event is added over the frozen spec's event-free anonymisation** (design D3). Frozen §15.1's Customer-family events name **no** anonymisation event (`CustomerCreated | Activated | Suspended | Reactivated | Closed | HoldPlaced | HoldLifted | SegmentChanged`); §8.2 records anonymisation as a **state write only**. We add `CustomerAnonymised` (payload = Customer id + `anonymised_at`, **PII-free**) because (i) it is the module's consistent emit-on-state-change pattern, (ii) it is the clean **seam** for the deferred HubSpot PII-removal sync (AC-K-J-9 *"in the same operation"* — HubSpot is unbuilt), and (iii) being PII-free it respects the event-substrate discipline. Recorded here as an addition-over-silence.

## Context: why this came up

This is **RM-01**, the Round-2 P0 compliance-floor headline: the GDPR right-to-erasure floor (`AC-K-J-9 / J-9a / FSM-16 / BR-Customer-2`) is **100% unbuilt** — no anonymise action, no `anonymised_at`, no PII-overwrite, no Address entity (2026-07-01 Module K validation, `docs/validation/Module_K_Verdict_v0.3-MVP.md` §§ 33–34, 64, 72, 159, 175). Only the erasure **seam** exists (PII-free `domain_events`; the `audit_records` before/after redaction path with the reserved PG `redactor` role; Customer PII confined to the module table). RM-01 is that erasure job — the very *"later change"* the event-substrate and identity-auth ADRs reserved the seam for.

Same **escalation-asymmetry** as RM-04/DEC-008 and RM-10/DEC-018: the canon corrections MVP-DEC-008..023 never flowed into our frozen snapshot, which **stops at DEC-007** (team-memory `spec-divergence-from-cmless-documentation`). `MVP-DEC-015` refines the erasure block-set to *"only `compliance` blocks; no separate `sanctions` Hold"* over the eight-type domain — it is absent from `spec/` and lives in our repo only as the validation **Canon Overlay** row (`Module_K_Verdict_v0.3-MVP.md:152`, 🔴 High / floor; `AC-K-J-9a:34` *"See Canon Overlay (DEC-015)"*). **Giovanni confirmed 2026-07-02** that the block-set is canon `compliance`-only and that J-9b ships minimal-synchronous. Because invariant #11 forbids editing `spec/**`, the resulting code↔frozen-spec divergence (and the two companion additions-over-silence, c/d) are recorded **here**. This mini-ADR **precedes** the implementation (tasks.md 1.1).

Per the standing rule (`lessons.md` 2026-07-02, confirmed 3× — RM-04/DEC-008, RM-10/DEC-018, RM-24/DEC-023), **adopting a canon MVP-DEC absent from the frozen spec always earns a mini-ADR**, regardless of the Remediation Tracker's advisory *"ADR? — mini"* column (`docs/validation/Remediation_Tracker.md` RM-01). Unlike RM-24 (a zero-behaviour codification of an already-satisfied invariant), this change **builds a floor feature from nothing**, so the ADR also fixes the frozen-spec contradiction the code must not inherit.

## Alternatives considered

- **Keep the frozen two-category rule (`compliance` + a `sanctions` Hold block).** Rejected — there is **no `sanctions` Hold type** to key on (the enum is six/eight, none `sanctions`), so it would need bespoke wiring of the separate `sanctions_status` FSM into the gate; and it contradicts canon MVP-DEC-015, which dropped the sanctions-OFAC block in favour of `compliance`-only. It also inherits, rather than resolves, the DEC-027-vs-§8.2 disagreement.
- **Keep greenfield DEC-027 as literal (`sanctions`-only blocks; `compliance` non-blocking).** Rejected — the **oldest** of the three frozen statements, contradicted by both §8.2 and AC-K-J-9a (which DEC-027 itself cites), and by canon. It would leave a `compliance`-Hold'd Customer wrongly anonymisable.
- **Event-free anonymisation (frozen §15.1 literal).** Rejected — leaves the deferred HubSpot PII-removal (AC-K-J-9 *"same operation"*) with **no seam** and breaks the module's emit-on-every-state-change pattern. A **PII-free** event costs nothing against the event-substrate discipline (point d).
- **A persisted / downloadable J-9b export.** Rejected — invents a document shape with no spec basis and **trips the undecided object-storage-for-documents ADR early** (gate = INV1 issuance). Minimal in-memory is the smallest thing that satisfies the compliance narrative (point c).
- **Defer J-9b entirely.** Rejected — the tracker's RM-01 Done-when lists export, and Giovanni chose minimal-now (2026-07-02). A synchronous read-only assembly is cheap and additive; a richer machine-readable format is a J-9b follow-up if canon later defines one.
- **Skip the mini-ADR (tracker column reads "mini").** Rejected — invariant #11 bans editing `spec/**`, so `decisions/` is the **only** home for a code↔canon divergence, and here the divergence is compound (canon block-set + a self-contradictory frozen spec + two additions-over-silence). RM-04/10/24 set the precedent: a canon adoption → a mini-ADR.

## Trade-offs accepted

- **The `sanctions_status` FSM is deliberately not a gate input.** A sanctioned Customer requiring data retention depends on Compliance operationally placing a `compliance` Hold. Accepted: it matches canon and code reality (no `sanctions` Hold type), and keeps the gate single-keyed and count-independent (immune to the RM-04 6→8 debt). The operational dependency is the compliance team's existing responsibility, not new machinery.
- **We knowingly diverge from a frozen spec that is itself internally inconsistent** on the block-set (DEC-027 vs §8.2 vs AC-K-J-9a). Accepted and recorded here; the delta spec's *Anonymisation Hold Precedence* requirement cites this ADR as the reconciliation.
- **The `CustomerAnonymised` consumer (HubSpot PII-removal) is an unexercised, deferred seam** until the HubSpot integration change — documented, not dead code, mirroring every other Module-E/S consumer seam in Parties (`PartiesServiceProvider::boot()` empty).
- **J-9b ships without a persisted, machine-readable export artifact.** Accepted — the in-memory payload satisfies the launch floor; persistence waits for a canon definition and the object-storage ADR, so we do not pre-commit a document shape.
