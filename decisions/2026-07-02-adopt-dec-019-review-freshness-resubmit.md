## Decision: adopt canon **MVP-DEC-019** locally — enforce the **review-freshness invariant** on the catalog approval flow (an un-remediated rejection **blocks** `reviewed → active`) and add an explicit **re-submit** operation (`reviewed → reviewed`, audit-only, the twin of `reject`) across all seven spine entities, both **derived from the audit trail** (no schema flag)

Canon **MVP-DEC-019** (review-freshness) is adopted. It sharpens what our own spec already required into an enforced invariant. Precise scope:

- **Block-gate on un-remediated rejection.** `activate` (`reviewed → active`) SHALL be rejected while the entity's **latest governance decision is a rejection**. This is enforced once, in the shared `ApprovalGovernance::guard()` `Activate` branch (beside the SoD floor), throwing `ApprovalGovernanceViolation::activationBlockedByPendingRejection($entity)`. One guard point → all seven catalog spine entities covered (the FSM is shared — `catalog-lifecycle-approval` **D1**).
- **Explicit `re-submit` operation.** A new `LifecycleTransition::resubmit(Model, string $entity)` — the twin of `reject()`: `reviewed → reviewed`, one `audit_records` row `catalog.<entity>.resubmitted` (before/after `{lifecycle_state}` + `decision: resubmitted`), operator-floor-gated, **no domain event**. Re-submission clears the rejection-pending condition and re-arms review. Seven thin `Resubmit{Entity}ForReview` actions (mirroring the seven `Reject*Review`).
- **Derive-from-audit, not a persisted flag** (design **D3**, below).
- **The edit-re-arms leg is NOT built here** — deferred to **RM-14** (below).

## Context: why this came up

Our **own** baseline already required this flow and we shipped it only **Partial** — canon then sharpened the same direction:

- **Our truth spec** — `openspec/specs/product-catalog/spec.md` *Approval Governance* (`:214`, rejection paragraph `:220`, scenario `:240`): a rejected entity "SHALL **stay in `reviewed`** … the Creator SHALL edit the entity **in place** … and **re-submission SHALL restart the approval flow**," with "the full rejection history … preserved."
- **The frozen spec** — `AC-0-J-7` (`spec/03-acceptance/Module_0_Acceptance_v0.3-MVP.md:78`), `AC-0-BR-Lifecycle-6` (`:133`), `BR-Lifecycle-6` / § 4.3 (`spec/02-prd/Module_0_PRD_v0.3-MVP.md:429`): same — reject stays `reviewed`, edit-in-place, **re-submit**, "approval restarts from review," history preserved.

But the **build was Partial** (2026-07-01 Module 0 validation): the reject-stays-`reviewed` + notes + append-only history half shipped, but there was **no re-submit operation** and **nothing enforced that a pending rejection blocks activation**. Worse, `tests/Feature/Modules/Catalog/ProductMasterLifecycleTest.php:363-387` ("rejection is not terminal") asserted a distinct approver activating a just-rejected entity **with no fresh review** as *correct* behaviour — encoding the Partial gap as if intended. Verdict: `docs/validation/Module_0_Verdict_v0.3-MVP.md:30` (AC-0-J-7 Partial), `:52` (BR-Lifecycle-6 Partial), `:117` (Probe 1 — "a distinct approver can approve directly post-rejection with no fresh review"), Canon Overlay `:102`. Tracked as **RM-06** (`docs/validation/Remediation_Tracker.md:119-124`) — also Paolo's "rejection round" walkthrough scenario (`docs/validation/README.md:48`).

Canon **MVP-DEC-019** is **absent from our frozen `spec/`** (which stops at MVP-DEC-007 — the same escalation-asymmetry as DEC-008/RM-04, DEC-018/RM-10, DEC-023/RM-24: the canon corrections DEC-008..023 never flowed into our snapshot; team-memory `spec-divergence-from-cmless-documentation`). It converts the already-required "restart from review" into an **enforced review-freshness invariant**: a pending rejection *blocks* activation; re-submission is an *explicit* audited action.

> ⚠️ **DEC-019 number collision.** Our frozen `spec/`'s own **DEC-019 is an unrelated Module-S decision** — club Offers must reference a single-producer constituent set (`spec/04-decisions/MVP_Decisions_Register_v0.1.md:50`; `BR-SKU-5` at `spec/02-prd/Module_0_PRD_v0.3-MVP.md:455`; D7 at `spec/02-prd/Module_S_PRD_v0.3-MVP.md:238`). It is **not** the same decision as canon MVP-DEC-019 (review-freshness). Same number, different registers — a collision, not a contradiction. This ADR's "DEC-019" means the **canon review-freshness** decision throughout.

**Honest nuance — which class of adoption is this?** DEC-023/RM-24 recorded three classes of canon adoption (spec self-contradicted; spec coherent-and-reversed; fills-a-silence, zero-behaviour). RM-06 is a **fourth blend**: it is primarily **build-completeness against our own spec** (re-submit + restart-from-review were *already* required; we were Partial), **sharpened by canon** (the explicit block-gate as an enforced review-freshness invariant). Unlike RM-24 it is **behavioural**: it **inverts a shipped, tested path** (`:363-387` "not terminal" → "rejection blocks until re-submit"). The canon adoption itself (the enforced invariant) rides on top of a gap-fill our own spec already mandated — so the mini-ADR is warranted both as a canon adoption *and* as the record of a deliberate behavioural inversion.

## Alternatives considered

- **Block-gate in `ApprovalGovernance::guard()`, scoped to `Activate` (chosen — design D1).** The guard already (a) runs on every commercial-impact transition before the state write, (b) is scoped to `Activate` for approval-specific checks (SoD), and (c) reads the audit trail as the system of record (`creatorOf`/`reviewerOf`). Adding `assertNotRejectionPending($entity, $entityId)` there is the minimal, cohesive home: one guard point, one audit read, all seven entities, `retire`/`reopen` untouched. Rejected: a check duplicated inside each of the seven `Activate*` actions (drift hazard — against D1's one-shared-mechanism); a check in `LifecycleTransition::transition()` itself (would need `type`-gating there too; `guard()` is exactly the governance seam that already reads the trail).
- **`re-submit` as a new `reviewed → reviewed` audit-only op, twin of `reject` (chosen — design D2).** A rejected entity is in `reviewed`, so `SubmitForReview` (`draft → reviewed`) cannot re-arm it (from-state guard). `resubmit()` mirrors `reject()` exactly. Rejected: reuse `submit` (impossible from `reviewed`); a boolean-flag toggle Action (needs a schema column — against D3).
- **Derive rejection-pending from the audit trail; NO schema flag (chosen — design D3).** See below.
- **A literal `rejection_pending` / `review_rejected_at` column on all seven spine tables.** Rejected — the frozen spec says "rejection **flag**," but a persisted column is schema churn ×7, duplicates the audit system-of-record, and contradicts `catalog-lifecycle-approval` **D5** (which already rejected ~21 persisted governance columns and derives creator/reviewer from the trail).
- **A dedicated `ReviewNotFresh` exception vs a new `ApprovalGovernanceViolation` factory.** Chose the **factory** (`activationBlockedByPendingRejection`) — family cohesion + free console surfacing (`ApprovalGovernanceViolation` already routes through the console kit's `surfaceLifecycleOutcome`, since it is what self-approval throws). A distinct exception would need its own notification wiring for no gain.
- **Skip the mini-ADR — the tracker §3 "ADR?" row says "mini" but is advisory.** Rejected. MVP-DEC-019 is a **canon decision absent from our frozen spec**; adopting it is the same class as DEC-008 ([[2026-07-01-adopt-dec-008-hold-types-8]]), DEC-018 ([[2026-07-02-adopt-dec-018-clubcredit-accrued]]) and DEC-023 ([[2026-07-02-adopt-dec-023-product-type-immutable]]) — each got a mini-ADR, and the discipline is now a firm rule (`lessons.md` 2026-07-02, confirmed 3×). Invariant #11 bans editing `spec/`, so `decisions/` is the only home for the code↔canon divergence.

## Trade-offs accepted

### D3 — "flag" interpreted as a derived audit condition, not a persisted column

The frozen spec's wording is "rejection **flag**" (`AC-0-J-7`, `BR-Lifecycle-6`, PRD § 4.3). We interpret **flag** as the **derived audit condition**, not a persisted attribute: `assertNotRejectionPending` reads the entity's latest governance action (the shipped `latestGovernanceAction` shape — `audit_records` filtered by `module`/`entity_type`/`entity_id`, `orderByDesc('id')`, `->value('action')`) and blocks iff it ends in `.rejected`. This reaffirms `catalog-lifecycle-approval` **D5** ("the audit trail is the system of record; no per-entity governance columns") and matches how `ApprovalGovernance` already derives creator (first `*Created` event) and reviewer (latest `.submitted` audit). **This ADR is the record of that interpretation** — the point to reconcile with the spec's literal wording if c-mless ever wants a materialised flag. Accepted cost: the block predicate must be *exact* — only an **un-remediated `.rejected`** blocks; a `.reopened` (after `retired → reviewed`), a `.resubmitted`, or a fresh never-rejected `.submitted` all leave the entity activatable (covered by scenarios in this change).

### Other trade-offs

- **A behavioural inversion of shipped, tested code** — `ProductMasterLifecycleTest.php:363-387` is *replaced* (not extended) by a "rejection blocks until re-submit" scenario. Accepted: the old test encoded the Partial gap as correct; the blast radius is grepped before editing (`grep -rn "not terminal\|latestGovernanceAction\|->reject(" tests/`) and any cross-entity chain that rejects-then-activates gets a re-submit inserted.
- **Seven near-identical `Resubmit*` actions** — intentional; mirrors the existing seven `Reject*Review` / `Submit*ForReview`. The shared mechanism (`resubmit()`) carries the logic; the actions stay thin.
- **`re-submit` carries no "notes" / "what-changed" argument at MVP** — the audit row records actor + `resubmitted`. A free-text re-submission note or a diff of the Creator's edit is RM-14's re-versioning concern, not folded in here (scope discipline).

### D4 — the edit-re-arms leg is deferred to RM-14

MVP-DEC-019 has a third leg: **editing review-governed content re-arms review** (an identity/quality edit auto-re-arms; observational edits don't gate). That leg is **deferred to RM-14** (re-versioning on identity edit). Reason: **there is no edit/Update path today** — no `Update*` action, the Filament catalog resources are read-only (verified in RM-24). The auto-re-arm-on-edit leg would have nothing to hook onto and would be dead code with no caller. RM-14 introduces the edit path and wires the re-arm into it (`docs/validation/Remediation_Tracker.md:181` — "RM-14 … also enables RM-06's edit path"). This change delivers the two legs buildable **without** an edit path: the explicit re-submit + the block-gate.
