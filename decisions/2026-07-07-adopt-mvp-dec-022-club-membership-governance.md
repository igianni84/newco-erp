## Decision: Adopt canon **MVP-DEC-022** (CML-89) locally — the four Module-K business-rule criteria **Club-6 / Identity-6 / Profile-5 / Producer-5**, each at its faithful **build-vs-deferred** boundary

> ⚠️ **Number collision — read first.** This is canon **MVP-DEC-022** (`MVP_Decisions_Register_v0.1.md:147`, *"CML-89 Membership, Onboarding & Club-Governance completion"*), **NOT** the greenfield **`DEC-022`** in our frozen spec (`spec/04-decisions/decisions.md:148`, *"NFC/NFT recovery policies — Claude proposes defaults in BMD"*). Unrelated. Use the full token **`MVP-DEC-022`** everywhere. Same trap the RM-03 ADR flagged for DEC-016, RM-06 for DEC-019, and the sibling [[2026-07-07-adopt-mvp-dec-009-agreement-club-active-scope]] / [[2026-07-07-adopt-mvp-dec-010-settlement-cadence-closed-set]] for DEC-009/DEC-010.

MVP-DEC-022 is **one canon decision structured by sub-question** (the MVP-DEC-014/021 batch precedent): the tech team (CML-89) proposed dev defaults on seven residual Module K product ambiguities; Paolo ratified all seven. **Four** of the seven land a Module-K **BR + AC** and are in this change's scope — **Club-6** (sub-2), **Producer-5** (sub-4), **Identity-6** (sub-5b), **Profile-5** (sub-7). The other three (waitlist-conversion confirm sub-1, credit-validity confirms sub-3, Profile-`role` confirm sub-6) add no BR/AC and are out of scope. All four target BRs are **absent from our frozen `spec/`** (which stops at MVP-DEC-007 — the escalation-asymmetry), so adopting them is **completeness**, not a reversal of any *shipped* rule of ours — canon reverses the *dev's stated default*, not our spec.

**What changes (Module K / Parties) — each criterion at its build-vs-deferred boundary:**

1. **Club-6 (`AC-K-BR-Club-6`, LIVE `:175`) — `registration_flow_type` is an entry channel, never an approval bypass.**
   - **BUILD:** (a) **drop the redundant `invite_only` boolean** (subsumed — `invite_only = true` ⟺ `invitation_only`), sweeping it out of the model, `CreateClub`, the `ClubCreated` payload, factory, seeder, console + lang; (b) make **`open_registration` latent / non-selectable** at launch (a create/update setting it is rejected) — the three launch values are `application_with_approval` (default) / `invitation_only` / `link_onboarding`; (c) **assert no value auto-approves** — every flow still requires the producer/operator approval write to reach `Active` (DEC-069, atomic approve = charge = activation §4.2.1). The 4-case `ClubRegistrationFlowType` enum is **not reshaped** — `OpenRegistration` stays a case, just latent.
   - **DEFER:** nothing material — `open` (auto-join) is *latent by canon design*, not deferred work.
2. **Identity-6 (`AC-K-BR-Identity-6`, LIVE `:146`) — a hard registration age-gate.**
   - **BUILD:** an age-gate in `CreateCustomer` — a self-attested `date_of_birth` implying an age below a **configurable platform min-age constant (default 18)** is rejected with a localized `BelowMinimumRegistrationAge` before any Customer / co-provisioned Account / `CustomerCreated`; **a missing (null) DOB is also blocked** (age attestation is *mandatory*, BMD §2.8 — no attestation fails the gate). The constant mirrors the RM-02 / MVP-DEC-014 enhanced-KYC threshold pattern (representation the dev's, DEC-073). Creation-only — existing Customers are grandfathered.
   - **DEFER:** per-shipping-jurisdiction higher floors (e.g. 21 for US destinations); the launch check is self-attest + payment-method-bound signal only — **no physical-document verification** (canon-specified).
3. **Profile-5 (`AC-K-BR-Profile-5`, LIVE `:164`) — `auto_renew` inheritance.**
   - **BUILD (K-side):** a new `parties_profiles.auto_renew` bool that **inherits the owning Club's new `parties_clubs.auto_renew_default` at creation**, plus an **operator override Action** (sole writer, one `DB::transaction`, audit-only — **no domain event**, the §15.2 Profile event family names none for `auto_renew`).
   - **DEFER:** the **customer self-toggle via the Consumer Portal** (BMD §2.4/B2) — the Consumer Portal does not exist at launch (a blocked frontend seam); and the fuller `renewal_policy` config blob (canon MVP-DEC-013) — we ship **only** its `auto_renew` element as a standalone `auto_renew_default` bool.
4. **Producer-5 (`AC-K-BR-Producer-5`, LIVE `:185`) — review-governed content lock (interim).**
   - **BUILD:** a model-level `Producer` `updating` guard (the RM-24 pattern) throwing `ProducerReviewGovernedContentLocked` when `name`/`description`/`region`/`website` is dirty while the persisted `status = active` — **the safety core** (unreviewed descriptive content never publishes on an `active` Producer). Path-complete (no Action-layer content-edit writer exists to guard). `draft` edits freely; a status/KYC/version-only transition passes.
   - **DEFER:** the **full "edit re-enters Creator → Reviewer → Approver and re-publishes on a fresh pass" UX** — the Producer FSM is linear `draft → active → retired` with **no `reviewed` sub-state and no content-edit path** today (grounding-confirmed: `ProducerStatus` = `Draft`/`Active`/`Retired`), so building the re-arm now is dead code (the RM-06 / RM-14 precedent). When that path lands, the hard lock is **replaced** by the edit-re-arms-review behaviour.

**Canon-adoption *kind* (the taxonomy — this batch is a mix):** Club-6 = **value-domain tightening + field collapse** (closes the selectable set + removes a redundant field — its wide *code* footprint is the 29-ref/16-file `invite_only` sweep, not a shipped-test inversion); Identity-6 = **erratum of omission** (adds a gate BMD §2.8 already mandated but the PRD had dropped); Profile-5 = **erratum of omission** (additive columns + inherit — canon reverses the dev's operator-only half per BMD §2.4); Producer-5 = an **interim safety lock** (canon mandates full re-arm; we ship a *stricter* interim block because the re-arm has no home, zero behaviour change today). Crucially **none is a behaviour inversion of a *shipped rule of ours*** (contrast RM-20): every one of the four BRs is absent from our frozen spec, so blast radius is add-a-negative-test (Identity-6, Producer-5) / add-columns (Profile-5) / a field-removal sweep (Club-6) — never a shipped-assertion flip.

## Alternatives considered

- **(A) Adopt all four at full canon fidelity now. ❌ REJECTED (two build dead code / a blocked seam).** Producer-5's full re-arm needs a `reviewed` sub-FSM + a content-edit path that **do not exist** (linear `draft→active→retired`) — building them now is speculative dead code (RM-06/RM-14). Profile-5's customer self-toggle needs the **Consumer Portal**, which does not exist at launch. Full fidelity would ship untested, unreachable machinery.
- **(B) Carve Producer-5 out of the batch entirely (defer the whole criterion). ❌ REJECTED (drops the safety core).** Giovanni, 2026-07-07: *"includi un guard minimo ora"* — the safety invariant (unreviewed content never publishes on an `active` Producer) is codifiable **today** as a zero-behaviour-change model lock without the re-arm UX. Deferring the whole criterion would leave the invariant unguarded.
- **(C) Adopt each criterion at its faithful build-vs-deferred boundary — ship the enforceable safety core, defer the frontend/UX seams behind a recorded posture. ✅ CHOSEN.** Club-6 fully (a self-contained field collapse + latent-value gate); Identity-6 fully (a creation-time gate on an existing column); Profile-5 K-side (inherit + operator override) with the self-toggle deferred; Producer-5 as an interim lock with the re-arm deferred. Every shipped element is testable against its AC; every deferral is a named seam, not a silent gap.

## Reasoning: why C won

- **Spec fidelity + Simplicity First.** Each criterion's shippable core maps to concrete AC verification paths (under-age reject, latent-value not-selectable, inherited default, locked content) using existing chokepoints (`CreateCustomer`, `CreateClub`/enum, `CreateProfile`, the `Producer` model). No speculative machinery.
- **The safety cores are exactly the enforceable half.** Identity-6's gate and Producer-5's lock are hard invariants (age floor; unreviewed-content-never-publishes); the deferred halves (per-jurisdiction floors; the re-arm workflow) are *refinements/UX*, not the invariant. Profile-5's K-side inheritance is the recordable half; the self-toggle is a Consumer-Portal write with no host.
- **Low blast radius (grounding-confirmed) — three of four.** `CustomerFactory` DOB `1990-01-01` (adult) keeps Identity-6 green; every existing Producer `->update()` touches only `status`/`kyc_status`/`version` so the Producer-5 lock false-positives on none; Profile-5 is additive columns. Only **Club-6** carries a wide code footprint — the `invite_only` sweep (a mechanical removal, one migration + 16 files), not a shipped-assertion inversion.
- **Deferrals are recorded, not lost.** Each deferred seam (Producer re-arm, Profile self-toggle, per-jurisdiction age floors) is named here + in the delta spec `_Source_` lines, so a future change re-arms from a documented boundary rather than rediscovering it.

## Trade-offs accepted

- **Producer-5 interim is *stricter* than canon (block vs re-arm).** Canon lets the edit proceed and re-queues it for review; our interim **blocks the edit outright** on an `active` Producer. Accepted — both satisfy the invariant (unreviewed content never publishes), the block is zero-behaviour-change today (no content-edit path fires it), and it is explicitly the deferred-until-replaced posture, not a permanent divergence.
- **Profile-5 ships half the canon behaviour.** The customer self-toggle (a BMD §2.4 consumer-protection requirement) is deferred to the Consumer Portal. Accepted — the portal is a hard frontend gate that does not exist; the K-side inheritance + operator override are the launchable half, and the deferral is a named seam.
- **Identity-6 blocks a null DOB — an interpretation, not verbatim AC text.** AC-K-BR-Identity-6 asserts the under-min reject; it does not literally enumerate the null case. We read BMD §2.8's *"Mandatory; gating signup"* as: no attestation ⇒ the mandatory gate fails ⇒ block. Accepted as the faithful reading; recorded so a reviewer sees it is a deliberate interpretation (D7 / R6). If any legitimate flow needs a null-DOB Customer, this is the line to revisit — grounding confirms none does.
- **Two net-new columns + one dropped column** (`parties_profiles.auto_renew`, `parties_clubs.auto_renew_default`, drop `parties_clubs.invite_only`). Accepted — additive/removal migrations, Postgres-truthful + SQLite-compatible; no production data (no PG/staging env yet).
- **New localized exceptions.** `ClubNotAcceptingMemberships` (shared with RM-21), `BelowMinimumRegistrationAge`, `ProducerReviewGovernedContentLocked` add to the Parties exception set (EN + IT). Accepted — the module idiom; each gives a console a surfaceable rejection.
