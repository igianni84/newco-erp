### F1 — ~~DemoSeeder is SQLite-only (PG `TRUNCATE` rejects FK-referenced tables)~~ → **prediction falsified, and exactly inverted**  ·  🟧 · ✅ **closed 2026-07-09** (RM-05 build) · found 2026-07-01 (RM-07) · **the real defect is [F11](#f11)**
- **What:** `DemoSeeder::reset()` truncates FK-referenced tables one-by-one (masters, customers, and — new in RM-07 — `domain_events`/`audit_records`). PostgreSQL's `TRUNCATE` refuses a table referenced by a FK unless every referencer is truncated in the **same** statement or `CASCADE` is used; Laravel's PG `withoutForeignKeyConstraints` emits only `SET CONSTRAINTS ALL DEFERRED` (no effect on `TRUNCATE`). So `db:seed --class=DemoSeeder` would fail on Postgres (`domain_events` ← `event_deliveries.domain_event_id` + self-FK `causation_id`).
- **Pre-existing, not a regression:** true of the existing truncations too — RM-07 only extended the sweep. Verified by reading Laravel's `PostgresGrammar::compileDisableForeignKeyConstraints` (= `SET CONSTRAINTS ALL DEFERRED`), not empirically on PG.
- **Why it's fine today:** dev + demo run on SQLite (`withoutForeignKeyConstraints` = `PRAGMA foreign_keys=OFF`, which allows it); suite green. No PG demo/staging env exists yet.
- **✅ FALSIFIED EMPIRICALLY 2026-07-09** (RM-05 build; `db:seed --class=DemoSeeder` run twice against a scratch database on each engine, then dropped):
  - **PG17: the seeder runs, and re-runs.** Laravel's `PostgresGrammar::compileTruncate` (`:706`) emits `truncate "x" restart identity cascade` whenever `static::$cascadeTruncate` is true — and it **defaults to `true`** (`:47`). `CASCADE` is the very escape hatch F1's own text names. The finding read `compileDisableForeignKeyConstraints` and inferred the rest; it never read `compileTruncate`. **A code-reading claim about a second method is a hypothesis, not a verification.**
  - **The `CASCADE` is not silently destructive:** it also truncates `event_deliveries` (FK `domain_event_id` → `domain_events`), which is **not** in `reset()`'s sweep list. Measured after a PG re-seed: `domain_events=2`, `event_deliveries=1`, **orphaned deliveries = 0**. The delivery ledger is emptied alongside the events it points at — correct, but by accident of grammar rather than by the sweep list's intent.
  - **The inversion: it is SQLite that breaks, not PG.** SQLite has no `TRUNCATE`; `truncate()` compiles to `DELETE FROM`, which fires the append-only row trigger from `2026_06_12_000004_add_immutability_triggers.php:94` → `SQLSTATE[23000] … domain_events is immutable: DELETE is rejected`. PG's equivalent is a **row-level** `FOR EACH ROW` trigger (`TG_OP = 'DELETE'`), and **PostgreSQL does not fire row triggers on `TRUNCATE`** — so PG both cascades *and* bypasses immutability, while SQLite refuses. Exactly backwards from F1. → **F11.**
  - **Why the suite never saw either:** every DemoSeeder test uses `DatabaseMigrations`, so `reset()` always meets **empty** tables — `DELETE FROM` removes zero rows, the row trigger never fires, and both engines go green. **A guard that only fires on non-empty state is invisible to a fresh-schema test.**
- **Disposition:** convert `reset()` to one `TRUNCATE … CASCADE` (or `delete()`s) **when a Postgres demo/staging env is stood up** — ties to the hosting/infra open-stack gate (`CLAUDE.md`). Small; promote to an RM row then.

### F11 — `DemoSeeder` is **not re-runnable on SQLite**, though its own docblock promises it is — and on PG it "works" only by bypassing the append-only trigger  ·  🟧 · 🔴 · found 2026-07-09 (RM-05 close gate, while verifying [F1](#f1))

- **What:** `DemoSeeder::reset()` sweeps `domain_events` and `audit_records` (added by RM-07, so the SoD fixture's lineage is deterministic per run). SQLite has no `TRUNCATE`, so Laravel compiles `truncate()` to **`DELETE FROM`** — which fires the append-only row trigger installed by `2026_06_12_000004_add_immutability_triggers.php:94`. **A second `php artisan db:seed --class=DemoSeeder` on a populated SQLite database aborts:** `SQLSTATE[23000] … domain_events is immutable: DELETE is rejected`. The *first* run on a freshly-migrated database succeeds only because it deletes **zero rows**.
- **Why it matters, and where:** `DemoSeeder.php:86-88` states *"Re-runnable: … Run it repeatedly while iterating on the demo."* `docs/development.md:59` documents exactly that invocation (`php artisan db:seed --class=DemoSeeder`), and SQLite is the documented dev/demo engine. **The promise is false precisely where it is meant to be used** — someone iterating on the pre-Paolo demo will hit this on their second run. Today's workaround is `migrate:fresh --seed`, which is not what the docblock or the docs say.
- **The PG side is not a clean bill of health either:** PostgreSQL **does not fire row-level triggers on `TRUNCATE`**, and the immutability guard is a `FOR EACH ROW` trigger (`TG_OP = 'DELETE'`). So on PG the demo reset **cascades past the FKs *and* bypasses the append-only guarantee** the trigger exists to enforce (invariant 4). It is safe today only because `run()` fails closed outside a demo environment. Verified empirically: PG re-seed succeeds, SQLite re-seed aborts.
- **Why the suite is green on both engines:** every DemoSeeder test uses `DatabaseMigrations`, so `reset()` always meets **empty** tables and the trigger never fires. **A guard that only fires on non-empty state is invisible to a fresh-schema test** — the same shape as F5's "counts rows, not statuses".
- **Disposition (do NOT fix silently — this is a demo-facing behaviour change):** three options, none free. (a) Drop `domain_events`/`audit_records` from the sweep and make `migrate:fresh --seed` the documented demo command — honest, cheapest, loses per-run event determinism unless the fixture tolerates it. (b) Have `reset()` **recreate** the two tables (drop + re-run their migrations) instead of deleting rows — preserves determinism on both engines, and stops relying on PG's TRUNCATE bypass. (c) Weaken the trigger outside production — **rejected**: invariant 4 is the reason the trigger exists, and the production guard already draws the boundary. Whichever wins, **add a re-runnability test that seeds twice** (the one assertion neither engine's suite makes today) and correct the `DemoSeeder` docblock. Promote to an RM row when the demo is rebuilt for Paolo.

### F12 — A **`Profile ↔ Customer` lock-order inversion** can deadlock `ApproveProfile` against `SuspendCustomer` / `ReactivateCustomer`  ·  🟨 · 🔴 · found 2026-07-09 (RM-05 §2.7 semantic-verify)

- **What:** `ApproveProfile` locks **Profile → Customer** (`ApproveProfile.php:125` then `:182`, the Originating-Club one-shot lock). `SuspendCustomer` (`:72` then `:102`) and `ReactivateCustomer` (`:70` then `:99`) lock **Customer → Profile** (the cascade over that Customer's Profiles). Two transactions touching the same (Customer, Profile) pair from opposite ends can each hold one row and wait for the other. On PostgreSQL the deadlock detector aborts one with `40P01`; the operator sees a raw 500, not a domain reason.
- **NOT introduced by RM-05.** The Customer lock is the Originating-Club one-shot that has shipped since `parties-membership-activation`; RM-05 only inserted the **`parties_clubs`** lock *between* the two (`ApproveProfile.php:139`), which lengthens the Profile-lock hold window but changes nothing about the relative Profile/Customer order. Recorded here because the §2.7 verify of RM-05 is where it was first seen, not because RM-05 caused it.
- **The new Club lock closes no cycle of its own:** every Module K Action that takes both a Profile and a Club takes them **Profile → Club** (`ApproveProfile.php:125/139`, `RenewProfile.php:101/124`, `IssueClubCredit.php:69/70`), and **nothing** locks Customer → Club. So the RM-05 lock is consistent with the house order and is not part of any cycle.
- **Why no test catches it:** a deadlock needs two live connections racing in opposite orders. The only two-connection test in the repo is `ProfileApprovalSeatRaceTest` (PG17), and it races two `ApproveProfile` calls — the *same* order — which is exactly the case that cannot deadlock. Every other suspension/approval test is single-connection, where lock order is unobservable.
- **Disposition (do not "fix" by reordering blind):** (a) impose a module-wide canonical lock order (Customer → Profile → Club) and make `ApproveProfile` take the Customer lock first — cheap, but it serialises every approval of a Customer against every other, and the OC lock is only *conditionally* needed (`originating_club_id IS NULL`); (b) leave the order and add a retry-on-`40P01` at the Action boundary — no other Action implements retry, and SQLite cannot express it; (c) accept it and document, since `SuspendCustomer` is an operator action and an approval racing a suspension of the same Customer is vanishingly rare in a single-operator console. **Needs a decision before the producer-facing HTTP surface lands** (DEC-083 defers it), because that is when approvals stop being operator-serialised. Not before.
