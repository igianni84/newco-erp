# Internationalisation (i18n)

How user-facing copy is localised in NewCo ERP. Scope at launch: the supported-locale
registry, the `lang/` translation scaffolding and its file convention. The request-time
locale-resolution chain (`Accept-Language` ‚Üí sticky cookie ‚Üí English), the manual switcher
and the translatable-entity primitive (`TranslatableText`) are documented where they land
(Module B / the F2 PIM). Authority: `spec/02-prd/Architecture_v0.3-MVP.md` ¬ß 5.1,
`spec/04-decisions/decisions.md` DEC-031 / DEC-127 / DEC-064,
`spec/03-acceptance/Module_0_Acceptance_v0.3-MVP.md` AC-0-XM-4, CLAUDE.md invariant 12.

## Supported locales

Six launch locales, with English the final fallback (DEC-031, DEC-127):

| Locale | Code (`lang/` dir) |
|---|---|
| English | `en` (fallback) |
| Italian | `it` |
| French | `fr` |
| German | `de` |
| Japanese | `ja` |
| Simplified Chinese | `zh_Hans` |

The single source of truth is the **`App\Platform\I18n\SupportedLocale`** enum (the typed
anchor); `config/i18n.php` derives `i18n.supported` / `i18n.fallback` from it so the two
cannot drift. Validate any locale through the enum (`SupportedLocale::isSupported()` /
`assertSupported()`) ‚Äî never against a hand-listed set. `config/app.php` already sets
`locale = en` and `fallback_locale = en`.

**Adding a locale post-launch is configuration, not a migration** (DEC-031): add a case to
`SupportedLocale`, create `lang/<code>/`, and translate. No schema change.

## File convention: PHP-array group files (not JSON)

Translations live in **PHP-array group files** ‚Äî `lang/{locale}/{group}.php` returning an
array ‚Äî and are referenced with **dotted keys**: `__('welcome.headline')`,
`trans('welcome.tagline')`.

```
lang/
‚îú‚îÄ‚îÄ en/welcome.php   # ['headline' => 'NewCo', 'tagline' => '‚Ä¶', 'coming_soon' => '‚Ä¶']
‚îú‚îÄ‚îÄ it/welcome.php   # ['tagline' => '‚Ä¶']   ‚Üê partial; other keys fall back to en
‚îú‚îÄ‚îÄ fr/welcome.php
‚îú‚îÄ‚îÄ de/welcome.php
‚îú‚îÄ‚îÄ ja/welcome.php
‚îî‚îÄ‚îÄ zh_Hans/welcome.php
```

Laravel also supports JSON string-key files (`lang/{locale}.json`, keyed by the full English
sentence). We **chose the PHP-array group convention** deliberately:

1. **Stable, namespaced keys.** A key is an identifier (`welcome.headline`), not the English
   copy itself. Editing the English wording does not change the key, so it does not churn the
   other five locales' diffs or break their lookups. JSON keys *are* the English string, which
   couples key to content.
2. **Native per-key fallback.** For group files, `Translator::get()` resolves the chain
   `[active-locale, fallback]` (`Translator::localeArray()`), so a key missing in `it` is
   served from `en` automatically ‚Äî exactly DEC-127's "English (final fallback)". This is why a
   locale file may cover only a subset (AC-0-XM-4 allows partial coverage).
3. **Matches the framework's own convention.** Laravel ships its own translations as
   `lang/{locale}/{group}.php` (`auth`, `validation`, ‚Ä¶); staying consistent keeps tooling and
   reader expectations aligned.

## Authoring rules

- **English is the authored baseline and must be complete.** Every key the app introduces is
  defined in `lang/en/<group>.php` in full ‚Äî it is the fallback for all other locales.
- **The other five may stagger.** Partial coverage is allowed (AC-0-XM-4); an undefined key
  falls back to English per key, not per page (DEC-127 item 4 ‚Äî per-attribute fallback).
- **No hardcoded user-facing strings** (CLAUDE.md invariant 12). Every visible string resolves
  through a translation key ‚Äî in Blade, in code, everywhere. The remediated
  `resources/views/welcome.blade.php` is the worked example.
- Translatable **entity attributes** (PIM product copy, etc.) are a different mechanism:
  i18n-keyed JSON per attribute via `TranslatableText` (DEC-064), not `lang/` files. `lang/`
  is for static UI chrome; `TranslatableText` is for per-row content. (Primitive lands in this
  change; first column attachment is F2.)

## Resolving copy

```php
use Illuminate\Support\Facades\App;

App::setLocale('it');
__('welcome.tagline');      // ‚Üí 'Vini pregiati, direttamente dai produttori.' (it)
__('welcome.coming_soon');  // ‚Üí English value (key absent in it ‚Üí falls back to en)

trans('welcome.tagline', [], 'fr');   // explicit locale, ignores the active one
```
