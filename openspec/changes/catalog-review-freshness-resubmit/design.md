## Context

This is a small, focused **completion** of the catalog lifecycle governance shipped by the archived `catalog-lifecycle-approval` change. That change's design **D5** ("Approval governance is audit-derived; the audit trail is the system of record") already stated the intent: *"'rejection-pending' is derived (the latest governance audit action for the entity is a rejection), so re-submission … clears it without a schema flag, and the append-only trail preserves the full history."* But two halves of that intent were never built:

1. **No re-submit operation exists.** D5 imagined re-submission as "a fresh `SubmitForReview` audit" — but `SubmitForReview` is `draft → reviewed`, and a rejected entity is in `reviewed`, so the from-state guard (`catalog-lifecycle-approval` D2) makes it un-runnable from a rejected entity. There is no `reviewed → reviewed` re-arm operation.
2. **No block-gate is enforced.** `ActivateProductMaster` → `LifecycleTransition::transition(Activate)` runs the from-state guard + approval governance (SoD) + Producer gate, but never reads the rejection-pending condition. The shipped test `ProductMasterLifecycleTest.php:363-387` ("lets the approval flow complete after a rejection (rejection is not terminal)") asserts a distinct approver can activate a just-rejected entity with **no** re-submit — the exact behaviour canon MVP-DEC-019 forbids.

Our truth spec (`product-catalog` → *Approval Governance*, `:220`) and the frozen spec (`AC-0-J-7`, `BR-Lifecycle-6`, PRD § 4.3) already require "re-submission SHALL restart the approval flow" with history preserved — so this is **build-completeness against our own spec**, sharpened by canon MVP-DEC-019 (absent from our frozen `spec/`, which stops at MVP-DEC-007). RM-06.

**The FSM is shared.** `LifecycleTransition` (`transition()` + `reject()`) is the sole `lifecycle_state` writer for all seven catalog spine entities (`catalog-lifecycle-approval` **D1**: one identical FSM). So the block-gate + re-submit belong in the shared mechanism and apply to all seven — a Product-Master-only fix would break the uniformity invariant.

## Goals / Non-Goals

**Goals:**
- Add an explicit `re-submit` operation (`reviewed → reviewed`, audit-only, no event; twin of `reject`) across all seven spine entities that re-arms review.
- Enforce the review-freshness block-gate: `activate` (`reviewed → active`) is rejected while the entity's latest governance action is an un-remediated rejection.
- Keep rejection-pending **derived from the audit trail** — no schema flag, no migration (reaffirm `catalog-lifecycle-approval` D5).
- Surface `re-submit` in the operator console; the block-gate rejection surfaces via the kit's existing outcome path.
- Adopt canon MVP-DEC-019 via a mini-ADR.

**Non-Goals:**
- **"Editing review-governed content re-arms review"** (the third MVP-DEC-019 leg) — deferred to **RM-14** (no edit path today; would be dead code). See proposal slice boundary.
- Any Filament Edit page / Update action — RM-14.
- Re-submit "change notes" / a diff of what the Creator edited — the audit row records actor + `resubmitted`; the *what-changed* history is RM-14's re-versioning concern.
- Touching the *Product Lifecycle State Machine* requirement's four state-changing transitions — `re-submit`, like `reject`, does not change `lifecycle_state`, so it lives in *Approval Governance*.

## Decisions

### D1 — The block-gate joins the approval-governance guard, scoped to `Activate` (reaffirms & completes catalog-lifecycle-approval D5)
The rejection-pending block is enforced in `ApprovalGovernance::guard()`, in the existing `if ($type === LifecycleTransitionType::Activate)` branch — right beside `assertSeparationOfDuties`. That guard already (a) runs on every commercial-impact transition before the state write, (b) is scoped to `Activate` for the approval-specific checks, and (c) reads the audit trail as the system of record (`creatorOf`/`reviewerOf`). Adding an `assertNotRejectionPending($entity, $entityId)` there is the minimal, cohesive home: one guard point, one audit read, all seven entities covered, retire/reopen untouched. On a breach it throws `ApprovalGovernanceViolation::activationBlockedByPendingRejection($entity)` (a new named factory + localized reason). `ApprovalGovernanceViolation` already surfaces through the console kit's `surfaceLifecycleOutcome` (it is what self-approval throws), so the block renders for free. _Alternative rejected:_ a check inside each of the seven `Activate*` actions (duplicated ×7, drift hazard — against D1's "one shared mechanism"); a check in `LifecycleTransition::transition()` itself (would need `type`-gating there too, and `guard()` is exactly the governance seam that already reads the trail).

### D2 — `re-submit` is a new `reviewed → reviewed` audit-only operation (twin of `reject`), NOT a re-run of `submit`
Because a rejected entity is in `reviewed`, `SubmitProductMasterForReview` (`draft → reviewed`) cannot re-arm it (from-state guard, D2 of the prior change). So we add `LifecycleTransition::resubmit(Model, string $entity)` mirroring `reject()`: one `DB::transaction`, `lockAndRefresh`, from-state assert `reviewed` (else `IllegalLifecycleTransition::cannotResubmit($state, $entity)`), `governance->requireOperator($entity)` (a re-submission is a human decision), then one `audit_records` row `catalog.<entity>.resubmitted` (before/after `{lifecycle_state}` + `decision: resubmitted`), **no** domain event. Seven thin per-entity actions `Resubmit{Entity}ForReview` call it (mirroring the seven `Reject*Review`). The mechanism's from-state is `reviewed` (not "must be rejection-pending") — symmetric with `reject`, and a redundant re-submit on a non-rejected `reviewed` entity is a harmless audit row that leaves it activatable; the console only *offers* re-submit when rejection-pending (D5). _Alternative rejected:_ reuse `submit` (impossible from `reviewed`); a boolean-flag toggle Action (needs a schema column — against D3).

### D3 — Rejection-pending is derived from the audit trail; no schema flag (over the frozen spec's literal "flag")
The frozen spec says "rejection **flag**" (`AC-0-J-7`, `BR-Lifecycle-6`, PRD § 4.3). We interpret "flag" as the **derived audit condition**, not a persisted column: `assertNotRejectionPending` reads the entity's latest governance action (the shipped `latestGovernanceAction` shape — `audit_records` filtered by `module`/`entity_type`/`entity_id`, `orderByDesc('id')`, `->value('action')`) and blocks iff it ends in `.rejected`. This reaffirms `catalog-lifecycle-approval` D5's rejection of ~21 persisted governance columns and matches how `ApprovalGovernance` already derives creator (first `*Created` event) + reviewer (latest `.submitted` audit). The **mini-ADR** records this interpretation as the point to reconcile with the spec wording. _Alternative rejected:_ a literal `rejection_pending`/`review_rejected_at` column on all seven spine tables — schema churn, duplicates the audit SoR, contradicts D5.

### D4 — Adopt canon MVP-DEC-019 via a mini-ADR; defer the edit-re-arms leg to RM-14
MVP-DEC-019 is a canon decision **absent from our frozen `spec/`** (⚠️ the frozen spec's own DEC-019 is unrelated — club single-producer composites, Module S). Per the standing rule (`lessons.md` 2026-07-02, confirmed 3× via RM-04/10/24), adopting a canon-DEC absent from the frozen spec always earns a mini-ADR regardless of the tracker's advisory "ADR? —". The ADR (`decisions/2026-07-02-adopt-dec-019-review-freshness-resubmit.md`) records: (a) this is **build-completeness** toward our own `AC-0-J-7`/`BR-Lifecycle-6` (re-submit + block-gate are already spec-required; we were Partial); (b) the canon adoption of the explicit review-freshness invariant; (c) the **derive-from-audit vs literal-"flag"** interpretation (D3); (d) the **edit-re-arms leg is deferred to RM-14** (no edit path today). _Alternative rejected:_ no ADR (the tracker "ADR? —") — invariant #11 bans editing `spec/`, so the `decisions/` trace is the only home for the code↔canon divergence-from-silence.

### D5 — Console surfaces `re-submit`; visibility gated to rejection-pending; the block surfaces via the kit
The operator console (Paolo's "rejection round" walkthrough) gains a `re-submit` header action on the catalog View pages, wired through the shared console kit's `lifecycleAction` factory + write-through-Action discipline (`decisions/2026-06-19` / `2026-06-20`). Its `->visible()` predicate is "the record is rejection-pending" (the same derived read). The block-gate rejection needs no new console code — `ApprovalGovernanceViolation` already routes to `surfaceLifecycleOutcome`'s `action_failed` notification. _Note (lessons.md 2026-06-23/24):_ a Filament action whose `->visible()` is false is undrivable via any test helper, and `assertTableActionExists`/isolated mounts prove definition, not render — so test the re-submit **visibility** with `assertActionVisible`/`assertActionHidden` on the page and the domain re-arm via the Action directly; live-verify the button if a render-suppression is suspected.

## Risks / Trade-offs

- **Inverting a shipped, tested behaviour** → `ProductMasterLifecycleTest.php:363-387` ("rejection is not terminal") is **replaced** by a "rejection blocks until re-submit" scenario. Grep the blast radius before editing (lessons.md 2026-06-18 — a handoff's file list is a hint): `grep -rn "not terminal\|latestGovernanceAction\|->reject(" tests/` (currently only `ProductMasterLifecycleTest` asserts the reject-then-activate-succeeds path; `CatalogLifecycleChainTest` + the console test also exercise reject). Any cross-entity chain test that rejects-then-activates must insert a re-submit.
- **The derived predicate must be exact** → "latest governance action ends in `.rejected`". Verify the interaction with `reopen`: after `retired → reviewed` the latest action is `.reopened` (not a rejection) → activatable, correct. After `resubmitted` → activatable. A freshly `.submitted` entity (never rejected) → activatable. Only an un-remediated `.rejected` blocks. Add a scenario that a reopen is not treated as rejection-pending.
- **Pest + PHPStan-max traps (lessons.md, many)** → for the audit-read assertions use the `latestGovernanceAction` query shape; one `->not` per `expect()` chain (re-anchor with `->and`); array membership via `expect(in_array(...,true))->toBeTrue()`; HTTP/instance methods via `Pest\Laravel\*` globals; class-absence via `glob` not `class_exists`. Run the FULL suite + PHPStan via `php -d memory_limit=-1 vendor/bin/{pest,phpstan analyse}` (bare `artisan test` OOMs at 128M).
- **`ApprovalGovernanceViolation` must extend `RuntimeException`** for the console to surface it → verify (it already surfaces self-approval, so it does; confirm before relying).
- **Seven near-identical `Resubmit*` actions** → intentional, mirrors the existing seven `Reject*Review` / `Submit*ForReview`; the shared mechanism carries the logic, the actions stay thin.
- **`SupplyLifecycleChainTest`-style exhaustive Action allow-lists** (lessons.md 2026-06-23) → a new non-`Create*` Action can red an exhaustive whitelist. Check for a catalog analogue (`CatalogLifecycleChainTest` / any test globbing `Actions/` and asserting an exact set) and register the seven `Resubmit*` there.

## Migration Plan

**No migration.** Rejection-pending is derived (D3); no schema change. Deploy is code-only; rollback is a revert. No data backfill (the audit trail already carries the `.rejected` rows).

## Open Questions

- **Exception shape:** a new `ApprovalGovernanceViolation::activationBlockedByPendingRejection()` factory (recommended — family cohesion + free console surfacing) vs a dedicated `ReviewNotFresh` exception. Default to the factory unless the console needs a distinct notification style.
- **Re-submit notes:** MVP omits a notes argument on `re-submit` (the audit row records actor + `resubmitted`). If Paolo wants "what changed" captured, that is RM-14's re-versioning diff, not a free-text note here — confirm at review if desired.
