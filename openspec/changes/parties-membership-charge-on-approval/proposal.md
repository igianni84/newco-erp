## Why

NewCo's frozen `spec/` (@ MVP-DEC-007) documents the membership flow **canon declared _wrong_** — Paolo's headline validation walkthrough scenario #1 (`docs/validation/README.md`, 🔴 High; Remediation_Tracker **RM-03**, Round-2 P1-canon). Today `ApproveProfile` writes a **durable** `Approved` ("approved-but-unpaid") and a **separate** `ActivateProfile` — gated on a **Module-E** `MembershipFeePaid` for an **INV0** charge — completes `Approved → Active`. Canon **MVP-DEC-016** (tech-team Q&A 2026-06-21, register `:141`) corrected this to a single **atomic approve = charge = activation**: `Approved` is a **transient** pass-through, never a durable resting state (`AC-K-FSM-2`); a charge that fails at approval leaves the Profile in `Applied`; and the `MembershipFeePaid` signal is **Module-S-emitted** (Module E records; Module K consumes — DEC-173), firing **INV1, no INV0** (DEC-157). Our frozen spec predates MVP-DEC-016 (canon `main` is +23 commits ahead), so **RM-03 adopts it**, per the committed authority ADR `decisions/2026-07-03-adopt-mvp-dec-016-membership-charge-on-approval.md`. This is a full-ADR behavioural change (tracker §3 "ADR? yes"), not a naming tweak.

## What Changes

- **Collapse the two-step flow to one atomic operation (Module K).** `ApproveProfile` on an `Applied` Profile SHALL, in one `DB::transaction`, write `Approved`, perform the conditional Originating-Club one-shot lock, then drive `Approved → Active` through the within-module `ActivateProfile` writer — leaving the Profile `Active` and recording `ProfileActivated`. **`Approved` becomes transient** (never durably rested-in — `AC-K-FSM-2`). `DeclineProfile` (`Applied → Rejected`, terminal, event-silent) is unchanged.
- **The charge is a documented no-op seam today.** Module S/E are single-file stubs (no mandate, no pull-capable instrument, no invoice entity), so there is no charge to run: the K-internal atomic activate-on-approval is **unconditional** now and **delegates to the Module-S `MembershipFeePaid` listener** when Module S lands (the tracker's own "temporary K-internal atomic activate-on-approval that later delegates to Module-S `MembershipFeePaid`"). The **charge-fail contract** (fail → stays `Applied`, no seat, no `OriginatingClubLocked`, re-attemptable) is specified as that Module-S target.
- **Re-home the `MembershipFeePaid` seam E → S (docblock/seam-name only).** No `MembershipFeePaid` event class exists — Module K only *consumes* it — so this is **zero behaviour change** for the seam-name part (the RM-10 / DEC-018 precedent): update `ActivateProfile`'s docblock + the truth-spec prose to "Module S emits; Module E records; Module K consumes (DEC-173); fires INV1, no INV0 (DEC-157)".
- **Collapse the operator console verbs.** The `activate` header verb on `ViewProfile` (visibility-gated `state == approved`) becomes **structurally dead** once `Approved` is never durable; remove it. The surviving **Approve** verb (the one retained producer write, L-PP) now yields `Active`; its success notification is reworded to reflect the atomic outcome. The i18n contract test (`ProfileConsoleI18nTest`) and the EN+IT `operator_console.profile.*` keys are updated to drop `actions.activate` / `notifications.activated`.
- **Invert the two-step tests.** Every test that pinned a durable `Approved` (or double-drove approve→activate as a precondition) is flipped to "approval yields `Active` in one operation" (~11 files); `ActivateProfile`'s own isolated contract (`ProfileActivationTest`, factory-forced `Approved → Active`, incl. `cannotActivate`) **stands unchanged**.
- **Truth-spec deltas** (`party-registry`): **MODIFY** *Profile Membership Approval* (atomic approve → active; transient `Approved`; charge-fail seam), *Profile Activation* (seam re-home E→S; INV1/no-INV0; the internal-step + future-listener framing), *Demand-Side Activation Events* (`ApproveProfile` now co-records `ProfileActivated`), and *Birth States Recorded, Lifecycle Transitions Deferred* (the demand-side activation sentence + the Club-Credit `MembershipFeePaid`-ownership clause).

## Impact

- **Modified code (Module K):** `ApproveProfile` (inject `ActivateProfile`; drive `Approved → Active` in the approval transaction; docblock rewrite — atomic, charge-fail seam, delegates to Module-S `MembershipFeePaid` later); `ActivateProfile` (docblock/seam re-home E→S — INV1/no-INV0; the two invocation modes). `DeclineProfile` unchanged.
- **Modified operator console:** `ViewProfile` (remove the dead `activate` verb); `lang/en/operator_console.php` + `lang/it/operator_console.php` (drop `profile.actions.activate` + `profile.notifications.activated`; reword `profile.notifications.approved`); `ProfileConsoleI18nTest` (`profileConsoleKitKeys()` drops the two keys); remove `ProfileActivationConsoleTest` (its verb is gone).
- **No new Action or Event class, none renamed/removed** — so the exact-set Action allow-list in `SupplyLifecycleChainTest` and the `ProfileApproved`-absent guard stay **green unamended**. `ComplianceIndependenceTest`'s "`ApproveProfile` writes `originating_club_id` exactly once, every other Action zero" guard holds because the internal `ActivateProfile` call writes **no** OC FK.
- **Tests inverted (~11):** `ProfileMembershipApprovalTest`, `MembershipActivationChainTest`, the four precondition helpers (`ProfileCancellationTest`, `ProfileSuspensionTest`, `ProfileLapseGraceTest`, `MembershipSuspensionChainTest` — drop the now-illegal `ActivateProfile` call), and the console `ProfileApprovalConsoleTest` / `ProfileMembershipChainTest` / `ProfileLifecycleConsoleTest`. **Stands unchanged:** `ProfileActivationTest` (the `ActivateProfile`-in-isolation contract).
- **No ADR gate tripped, no new dependency, no migration:** the `Approved` enum case is retained (canon keeps it transient); no `fee_paid_at` column, no invoice entity, no schema change. i18n: EN+IT console copy only (invariant 12).
- **Cross-engine:** the full Parties suite is verified green on SQLite **and** PostgreSQL 17 before the change is declared complete (`knowledge/testing/rules.md`).
