# Design — catalog-lifecycle-approval

## Context

`catalog-product-spine` (archived) stood up the seven Module 0 spine entities with their full four-state `lifecycle_state` enum (`LifecycleState{Draft,Reviewed,Active,Retired}`, with a driver-guarded `CHECK`) but wrote **no transition** — every entity is born and frozen in `draft`, and `product-catalog`'s "Lifecycle State Recorded, Transitions Deferred" requirement names **this** change as the home for the transitions, the Creator → Reviewer → Approver workflow, rejection handling, the cascades, and the Producer-activation gate. This change implements all of them, plus the cross-module event consumption that the just-archived `parties-producer-lifecycle` unblocked.

The platform substrate this change builds on (verified in `app/Platform`):

- **`DomainEventRecorder::record(name, module, actorRole, actorId, entityType, entityId, payload, correlationId?, causationId?): DomainEvent`** — appends the event + one `pending` `event_deliveries` row per registered consumer inside the caller's open transaction (throws `NotInTransactionException` at `transactionLevel() === 0`), returns the persisted `DomainEvent`. The `*Created` Catalog events already use it (`CreateProductMaster`).
- **`AuditRecorder::record(action, module, actorRole, actorId, entityType, entityId, before, after, authorizationBasis, correlationId?): AuditRecord`** — appends an append-only `audit_records` row (immutability triggers reject DELETE / structural UPDATE); also requires an open transaction. This is where the governance steps are recorded.
- **The consumer substrate** — `DomainEventConsumer` (one method `handle(DomainEvent): void`), `ConsumerRegistry::register(eventName, consumerClass, mode = Inline)` (a container singleton), `InlineDeliveryExecutor` (inline post-commit + the scheduled `events:sweep` at-least-once retry/dead-letter), `EventDelivery` ledger. **Built but never used by a module** — `DemoConsumer` (registered in a console command) is the only existing consumer; both module providers are empty stubs. This change registers the **first real module consumer**.
- **`ActorContext::role()/actorId()`** — `newco_ops` + `Operator.id` when authenticated on the `operator` guard, else `system`/null. `Module::Catalog->value === 'catalog'`.
- **Boundary law** — `ModuleBoundariesTest` forbids cross-module imports; the only Catalog ↔ Parties coupling permitted is the domain-event payload (`producer_id`) read through the consumer.

## Goals / Non-Goals

**Goals:**
- Implement the uniform four-state FSM (`draft → reviewed → active → retired`, plus `retired → reviewed` reopen) for all seven spine entities, with from-state guards and a single localized transition exception.
- Enforce the Creator → Reviewer → Approver separation-of-duties floor (no self-approval) at the configured role-count, with rejection handling, every step audited.
- Gate Product Master activation on the linked Producer being `active`, read from a Catalog-owned projection; re-activation re-checks.
- Consume `ProducerActivated`/`ProducerRetired` into that projection (the first module consumer): enable / block-new, preserve existing actives, no auto-activate, idempotent + order-tolerant.
- Enforce the parent-before-child activation cascade per entity, and the operator-driven parent-before-child retirement cascade + within-catalog reference-integrity guard.
- Record the fourteen verbatim `*Activated`/`*Retired` events transactionally, PII-free, parent-before-child in cascades.

**Non-Goals:**
- The Filament catalog-approval-queue UI (`catalog-operator-console`); the KYC four-state lifecycle and the gate on `ActivateProducer` (`parties-compliance`); the cross-module retirement-blocking references and BR-Identity-4 / BR-SKU-3/4 (Phase-3 referencers); demand-side Parties.
- Any schema change beyond the one `catalog_producer_states` projection — every `lifecycle_state` column + `CHECK` ships from the spine; `version` stays reserved for identity revisions and is not bumped by a state transition.

## Decisions

### D1 — One shared lifecycle mechanism, applied per entity; the Action is the sole `lifecycle_state` writer
All seven entities share the **identical** four-state FSM, guards, governance and audit shape; they differ only in (a) the **activation-gate predicate** (which parents must be `active`) and (b) the **event classes** recorded. So the generic mechanics live in **one** shared place — a `LifecycleTransition` service (or a `HasLifecycle` concern) under `app/Modules/Catalog/Lifecycle/` — that takes the model, the target state, the resolved actor, and (for activation) a per-entity gate closure + the event factory. Per-entity thin Actions (`SubmitForReview`, `Activate{Entity}`, `Retire{Entity}`, `Reopen{Entity}`, `RejectReview`) call it. The Models stay **persistence-only** (`$guarded = []`, no transition methods) — the immutability discipline the spine rests on ("the Action is the only writer") extends to `lifecycle_state`; no transition Action writes identity columns (`producer_id`, the parent FKs), so those stay immutable by construction. _Alternative rejected:_ 7 × ~5 bespoke Action classes with copy-pasted FSM/guard/governance — ~35 near-identical classes, a maintenance and drift hazard; the per-entity variation is small enough to parameterize.

### D2 — From-state guard via a transaction-locked re-read; one localized `IllegalLifecycleTransition`
Every transition runs `DB::transaction(fn () => …)`, re-reads the target row with `->lockForUpdate()`, asserts the expected from-state, and only then writes + records. The lock serializes concurrent attempts on PostgreSQL (a harmless no-op on SQLite, where the from-state assert carries correctness). An out-of-state call throws a single **`IllegalLifecycleTransition`** (`extends RuntimeException`) with named factories (`::cannotSubmit/cannotActivate/cannotRetire/cannotReopen(LifecycleState $from, string $entity)`) resolving copy from a new `lang/en/catalog.php` group (dotted keys, `:state`/`:entity` placeholders — neither is PII). One exception (not seven) because the FSM is uniform; the entity name is a parameter. _Alternative rejected:_ per-entity exception classes — the FSM is identical across entities, unlike Module K's three genuinely different state machines, so one parameterized class is the faithful analogue.

### D3 — The Producer-state projection: Catalog's first cross-module read model
The gate needs "is producer X `active`?" but invariant 10 forbids querying Module K. The spec-clean mechanism is a **Catalog-owned read model** fed by the consumer: a new table **`catalog_producer_states`** — `producer_id` (PK / unique), `status` (a small enum mirroring the producer lifecycle: `active`/`retired`), `last_event_id` (unsigned bigint watermark = the applied `domain_events.id`), `timestamps`. The *Producer Activation Gate* reads exactly this table; the consumer is its sole writer. This is the codebase's **first projection**, and it sets the reusable pattern every future cross-module gate follows (consume → project → read-locally). _Alternatives rejected:_ (a) querying the shared `domain_events` log at gate time ("latest Producer event for this id") — couples business logic to the audit store, which is the append-only 10-year record, not a query-optimized read surface, and re-derives state on every gate check; (b) a boolean on the Product Master — there is no per-producer row to flip except a projection, and many Masters share a producer. The projection is the conventional read model. _(This decision is documented here rather than as a standalone ADR because the event-substrate ADR already blessed the consumer-maintains-its-own-state pattern; flag for elevation to `decisions/` if the team wants the projection precedent crystallized.)_

### D4 — `ProducerLifecycleProjector`: the first registered module consumer, inline + idempotent
A `App\Modules\Catalog\Consumers\ProducerLifecycleProjector implements DomainEventConsumer`. `handle(DomainEvent $event)` switches on `$event->name` (`ProducerActivated` → `active`, `ProducerRetired` → `retired`), reads `$event->payload['producer_id']`, and **upserts** `catalog_producer_states` **only when `$event->id` advances the row's `last_event_id`** (the substrate's per-entity watermark — latest-wins, so at-least-once + out-of-order delivery converge). DB-work-only; the handler's write and the delivery-status flip share the executor's transaction. Registered in **`CatalogServiceProvider::boot()`** via the injected `ConsumerRegistry` singleton: `register('ProducerActivated', ProducerLifecycleProjector::class)` and `register('ProducerRetired', …)` (default `inline` mode — no `queued`, no ADR gate). It imports **no** `App\Modules\Parties\*` type (only `App\Platform\Events\DomainEvent` + its own model). Consuming `ProducerRetired` writes **only** the projection — it never touches any `ProductMaster` row (block-new, never cascade-retire); consuming `ProducerActivated` never auto-activates a Master (no auto-replay — AC-0-BR-BulkImport-4). _Alternative rejected:_ registering in a console command like the demo — the registry docblock + the empty provider stubs designate `{Module}ServiceProvider::boot()` as the production seam.

### D5 — Approval governance is audit-derived; the audit trail is the system of record for "who did each step"
§ 4.2 states the review/approval steps are "recorded in the audit trail with actor identity, timestamp, and decision" — so the **audit trail is the SoR** for the separation-of-duties guard, and no per-entity governance columns are added (no churn across 7 tables). At a governance step the acting operator is `ActorContext::actorId()`; the guard queries `audit_records` for this entity's prior governance steps (this `module`/`entity_type`/`entity_id`, the governance actions) and reads the **creator** from the entity's `*Created` domain event `actor_id` (vacuous if the entity was system/seed-created with a null actor), collects those `actor_id`s, and rejects if the current operator is among the actors the configured separation requires to be distinct — concretely, the **approver (`reviewed → active`) must differ from the creator and, in the three-step configuration, the reviewer (the `draft → reviewed` submitter)**, and in three-step the **reviewer must differ from the creator**. The **role count** is config (`config('catalog.approval.role_count')` ∈ {2, 3}, default 3 — `feedback_prd_rr_approval`); the floor (distinct actor per step, no self-approval, every step audited) holds at any depth. A governance transition **requires an operator principal** (`newco_ops` + non-null `actor_id`) — a `system`/null actor cannot satisfy distinctness and is rejected (creation may be system-driven; approval is inherently a human decision). **Rejection** is a `RejectReview` action: the entity stays `reviewed`, an audit record captures the actor + notes + `decision: rejected`; "rejection-pending" is **derived** (the latest governance audit action for the entity is a rejection), so re-submission (a fresh `SubmitForReview` audit) clears it without a schema flag, and the append-only trail preserves the full history. _Alternative rejected:_ persisted `created_by`/`reviewed_by`/`approved_by`/`rejection_note` columns on all 7 tables (~21 columns) — duplicates what the audit trail already records as the SoR, and churns the minimal spine entities; if audit-derived querying proves awkward operationally a follow-on can project it, but launch follows the spec's "the audit trail records it" literally.

### D6 — The KYC conjunct is enforced transitively upstream, not in Module 0
§ 5.4 / BR-Producer-1 phrase the gate as "linked Producer is `active` **and** KYC-verified." But the **KYC four-state lifecycle is unmodelled** at launch — `parties-producer-lifecycle` shipped `ActivateProducer` **ungated** (KYC deferred to `parties-compliance`, DEC-071), and `ProducerActivated`'s payload is exactly `{producer_id, status}` with **no KYC bit**. The faithful resolution: Module 0 gates on the linked Producer being **`active`**, and the KYC conjunct is satisfied **upstream** — `parties-compliance` will tighten `ActivateProducer` so a Producer cannot reach `active` without a clear KYC verdict, at which point "producer `active`" transitively implies "KYC-verified" with **no change to Module 0's gate**. This is the same "seam now, behaviour later" discipline the spine and the supply-side lifecycle used; it keeps the KYC bit out of the cross-module event contract entirely (the producer simply isn't activatable un-KYC'd). Documented in the gate requirement + the Catalog contract note. _Alternative rejected:_ widening `ProducerActivated`'s payload with a `kyc_verified` flag and projecting it — invents a contract field the emitting spec does not define, and couples Module 0 to a KYC model that does not exist; the upstream-tightening path needs no Module-0 change ever.

### D7 — Activation cascade: within-module parent reads, checked at the child's activation moment only
Each child's `Activate` gate reads its parent(s)' `lifecycle_state` **within Module 0** (Variant→`product_master_id`; PR→`product_variant_id`+`format_id`; Sellable SKU→`product_reference_id`+`case_configuration_id`; Composite SKU→every `catalog_composite_sku_constituents.product_reference_id`). The Master→Producer parent is the one cross-module case and goes through the projection (D3). The gate is evaluated **only at the child's `reviewed → active` transition** — retiring a parent later does **not** auto-retire active children (§ 4.5, block-new). Parent-before-child **emission** ordering falls out for free (a child cannot reach `active` before its parent). _Alternative rejected:_ maintaining a denormalized "parent active" flag on children — redundant with the parent's own `lifecycle_state` and stale-prone; a direct within-module read at activation time is correct and simple.

### D8 — Retirement cascade: operator-driven, parent-before-child; within-catalog reference integrity scoped to the terminal sellable edge
A single-entity `Retire` is **blocked only while the entity is referenced by an `active` terminal sellable object** — a Product Reference referenced by an `active` Sellable/Composite SKU, or a Case Configuration referenced by an `active` Sellable SKU — surfaced, not silently cascaded (the within-catalog, launch-realizable subset of BR-Lifecycle-5). A **hierarchy parent is not blocked on its children**: retiring a Master with `active` Variants (or a Variant with `active` PRs) **succeeds and preserves** them (§ 4.5 / BR-Lifecycle-4 — existing actives stay valid; only new activation under the retired parent is prevented). The **operator-driven cascade** (§ 4.7) is a distinct workflow that retires a Master + descendants in **one transaction**, parent-before-child (Master → Variants → PRs → SKUs), recording each `*Retired` in order. The **cross-module** reference leg (Allocations/vouchers/Offers) is a **seam** — those referencers arrive in Phase 3; this change enforces within-catalog only and the referencer changes extend the guard. _Decision (2026-06-16, `decisions/2026-06-16-catalog-retirement-reference-integrity-scope.md`):_ the spec delta as first authored gave "a Master with `active` Variants" as a *block* example, which contradicted this requirement's own *preserve* scenario and the immutable PRD § 4.5; the founder resolved it to this scope (block only the terminal-sellable PR/CaseConfig ← `active` SKU edge; preserve hierarchy parents). _Alternatives rejected:_ (a) a **symmetric** block on every parent-with-active-children — makes the PRD § 4.5 *preserve* behaviour unreachable within catalog and diverges from the immutable PRD; (b) **no** within-catalog block at all (defer the whole guard to Phase 3) — spec-literal, but drops the only launch-meaningful integrity check (an operator silently orphaning an `active` sellable SKU by retiring its PR out from under it); (c) auto-cascading a single-entity retire to all descendants — the spec separates the guarded single retire (§ 4.6) from the explicit operator cascade (§ 4.7); conflating them would retire children the operator did not intend to.

### D9 — Fourteen `final` event classes mirroring `*Created`; no `*Reviewed`
Each event is a `final` class under `Events/` with `const NAME` (verbatim § 14.1, category-neutral per § 18), `const ENTITY_TYPE`, and a static PII-free `payload(Model): array` — exactly the `ProductMasterCreated` shape. Payload sketch: `*Activated`/`*Retired` → `{<entity>_id, lifecycle_state, …ids}` (e.g. `ProductMasterActivated` → `{product_master_id, producer_id, lifecycle_state}`; `ProductReferenceActivated` → `{product_reference_id, product_variant_id, format_id, lifecycle_state}`). Recorded via `DomainEventRecorder` inside the transition's transaction (`Module::Catalog->value`, `ActorContext` actor, stringified id); in a cascade, recorded parent-before-child so `domain_events.id` order encodes it (§ 14.3). **No `*Reviewed` event** — `draft → reviewed` and `retired → reviewed` are audit-only. _Alternative rejected:_ a single generic `LifecycleTransitioned` event — the spec names fourteen distinct verbatim events that downstream modules key off; genericizing would break the inter-module contract.

### D10 — One migration; transitions update `lifecycle_state` in place; `version` untouched
The only schema addition is **`catalog_producer_states`** (additive, Postgres-truthful/SQLite-compatible; a `status` `CHECK` derived from the producer-state enum, mirroring the spine's pattern). No `lifecycle_state` column migration (all seven ship from the spine). A transition `update`s `lifecycle_state` in place; the **domain event + audit record are the immutable history**, so transitions do **not** bump `version` (it stays the spine's identity-revision counter).

### D11 — Operator transitions via `ActorContext`; tests use distinct `Operator`s under `DatabaseMigrations`
Transitions resolve the actor from `ActorContext` (the governance guard needs the operator `actor_id`). Tests authenticate distinct operators with `actingAs(Operator::factory()->create(), 'operator')` and switch principal between steps to exercise separation-of-duties. **Consumer-delivery tests use `DatabaseMigrations`, not `RefreshDatabase`** — the recorder's `DB::afterCommit` inline hook and the executor's commit only fire at `transactionLevel 0`, which `RefreshDatabase`'s wrapping transaction suppresses; the reusable `tests/Support/Platform` consumer doubles and the `domain_events`/`event_deliveries` assertions from `InlineDeliveryTest` are the template.

## Risks / Trade-offs

- **SQLite-green is necessary, never sufficient** → every DB-touching task is verified on **PostgreSQL 17** before close (`knowledge/testing/rules.md`). Live traps: `lockForUpdate` is a real lock on PG / no-op on SQLite (the from-state assert must carry correctness); the `catalog_producer_states.status` `CHECK` must hold on both engines; `timestamptz` `+00` if any raw timestamp is asserted; the consumer must run under `DatabaseMigrations` so the inline post-commit hook fires.
- **First-consumer wiring is subtle** → if `CatalogServiceProvider::boot()` does not register the consumer, `ProducerActivated`/`ProducerRetired` fan out **zero** deliveries and the projection never updates (the gate would then block every Master forever). A test asserts the registration (a delivered `ProducerActivated` produces a `done` `event_deliveries` row for `ProducerLifecycleProjector` and a projection row).
- **Audit-derived separation-of-duties** → the guard depends on the governance steps being faithfully audited; a missing audit write would weaken the floor. Mitigation: the shared mechanism (D1) records the audit row and evaluates the guard in the **same** transaction as the state write, so a step can never transition without its audit row, and the guard reads a consistent snapshot.
- **Gate vs. eventual consistency** → a `ProducerActivated` delivered late (sweep, after a crash) means a Master is briefly un-activatable; this is correct (operator retries the activation; no auto-replay). The watermark guarantees the projection converges; the gate is read-time, so it reflects the latest applied state.
- **Cascade over-reach / scope creep** → the retirement cascade is Master→descendants within catalog; the cross-module reference leg and the demand-side are explicitly out. A test asserts no `App\Modules\Parties\*`/`Allocation`/`Commerce` reference appears in the new code (arch test stays green).
- **Inventing events/states for symmetry** → rejected: exactly the fourteen § 14.1 names, no `*Reviewed`, no new `lifecycle_state` value. The producer-state projection's enum is a Catalog-local read model, not a new cross-module contract.

## Migration Plan

Additive and reversible at the change level. The single new migration creates `catalog_producer_states` (no data backfill — the projection starts empty and is populated as `ProducerActivated`/`ProducerRetired` are consumed; any Producer activated **before** this change deployed will re-converge on its next event, and the gate correctly blocks Masters under producers with no projection row until then). No existing table is altered; no `lifecycle_state` column changes. Rollback = drop `catalog_producer_states` + unregister the consumer; no destructive effect on the spine. The consumer registers in `inline` mode, so no queue infrastructure is required; the existing scheduled `events:sweep` provides at-least-once recovery for free.

## Open Questions

None blocking. The two scope forks (include the approval governance now; all seven entities + cascades) were resolved by the founder before authoring (AskUserQuestion 2026-06-16). Two seams are recorded explicitly and traced to named future changes: the **KYC conjunct** (enforced upstream by `parties-compliance`, D6) and the **cross-module retirement-blocking references** (extended by the Phase-3 referencer changes, D8). Whether the producer-state projection precedent (D3) is elevated from this design note to a standalone `decisions/` ADR is a documentation choice for the team — flagged, not blocking.
