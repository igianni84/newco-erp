## Context

The Customer console kit is settled across five consoles: a read-only `OperatorConsoleResource` for list/infolist, write-through via domain Actions (never `$model->save()` — `NoEloquentWriteInOperatorPanelRule` guards it), and a `ViewRecord` page that `use`s the `SurfacesDomainActions` trait and assembles its own verbs. The Customer console (`archive/2026-06-22-operator-console-parties-customer`) shipped the four direct status verbs and renders `status` / `kyc_status` / `sanctions_status` / Account status / Profiles **read-only**. The Holds slice (`archive/2026-06-22-operator-console-parties-holds`) shipped the first compliance **write** surface (place / lift) plus a per-row Holds table on `ViewCustomer`, and left a forward note: KYC + sanctions writes are the next slice, and their cross-slice link is that `RequireKyc` auto-places a `kyc` Hold and `RecordKycVerified` auto-lifts it — verifiable against the Holds table this console already renders.

The Module K KYC + sanctions backend is shipped and **unchanged** by this slice. Verified signatures (zero-invention, re-checked 2026-06-22):

- `RequireKyc::handle(int $customerId): Customer` — `kyc_status` `not_required | null → pending`, sets `kyc_required`; in the same transaction **auto-places a Customer-scope `kyc` Hold** via `PlaceHold` (records `CustomerHoldPlaced`; the Hold coupling suspends an `active` Customer → `CustomerSuspended`). Rejects with `IllegalKycTransition::cannotRequire` out of from-state. **Records no KYC-specific event.**
- `RecordKycVerified::handle(int $customerId): Customer` — `pending → verified`; **auto-lifts** every active `kyc` Hold (one `CustomerHoldLifted` each) and reactivates the Customer when it was `suspended` and no other active Hold covers it (`CustomerReactivated`). Rejects with `IllegalKycTransition::cannotVerify`. **Records no KYC-specific event.**
- `RecordKycRejected::handle(int $customerId): Customer` — `pending → rejected`; **audit-only, records no event**, the active `kyc` Hold **remains**. No `reason` parameter. Rejects with `IllegalKycTransition::cannotReject`.
- `RecordCustomerScreening::handle(int $customerId, SanctionsStatus $verdict, ScreeningTriggerSource $source): Customer` — sets `sanctions_status`, `last_screening_at`, `next_rescreen_at` (= `last_screening_at + 12 months`), `screening_trigger_source`; records exactly one of `CustomerOnboardingScreeningPassed` / `CustomerOnboardingScreeningFailed` / `CustomerRescreeningPassed` / `CustomerRescreeningFailed` by verdict × source (`Passed|Failed` × `Onboarding` → onboarding-pass/fail; `Passed|Failed` × non-onboarding → rescreen-pass/fail); a `Pending` / `UnderReview` verdict records **no** event. Rejects with `IllegalSanctionsTransition::onboardingAlreadyScreened` when `source = Onboarding` and `last_screening_at` is already set.
- `KycStatus` { NotRequired=`not_required`, Pending=`pending`, Verified=`verified`, Rejected=`rejected` } with `clears(): bool` (true for `Verified` / `NotRequired`) — a **state** enum. `SanctionsStatus` { Pending, Passed, Failed, UnderReview }, `ScreeningTriggerSource` { Onboarding, Cadence, AmlThreshold, ComplianceAdHoc } — **operand** enums for `RecordCustomerScreening`.
- `IllegalKycTransition` and `IllegalSanctionsTransition` both extend `RuntimeException` (so `surfaceLifecycleOutcome`'s base-`RuntimeException` catch surfaces them as `action_failed` — never imported). There is **no** Customer KYC waive Action (only `WaiveProducerKyc`).

## Goals / Non-Goals

**Goals:**

- Surface `RequireKyc` / `RecordKycVerified` / `RecordKycRejected` as three **form-less** header verbs on `ViewCustomer`, each **visibility-gated** to its legal from-state.
- Surface `RecordCustomerScreening` as one **form-bearing** header verb (verdict × restricted trigger-source), reusing the bespoke-form recipe the Holds slice proved.
- Let the KYC-driven Hold coupling (auto-place on require / auto-lift on verify) run as the domain's own behaviour, **additive** to the shipped surfaces; prove it **through** the Holds table the previous slice shipped.
- EN + IT copy + completeness test; a PG17 closing-chain test.

**Non-Goals (deferred — see proposal Impact):**

- **Enhanced-KYC** toggle (no domain Action; auto-detected per §9.1 — a console slice can't write it without breaching the no-Eloquent-write invariant). **Automated sanctions sources** (`cadence` / `aml_threshold`, §9.5 manual-first). **Direct Account / Profile verbs** (account / profile slices). **The order-completion purchase gate** (Module S, §9.3). Anonymisation / GDPR.
- Any change to the KYC / sanctions domain backend, a new read contract, a migration, or a composer dependency. Extending the shared `SurfacesDomainActions` trait's contract.

## Decisions

**D1 — Trait level, on the existing `ViewCustomer` page; no new base class.** The KYC + sanctions verbs are header actions on `ViewCustomer` (`extends ViewRecord`, `use SurfacesDomainActions`). `CustomerResource` is untouched — the `kyc_status` / `sanctions_status` badges already render. _Per ADR 2026-06-21-rule-of-three (CLOSED): stay at the trait level._

**D2 — The three KYC verbs reuse `lifecycleAction()` directly (form-less), unlike the Holds `placeHold`.** Each KYC Action takes only `int $customerId`, so the trait's `lifecycleAction($verb, $successKey, $invoke)` — shaped for the bare-`int $id` status verbs — fits exactly: `$invoke = fn (Customer $record) => app(RequireKyc::class)->handle($record->id)`, `$form = null`. The verbs chain `->visible(...)` for the from-state gate (D4). The trait is **not** modified. _This is the cheaper path the Holds slice could not take (its multi-operand `placeHold` needed the bespoke recipe); the KYC verbs are simpler than the status verbs (no `notes`)._

**D3 — `RecordCustomerScreening` is a bespoke form header action reusing `surfaceLifecycleOutcome()`.** It needs two operands (`verdict`, `trigger_source`), so — exactly like `placeHold` (Holds D3) — it is built directly (`Action::make('recordScreening')->schema([...])->action(fn (array $data) => $this->surfaceLifecycleOutcome(fn () => app(RecordCustomerScreening::class)->handle($record->id, SanctionsStatus::from($v), ScreeningTriggerSource::from($s)), $successTitle))`), **reusing** the kit's `surfaceLifecycleOutcome()` for the uniform success / base-`RuntimeException` → `action_failed` notification. Narrow each `$data` value with `is_string` before `Enum::from()` (the Holds form-data discipline). The trait is **not** forked.

**D4 — Visibility = the complement of the domain from-state guard (the lift-discipline pattern), so a rejected transition is unreachable through the surface.** Each KYC verb is visible **iff** `kyc_status` sits in its legal from-state — `requireKyc` iff `kyc_status` is `not_required` or unset; `recordKycVerified` / `recordKycRejected` iff `kyc_status` is `pending`. Consequence (the **Filament hidden-action landmine**, lessons.md 2026-06-22): a hidden action can't be mounted/invoked server-side, so the verb's `action_failed` reject branch is **unreachable through the surface**. Test the reject the way the Holds lift did — `assertActionHidden('verb')` (the surface) **+** `expect(fn () => app(Action::class)->handle($id))->toThrow(IllegalKycTransition::class)` (the domain floor), state + event-log unchanged — **not** a notification the page can't raise. Extract each predicate to a `private` helper (`kycRequirable(Customer): bool`, `kycPending(Customer): bool`) so the magic-string token + the D4 rationale live in one place. _Note: these header verbs are visibility-gated, unlike the always-present status verbs (whose reject IS surfaced) — the brief's deliberate UX choice; the page-action visibility-test API (`assertActionVisible/Hidden`) is the header-action twin of the table API the Holds row-lift used, pinned against the installed Filament version in task 1._

**D5 — State enum cast-only; operand enums imported; `ModuleBoundariesTest` UNCHANGED.** `KycStatus` is a **state** enum — predicated and rendered through the model cast (`$record->kyc_status?->value === 'pending'`, the literal token), **never imported** (even in predicates — Holds D2). `SanctionsStatus` and `ScreeningTriggerSource` are **operand** enums for `RecordCustomerScreening` — imported and constructed with `::from($formValue)`. Both ride the existing operand-enum carve-out by the `Parties\Enums` namespace prefix (ADR 2026-06-21), so `tests/Architecture/` stays **untouched** — exercising the carve-out, never widening it. _Confirm mechanically: `git diff --name-only main | grep '^tests/Architecture/'` → none._

**D6 — The sanctions form offers only `{onboarding, compliance_ad_hoc}`, with `onboarding` dropped once already screened.** The `trigger_source` Select offers the two operator-selectable sources; the automated `cadence` / `aml_threshold` are never offered (§9.5 manual-first). `onboarding` is **first-screening-only**: the option set is record-dependent — `onboarding` is present iff `$record->last_screening_at === null`, so the complement-of-guard discipline (D4) applies at the **form-option** level too. The domain `IllegalSanctionsTransition::onboardingAlreadyScreened` remains the floor for any out-of-band re-onboarding. The `verdict` Select offers all four `SanctionsStatus` values. Record-dependent options resolve at render time via a Closure (`->options(fn () => $this->screeningSources($this->getRecord()))`), the Holds `profile_id` precedent. _Decision (from the interview): the founder confirmed `{onboarding, compliance_ad_hoc}` over offering all four._

**D7 — Assert the events that exist, never a KYC event.** The KYC verbs record **no** KYC-specific domain event — the only events are the coupled `CustomerHoldPlaced` / `CustomerHoldLifted` (from the auto-Hold) and the status events `CustomerSuspended` / `CustomerReactivated` (from the coupling). `RecordKycRejected` records **nothing**. Tests assert exactly these and assert the **absence** of any KYC event (the catalog names none — do not invent `CustomerKycVerified`). The sanctions verb records exactly one screening event by verdict × source (or none for `pending` / `under_review`).

**D8 — No Customer KYC waive verb.** No `WaiveCustomerKyc` Action exists (only `WaiveProducerKyc`, producer-only). The console exposes no waive; the i18n + absence tests assert it stays absent.

**D9 — Enhanced-KYC deferred.** §9.1 sets the €10k / €50k flag by auto-detection with no separate state machine and no operator Action; surfacing a manual toggle needs new Module K domain code (its own slice) — out of a console slice's scope and impossible without breaching `NoEloquentWriteInOperatorPanelRule`. Named out-of-scope in the proposal.

**D10 — Spec delta: two ADDED + two minimal MODIFIED, plus a stale-guard fix.** ADD _Operator manages a Customer's KYC verification through the console_ and _Operator records a Customer's sanctions screening through the console_ (mirroring the spec's own §9.1 / §9.2 split). MODIFY the two Customer requirements to drop **KYC + sanctions** from their negative-scope assertions (cross-referencing the new requirements); Account / Profile negatives remain. Update `CustomerLifecycleConsoleTest`'s `assertActionDoesNotExist('requireKyc' / screening)` guards (keep the **account** / **profile** guards — still deferred). The read-only KYC / sanctions **badge rendering** is unchanged.

**D11 — PG17 closing-chain test, driven through the surface, proving the cross-slice Hold coupling.** `uses(DatabaseMigrations::class)` (not `RefreshDatabase` — each console action opens its own `DB::transaction`; the in-transaction event append must commit). Drive the chain through the page actions / Holds table widget: on an `active` Customer, `requireKyc` → `kyc_status = pending`, a `kyc` Hold row appears in the Holds table **exposing no per-row Lift** (auto-managed, the Holds slice's discipline), the Customer is `suspended` (`CustomerHoldPlaced` + `CustomerSuspended`); `recordKycVerified` → the `kyc` row flips to `lifted`, the Customer is `active` (`CustomerHoldLifted` + `CustomerReactivated`); then a `recordScreening` (onboarding/passed) records `CustomerOnboardingScreeningPassed`; assert the emergent `DomainEvent` set via `pluck('name')->toEqualCanonicalizing([...])` (a multiset — **no** KYC event in it) + the per-event envelope (strict `module` `parties` / `entity_type` / `entity_id`, `actor_role` `NewcoOps`, representative `actor_id` loose `toEqual` — PG bigint-as-string). The envelope is **heterogeneous** (Hold events carry `entity_type=Hold`; status + screening events carry `entity_type=Customer`).

## Risks / Trade-offs

- **Asserting an event that isn't emitted.** The KYC verbs are event-silent; a naive test asserting a `CustomerKycVerified` would fail forever. → D7: assert only the coupled Hold/status events + the **absence** of any KYC event; the chain-test multiset names none.
- **The hidden-action reject is structurally untestable through the widget.** → D4: the visibility predicate is the exact complement of the domain guard, so the reject is proven by a domain `toThrow` + an `assertActionHidden`, never an `assertNotified(action_failed)` the page can't raise (the lesson the Holds slice already paid for).
- **`onboarding` re-selection.** A second onboarding screening throws. → D6: drop `onboarding` from the options once `last_screening_at` is set (record-dependent option set) **and** let the domain guard be the floor; the test covers both the option-absence and the domain throw.
- **State enum leaking into an import.** Reaching for `use KycStatus` to write a predicate would needlessly test the boundary. → D5: predicate via the cast `->value`; only the two sanctions **operand** enums are imported.
- **Full suite OOMs at PHP's default 128 MB.** → run the full / PG17 suite as `php -d memory_limit=-1 vendor/bin/pest` (the ritual flag); `--filter` runs and `phpstan` are fine at the default.
