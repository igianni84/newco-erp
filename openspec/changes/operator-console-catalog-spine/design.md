# Design — operator-console-catalog-spine

## Context

`operator-console-catalog-master` founded the `operator-console` capability and shipped one concrete, idiomatic console (Product Master), **deliberately deferring the shared abstraction** (its design **L9**). This change operates the other six Module 0 spine entities — **Product Variant, Format, Product Reference, Case Configuration, Sellable SKU, Composite SKU** — and resolves L9 by extracting the shared **operator-console kit** and retrofitting Master onto it.

It is governed by two ADRs: `decisions/2026-06-19-operator-console-read-binding-write-through-actions.md` (read-bind / write-through-actions; the `NoEloquentWriteInOperatorPanelRule` PHPStan rule; the `ModuleBoundariesTest` `{Models, Actions}` carve-out) and the new `decisions/2026-06-20-operator-console-shared-resource-kit.md` (the kit + the Master retrofit). The Catalog backend is **not** modified — this is a pure operator-surface layer over shipped behaviour (Admin Panel §1.2).

The six backends it operates (verified in code): each entity has a model under `app/Modules/Catalog/Models/` (`ProductVariant`/`catalog_product_variants` with `product_master_id`; `Format`/`catalog_formats` standalone; `ProductReference`/`catalog_product_references` with `product_variant_id` + `format_id`, unique on the pair; `CaseConfiguration`/`catalog_case_configurations` standalone, no breakability column; `SellableSku`/`catalog_sellable_skus` with `product_reference_id` + `case_configuration_id`; `CompositeSku`/`catalog_composite_skus` with the `catalog_composite_sku_constituents` ordered junction), the four-state FSM through `LifecycleTransition`, and the `Create / Submit*ForReview / Reject*Review / Activate / Retire / Reopen <Entity>` Action set. There is **no** update Action, **no** bulk-import Action, and **no** `Retire<Entity>Cascade` for any of the six (cascade-retire is Master-only).

## Goals / Non-Goals

**Goals:**
- Extract the **operator-console kit** (base read-only Resource + the operator-action wrapper + base Create/View pages) per ADR 2026-06-20, and **retrofit `ProductMasterResource`** onto it with the shipped Master suite + arch/boundary tests green (no behaviour change).
- Six complete, demoable consoles: list + view, manual create (scalar / parent-picker / N≥2 constituents), submit·reject·activate (SoD affordance), retire·reopen — all read-binding for display, all writes through Catalog Actions.
- Surface the within-catalog **activation-cascade** gate (the four child entities) and the **retire reference-integrity** block (Product Reference, Case Configuration) — by rendering the domain rejection, never reimplementing it.
- Surface the two create guards as **form validation errors**: Composite SKU `< 2 constituents` and Product Reference duplicate `(variant, format)`.
- EN + IT localization for all spine copy, per-key IT fallback, no hardcoded copy.
- Green on SQLite **and** PostgreSQL 17; Pint + PHPStan green.

**Non-Goals:**
- Field-editing / updating any entity in place; bulk import; enrichment-metadata update — **no shipped backend** (deferred-pending-backend, proposal slice-boundary table).
- Cascade-retire for the six — **Master-only**; no Action ships (deferred-pending-backend).
- A producer picker or Producer-gate handling on any of the six — **none binds a producer** (Master-only).
- The Parties / A–E consoles (future changes); the cross-module retire leg (Phase-3 seam); authority-tier RBAC (`feedback_prd_rr_approval`); UX/IA polish (DEC-073); Producer/Consumer SPA portals.

## Decisions

**L1 — The shared console kit (ADR 2026-06-20).** Extract, inside `app/Modules/OperatorPanel/Filament/`: a base `OperatorConsoleResource` (read-only conventions — `getModelLabel/getPluralModelLabel` from a per-entity `i18nKey()`; a shared `lifecycle_state` badge + `version` column helper; adds no mutating action); an **operator-action wrapper** trait promoting `surfaceLifecycleOutcome(Closure $run, string $successTitle)` (today a private static on `ViewProductMaster`) plus a factory for the standard lifecycle `Filament\Actions\Action`; a base `OperatorConsoleCreateRecord` (`$canCreateAnother=false`; a `handleRecordCreation` template calling a per-entity `createViaAction(array $data): Model` wrapped in the create-rejection→form-error catch); and a base `OperatorConsoleViewRecord` providing the **five uniform** lifecycle header actions. **Names are indicative — confirm the Filament 5 base-class extension points in `vendor/` before writing** (`CreateRecord::handleRecordCreation(array $data): Model`, `ViewRecord::getHeaderActions(): array`, `Filament\Actions\Action` builder). The kit is OperatorPanel-internal; it adds no dependency.

**L2 — Build order: extract from Master, retrofit Master FIRST, then the six are reuse.** The kit is extracted from the one concrete example (Master) and immediately proven against it (the shipped Master suite is the regression net). Only then are the six built on the proven base. Retrofitting the **richest** entity first front-loads the risk where the net is strongest; deferring it would design the base against six simpler entities and risk a base that cannot express Master.

**L3 — Per-entity create form + field→Action mapping** (verified signatures — do not invent or rename; re-confirm in `app/Modules/Catalog/Actions/` before use):
- **Format** → `CreateFormat::handle(string $name, string $sizeLabel, int $volumeMl): Format` — scalar form.
- **Case Configuration** → `CreateCaseConfiguration::handle(string $name, int $unitsPerCase, string $packagingType): CaseConfiguration` — scalar form; **no breakability field**.
- **Product Variant** → `CreateProductVariant::handle(int $productMasterId, string $variantIdentifier, ?int $vintageYear = null, bool $nonVintage = false, ?TranslatableText $tastingNotes = null): ProductVariant` — Master picker + wine attrs.
- **Product Reference** → `CreateProductReference::handle(int $productVariantId, int $formatId): ProductReference` — Variant + Format pickers.
- **Sellable SKU** → `CreateSellableSku::handle(int $productReferenceId, int $caseConfigurationId, string $commercialName, ?string $marketingCopy = null): SellableSku` — PR + Case Configuration pickers + commercial attrs.
- **Composite SKU** → `CreateCompositeSku::handle(array $productReferenceIds): CompositeSku` — an **ordered, N≥2** Product-Reference picker (`list<int>`, bundle order). Verify the Filament 5 ordered-multi-input API (repeater / `Select::multiple` + order) in `vendor/` before writing.

**L4 — Lifecycle actions are uniform; the gate / dedup / reference-integrity divergences are surfaced FOR FREE.** The five lifecycle Actions (`Submit*ForReview` audit-only, `Reject*Review($record,$notes)` audit-only, `Activate*` → `*Activated`, `Retire*` → `*Retired`, `Reopen*` audit-only) are identical in shape across the six. Because the console **routes to the Action and surfaces whatever it throws** (the wrapper renders a thrown `RuntimeException` as a danger notification), the activation-cascade gate (`catalog.gate.parent_not_active`), the SoD floor (`catalog.approval.*`), the illegal-from-state guard (`catalog.lifecycle.*`), and the retire reference-integrity block (`catalog.retirement.blocked_by_active_references`) all surface with **no per-entity console branching** — they are already-localized domain rejections. The console never re-checks parent-active, actor identity, or open references.

**L5 — Create-rejection surfacing: one common path + one special case.** The base `createViaAction` catch handles the **localized domain `RuntimeException`** case — Composite SKU's `InsufficientCompositeConstituents` (`catalog.composite_sku.insufficient_constituents`) — by mapping `$e->getMessage()` to a `data.<field>` form error (the Master-dedup pattern). **Product Reference is the one special case:** a duplicate `(variant, format)` is a structural DB `Illuminate\Database\UniqueConstraintViolationException` with **no** localized domain message (verified: `CreateProductReference` has no app-layer dedup — the DB unique index owns it). Its Create page catches that **framework** exception specifically and maps a **console-owned** localized message to the form error — catching the generic `RuntimeException` and rendering `$e->getMessage()` here would leak a raw SQL string. `UniqueConstraintViolationException` is a framework class, so importing it does not touch the `{Models, Actions}` carve-out. **Format / Case Configuration / Product Variant / Sellable SKU ship no create guard** — no rejection surfacing for them (Sellable SKU has no uniqueness — a PR+Case-Config pair may back multiple SKUs; do not add a guard).

**L6 — Master-only extensions stay Master-only.** The **producer picker** (a create-form `Select` sourced from Catalog's own `catalog_producer_states` / `ProducerState` projection — never Module K) and the **cascade-retire** header action (`RetireProductMasterCascade`, with its confirmation affordance) are extensions on Master's Resource/View page, not in the base and not on any of the six. The Producer-activation **gate** needs no special support — like the cascade gate it is a domain rejection the wrapper surfaces. **Do not** copy `producerOptions/producerLabel` into a spine resource; **do not** add a cascade-retire action to any of the six.

**L7 — i18n.** Add per-entity copy to `lang/{en,it}/operator_console.php` mirroring the Master shape (`operator_console.<entity>.{label,plural_label,columns,fields,actions,affordance,notifications}`). Reuse `lang/en/catalog.php` for the domain rejection messages (`composite_sku.*`, `lifecycle.*`, `approval.*`, `gate.parent_not_active`, `retirement.blocked_by_active_references`) shown via `$e->getMessage()`. The **Product Reference duplicate** message is console-owned (no domain copy exists) — add `operator_console.product_reference.duplicate_reference` (or similar). EN baseline + per-key IT fallback (DEC-127); the English-invariant `label`/`plural_label` may be omitted from `it`. Reuse the predecessor's **sink-anchored `token_get_all` scan** to prove no hardcoded user-facing string.

**L8 — No backend changes (scope guard).** Zero new Catalog Actions/events/migrations and **no** Composer dependency (`git diff main -- composer.json composer.lock` stays empty; no `database/migrations/*`). Event names are the verbatim category-neutral families: `ProductVariant*`, `Format*`, `ProductReference*`, `CaseConfiguration*`, `SellableSKU*`, `CompositeSKU*` (SKU upper-case). Submit/reject/reopen record **no** event. Coin **no** new event.

**L9 — Boundary & arch: expect NO amendment.** The `NoEloquentWriteInOperatorPanelRule` (scoped to `Modules/OperatorPanel/Filament/`) and the `ModuleBoundariesTest` `{Models, Actions}` carve-out already cover the whole OperatorPanel namespace — the six new Resources importing `App\Modules\Catalog\Models\*` + `App\Modules\Catalog\Actions\*` are already permitted, and the kit's base classes live under the same namespace. No boundary edit is expected; if one seems necessary, STOP and re-read the carve-out before widening it. Keep the cross-module surface at exactly `{Models, Actions}`: catch domain rejections via base types (and PR's via the framework `UniqueConstraintViolationException`), render enum casts via the instance (`$record->lifecycle_state->value`) — no `Catalog\Exceptions`/`Catalog\Enums` import.

**L10 — Tests + PG17.** Pest under `tests/Feature/Modules/OperatorPanel/Catalog/`, one file per entity (`<Entity>ResourceTest`, `<Entity>LifecycleConsoleTest`) plus the i18n + closing-chain files, driving pages via Livewire/Filament helpers and `actingAs(Operator::factory()->create(), 'operator')`. Assert **domain effects** (state + recorded events + `actor_role: newco_ops`/`actor_id`) — the UI is the trigger, the Action is the contract. The closing-chain test builds a full `Master → Variant → Product Reference → {Sellable SKU, Composite SKU}` active tree **through the consoles**, exercising the cascade gate (activate a child under a non-active parent), the reference-integrity blocks (retire a PR/Case Config under an active SKU), and the two create form-errors (PR duplicate, Composite `<2`). Every DB-touching test is verified on **PostgreSQL 17** before close (`knowledge/testing/rules.md`); record the PG run in `progress.md`. Run the full suite as `php -d memory_limit=-1 vendor/bin/pest` and PHPStan as `php -d memory_limit=-1 vendor/bin/phpstan analyse` (the suite blows the 128M default — Codebase Patterns).

## Risks / Trade-offs

- **The kit's base CreateRecord could silently fall back to the default `new Model($data); save()`** → the base `handleRecordCreation` MUST delegate to the per-entity `createViaAction`; the `NoEloquentWriteInOperatorPanelRule` fails on any Eloquent write in the namespace (kit base classes included). Part of DoD.
- **Retrofit regresses Master** → run the entire shipped Master test file + the arch/boundary tests after the retrofit; the retrofit changes no behaviour. If the base cannot express Master's producer picker / cascade-retire / dedup, the base is wrong — fix the base, do not special-case Master back out.
- **PR's duplicate leaks a raw SQL message** → L5: catch `UniqueConstraintViolationException` specifically and render a console-owned localized message, never `$e->getMessage()` for that path.
- **Composite ordered-constituent input** → verify the Filament 5 ordered-multi-input API in `vendor/` before writing; the Action takes a `list<int>` in bundle order, and the `<2 distinct` guard runs before the transaction (surfaces as a form error on the constituents field).
- **Accidental Master-ism on a spine entity** → no producer picker, no cascade-retire, no producer-gate handling on any of the six. A spine activate that is gate-blocked is blocked by the *parent-active cascade*, not a producer gate.
- **Over-coupling the kit to Catalog** → the kit is shaped from Catalog's seven entities but is the template for nine modules; keep its seams Catalog-agnostic where cheap (the lifecycle wrapper, the read-only conventions), and let genuinely Catalog-specific bits (the producer picker) live on the Master resource, not the base.

## Migration Plan

No database migration. Deploy is code-only. Rollback = revert the change. The Master retrofit is behaviour-preserving (its tests are the proof). No data backfill, no feature flag.

## Open Questions

- **Exact kit class names / trait vs base-class split** — indicative here (`OperatorConsoleResource`, `OperatorConsoleCreateRecord`, `OperatorConsoleViewRecord`, a `SurfacesDomainActions` trait); the implementer confirms the Filament 5 extension points in `vendor/` and may adjust shape (e.g. a concern vs a base page) — not a blocker, resolved in task 1.
- **Composite constituent picker UX** (repeater of PR selects vs an ordered multi-select) — resolved during implementation against the verified Filament 5 API; the contract is "an ordered `list<int>` of ≥2 distinct PR ids reaches `CreateCompositeSku`".
- **Whether to seed a small dev demo tree** (a human-navigable Master→…→SKU chain) — a convenience, resolved during implementation; tests self-seed through the real Catalog Actions.
