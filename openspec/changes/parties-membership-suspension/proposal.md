# Proposal — parties-membership-suspension

## Why

This is the **demand-side status-lifecycle slice of Module K (Parties)** — the direct successor to `parties-membership-activation` (archived). Activation shipped the `→ Active` edges (Customer `pending → active`, Profile `Approved → Active`) and explicitly deferred everything that happens to a Customer/Profile/Account **after** it reaches `Active`. This change closes that whole remaining block in one coherent slice: it is the lifecycle's other half, coupled end-to-end by a single mechanism — a Hold drives `suspended`, lifting it restores — so it ships as one change rather than fragments.

- It closes the seams the shipped *Birth States Recorded, Lifecycle Transitions Deferred* requirement still holds open after activation: Profile `Active → Suspended | Lapsed | Cancelled | Inactive` and `Lapsed → Active` grace (§ 4.2.1 / AC-K-FSM-2, AC-K-FSM-12), Customer `active → suspended | closed` with cascade (§ 4.1 / AC-K-FSM-1), Account `active → suspended → closed` (§ 4.7 / AC-K-FSM-9), and the **Hold→`suspended` coupling** (§ 10.1 / AC-K-FSM-2a) — *"placing a Hold records the Hold but performs no status transition"* is the precise sentence this change deletes.
- It introduces **one new ADR** — `decisions/2026-06-19-hold-status-coupling.md` — resolving how a Hold drives status: a scope is `suspended` **iff** covered by ≥1 active Hold (coverage-recompute, not provenance), restored on lift **iff no other active Hold still covers it**. The transitions are explicit `Suspend*`/`Reactivate*` Actions; `PlaceHold`/`LiftHold`/`RecordKycVerified` invoke them.
- It steps through **no open ADR gate**: domain events deliver **inline** at launch (`decisions/2026-06-12-event-substrate-and-audit-store.md`), so the new status events record like the activation events — no `queued` consumer. No object storage, no payment provider, no frontend, no new dependency. Every transition follows the established Module K lifecycle-Action idiom (`ActivateProducer` / `ActivateCustomer` are the verbatim templates; `RecordKycVerified` is the cross-entity-in-one-transaction template).

## What Changes

- **Profile suspend / restore (Active ↔ Suspended)** (PRD § 4.2.1 / § 10.1; AC-K-FSM-2, AC-K-FSM-2a) — `SuspendProfile` (`Active → Suspended`) records `ProfileSuspended`; `ReactivateProfile` (`Suspended → Active`) records `ProfileReactivated`. Both from-state guarded against a transaction-locked re-read. Suspension is **state-preserving** (AC-K-FSM-2a): it writes **only** `Profile.state` — it does not cancel vouchers/orders/reservations or touch Club Credit (those entities are Module S/B/E, unbuilt — the freeze is a downstream seam, vacuous here).
- **Profile lapse / grace renewal (Active → Lapsed → Active | Cancelled)** (PRD § 4.2.1; AC-K-FSM-12, AC-K-BR-Profile-3, DEC-034) — `LapseProfile` (`Active → Lapsed`) records `ProfileExpired` (the § 15.2 event for this edge — **not** a `ProfileLapsed`, which the catalog does not name) and stamps `lapsed_at`. `RenewProfile` (`Lapsed → Active`, **within the 30-day grace** measured from `lapsed_at`) records `ProfileRenewed`; past 30 days it is rejected (`IllegalProfileTransition`). The validity-period-expiry trigger (lapse) and Module E's `MembershipFeePaid` (renewal) are **deferred seams** — the Actions are the within-module writers, invoked directly now (no scheduler, no Module E fabricated).
- **Profile cancel / deactivate (Active | Lapsed → Cancelled; Active → Inactive)** (PRD § 4.2.1 / § 10.2; AC-K-FSM-13, AC-K-BR-Profile-2) — `CancelProfile` (`Active | Lapsed → Cancelled`, terminal soft-delete) is **audit-only** (§ 15.2 names **no** `ProfileCancelled`; the state write IS the record — the `ApproveProfile`/`DeclineProfile` precedent), stamping a `cancellation_reason`. `DeactivateProfile` (`Active → Inactive`, terminal soft-delete) records `ProfileInactive`. The **per-Profile cancellation signal** Module S consumes for Club-Credit conversion at Producer offboarding (AC-K-EVT-14 / § 15.7) is a **deferred Module-S seam** — its event shape is "a downstream consumer concern" (§ 15.7) and Module S is unbuilt; `CancelProfile` ships the within-module transition + reason, not the offboarding orchestration.
- **Customer suspend / restore / close with cascade** (PRD § 4.1 / § 10.1; AC-K-FSM-1, AC-K-BR-Customer-1/2) — `SuspendCustomer` (`active → suspended`) records `CustomerSuspended` and **cascades** to each of the Customer's `Active` Profiles (one `ProfileSuspended` each — § 15.1 "Cascades to all the Customer's Profiles"); `ReactivateCustomer` (`suspended → active`) records `CustomerReactivated` and **cascade-restores** each Profile **no longer covered** by any active Hold; `CloseCustomer` (`active | suspended → closed`, terminal) records `CustomerClosed`. Suspension is **explicit** (manual or via Hold), never auto-driven by Profile state or a compliance verdict (AC-K-BR-Customer-1; § 9.4). Closure does **not** cascade-cancel Profiles (§ 15.1 `CustomerClosed` names no cascade — zero-invention). `closed` and anonymised stay orthogonal (AC-K-BR-Customer-2; anonymisation out of scope).
- **Account suspend / restore / close (audit-only)** (PRD § 4.7; AC-K-FSM-9) — `SuspendAccount` (`active → suspended`), `ReactivateAccount` (`suspended → active`), `CloseAccount` (`active | suspended → closed`), each from-state guarded. **No Account domain event** — § 15 names none for the Account family (the Account is event-silent at creation too); the transitions are audit-only. `ActivateAccount` is **not** created (the Account is born `active` — there is no `pending → active`; its only `→ active` edge is `ReactivateAccount`).
- **Hold → `suspended` coupling (the headline, ADR 2026-06-19)** (PRD § 10.1 / § 4.2.1 / § 4.7; AC-K-FSM-2a, AC-K-FSM-9, AC-K-BR-Hold-1/3/4) — `PlaceHold` now drives every **covered scope in its suspendable from-state** to `suspended` (customer-scope → `SuspendCustomer` + cascade; account-scope → `SuspendAccount`; profile-scope → `SuspendProfile`), invoked in the placement transaction. `LiftHold` (operator) and `RecordKycVerified` (system `kyc`-lift) now restore every **covered `suspended` scope** to `active` **iff no other active Hold still covers it** (coverage re-queried via the read-API's cascade logic). A Hold on a **non-suspendable** scope (the `kyc` Hold auto-placed on a `pending` Customer at onboarding) records the Hold and drives **no** transition — the from-state guard preserves compliance-FSM independence.
- **Eight demand-side status events** (verbatim PRD § 15.1 / § 15.2) — `CustomerSuspended` / `CustomerReactivated` / `CustomerClosed` (`entity_type = Customer`), `ProfileSuspended` / `ProfileReactivated` / `ProfileExpired` / `ProfileRenewed` / `ProfileInactive` (`entity_type = Profile`), each recorded transactionally through `DomainEventRecorder`, module `parties`, actor from `ActorContext`, **PII-free** payload (entity ids + enum/business values).
- **One additive migration** — two nullable columns on `parties_profiles`: `lapsed_at` (the grace-window anchor, § 4.2.1 *"the Profile carries a `lapsed_at` timestamp"*) and `cancellation_reason` (§ 4.2.1 / § 10.2 — the Producer-initiated cancellation reason). **No change to the partial-unique index** — it already excludes `{rejected, cancelled, inactive}`, so the now-reachable terminal states need no index migration (only tests exercising that a cancelled Profile does not block a fresh application). New `IllegalProfileTransition` / `IllegalCustomerTransition` factories + a new `IllegalAccountTransition`; new `account` lang group; eight new event classes.
- **Guard-test realignment** — `SupplyLifecycleChainTest` (exact-set transition whitelist + event-non-existence list) and `ComplianceIndependenceTest` (forbidden-Action list) currently assert the **absence** of exactly this slice's Actions; they are narrowed to what **remains** deferred (`ActivateAccount`, `LockOriginatingClub`/`SetOriginatingClub`, `WaitingList`, segments stay forbidden/absent). `HoldRegistryTest` / `HoldChainTest` assert *"placing a Hold performs no status transition"* — those scope-guards are **inverted** (in the coupling tasks only) where a Hold now suspends an `active`-born scope.
- **Docs** — extend `CONTEXT.md` with the resolved suspension terms (the coverage-recompute coupling; the `ProfileExpired`=lapse / no-`ProfileLapsed` and audit-only-cancel naming traps; Account audit-only; the deferred seams) and the eight event payloads.

## Capabilities

### New Capabilities

_None._ This change extends the existing `party-registry` capability with the demand-side status transitions (the Module-K-is-one-capability pattern the prior slices established); it introduces no new capability.

### Modified Capabilities

- `party-registry`: gains the demand-side status FSMs (Customer / Profile / Account) and the Hold→`suspended` coupling. **ADDED** — *Profile Suspension and Restoration* (`SuspendProfile` / `ReactivateProfile`), *Profile Lapse and Grace Renewal* (`LapseProfile` / `RenewProfile`), *Profile Cancellation and Deactivation* (`CancelProfile` audit-only / `DeactivateProfile`), *Customer Suspension and Closure* (`SuspendCustomer` cascade / `ReactivateCustomer` / `CloseCustomer`), *Account Status Lifecycle* (`SuspendAccount` / `ReactivateAccount` / `CloseAccount`, audit-only), *Hold-Driven Status Coupling* (place ⇒ suspend, lift ⇒ restore-iff-uncovered), *Demand-Side Status Events* (the eight events). **MODIFIED** — *Profile — Multi-Profile Membership* (the uniqueness constraint's terminal states are now reachable — the refinement its own text defers to "the deferred Profile lifecycle, whose change refines the constraint accordingly"), *Birth States Recorded, Lifecycle Transitions Deferred* (narrowed — suspension / lapse / cancel / inactive / close + the Hold coupling are now implemented, while the Hero Package invariant, WaitingList and Customer segments remain the only deferred demand-side seams).

## Impact

- **New code** — `app/Modules/Parties/Actions/{SuspendProfile,ReactivateProfile,LapseProfile,RenewProfile,CancelProfile,DeactivateProfile,SuspendCustomer,ReactivateCustomer,CloseCustomer,SuspendAccount,ReactivateAccount,CloseAccount}.php`; `app/Modules/Parties/Events/{ProfileSuspended,ProfileReactivated,ProfileExpired,ProfileRenewed,ProfileInactive,CustomerSuspended,CustomerReactivated,CustomerClosed}.php`; `app/Modules/Parties/Exceptions/IllegalAccountTransition.php` + new factories on `IllegalProfileTransition`/`IllegalCustomerTransition` + new `profile`/`customer`/`account` lifecycle keys in `lang/en/parties.php`; one additive migration on `parties_profiles`; tests under `tests/Feature|Unit/Modules/Parties/`.
- **Modified code** — `app/Modules/Parties/Models/Profile.php` gains the `lapsed_at` (`immutable_datetime`) + `cancellation_reason` casts + `@property`; `app/Modules/Parties/Actions/{PlaceHold,LiftHold,RecordKycVerified}.php` gain the status coupling (and lose the *"performs no status transition"* docblock seam-note); the guard tests `SupplyLifecycleChainTest` / `ComplianceIndependenceTest` / `HoldRegistryTest` / `HoldChainTest` are narrowed/inverted to the still-deferred set.
- **One migration, no new dependency** — `git diff main -- composer.json composer.lock` stays empty; the only schema change is the **two additive nullable** columns on `parties_profiles` (no backfill, reversible `down()`); the partial-unique index is untouched (already excludes the terminal set).
- **Reuses, does not modify** — `DomainEventRecorder` (open-transaction guard), `ActorContext` / `ActorRole`, the `Module` enum, the `CustomerStatus` / `ProfileState` / `AccountStatus` / `HoldStatus` / `HoldScope` / `HoldType` enums (all states already enumerated — **no enum/state migration**), and the read-API coverage logic `DatabaseComplianceStatusReader::activeHoldTypesForScopes()` (the Customer→Profile cascade shape the coupling re-queries).
- **Boundary & arch** — every reference is within Module K (`ModuleBoundariesTest`); no cross-module DB access (Module A capacity, Module E `MembershipFeePaid`, Module S Club-Credit conversion are deferred seams, not reads). The Customer→Profile cascade and the Hold→status coupling are cross-**entity** writes within one Action (all Module K) — the `RetireProducer` / `RecordKycVerified` in-module pattern.
- **DB engines** — every transition + the two columns + the eight events touch the database, so every DB-touching task is verified on **PostgreSQL 17** before close, in addition to SQLite (`knowledge/testing/rules.md`).
- **Deliberate traceability gaps** — each deferred concern above maps to a named future change/module. The §13 capacity cap, WaitingList, Customer segments, the Producer-offboarding orchestration + the AC-K-EVT-14 signal, the `MembershipFeePaid` listener, the Club-Credit freeze, anonymisation, and the operator/producer surfaces are recorded as named seams; the Actions they will drive / consume ship ready.
