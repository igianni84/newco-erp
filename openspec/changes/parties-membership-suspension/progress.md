## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Additive-column migration on a Parties table = the `2026_06_18_000002` template.** Plain `Schema::table(...)` with nullable columns, **no default, no backfill**, reversible `down()` that only `dropColumn([...])`s (leaves the table + indexes intact — proven by rolling back one step and asserting `to_regclass` still returns the table). A `timestamptz` carries no value-set CHECK (its domain is the type) → `timestampTz()->nullable()` + an `immutable_datetime` cast + `@property CarbonImmutable|null`. A free-text column with no enum (e.g. `cancellation_reason`) is a plain `string()->nullable()` with **no cast** and `@property string|null`. The partial-unique index on `parties_profiles` already excludes `{rejected,cancelled,inactive}` — never re-migrate it when making a terminal state reachable.
- **Schema-column test = the `CustomerOnboardingAcceptanceTest` template** (3 `it()`s, `uses(RefreshDatabase::class)`): (1) default factory row → new columns NULL by birth; (2) round-trip via a re-fetch (`Model::findOrFail($created->id)`) so hydration casts run — read a `timestamptz` back THROUGH the `immutable_datetime` cast + `->format('Y-m-d H:i:s')` (testing-rule #4: never byte-compare a raw timestamptz, `+00` on PG); (3) `Schema::hasColumn(...)` for each column (positively proves the column on whatever engine the run is on, incl. the PG17 cross-engine pass).
- **PG17 cross-engine close** (knowledge/testing/rules.md): `docker run -d --name pg … -p 55432:5432 postgres:17`; full suite via `DB_CONNECTION=pgsql DB_HOST=127.0.0.1 DB_PORT=55432 … php -d memory_limit=512M vendor/bin/pest` (artisan test OOMs at 128M + pao swallows the PG summary). `docker rm -f pg` after.
- **Transition-exception (`Illegal*Transition`) factory = the verbatim `cannotApprove` template:** a `public static fn cannotX(SomeEnum $from): self` returning `new self((string) __('parties.<group>.cannot_x', ['state' => $from->value]))` — the `(string)` cast tames Larastan's `mixed` `__()` return; the `:state` token is a business enum value (NOT PII), so it IS interpolated (PII like an email/timestamp would be named-not-interpolated, the `gateNotMet`/`DuplicateCustomerEmail` discipline). Copy lives in `lang/en/parties.php` only (English baseline + per-key fallback, DEC-127); a new entity-family gets a new top-level group (`account`). Adding factories/keys is purely additive — moves NO guard test (1.x).
- **Type-clean Pest test for a localized static factory (no-DB, `uses(TestCase::class)` to boot the translator):** call the factory in the test BODY (`$e = Foo::cannotX(Enum::Case); expect($e)->toBeInstanceOf(Foo::class);`) — NOT via a `->with([fn () => Foo::cannotX(...)])` dataset whose `callable` param makes `$factory()` return `mixed` (→ PHPStan `method.nonObject` on `$e->getMessage()`). A `@param class-string` PHPDoc on a *variable-assigned* closure is NOT honored by PHPStan — feed `toBeInstanceOf` a literal `::class` (already a `class-string`), never a `string` var. Prove `:state` was interpolated by choosing a from-state token ABSENT from the literal template (e.g. `Lapsed` against a "suspends only from active" message; for an all-tokens-named template like Account `cannot_close`, only `closed` is absent — and not a substring of "closes"). Prove a subclass via `(new ReflectionClass(X::class))->isSubclassOf(Y::class)`, never `is_subclass_of()`/`instanceof` (PHPStan constant-folds those to `alreadyNarrowedType`). PII guard mirrors `gateNotMet`: `preg_match('/\d/', $msg) === 0` + no `'@'` (so keep business digits like "30-day" OUT of copy — say "the grace window").
- **Self-edge status-transition Action = the verbatim `ActivateProfile` template.** A `Suspend*`/`Reactivate*`/`Lapse*`/`Deactivate*` Action (`handle(int $id): Model`) injects `DomainEventRecorder` + `ActorContext`; one `DB::transaction` → `Model::query()->whereKey($id)->lockForUpdate()->firstOrFail()` → from-state `if ($m->state !== ProfileState::X) throw Illegal*Transition::cannotY($m->state)` (FIRST, before any write) → `update(['state' => …])` (state-preserving — write ONLY the status field, L9) → `record(Event::NAME, Module::Parties->value, $this->actor->role(), $this->actor->actorId(), Event::ENTITY_TYPE, (string) $m->id, Event::payload($m))`. Directly-invoked = ROOT event (omit causationId/correlationId — the recorder self-correlates). Reference forward-task classes (`SuspendCustomer`, `RenewProfile`) in plain backticks in docblocks, `{@see}` only for already-shipped ones, so no broken `@see` accumulates as the slice lands tasks out of order.
- **Guard-narrowing for a new transition Action (`SupplyLifecycleChainTest`, the ONLY guard a Profile self-edge touches).** Add the Action name to the exact-set whitelist or it goes red on `toEqualCanonicalizing`: cleanest is a NEW sub-array (`$demandSideStatusTransitions = ['SuspendProfile','ReactivateProfile']`) spread into the assertion (design L10). `ComplianceIndependenceTest` stays GREEN unamended for Profile self-edges — its forbidden list is Customer/Account/`*OriginatingClub` only, and its per-Action `'originating_club_id' =>` write-count check puts a new Action in the 0-writes `else` branch. Always `grep -rn '<Action>' tests/` first to re-derive the real blast radius (only `SupplyLifecycleChainTest` + `ComplianceIndependenceTest` glob `Actions/*.php`).
- **State-preservation proof = the event-delta + table-count snapshot.** To prove a transition "writes only `state`" (L9) when the downstream tables (voucher/order/credit) don't exist yet: snapshot `DomainEvent::count()` + each sibling table count BEFORE the call, then assert the DomainEvent delta is exactly 1 (the single status event) and every other count is unchanged, plus the row's non-status columns (`customer_id`/`club_id`) are untouched. Drive the happy-path subject through the REAL `CreateProfile → ApproveProfile → ActivateProfile` chain (a top-level Pest helper, uniquely named per file — `createActiveProfile()`); the guard/restore cases use `factory()->create(['state' => …])` to pin a from-state in isolation. **PG id-type safety (trap #6):** compare a jsonb-stored id (written by the Action's re-read) against a value from the SAME hydration source (the helper's `findOrFail`, or a factory model — the `ProfileActivationTest` pattern proven on PG) so `->toBe()` stays type-matched on both engines; the PK `id` round-trips int on PG, FKs read back as numeric strings.
- **A time-windowed transition Action (lapse/grace) = the self-edge template PLUS a frozen-clock test.** `LapseProfile` stamps `update(['state' => …, 'lapsed_at' => CarbonImmutable::now()])`; `RenewProfile` guards a 30-day INCLUSIVE window — `$deadline = $profile->lapsed_at?->addDays(30); if ($state !== Lapsed || $deadline === null || CarbonImmutable::now()->greaterThan($deadline)) throw …` (strict `greaterThan` ⇒ exactly-on-the-edge still renews; both wrong-from-state AND past-grace throw the same `cannotRenew`), then clears `lapsed_at` on success. **Test the boundary with a frozen clock** (`afterEach(fn () => CarbonImmutable::setTestNow())` + per-test `setTestNow($now)`): factory a `Lapsed` Profile with `lapsed_at = $now->subDays(30)` (renews) vs `$now->subDays(30)->subSecond()` / `subDays(31)` (rejects) — proving both sides of the inclusive edge deterministically on PG. Assert a persisted `timestamptz` as an INSTANT — `$row->lapsed_at?->equalTo($now)` (null-safe `?->`, tz-robust across the SQLite/PG asymmetry, trap #4), never a byte-compare.
- **Two traps that fail ONLY at PHPStan / isolated-run (not in the green full suite):** (1) **`CarbonImmutable::create(...)` is typed `CarbonImmutable|null`** in this project's stubs → any `->subDays()`/`equalTo($it)` on the result is a `method.nonObject`/`argument.type` error. Use **`CarbonImmutable::parse('2026-06-19 12:00:00', 'UTC')`** for a non-null instant (the `SupplyLifecycleChainTest` idiom). `setTestNow(create(...))` is fine only because `setTestNow` accepts null. (2) **A Pest helper from a sibling file (`createActiveProfile()` in `ProfileSuspensionTest`) makes the new file pass in the FULL suite but FAIL run-in-isolation** (`Call to undefined function`). The global-helper rule's "reuse OR uniquely-name" → prefer a **uniquely-named file-local helper** (`lapseGraceActiveProfile()`) so the file is self-contained; never redeclare the sibling's name (fatal redeclare on full-suite load). Always run the new test file ALONE once, not just in the suite.
- **Cascade Action (cross-entity, causation children) = the `RetireProducer`/`RecordKycVerified` template.** A `Suspend*`/`Reactivate*` parent that fans out to child rows: guard + `update` the parent → `$root = record(ParentEvent::NAME, …)` (ROOT — omit causation/correlation) → re-read the children `->lockForUpdate()->get()` via a `hasMany` on the parent (add it mirroring `Producer::clubs()` — within-module, so `ModuleBoundariesTest` stays green; Pint's `fully_qualified_strict_types` will hoist the docblock `{@see}` FQNs into `use` imports, consistent with the model's existing `{@see CreateCustomer}`) → for each child in the right from-state, `update(['state' => …])` + `record(ChildEvent::NAME, …, correlationId: $root->correlation_id, causationId: $root->id)`. The cascade **inlines** the child write+record — it does NOT delegate to the child's self-edge Action (`SuspendProfile`/`ReactivateProfile`), which records a ROOT (the cascade needs causation CHILDREN — design L11; the same reason `RecordKycVerified` inlines its Hold lift instead of reusing `LiftHold`). Resolve the actor ONCE before the root and reuse it for every child. **Causation-child assertion (trap #6 extended):** `domain_events.causation_id` is UNCAST (`@property int|null` is a hint, NOT a `casts()` entry) → it reads back a **numeric string on PG** while the `id` PK round-trips int — assert `(int) $child->causation_id === $root->id`; `correlation_id` is a uuid string → compare directly. Pin the cascade event counts with factory-built subjects (factories record no event, so every counted event is the Action's).
- **Coverage-recompute under lock** = the `(Profile-scope) OR (its Customer-scope)` union of `DatabaseComplianceStatusReader::activeHoldTypesForScopes()`, but re-read as Hold ROWS `->lockForUpdate()->get()` (the write path locks the coverage rows) and reduced with `return count($holds) > 0` — NOT `->isNotEmpty()` (trips larastan `noUnnecessaryCollectionCall`, which steers you to `->exists()`), and NOT `->lockForUpdate()->exists()` (would lift `FOR UPDATE` into an `EXISTS(...)` subquery — avoided for PG cross-engine safety; the row-lock via `->get()` is the proven `RecordKycVerified` idiom). Profile-scope Holds isolate (BR-K-Hold-4); a Customer-scope Hold covers all its Profiles (BR-K-Hold-3). `PlaceHold` at tasks ≤3.x records the Hold WITHOUT a status transition (the coupling lands at 4.x), so tests can place a residual Hold to set up coverage without auto-suspend.
- **No-cascade root transition** (`CloseCustomer`) = the `SuspendCustomer` template MINUS the cascade: recorder+actor injected, one `DB::transaction` → `lockForUpdate` re-read → from-state guard (`in_array([Active, Suspended], strict)` for a two-from-state edge, mirroring `CancelProfile`'s guard) → `update(['status' => …])` → root `record(Event::NAME, …)` and STOP (no child re-read/loop). The no-cascade proof in the test: snapshot table counts + assert the Profile is untouched + `DomainEvent::count()` delta is exactly 1 (the lone root) + `where('entity_type','Profile')->count() === 0`. § 15.1 names a cascade only for `CustomerSuspended`/`CustomerReactivated`, NOT `CustomerClosed` (design L7) — zero-invention skips it.
- **Audit-only FSM trio** (Account `SuspendAccount`/`ReactivateAccount`/`CloseAccount`) = the verbatim `CancelProfile` no-recorder shape (inject NEITHER `DomainEventRecorder` NOR `ActorContext`; just `handle(int): Model { DB::transaction(lockForUpdate → guard → update) }`). Audit-only proof in the test = the event-count DELTA is **0** across the whole walk (`DomainEvent::count() === 0` after a factory-built subject + `where('entity_type','Account')->count() === 0`). The Account is born `active` (no `pending`), so there is NO `ActivateAccount` — its only `→ active` edge is the restore `ReactivateAccount`; assert absence via `glob(Actions/*.php)` basenames `not->toContain('ActivateAccount')` + `class_exists(...)->toBeFalse()`.
- **Hold→status coupling on PLACE** (`PlaceHold`, 4.1) = inject the three Suspend Actions in the constructor (the `RequireKyc → PlaceHold` Action-calling-Action precedent — no DI cycle: none of `SuspendCustomer`/`SuspendAccount`/`SuspendProfile` inject `PlaceHold`). After recording `CustomerHoldPlaced` and **inside the same transaction**, `match($scope)` dispatches to a per-scope helper that **pre-checks the from-state** on a lock-free `->first()` read (`$customer?->status === CustomerStatus::Active`, `$account?->status === AccountStatus::Active`, `$profile?->state === ProfileState::Active` — `?->` so an absent scope is a clean no-op) and invokes the matching Suspend Action **only when suspendable**. The pre-check is load-bearing: the Suspend Action's OWN from-state guard *throws* on a wrong from-state, which would roll back the whole placement (incl. the Hold) — so a Hold on a non-suspendable scope (onboarding `kyc` Hold on a `pending` Customer, a Hold on an `Applied` Profile / already-`suspended` scope) must record the Hold and invoke nothing. **The writer locks:** `PlaceHold` does NOT lock the scope (it doesn't write the status); the invoked Suspend Action does its own `lockForUpdate` re-read inside a nested transaction = **a SAVEPOINT under the placement transaction** (the `RequireKyc → PlaceHold` nesting, one level deeper — verified on PG17). The cascade/audit-only behaviour is the Suspend Action's, unchanged — `PlaceHold` only decides *whether* to invoke.
- **Guard inversion on a coupling task = flip ONLY what actually breaks; the `active`-born Account breaks first.** A Hold on a `pending` Customer or `Applied` Profile (birth states) is non-suspendable, but the Account is born `active`, so it is the FIRST scope a placed Hold suspends. A "Hold drives no status transition" guard therefore breaks ONLY on its Account assertion → flip that one to `suspended`, KEEP the Customer-`pending`/Profile-`applied` assertions, and the demand-side-event-absence loop stays valid **because the Account suspend is audit-only** (records no event — the "only Hold events recorded" assertion holds even as the Account flips). A guard whose Holds are all Customer-scope on a `pending` Customer (`HoldChainTest`) needs NO amendment. `grep -rn '<symbol>' tests/` + RUN the suite to see what truly breaks — do NOT pre-amend a green guard (minimal-impact; over-amending a green test only adds churn). Confirmed 4.1: of the four PlaceHold-observer guards, only `HoldRegistryTest`'s one Account assertion broke.
- **Hold→status coupling on LIFT** (`LiftHold`/`RecordKycVerified`, 4.2) = the mirror of 4.1. Inject the three Reactivate Actions in `LiftHold` (the Suspend-injection precedent — no DI cycle: no `Reactivate*` injects `LiftHold`); `RecordKycVerified` injects only `ReactivateCustomer` (it only ever lifts Customer-scope `kyc` Holds). After recording `CustomerHoldLifted` (or the system kyc-lift loop) and **inside the same transaction**, dispatch on `scope_type` → a per-scope `restore*IfUncovered` helper that (a) **pre-checks the from-state** lock-free (`$customer?->status === CustomerStatus::Suspended` / `Account…Suspended` / `Profile…ProfileState::Suspended` — `?->` so an absent scope is a clean no-op) AND (b) **re-queries coverage** under lock, invoking the matching `Reactivate*` only when `suspended && ! stillCovered`. **The just-lifted Hold is already `Lifted` before the re-query** (the lift `update` runs first), so it never counts as its own residual coverage — the "re-query after lift" model needs no special-casing. **Three coverage shapes** (NOT one): Customer = active **Customer-scope** Holds on it (Customers aren't covered by Profile/Account Holds); Account = active **Account-scope** Holds on it (isolate); Profile = the **`(Profile-scope) OR (its Customer-scope)` union** (BR-K-Hold-3 cascade) — the same shape `ReactivateCustomer::stillCovered` uses. All re-read as Hold ROWS `->lockForUpdate()->get()` + `count() > 0` (the writer locks the coverage rows; never `->exists()` — PG cross-engine). `ReactivateCustomer` (invoked for a Customer lift) does the **per-Profile** cascade-restore coverage check itself, so `LiftHold` only checks **Customer-level** coverage before invoking it. The nested lift→Reactivate is a SAVEPOINT under the lift tx (PG-verified). **DRY note:** `customerStillCovered`+`restoreCustomerIfUncovered` are duplicated in `LiftHold`+`RecordKycVerified`, and `profileStillCovered` duplicates `ReactivateCustomer::stillCovered` — consistent with the established "the writer duplicates the coverage query" convention (`ReactivateCustomer` already duplicates the read-API shape); a shared `HoldCoverage` reader across the three Actions is a clean future `improve-codebase-architecture` refactor (deferred — extracting mid-slice risks the `Actions/*.php`-globbing arch guards).
- **The LIFT coupling inverts NO guard test (verified empirically, like PLACE).** Of the 9 `LiftHold`/`RecordKycVerified`-observing test files, **none** broke: every existing lift operates on a `pending` Customer (`HoldChainTest`/`HoldLifecycleTest`/`ComplianceIndependenceTest`/`ComplianceReadApiTest`/`CustomerKycLifecycleTest`) — never suspended, so the from-state pre-check makes the restore a no-op — and the one `active`-born Account suspended in `HoldRegistryTest` test 4 never has its Hold **lifted** in that test (only a Customer Hold is lifted, off a `pending` Customer). So the design's "amend the remaining `HoldRegistryTest`/`HoldChainTest` lift-side scope-guards" did **not** materialise — the same lesson as 4.1: `grep` + RUN before amending; the from-state guard keeps birth-state lifts inert. Both 4.x coupling tasks shipped touching ZERO guard tests.
- **Guard-realignment reached its terminal shape at 3.2** (the LAST task to add new Action classes). `SupplyLifecycleChainTest`'s `$demandSideStatusTransitions` whitelist now holds all twelve demand-side status Actions; `ComplianceIndependenceTest`'s forbidden list is down to its **permanent floor** — `ActivateAccount` (born active, never exists) + `LockOriginatingClub`/`SetOriginatingClub` (the OC lock lives in `ApproveProfile`, never a standalone Action).
