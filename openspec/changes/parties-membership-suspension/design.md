# Design — parties-membership-suspension

## Context

`parties-membership-activation` shipped the `→ Active` edges and deferred the rest of the demand-side status lifecycle to **this** change (named in its boundary table). This slice closes that block: Profile `Active → Suspended | Lapsed | Cancelled | Inactive` + `Lapsed → Active` grace, Customer `active → suspended | closed` with cascade, Account `active → suspended → closed`, and the **Hold→`suspended` coupling** the shipped *Birth States* requirement defers with the sentence *"placing a Hold records the Hold but performs no status transition"*. It is **additive** — one migration adding two nullable columns to `parties_profiles`, twelve new Actions, eight new events, one new exception + factories — plus narrowing/inverting four archived guard tests, and wiring the coupling into three shipped Actions. **No new dependency**, **no open ADR gate** (events deliver inline). One **new ADR** governs the coupling: `decisions/2026-06-19-hold-status-coupling.md`.

The substrate this builds on (verified in `app/Modules/Parties` + `app/Platform/Events`):

- The transition-Action idiom (`app/Modules/Parties/Actions/ActivateCustomer.php` / `ActivateProducer.php` are the verbatim templates): `DB::transaction` → `Model::query()->whereKey($id)->lockForUpdate()->firstOrFail()` → **from-state guard** throwing a localized `Illegal*Transition` (**before** any other gate) → `update([...])` → `record(...)`, all in the one transaction. Models stay persistence-only (`$guarded = []`); the **Action is the sole writer**. `version` is **not** bumped (the domain event is the audit record). `RecordKycVerified` is the **system-driven cross-entity-in-one-transaction** precedent (writes `kyc_status`, then queries active `kyc` Holds `->lockForUpdate()->get()` and lifts each + records `CustomerHoldLifted`, inline — it deliberately does **not** reuse `LiftHold` because the per-type discipline (ADR 2026-06-18) forbids an operator path from the `kyc` Hold).
- `DomainEventRecorder::record(string $name, string $module, ActorRole $actorRole, ?int $actorId, string $entityType, string $entityId, array $payload, ?string $correlationId = null, ?int $causationId = null): DomainEvent` — appends inside the caller's open transaction (throws at `transactionLevel() === 0`); a **root** event omits the last two. `ActorContext::role()`/`actorId()` (default `ActorRole::System`); `Module::Parties->value === 'parties'`.
- The enums **already enumerate every state** this slice transitions to (so there is **no state/enum migration**): `CustomerStatus` {`Pending`,`Active`,`Suspended`,`Closed`}; `ProfileState` {`Applied`,`WaitingList`,`Approved`,`Rejected`,`Active`,`Suspended`,`Lapsed`,`Cancelled`,`Inactive`} (column is **`state`**, not `status`); `AccountStatus` {`Active`,`Suspended`,`Closed`}; `HoldStatus` {`Active`,`Lifted`}; `HoldScope` {`Customer`,`Account`,`Profile`}; `HoldType` {`Admin`,`Kyc`,`Payment`,`Fraud`,`Compliance`,`Credit`} with `autoLiftable()` true for `Kyc`/`Payment`.
- The Hold registry (`PlaceHold` records `CustomerHoldPlaced`; `LiftHold` records `CustomerHoldLifted` — **every** scope uses these two names, the payload's `scope_type`/`scope_id` distinguishing scope). The read-API `DatabaseComplianceStatusReader::activeHoldTypesForScopes(array $scopes): array` computes the distinct active `HoldType`s across one-or-more `(HoldScope, int)` pairs; `forProfile` passes **both** `[Profile,id]` and `[Customer,customerId]` — the BR-K-Hold-3 cascade / BR-K-Hold-4 isolation shape the coupling re-queries (writing, so it re-reads Hold rows under lock, not the type-only DTO).
- The partial unique index `parties_profiles_customer_club_nonterminal_unique` = `(customer_id, club_id) WHERE state NOT IN ('rejected','cancelled','inactive')` is created with **one raw-DDL statement on both engines** (the predicate is valid identically on SQLite + PG); it **already excludes** the terminal set, so making `Cancelled`/`Inactive` reachable merely *exercises* it. The additive-migration idiom: nullable columns, no backfill, reversible `down()`, casts + `@property` on the model.
- i18n: only `lang/en/parties.php` (English baseline + per-key fallback — DEC-127); existing groups `producer`/`club`/`producer_agreement`/`customer`/`profile`/`kyc`/`sanctions`/`hold` — **no `account` group yet**.

## Goals / Non-Goals

**Goals:**
- Profile `Active ↔ Suspended` (`SuspendProfile`/`ReactivateProfile` → `ProfileSuspended`/`ProfileReactivated`), state-preserving.
- Profile `Active → Lapsed → Active|Cancelled` (`LapseProfile` → `ProfileExpired` + `lapsed_at`; `RenewProfile` → `ProfileRenewed`, 30-day grace; lapse → cancel after grace).
- Profile `→ Cancelled` (`CancelProfile`, audit-only, `cancellation_reason`) and `→ Inactive` (`DeactivateProfile` → `ProfileInactive`).
- Customer `active → suspended | closed` (`SuspendCustomer` cascade / `ReactivateCustomer` cascade-restore / `CloseCustomer`) → `CustomerSuspended`/`CustomerReactivated`/`CustomerClosed`.
- Account `active → suspended → closed` (`SuspendAccount`/`ReactivateAccount`/`CloseAccount`), audit-only.
- The **Hold→`suspended` coupling** (ADR 2026-06-19): coverage-recompute, `PlaceHold`/`LiftHold`/`RecordKycVerified` drive the explicit Actions, from-state guarded, cascading.
- Every DB-touching task verified on PostgreSQL 17.

**Non-Goals (named seams — proposal boundary table):**
- The **Hero Package Capacity Invariant** + `Applied → WaitingList` + `WaitingListJoined` → `parties-hero-package` (Module A). **Customer segments** → `parties-customer-segments`.
- The **Producer-offboarding cascade** orchestration + the AC-K-EVT-14 per-Profile signal → Module S; the **`MembershipFeePaid` listener** → Module E; **Club Credit** (incl. the suspension freeze) → `club-credit`.
- **Anonymisation** (`closed` ≠ anonymised) → `parties-anonymisation`. **Enhanced-KYC-at-order** → Module S. The **operator/producer UI** → `parties-operator-console` / portal.

## Decisions

### L1 — One additive migration on `parties_profiles` (`lapsed_at` + `cancellation_reason`); the unique index is UNTOUCHED
The only schema change is two **nullable** columns on `parties_profiles`: `lapsed_at` (`timestampTz`, the grace-window anchor — § 4.2.1 *"the Profile carries a `lapsed_at` timestamp"*) and `cancellation_reason` (`string`, nullable — the Producer-initiated cancellation reason § 4.2.1 / § 10.2). `immutable_datetime` cast on `lapsed_at`, `@property` for both on `Profile`. No backfill, reversible `down()`. **The partial unique index is not migrated**: it already excludes `{rejected, cancelled, inactive}`, so making `Cancelled`/`Inactive` reachable only *exercises* the predicate (a regression test proves a `Cancelled` Profile does not block a fresh `Applied` one). _Alternative rejected:_ a `cancelled_at` timestamp too — the cancellation event/audit-record carries the moment (and cancellation is audit-only — L2 — so the `domain_events`/`audit_records` row IS the time); only the **reason** needs a column (it is domain data a future Module-S consumer reads), and only `lapsed_at` needs persistence on the row (the grace check reads it). _Alternative rejected:_ a membership-validity-end column for `LapseProfile` to key off — the validity-period mechanic is a deferred renewals/Module-E concern; `LapseProfile` ships as the within-module writer invoked directly (L5), reading no validity field.

### L2 — Cancellation is AUDIT-ONLY; the per-Profile cancellation signal is a deferred Module-S seam
§ 15.2's Profile event family names **no `ProfileCancelled`** (it names `ProfileExpired`/`ProfileRenewed`/`ProfileSuspended`/`ProfileReactivated`/`ProfileInactive` — not a cancellation event), and § 15.7 explicitly defers the cancellation **signal** shape as *"a downstream consumer concern … a NewCo-specific name or a reason-coded `ProfileExpired` variant"*. So — exactly as `ApproveProfile`/`DeclineProfile` write `state` and record **no** event (the shipped audit-only precedent) — `CancelProfile` writes `state = Cancelled` (+ `cancellation_reason`) and records **no** domain event; the append-only audit trail is the record. The AC-K-EVT-14 per-Profile cancellation signal Module S consumes for Club-Credit conversion at Producer offboarding (§ 10.2, DEC-043) is a **deferred Module-S seam** — Module S is unbuilt and the event shape is its concern; this change ships the within-module `→ Cancelled` transition + the Producer-initiated reason, not the offboarding orchestration. Inventing a `ProfileCancelled` event is forbidden (it would coin a name the catalog leaves open — zero-invention). _Alternative rejected:_ emit a NewCo `ProfileCancelled` now "to satisfy AC-K-EVT-14 early" — § 15.7 defers the shape and the consumer (Module S) is absent; eager invention.

### L3 — Verbatim § 15.2 event↔transition mapping (the naming traps); invent no event
The event names do **not** track the state names — copy § 15.2 exactly:
- `Active → Lapsed` records **`ProfileExpired`** (§ 15.2). There is **no `ProfileLapsed`** — the state is `Lapsed`, the event is `ProfileExpired`. (`ProfileState::Lapsed` is the enum value; `ProfileExpired::NAME = 'ProfileExpired'`.)
- `Lapsed → Active` (grace) records **`ProfileRenewed`** (§ 15.2 — "a renewal cycle's `MembershipFeePaid` extends validity"). It does **not** reuse `ProfileReactivated`.
- `Suspended → Active` records **`ProfileReactivated`** (§ 15.2) — **only** this edge; never the grace edge.
- `Active → Inactive` records `ProfileInactive`; `Active → Suspended` records `ProfileSuspended`.
- Customer: `CustomerSuspended` (`active → suspended`, "cascades to all Profiles"), `CustomerReactivated` (`suspended → active`), `CustomerClosed` (`active|suspended → closed`, terminal).
- Account: **no event** (§ 15 names none — L8).
The guard tests pin specific never-existing names; coining any event outside the eight is forbidden.

### L4 — Twelve Actions are the sole writers; mirror `ActivateCustomer`; no enum/state migration
Each of the twelve Actions follows the `ActivateCustomer`/`ActivateProducer` shape (inject `DomainEventRecorder` + `ActorContext`; one `DB::transaction`; `lockForUpdate` re-read; from-state guard → localized exception **first**; `update([...])`; `record(...)` for the evented transitions). From-states: `SuspendProfile` Active→Suspended; `ReactivateProfile` Suspended→Active; `LapseProfile` Active→Lapsed; `RenewProfile` Lapsed→Active (grace — L5); `CancelProfile` {Active,Lapsed}→Cancelled; `DeactivateProfile` Active→Inactive; `SuspendCustomer` active→suspended (cascade — L7); `ReactivateCustomer` suspended→active (cascade — L7); `CloseCustomer` {active,suspended}→closed; `SuspendAccount` active→suspended; `ReactivateAccount` suspended→active; `CloseAccount` {active,suspended}→closed. Because every target state is **already** in the enums, this is `update()` on the existing locked row — **no enum/state migration**. The models stay persistence-only. _Alternative rejected:_ an FSM-library / a `transitions()` map on the enums — over-engineering; the linear from-state `if` guard is the shipped idiom across all prior transition Actions.

### L5 — Lapse/renew/cancel triggers are deferred seams; `RenewProfile` enforces the 30-day grace from `lapsed_at`
§ 4.2.1 drives `Active → Lapsed` from "the membership validity period passes without a successful renewal" (time) and `Lapsed → Active` from a "successful renewal payment" (Module E `MembershipFeePaid`); `Lapsed → Cancelled` at "30 days post-lapse" (time). No scheduler and no Module E exist, so all three **triggers** are documented seams — `LapseProfile`, `RenewProfile`, `CancelProfile` are the within-module **writers**, invoked directly now (operator/test). The **grace window is enforced in code**: `RenewProfile` guards `state === Lapsed` **and** `now ≤ lapsed_at + 30 days` (DEC-034) — past grace it throws `IllegalProfileTransition::cannotRenew` (the deferred scheduler will instead call `CancelProfile`). `RenewProfile` clears `lapsed_at` on success. The 30-day boundary is a fixed constant (DEC-034), not config. _Alternative rejected:_ fabricate a `MembershipFeePaid` event/listener to drive `RenewProfile` — Module E unbuilt; zero-invention (activation's L5 precedent).

### L6 — The Hold→`suspended` coupling is coverage-recompute, invoking the explicit Actions (ADR 2026-06-19)
The coupling (full reasoning + alternatives in `decisions/2026-06-19-hold-status-coupling.md`): a scope is `suspended` **iff** covered by ≥1 active Hold. `PlaceHold`, after recording the Hold in its transaction, drives every **covered scope currently in its suspendable from-state** to `suspended` by **invoking** the matching explicit Action — `scope_type = customer` → `SuspendCustomer` (cascade — L7); `account` → `SuspendAccount`; `profile` → `SuspendProfile`. `LiftHold` (operator) and `RecordKycVerified` (system `kyc`-lift), after lifting, **restore** every covered scope **currently `suspended`** by invoking `Reactivate*` — **iff** re-querying coverage shows **no other active Hold** still covers it (the read-API's `(scope) OR (its Customer)` shape, re-read under lock). The invocation pre-checks the from-state so it never throws on a legitimately-unaffected scope: a Hold on a **non-`active`** scope — the canonical case is the `kyc` Hold auto-placed on a **`pending`** Customer at onboarding — records the Hold and drives **no** transition. This preserves the shipped `ComplianceIndependenceTest` guarantee: the FSM verdict drives the **Hold**, the Hold (only from `active`) drives status — there is no direct verdict→status edge. _Alternative rejected:_ inline status-writes in `PlaceHold` (no explicit Action) — the guard tests pre-name the Actions and AC-K-BR-Customer-1 needs a manual path; the coupling invokes, never duplicates. _Alternative rejected:_ provenance tracking — worse under BR-K-Hold-1 multi-Hold; see ADR.

### L7 — Customer cascade: suspend `Active` Profiles; restore the uncovered ones; closure does NOT cascade
`SuspendCustomer` writes `status = suspended`, records `CustomerSuspended`, **then** (same transaction) re-reads the Customer's Profiles `->lockForUpdate()`, and for each Profile **in `Active`** transitions it to `Suspended` + records `ProfileSuspended` (§ 15.1 "Cascades to all the Customer's Profiles"; non-`Active` Profiles — `Applied`/`Approved`/`Lapsed`/terminal — are skipped: the FSM has only `Active → Suspended`, and the read-API already blocks them logically via the Customer Hold cascade). `ReactivateCustomer` writes `status = active`, records `CustomerReactivated`, then for each `Suspended` Profile restores it to `Active` + records `ProfileReactivated` **iff** that Profile is no longer covered by any active Hold (its own Profile-scope Hold — or, if the Customer still has another active Hold, none restore). `CloseCustomer` writes `status = closed`, records `CustomerClosed`, and **does not** cascade to Profiles — § 15.1 `CustomerClosed` names no cascade (unlike `CustomerSuspended`); zero-invention leaves Profile resolution at closure to the spec's silence. The cascade is a within-module cross-entity write (Customer + its Profiles, all Module K) — the `RetireProducer` (Producer+Club) precedent; all rows `lockForUpdate` in the one transaction. _Alternative rejected:_ cascade-cancel Profiles on close — the catalog/§10 do not specify it; inventing a destructive cascade.

### L8 — Account transitions are AUDIT-ONLY; `ReactivateAccount`, never `ActivateAccount`
§ 15 names **no** Account-family event (the Account is event-silent at creation too — the shipped *Account — Billing Container* requirement records no `AccountCreated`). So `SuspendAccount`/`ReactivateAccount`/`CloseAccount` write `status` and record **nothing** — the audit trail is the record (the `CancelProfile` audit-only shape — L2). The Account's only `→ active` edge is `suspended → active` = **`ReactivateAccount`**; there is **no** `pending → active` (the Account is born `active`), so **`ActivateAccount` is never created** and stays in the forbidden-Action guard list forever. _Alternative rejected:_ emit an `AccountSuspended`/`AccountClosed` event for symmetry with Customer — § 15 names none; vacuous invention (the guard test pins `AccountActivated` as never-existing — the same discipline forbids any Account event).

### L9 — Suspension is state-preserving (AC-K-FSM-2a); the Club-Credit/voucher freeze is a downstream seam
`SuspendProfile`/`SuspendCustomer` write **only** the status field(s) (and the cascade) — they do **not** cancel vouchers, orders, allocation reservations, or touch Club Credit (AC-K-FSM-2a: "active vouchers stay ACTIVE; reservations reserved; Club Credit frozen"). Those entities live in Module S/B/E and are unbuilt, so the "freeze" is **vacuous** here — there is nothing to freeze, and crucially nothing destructive must happen on suspend. The guarantee is expressed as a spec scenario asserting a suspend touches only `Profile.state`/`Customer.status` (+ the cascade) and emits only the status event(s); the Club-Credit-freeze-on-suspend / unfreeze-on-restore is a documented `club-credit` seam (the credit entity will read `state` when it exists). _Alternative rejected:_ pre-emptively model a "frozen" flag — no Club Credit entity to flag; the status IS the signal a future consumer reads.

### L10 — Guard-test realignment: narrow/invert four archived guard tests (names pre-fixed by them)
The load-bearing trap (grep the blast radius per task — `grep -rn '<symbol>' tests/` — don't trust this list):
- `tests/Feature/Modules/Parties/SupplyLifecycleChainTest.php` — the **exact-set** transition whitelist (the four sub-arrays asserted via `toEqualCanonicalizing`, ≈ L289-339) must **gain all twelve** new Actions (a new `demandSideStatusTransitions` sub-array, or extend `demandSideTransitions`). The event-non-existence `glob` loop (≈ L349-354) names `['AccountActivated','ProfileApproved','CustomerSegmentChanged']` — **keep all three** absent (this slice ships none of them; `AccountActivated` never exists — L8). The runtime "zero demand-side events" loop (≈ L264-272) flows create only — stays green.
- `tests/Feature/Modules/Parties/ComplianceIndependenceTest.php` — the **forbidden-Action** list (≈ L153-159) names `['SuspendCustomer','CloseCustomer','ActivateAccount','SuspendAccount','CloseAccount','LockOriginatingClub','SetOriginatingClub']`. Remove the **four now-shipped** (`SuspendCustomer`,`CloseCustomer`,`SuspendAccount`,`CloseAccount`); **keep** `ActivateAccount` (born active — L8) and `LockOriginatingClub`/`SetOriginatingClub` (activation's in-tx-side-effect — not ours). Its runtime independence assertions (KYC/sanctions verdict on a **`pending`** Customer leaves `status` `pending`; the Hold place/lift records no demand-side event) **stay green** because that test operates on **birth-state** entities — the from-state guard (L6) means a Hold on a `pending` Customer / `applied` Profile drives no transition; verify, don't assume.
- `tests/Feature/Modules/Parties/HoldRegistryTest.php` (≈ L149-160) and `HoldChainTest.php` (≈ L128-132) — the scope-guards asserting "after place/lift every scope is in its birth state". These are **inverted only in the coupling tasks (4.x)**: the **Account** assertion breaks first (Account is born `active`, so an account-scope Hold now suspends it); a Customer/Profile Hold on a `pending`/`applied` entity still drives nothing, but any test placing a Hold on an `active` Customer / `Active` Profile now suspends it. Each coupling task amends the specific assertions + the demand-side-event-absence loops in the file it touches. Through tasks 1-3 (before the coupling is wired) both files stay green unamended — `PlaceHold` still performs no transition until 4.x.

### L11 — Eight events, verbatim names, PII-free, root
`CustomerSuspended`/`CustomerReactivated`/`CustomerClosed` (`ENTITY_TYPE = 'Customer'`, payload `{customer_id, status}`); `ProfileSuspended`/`ProfileReactivated`/`ProfileExpired`/`ProfileRenewed`/`ProfileInactive` (`'Profile'`, payload `{profile_id, state}`). Each a `final` class with an untyped `const NAME` (verbatim § 15 name), `const ENTITY_TYPE`, and a static `payload(...): array` (enum→`->value`; ids + business values only; no name/email/phone/DOB) — the `CustomerActivated`/`ProfileActivated` shape. All are **root** events except the cascade children: a `ProfileSuspended`/`ProfileReactivated` recorded **inside** `SuspendCustomer`/`ReactivateCustomer` is a **causation child** of the `CustomerSuspended`/`CustomerReactivated` root (pass its `event_id` as `causationId` + `correlationId`) — the cascade is one causal chain; a directly-invoked `SuspendProfile`/`ReactivateProfile` records a root `ProfileSuspended`/`ProfileReactivated`. _Alternative rejected:_ all-root even in the cascade — the cascade has a genuine parent (the Customer transition in the same transaction), and the substrate supports causation; threading it keeps the audit chain honest (the only place this slice has a non-root event).

## Risks / Trade-offs

- **Inverting two shipped, protected-by-convention guard tests (L10).** `HoldRegistryTest`/`HoldChainTest` assert the *old* "Hold performs no status transition" contract; the coupling tasks invert exactly the assertions where a Hold now suspends an `active`-born scope, and must **keep** the birth-state cases (the from-state guard leaves them green). A reviewer reads precisely: the slice un-forbids twelve Actions, ships eight events, and keeps `ActivateAccount`/`Lock`/`Set`OriginatingClub`/WaitingList/segments forbidden/absent. The full Parties suite on SQLite + PG17 is the backstop. → Mitigation: each task greps its symbol's blast radius and narrows in the same task; task 5.1 audits all guard files green.
- **The Customer→Profile cascade restore is the subtlest path (L6/L7).** Restoring a Customer must restore only Profiles no longer covered by *any* active Hold — a Profile with its own active `admin` Hold stays `Suspended` when the Customer's Hold lifts. → Mitigation: re-query coverage per Profile (reuse the read-API cascade shape), under lock; an explicit spec scenario + test (Customer Hold + Profile Hold; lift the Customer Hold; assert the Profile stays `Suspended`).
- **Compliance-independence under the coupling (L6).** The coupling makes a Hold drive status; a careless reading sees "KYC now auto-suspends". → Mitigation: the from-state guard means onboarding KYC (a `pending` Customer) never suspends; post-activation re-screen suspends **via the Hold**, not the verdict — the independence (DEC-071) is "no direct verdict→status write", preserved. `ComplianceIndependenceTest` stays green because it uses birth-state entities.
- **Cancellation ships event-less (L2).** A future Module S expecting an AC-K-EVT-14 signal finds an audit record, not a domain event, until the offboarding/Module-S slice defines the shape. → Mitigation: the `cancellation_reason` column carries the domain data; the seam is documented in the spec delta, the `CancelProfile` docblock, `CONTEXT.md`, and the ADR.
- **Shipping suspension before the Club-Credit freeze (L9).** A suspended Profile's (future) Club Credit is not frozen because the entity does not exist. → Mitigation: vacuous today (no credit); the suspend writes only status, so the freeze lands cleanly when `club-credit` reads `state`. No live surface consumes it.
- **Nested transactions for the coupling (L6).** `PlaceHold` invoking `SuspendCustomer` opens a nested `DB::transaction` (a savepoint). → Mitigation: Laravel savepoints are atomic with the outer transaction; the recorder's open-transaction guard still holds; both Actions are Module-K, both lock their rows. Verified on PG17 (savepoint semantics differ from SQLite — the cross-engine close is the proof).

## Open Questions

None blocking. Two items surfaced for the reviewer (both resolved from spec/AC, recorded here): (1) **`CloseCustomer` does not cascade-cancel Profiles** — § 15.1 `CustomerClosed` names no cascade (contrast `CustomerSuspended`); the zero-invention reading ships closure as a Customer-only transition, leaving Profile resolution-at-closure to the spec's silence (a future change refines it if the spec gains a rule). (2) **Multi-Hold restore ordering** — § 10.1 says restore on "the triggering Hold" lift (singular); BR-K-Hold-1 admits many; the ADR reads it as "the last covering Hold" (restore iff no residual coverage) — the only reading consistent with both, encoded as the coverage-recompute (L6).
