## Context

The operator-console capability is built as read-projection Filament resources that **drive shipped domain Actions** (never an Eloquent write), each carrying the `actor_role: newco_ops` audit envelope, localized EN+IT. Four Parties consoles already ship on this kit: Customer, Club, Producer, ProducerAgreement. This slice adds the **demand-side membership** surface — Profile lifecycle + Account lifecycle — that the Customer slice (`2026-06-22`) explicitly deferred to "the account and profile slices."

Everything domain-level is settled by the frozen spec and verified on disk; this is a **pure console slice** — no new domain code, migration, dependency, or `party-registry` truth-spec change. The backend it drives:

- **Profile FSM** (`Parties\Enums\ProfileState`: Applied, WaitingList, Approved, Rejected, Active, Suspended, Lapsed, Cancelled, Inactive) with ten Actions: `CreateProfile`, `ApproveProfile`, `DeclineProfile`, `ActivateProfile`, `SuspendProfile`, `ReactivateProfile`, `LapseProfile`, `RenewProfile`, `CancelProfile`, `DeactivateProfile`.
- **Account FSM** (`Parties\Enums\AccountStatus`: Active, Suspended, Closed) with three Actions: `SuspendAccount`, `ReactivateAccount`, `CloseAccount` (no `ActivateAccount` — born active).
- **Event surface** (verified in the Action bodies): `CreateProfile` records one `ProfileCreated`; `ApproveProfile` records **no** Profile event except the conditional `OriginatingClubLocked` on the **first-ever** approval (idempotent NULL-gate on `Customer.originating_club_id`); `DeclineProfile` records **nothing** (the `state=rejected` write is the audit record). Status verbs record `ProfileActivated` / `ProfileSuspended` / `ProfileReactivated` / `ProfileExpired` (lapse) / `ProfileRenewed` / `ProfileInactive`. `CancelProfile` is **audit-only** (no `ProfileCancelled` event exists). All three Account verbs are **audit-only** (no Account events on disk).

## Goals / Non-Goals

**Goals:**
- A standalone `ProfileResource` whose list is the cross-Customer **membership approval queue** (pending `Applied` surfaced first), with `ViewProfile` carrying the full 9-verb lifecycle and a write-through create surface.
- Account suspend/reactivate/close added to `ViewCustomer` (the Account is 1:1 with the Customer; no independent queue).
- Every write routes through a domain Action with `actor_role: newco_ops`; no Eloquent write under `App\Modules\OperatorPanel`; EN+IT localized; full Profile-FSM PG17 closing-chain proven through the Filament pages.

**Non-Goals (the slice boundary — restated for the loop):**
- **No `Applied → WaitingList` transition** — no writer exists (awaits `parties-hero-package`). Render the state read-only if present; author no edge into it.
- **No capacity cap at activation** — `ActivateProfile` ships **uncapped** (Hero-Package capacity is a deferred Module-A seam). Drive activation as-is; **invent no cap**.
- **No producer-facing (TanStack) approve/decline UI** — that is the deferred Producer Portal. The operator exercises the one producer write via `newco_ops` (admin-parity).
- **No `MembershipFeePaid` listener / renewal trigger** — deferred Module-E seam; the console drives activation/renewal directly.
- **No new domain Action, migration, composer dep, or `openspec/specs/party-registry` edit.**

## Decisions

**D1 — Standalone `ProfileResource`; pages reuse the trait, not `OperatorConsoleViewRecord`.** `ProfileResource extends OperatorConsoleResource`. `ViewProfile extends \Filament\Resources\Pages\ViewRecord` and `use SurfacesDomainActions` — **not** `OperatorConsoleViewRecord` (that base is the catalog 5-verb submit/reject/activate/retire/reopen shape; the membership FSM is a different, larger verb set). `CreateProfile extends OperatorConsoleCreateRecord`. This mirrors the Customer slice exactly (ADR `2026-06-20-operator-console-non-catalog-lifecycle-trait-reuse`, ADR `2026-06-21` trait-level for non-catalog consoles). _Alternative rejected:_ subclassing `OperatorConsoleViewRecord` — its fixed 5-verb header set neither matches nor extends to the 9-verb Profile FSM.

**D2 — State enums rendered through the cast, never imported.** Profile `state` and Account `status` columns/badges render via `getStateUsing` + `instanceof BackedEnum` reading `->value`; **no** `Parties\Enums` import enters `OperatorPanel` (operand-enum carve-out, ADR `2026-06-21`). Do **not** call `lifecycleStateColumn()` — that renders the catalog `lifecycle_state`, a different column. _This keeps `ModuleBoundariesTest` green._

**D3 — The approval queue is the `ProfileResource` list; all 9 verbs are header actions on `ViewProfile`.** The list defaults to surfacing `Applied` (the operationally central queue) and offers a `state` filter for the other states. Approve/decline — and every other verb — live as **header actions on `ViewProfile`** (the row → view → act pattern of every sibling Resource), built with the kit's `lifecycleAction($verb, $successKey, $invoke)`, each with the uniform `(Model $record, string $notes)` closure signature. _Alternative rejected:_ per-row approve/decline table actions on the queue — ergonomic for triage but splits the verb surface across two surfaces and departs from the kit's header-action shape; revisit post-launch if triage volume demands it (low at launch — single aggregator, "minimal approve/decline surface", PRD §3.1).

**D4 — Visibility gating = exact complement of the domain from-state guard.** Each `lifecycleAction(...)->visible(fn () => …)` predicate is the exact complement of its Action's from-state precondition, so the rejection branch is structurally unreachable through the UI (the established defense-in-depth, lesson `2026-06-22`): approve/decline visible iff `Applied`; activate iff `Approved`; suspend/lapse/deactivate iff `Active`; reactivate iff `Suspended`; renew iff `Lapsed`; cancel iff `Active|Lapsed`. **Test consequence (lesson `2026-06-22`):** a hidden action cannot be driven through any Filament test helper — assert the domain rejection **directly** (`expect(fn () => app(X::class)->handle($id))->toThrow(IllegalProfileTransition::class)`) **plus** `assertActionHidden(...)`, never an `action_failed` notification the widget can never send.

**D5 — `renew` is the one verb whose reject branch IS UI-reachable.** `RenewProfile`'s gate is compound — `Lapsed` **and** within the 30-day grace of `lapsed_at` (DEC-034). The visibility predicate can only check `state == Lapsed`; the grace sub-gate is domain-internal. So a **Lapsed-but-past-grace** renew is reachable through the surface and the domain rejects it → `surfaceLifecycleOutcome` sends the `action_failed` notification. This branch IS testable through the page (drive `renew` on a Profile lapsed > 30 days; assert the danger notification + state unchanged). All other rejections stay domain-direct (D4).

**D6 — Profile create = Customer + Club selects → `CreateProfile`, born `Applied`.** `CreateProfile.createViaAction()` routes to `app(CreateProfile::class)->handle($customerId, $clubId)`. The form is two within-module selects (Customer, Club — both Parties models, no cross-module import); `tier` is omitted (single-tier at launch, DEC-062 — the Profile carries the Club's single tier conceptually, not an operator input), `role`/`invitedBy` omitted (the invitation flow is a deferred surface). Born `Applied`, no `state` field exposed, records one `ProfileCreated`. A duplicate non-terminal Profile per Customer–Club pair → `DuplicateProfileForClub`, surfaced on the create-rejection field. **Implementer must verify `CreateProfile`'s own gate** (read the Action) before wiring: surface exactly the domain's rejection(s); if Club-`sunset` blocks new membership at the domain (AC-K-BR-Club-3), let that rejection surface — do **not** invent a console-side Club filter the domain doesn't enforce.

**D7 — Account verbs live on `ViewCustomer`; MODIFY the two Customer requirements.** The 3 Account verbs are form-less `lifecycleAction`s on `ViewCustomer.getHeaderActions()`, invoking `app(SuspendAccount::class)->handle($customer->account->id)` etc., visibility-gated by `account->status` (suspend iff `active`, reactivate iff `suspended`, close iff `active|suspended`). Account labels/notifications go under `operator_console.customer.*` (ViewCustomer's `i18nKey` is `customer`). This requires **MODIFY**ing the shipped "_status lifecycle_" requirement (its negative "no Account-lifecycle … verb" is now false) and the "_…read-only_" requirement (drop the Account deferral). The Profile-on-ViewCustomer negative stays true (Profile writes live on the new `ProfileResource`, not here) but gains a cross-reference. _This is the extensional-spec discipline the KYC/sanctions slice used._

**D8 — No new Action ⇒ no `SupplyLifecycleChainTest` change.** This slice adds **zero** new `Parties\Actions\*` classes (it drives existing ones), so the exhaustive non-`Create` Action whitelist in `SupplyLifecycleChainTest` needs no update (contrast the club-credit writers, lesson `2026-06-23`). Confirm by grep before the final gate.

**D9 — `actor_role: newco_ops` from `ActorContext`.** Every operator write records the envelope on its domain event (or, for audit-only transitions — decline, cancel, all Account verbs — on the audit record the Action drives) via the existing `ActorContext` operator guard; the console constructs no envelope itself.

## Risks / Trade-offs

- **[Hidden-action test trap]** Asserting an `action_failed` notification for a from-state-gated verb → unreachable, false test (lesson `2026-06-22`). → **Mitigation:** D4 — domain-direct throw + `assertActionHidden`; only `renew`-past-grace (D5) asserts the notification, and only through the page.
- **[Pint `{@see}` boundary breach]** A `{@see \App\Modules\Parties\Exceptions\IllegalProfileTransition}` docblock makes Pint auto-add a `use` that violates the `{Models, Actions}`-only import boundary and reds `ModuleBoundariesTest` (lesson `2026-06-20`). → **Mitigation:** reference `IllegalProfileTransition` / `IllegalAccountTransition` / `DuplicateProfileForClub` only as **prose/backticks**, never as a `{@see FQCN}`/`@throws FQCN` type-position ref. Catch base rejections by `\RuntimeException`.
- **[Pest + PHPStan-max]** Use `Pest\Laravel\*` global `get/post/actingAs` (never `$this->get()`); one `->not->…` per `expect()` (re-anchor with `->and()`); `expect(in_array(...,true))->toBeTrue()` over `toContain` chains; scalar dataset params, never `...$spread` of a bare-`array` param into a string-keyed literal (lessons `2026-06-13` / `06-18` / `06-20`). → **Mitigation:** run `php -d memory_limit=-1 vendor/bin/phpstan analyse`, not just a green suite, before each commit.
- **[i18n suite-wide scanner load order]** `ProfileConsoleI18nTest` reuses the suite-wide `scanOperatorConsoleHardcodedSinks` helper declared in the Catalog folder behind a `function_exists` guard — verifying by a single **file path** (or a Parties-only **directory** path) leaves it undeclared → false red (lesson `2026-06-20` + corollary). → **Mitigation:** verify via `--filter=ProfileConsoleI18nTest` or the full suite; for the Parties-only PG17 run, **append** `tests/Feature/Modules/OperatorPanel/Catalog/ProductMasterConsoleI18nTest.php`.
- **[Account relation null]** `$customer->account` is non-null by 1:1 co-provisioning, but if any nullable relation read arises, use the local + null-check idiom (`$a = $record->relation; $a === null ? … : $a->attr`) — never `?->` (Larastan `nullsafe.neverNull`) nor an FK-column guard (lesson `2026-06-21`).
- **[Full suite OOM]** `php artisan test` OOMs at 128 MB on this box → run `php -d memory_limit=-1 vendor/bin/pest` (lesson `2026-06-20`).

## Migration Plan

Additive only — a new Resource + 3 pages, 3 header actions appended to an existing page, and i18n keys. No migration, no data backfill, no rollback concern beyond reverting the new/modified files. PG17 truth is proven at close via the `postgres:17` Docker recipe (GUIDE §2.7), as the prior console slices were.
