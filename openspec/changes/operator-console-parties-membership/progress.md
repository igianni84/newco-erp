    - **OVERLAPPING-from-state corollary (group 5; reused by group 6).** When verbs SHARE a from-state (group 5: `active` surfaces lapse+cancel+deactivate, `lapsed` surfaces renew+cancel), the single-verb maps don't fit — use SETS: the visibility test maps each state to an `array $visibleVerbs` and loops `['lapse','renew','cancel','deactivate']` asserting `assertActionVisible`/`assertActionHidden` by `in_array($verb, $visibleVerbs, true)`; the reject-floor maps each verb to a `list<ProfileState>` of its legal-froms (cancel = `[Active, Lapsed]`) and `continue`s when `in_array($from, $legalFroms, true)`. `array $visibleVerbs` as a positional dataset param is PHPStan-fine (NOT the string-keyed `...$spread` trap).
    - **RELATED-ENTITY verb corollary (group 6 — ViewCustomer Account verbs).** When the lifecycle verbs act on an entity REACHED THROUGH a relation off the page record (the Customer's co-provisioned 1:1 `Account`), the wiring shifts one hop but the shape is identical: the invoke routes by the NESTED id `app(<Action>::class)->handle((int) $this->recordOf(Customer::class, $record)->account?->id)` (nullsafe `?->id` + `(int)` cast — the `holdScopeId` precedent), and the visibility helper reads the NESTED cast value `private function accountStatusIs(string $status): bool => $this->recordOf(Customer::class, $this->getRecord())->account?->status->value === $status` (two-hop nullsafe `?->status->value` — PHPStan-max-clean, the `?->` short-circuits the whole chain; a NULL relation hides every verb). NO `Parties\Enums` import (status read via `->value`, design D2); the related Model is referenced in **prose/backticks**, never `{@see Account}` with a leading qualifier (a bare `{@see Account}` is import-safe but prose is the `holdScopeId` convention). The OVERLAPPING-from-state TEST corollary GENERALIZES verbatim across resources (ViewProfile group 5 → ViewCustomer group 6): SET-based visibility sweep + `list<Enum>` reject-floor, just keyed on `AccountStatus` and seeding the related entity (`Account::factory()->create(['customer_id' => $c->id, 'status' => $from])` — the Customer factory co-provisions NO Account). AUDIT-ONLY verbs (no event of ANY kind on disk) → assert `DomainEvent::count() === 0` (clean, factories record nothing) + ORTHOGONALITY (the Account FSM moves only `Account.status`, never the Customer or its Profiles — AC-K-FSM-9). The absent inverse verb (no `activateAccount`, born active) = `assertActionDoesNotExist` (the `waiveKyc` precedent), not merely hidden.
    - **`renew` past-grace = the SOLE UI-reachable reject (design D5).** `RenewProfile`'s gate is compound (`lapsed` AND within 30d of `lapsed_at`); the visibility predicate can only see `state == lapsed`, so a past-grace renew is VISIBLE → driven through the page → domain rejects → `action_failed` danger. Test it like the catalog `action_failed` shape (`callAction('renew')->assertNotified(…action_failed)` + state unchanged + event count 0), but make it past-grace by **anchoring `lapsed_at` in the past** (`CarbonImmutable::now()->subDays(31)`) — NOT `$this->travelTo()`, which fails PHPStan-max inside a Pest `it()` closure (`$this` binds to `Pest\PendingCalls\TestCall`, not the TestCase → `method.notFound`). Anchoring the domain time-field is the equivalent past-grace condition with no test-now mutation/cleanup. `cancel` is AUDIT-ONLY (no event) yet STILL throws `IllegalProfileTransition` out-of-band, so it belongs in the reject-floor `toThrow` set; its terminal soft-delete (AC-K-FSM-13) is asserted by `Profile::query()->whereKey($id)->exists()` (Profile has no `SoftDeletes` scope — the row simply persists with `state = cancelled`).
- **Closing chain = the `CustomerConsoleChainTest` shape, COMPOSED across two FSMs (group 8 — `ProfileMembershipChainTest`).** ONE `it`, `DatabaseMigrations` (each console verb drives a real action with its OWN `DB::transaction` → the in-transaction event append commits; factories seed event-free). Drive the WHOLE happy-path FSM through the PAGES (re-mount `Livewire::test(Page::class, ['record'=>$id])` per verb so the fresh record read re-evaluates the visibility gate), asserting the state at every leg. **Assert the ORDERED event sequence** via `DomainEvent::query()->orderBy('id')->pluck('name')->all()` `->toEqual([...plain strings...])` (the `id` append order is chronological across the per-verb transactions; plain-string names are the chain idiom, each verified == its `*::NAME`) — pin it after the first FSM to localise a regression to that walk. A **second, orthogonal audit-only FSM** (Account suspend→reactivate→close on the SAME Customer) is proven by re-asserting the SAME sequence after it → byte-identical = +0 events AND no cascade (the strongest orthogonality proof, AC-K-FSM-9). Envelope set-wide (`module`/`actor_role`/non-null `actor_id`) + representative `actor_id->toEqual($operator->id)` (loose — PG bigint→numeric-string) on writes spanning BOTH surfaces (create page + a view verb). Pick the **evented terminal** (`deactivate`→ProfileInactive) for the linear walk so the sequence is non-vacuous; the audit-only sibling (`cancel`) rides its per-task test. **PG17 ritual** (a self-contained chain test with NO shared top-level helper is safe via `--filter` alone): throwaway `postgres:17` Docker, wait `pg_isready`, `DB_CONNECTION=pgsql DB_HOST=127.0.0.1 DB_PORT=55432 … php -d memory_limit=-1 vendor/bin/pest --filter=<Test>`, `docker rm -f`.
