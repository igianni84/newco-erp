# Progress — operator-console-catalog-master

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Running the FULL test suite:** `php artisan test` (and `composer test`) blow the box's default 128M PHP memory limit during `laravel/pao` result aggregation over the ~950-test suite — the `-d memory_limit` flag does NOT reach the pao worker. Run the full suite as **`php -d memory_limit=-1 vendor/bin/pest`** and PHPStan as **`php -d memory_limit=-1 vendor/bin/phpstan analyse`**. A *filtered* run (`php artisan test --filter=Foo`) stays under 128M and is fine for step 2.
- **Asserting Filament panel discovery (no resource needed):** `Filament::getPanel('admin')` exposes public getters `getResourceNamespaces()/getResourceDirectories()`, `getPageNamespaces()/getPageDirectories()`, `getWidgetNamespaces()/getWidgetDirectories()` — each returns the verbatim `for:` / `in:` args passed to `discoverResources/Pages/Widgets`. Use these to assert the discovery repoint directly (`->toContain('App\Modules\OperatorPanel\Filament\Resources')`) without scaffolding a real resource. Filament's `discoverComponents` silently no-ops on a missing directory (`HasComponents.php:496`), so an empty `.gitkeep` skeleton is a valid discovery target.
- **Operator-panel test auth:** authenticate with `actingAs(Operator::factory()->create(), 'operator')` (the `operator` session guard); `Operator` = `App\Modules\OperatorPanel\Models\Operator`. `get('/admin')->assertOk()` proves the panel boots + `canAccessPanel`.
- **Repo-local PHPStan custom rule (type-aware arch test):** the stack ships phpstan as a phar BUT `PHPStan\Testing\RuleTestCase` is autoloadable (phpstan's `bootstrap.php` is composer-`files`-loaded → `class_exists` true). Pattern: (1) rule class under `tests/PHPStan/Rules/` (namespace `Tests\…`, already dev-autoloaded — **no composer.json change**, so the "empty composer diff" rule holds); implement `Rule<TNode>` with `getNodeType()` + `processNode()`; resolve receiver types via `$scope->getType($node->var)` and `(new ObjectType(Model::class))->isSuperTypeOf($t)->yes()`; build errors with `RuleErrorBuilder::message()->identifier('x.y')->line($node->getStartLine())->build()`. (2) Register in `phpstan.neon` under top-level `services:` with `tags: [phpstan.rules.rule]` — this wires it into `type_check`. (3) **Scope a rule to a directory via a constructor `scopedPathNeedles` arg** (checked against `$scope->getFile()`), NOT a hardcoded path: production registration passes the real needle, the RuleTestCase passes the fixture's needle — same rule, both testable. (4) Test with a class extending `RuleTestCase` under `tests/Architecture/` (Pest binds the Laravel `TestCase` only to `Feature/`, so a plain PHPUnit class elsewhere runs clean); `gatherAnalyserErrors([$fixture])` is public and returns `Error[]` with `getLine()/getIdentifier()`. (5) **`excludePaths` the fixture dir** in phpstan.neon so its deliberate violations don't red the main run. (6) Verify API surface in `vendor/` by reflection before writing — `$class_exists`/`ReflectionClass` dump (lessons.md). Two-call shapes cover the writes: `MethodCall` (`$model->save()`) on a Model-typed `->var`; `StaticCall` (`Model::create()`) on a Name resolved via `$scope->resolveName()`.
- **Marker-driven fixture line assertions:** when a test asserts which fixture lines a rule flags, derive the expected lines from a `// flag`-style trailing marker scanned out of the fixture source (`file()` + `str_contains`), not hardcoded numbers — Pint's `fully_qualified_strict_types` can silently add a `use` import (e.g. from a `{@see FQCN}` docblock) and shift every line. Keep the literal marker token OUT of the fixture's own docblock/prose or the scan self-matches.
- **Module-boundary carve-out (`tests/Architecture/ModuleBoundariesTest.php`):** the privacy law forbids `App\Modules\{M}` importing any other module except the namespaces fed to `->ignoring()`, now built by the top-level helper `moduleBoundaryAllowedImports(Module $source)` (single source of truth for the law + its guard test). `->ignoring()` matches by namespace PREFIX (`str_starts_with`), so listing `…\Models` permits `…\Models\Foo`. To permit a new cross-module import for ONE source module, add the prefix inside that helper under a `$source === Module::X` guard — NEVER unconditionally (that widens it to all nine). Pin the scoping with a logic-level guard test over the helper's output (positive: the source's list contains the prefix; negative: a lateral module's list does NOT and never the whole-module namespace) — the permanent analog of "plant a violation". Live red-proof is transient: drop a temp class importing the target under `app/Modules/{M}/…` (NOT under a discovered Filament dir), run the boundary test, observe red→green, delete. **Assert membership via `in_array($x, $list, true)` + `->toBeTrue()/->toBeFalse()`, NOT a chained `expect($list)->toContain()->toContain()->not->toContain()` — multi-method Pest expectation chains degrade to `mixed`/nullable under the larastan Pest extension and red `type_check`; `toContain` is only phpstan-clean as a single modifier (e.g. `->and($s)->toContain('x')`).**
- **Console cross-module surface = exactly {Models, Actions}:** the operator console reaches each operated module through only two sub-namespaces — `Models\*` (read-bind Filament resources, read-only) and `Actions\*` (`app(<Action>)->handle()` write-through). To keep it that tight for the lifecycle tasks (3.1–5.2): **catch domain rejections via base types** (`\Throwable` / `\DomainException` / `Illuminate\Validation\ValidationException`, then `$e->getMessage()`), do NOT `use App\Modules\Catalog\Exceptions\…`; and **render enum casts via their instance** (`$record->lifecycle_state->…`), do NOT `use App\Modules\Catalog\Enums\…`. Following this, no later console task needs to re-amend the boundary carve-out. If a task genuinely must reference a third sub-namespace, extend the helper (guarded) + the guard test together.
