# Progress — parties-producer-lifecycle

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Run the full suite with `php -d memory_limit=512M vendor/bin/pest`** — NOT `php artisan test`. The default 128M OOMs in the Pest arch plugin (`nikic/php-parser` scans every module class via `ModuleBoundariesTest`); the crash surfaces as a misleading `NoTestCaseObjectOnCallStackException` / `stream_filter_remove` shutdown trace, but the head of the output reveals `Allowed memory size … exhausted`. The 512M runner is green (441/441). PHPStan + Pint run fine at default memory.
- **Unit tests that call `__()` (or any container service) must `uses(Tests\TestCase::class)`** at the top of the file. `tests/Pest.php` binds `TestCase` (+ optional `RefreshDatabase`) to `Feature` ONLY; files under `tests/Unit/**` default to plain PHPUnit and DON'T boot Laravel, so the translator is absent. For a translator-only test add `uses(TestCase::class)` WITHOUT `RefreshDatabase` (no DB needed → fast, and honours "no DB" tasks).
- **PHPStan max rejects Pest closure-dataset typing.** Threading a `Closure $factory` + a `class-string` (declared `string`) through `->with([...])` loses type info: `$factory()` is `mixed` (so `->getMessage()` fails) and `toBeInstanceOf(string)` wants `class-string`. For factory/exception assertions prefer **explicit per-case `it()` blocks** where the concrete return type and `::class` are visible — type-clean and reads like the repo's `EnumsTest`. (Don't eager-build exceptions inside a plain-array dataset either: datasets evaluate at collection time, before the app boots, so `__()` won't resolve.)
- **Transition-guard exception shape** (this change): `class Illegal{Entity}Transition extends RuntimeException` (non-`final`, matching the parties-core creation guards) with named static factories `::cannotX({Entity}Status $from): self` that return `new self((string) __('parties.<group>.<key>', ['state' => $from->value]))`. The `:state` placeholder is the enum **backing value** (e.g. `'active'`), a business token — NOT PII — so it is interpolated (cf. `DuplicateCustomerEmail`, which omits the PII email). Lang lives in grouped keys in `lang/en/parties.php`; ADD keys to existing groups (`club`, `producer_agreement`) rather than replacing them, and a new `producer` group.
- **Within-module inverse `hasMany` shape** (first one in the repo — `Producer::clubs()`): both entities share the `App\Modules\Parties\Models` namespace, so the related model needs **no `use` import** (`Club::class` resolves directly); type it `@return HasMany<Club, $this>` on the method + `@property-read Collection<int, Club> $clubs` (import `Illuminate\Database\Eloquent\Collection`) on the class docblock — PHPStan max is then clean and the dynamic property is typed. It's boundary-clean (the `ModuleBoundariesTest` ban is on CROSS-module relations only); the full suite covers the arch test, so no amendment. Mirror the sibling `belongsTo` docblock style and keep the model persistence-only (the relation is a read, no write surface).
- **Pint `fully_qualified_strict_types` rewrites `{@see \Fully\Qualified\Class}` in a docblock into a real `use` import + short `{@see Class}`.** If that class does NOT exist yet (a forward-reference to a later task's Action, e.g. `RetireProducer` from task 3.2), Pint silently adds a `use` for a non-existent class — a latent PHPStan/autoload trap. **Reference a not-yet-built class in PROSE only** (plain backticked name, no `{@see}`), and `{@see}` only classes/methods that already exist. Re-run `vendor/bin/pint <file>` after a docblock edit to confirm it's stable (no re-added import). _Note: an UNQUALIFIED `{@see ClassInOtherNamespace}` (no leading `\`, not imported) is left untouched by Pint — that's how `ClubCreated` cites `{@see CreateClub}` without importing it; the trap is specifically the FULLY-QUALIFIED form._
- **Transition Action template** (established by `SunsetClub` 2.1; every 2.x–4.x transition follows it). **Signature rule (refined by `CloseClub` 2.2):** the threading params `?int $causationId = null, ?string $correlationId = null` appear ONLY on an Action that can be invoked as a cascade/derived STEP — in this change that is *only* `SunsetClub` (the sole cascade target). A transition that is only ever standalone uses the simpler `handle(int $id): Model` and omits both `record()` linkage args (→ root event), EVEN IF it is itself a cascade/supersession SOURCE: `RetireProducer` (3.2) and `ActivateProducerAgreement` (4.1) GENERATE linkage from `record()`'s return value to thread their derived event, they don't RECEIVE it, so they too are `handle(int $id): Model`. So `CloseClub`/`ActivateProducer`/`RetireProducer`/`ActivateProducerAgreement`/`TerminateProducerAgreement` = simple signature; `SunsetClub` = the only one with threading params. The body is `handle(...): Model` wrapping ONE `DB::transaction(fn () => …)` that (1) re-reads `Model::query()->whereKey($id)->lockForUpdate()->firstOrFail()` (real row-lock on PG, no-op on SQLite — the from-state assert carries correctness), (2) `if ($m->status !== Expected) throw Illegal{Entity}Transition::cannotX($m->status)`, (3) `$m->update(['status' => New])` (NEVER bump `version` — design L3; the immutable event is the audit record), (4) `$recorder->record(name: Evt::NAME, module: Module::Parties->value, actorRole: $actor->role(), actorId: $actor->actorId(), entityType: Evt::ENTITY_TYPE, entityId: (string) $m->id, payload: Evt::payload($m), correlationId: $correlationId, causationId: $causationId)`. The event payload reflects the POST-transition `status` (record AFTER the `update`). A standalone caller passes null/null → the recorder makes it a ROOT event (`causation_id` null, `correlation_id` defaults to its own `event_id`); a cascade/derived caller threads the root's `->id` (int!) + `->correlation_id` (string!). The transition Action is the SOLE `status` writer and the SOLE writer of its event (single-writer rule — design L1/L6).
- **`domain_events.causation_id` is a self-referencing FK to `domain_events.id`** (migration `2026_06_12_000001`, line 55) — so a test that threads a `causationId` MUST pass a REAL prior event's `id`, not an arbitrary int, or PG rejects the insert (SQLite with FKs on does too). Pattern for an isolated threading test: record a real root event first (a standalone transition), `sole()` it, then pass `causationId: $root->id, correlationId: $root->correlation_id` to the second call and assert the derived event carries both. Assert the standalone-root invariant separately: `causation_id` null AND `correlation_id === event_id`.
- **Cascade-source orchestration** (first instance — `RetireProducer` → `SunsetClub`, design L5/L6). A transition Action that is a cascade SOURCE: (1) **constructor-injects the target Action** (`private readonly SunsetClub $sunsetClub`) — NOT `app(SunsetClub::class)` — so the cascade is DI-idiomatic and testable (mirrors how the sibling actions inject the recorder/actor); (2) records the **root** event first and CAPTURES it (`$retired = $this->recorder->record(...)` with no linkage args → root), then walks the within-module relation and calls the target threading `causationId: $retired->id` (int!) + `correlationId: $retired->correlation_id` (string!); (3) runs the whole thing in ONE `DB::transaction` — the target Action's own `DB::transaction` **nests as a savepoint**, so the source + every cascade step commit or roll back together (all-or-nothing). Filter the relation by the enum column with the **backing `->value`** (`->where('status', ClubStatus::Active->value)`) — there is NO codebase precedent for passing a bare enum to `where()`, so `->value` is the engine-safe, unambiguous binding. The target's own `lockForUpdate` + from-state guard carries per-row correctness (no need to lock the enumeration); filter to the live from-state so already-transitioned rows are skipped (idempotent). **No Mockery in the suite** — prove cascade atomicity exactly as the siblings prove "same transaction / rollback": the happy path asserts ALL expected events present + linked (= recorded together), the guard-reject asserts ZERO events + untouched rows (= rolls back together).
- **Inline derived-event SOURCE** (established by `ActivateProducerAgreement` 4.1, design L5/L7 — the supersession variant of the cascade source). When the derived transition is on the **same entity type** and has **no dedicated Action** (supersession is not a direct operator call — it is driven only by activating a replacement), the source records the derived event INLINE via the injected recorder rather than injecting a target Action: capture the root (`$activated = $this->recorder->record(<Activated>, …)` with no linkage), do the prior row's `update(['status' => Superseded])` BEFORE the self `update(['status' => Active])` (so the scope is never momentarily double-active) and BEFORE recording (so each payload reflects post-transition status), then `if ($prior !== null)` record the `<Superseded>` threading `causationId: $activated->id` + `correlationId: $activated->correlation_id`. The Action still takes the **simple standalone signature** `handle(int $id): Model` (it GENERATES linkage, doesn't receive it). **Scope-aware NULL-safe lookup (the live PG trap):** a partition key that is a NULLABLE column (`club_id`) needs a branch — `whereNull('col')` when the activating row's value is null, `where('col', $v)` otherwise — because `where('col', null)` binds `col = NULL` (never true) and would silently match nothing; SQLite is green either way, so the bug is invisible until PG17. Lock the prior with `->lockForUpdate()` too (serialize concurrent same-scope activations). Prove pairing both directions: the activation payload's `supersedes` = prior id (null when none), the supersession payload's `superseded_by` = the new id, and the supersession's `causation_id` = the activation event id. Two scope-isolation tests pin BOTH NULL-trap directions on PG17 (a 2nd Producer-wide activation supersedes the 1st via `whereNull`; a Producer-wide + a Club-narrowed coexist as both active and a new activation in either scope supersedes ONLY the same-scope prior).
- **Transition-event payload key naming** (4.1): the supply-side transition events use the **fully-qualified** id key `producer_agreement_id` (design L4, matching `producer_id`/`club_id`), which DIVERGES from the parties-core creation event `ProducerAgreementCreated`'s shorter `agreement_id`. This is deliberate and traceable (design L4 + tasks 4.1 spell `producer_agreement_id` verbatim); the creation event is an archived-change choice left unchanged. If a future change wants one family-wide key, normalize there — do not silently rename the creation event here. Pin the exact key set with `expect(array_keys($event->payload))->toEqualCanonicalizing([...])` so the PII-free contract can't widen.
