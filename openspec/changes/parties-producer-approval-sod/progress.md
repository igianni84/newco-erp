## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Full suite = `php -d memory_limit=-1 vendor/bin/pest`, NOT `php artisan test`.** The `artisan test` path (laravel/pao driver) spawns the runner in a subprocess capped at 128 MB and OOMs during result collection on the full ~1956-test suite (fatal in `DemoSeeder.php`, misleading line number — it dies while autoloading a PHPUnit class, not in a test). `-d memory_limit=1G` on `artisan` does NOT propagate to the subprocess. Run Pest directly with uncapped memory; per-`--filter` runs are fine under `artisan test`.
- **Parties copy is per-key EN-fallback (DEC-127).** `lang/en/parties.php` is the authored baseline + final fallback; per-locale files (e.g. `lang/it/parties.php`) may cover a *subset* and fall back to EN per key (`config('app.fallback_locale') = 'en'`). A new IT file need only carry the keys it translates. **Invariant:** every authored `it` key MUST have an `en` counterpart (`array_diff(Arr::dot(it), Arr::dot(en)) === []`) — assert it or a typo'd IT key dangles without a fallback.
- **i18n copy test idiom (PHPStan-max-clean).** `trans($key, $replace, $locale)` is typed `array|string|null`; do NOT chain `->toBeString()->not->...` off it (PHPStan can't resolve `->not` on the union). Use `expect(trans(...))->not->toBe($key)->and(trans(...))->toContain(...)`, and for authored-IT proof `expect(Lang::has($key,'it',false))->toBeTrue()->and(trans($key,[],'it'))->not->toBe(trans($key,[],'en'))` (the ProductMasterConsoleI18nTest pattern). Unit copy tests must `uses(TestCase::class)` explicitly (Pest auto-binds TestCase only in `Feature`).
- **Copy keys mirror Catalog's `approval` group, minus the reviewer leg.** Catalog uses `requires_operator` / `self_approval_creator` / `self_approval_reviewer` / `insufficient_separation`; Parties (linear FSM, no reviewer) uses only `requires_operator_principal` / `creator_may_not_approve`. Keep `:entity` parameterized (the guard passes `'Producer'`), name only the violated rule, never PII.
- **The SoD guard resolves via `app(ProducerApprovalGovernance::class)` — no container binding.** It's a concrete class whose ctor injects the `ActorContext` **singleton**, so `app()`/constructor-injection just works (task 2.1 injects it into `ActivateProducer`). Test it directly by seeding the creator lineage through the **real `CreateProducer`** Action inside `ActorContext::runAs(ActorRole::NewcoOps, $creatorId, fn () => …)` (the RM-07 pattern; `runAs` restores the prior context on exit, so the system-actor case sees **no lingering auth** → resolves `(System, null)`), then approving inside a second `runAs`. A `Producer::factory()->create()` records no `ProducerCreated` → **null creator = vacuous** distinctness (fourth case). Assert the two reject legs by the resolved message (`(string) __('parties.approval.<key>', ['entity' => 'Producer'])`) to prove the RIGHT factory fired; pass legs via `->not->toThrow(SeparationOfDutiesViolation::class)`.
- **Cross-module `{@see}` in a module class docblock is a Pint→boundary landmine (lessons.md 2026-07-06).** A `{@see \App\Modules\Catalog\…}` FQCN in the guard/its ctor docblock makes Pint's `fully_qualified_strict_types` auto-add a `use` → Parties→Catalog import → `ModuleBoundariesTest` red (invariant 10). Name Catalog (and any other module) in **backticked prose**; keep only code-used `use`s. The arch test is ABSENT from a `--filter` run, so re-run Pint + `grep -n Catalog` the file before the full suite. **Standing trap for tasks 3.x** — `ViewProducer` docblock will describe the mirrored Catalog behavior. (In `ActivateProducer` the SoD paragraph avoided naming Catalog at all — it references the Parties `{@see ProducerApprovalGovernance}` and names the exception in backticked prose.)
- **The SoD floor's migration blast radius = EVERY `ActivateProducer` call site, not the task's named files (lessons.md 2026-07-06 floor-blast-radius).** Wiring the operator-principal floor into the shared Action broke SIX test files, only ONE named in tasks 2.1/2.2. Two migration shapes: **(1) real-lineage** — a Producer built by the real `CreateProducer` (as operator A) MUST be activated by a **distinct** operator B (`producerCreatedByOperator($a)` + `activateProducerAsOperator($id, $b)` in `ProducerLifecycleTest`; distinct `actingAs` for `ProducerConsoleChainTest`; else creator===approver self-approval, the F2 case). **(2) vacuous** — a `factory()->create()` Producer has a null creator, so a SINGLE `actingAs`/`runAs(operator)` clears the floor (`CatalogLifecycleChainTest`, the `ComplianceChainTest` matrix). And **SET-WIDE `->every(actor_role === System)` chain assertions must split**: only `ProducerActivated` is operator-gated (→ `NewcoOps`); creation/KYC-Holds/sanctions/agreements/clubs/retire stay `System`. Pest helpers are one shared global namespace but do NOT cross single-file runs — define per-file (unique name) or inline the `runAs`, never call another test file's helper. Applies again to `ActivateProfile`/`ApproveProfile` if a floor lands there.
