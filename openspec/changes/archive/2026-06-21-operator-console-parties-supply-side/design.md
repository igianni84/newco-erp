# Design — operator-console-parties-supply-side

## Context

This change completes the **Parties supply-side console trio** begun by `operator-console-parties-producer` (merged + archived 2026-06-20): the **Club** and **ProducerAgreement** operator consoles. Both entities have fully-shipped backends — domain Actions, FSMs, events, exceptions, models, enums, factories — and **no operator surface**. This is a pure operator (Filament) layer over those shipped Actions: no new domain code, no migration, no composer dep.

The implementation recipe is the predecessor's **non-catalog trait-reuse** pattern, consolidated verbatim in `openspec/changes/archive/2026-06-20-operator-console-parties-producer/progress.md` → "## Codebase Patterns" (which self-identifies as "the trait-reuse TEMPLATE the rest of Parties reuses", "the Parties i18n-guard template", and "the Parties console-slice closing proof template"). **Read that block first each iteration.** The governing ADR is `decisions/2026-06-20-operator-console-non-catalog-lifecycle-trait-reuse.md`. The read/write boundary is `decisions/2026-06-19-operator-console-read-binding-write-through-actions.md`.

The two FSM deltas vs Producer: **Club** is `active → sunset → closed` (born `active`, **no** activate verb); **ProducerAgreement** is `draft → active → superseded | terminated` (born `draft`; `superseded` is a *side-effect of activation*, never a direct verb). Neither has a KYC FSM.

## Goals

- A read-only `ClubResource` + a create surface + a `ViewClub` lifecycle page (sunset/close), EN+IT localized, fully tested incl. the PG17 closing chain.
- A read-only `ProducerAgreementResource` + a create surface + a `ViewProducerAgreement` lifecycle page (activate/terminate, with in-domain supersession), EN+IT localized, fully tested incl. the PG17 closing chain.
- The OperatorPanel module-boundary carve-out widened to admit **operand enums** (ADR `2026-06-21-operator-console-operand-enum-carveout.md`), so the Club create can construct `ClubRegistrationFlowType`.
- Every quality guard stays green: `NoEloquentWriteInOperatorPanelRule`, `ModuleBoundariesTest` (now admitting `Enums` for OperatorPanel), phpstan-max, Pint, the full suite on SQLite and the PG17 gate.

## Non-Goals (deferred — see proposal slice-boundary table)

- **Generalizing `OperatorConsoleViewRecord`** into a verb-list base (the rule-of-three — see Decision D10).
- **Field-editing** a Club or ProducerAgreement in place ("Club config" beyond creation) — no update Action ships; deferred-pending-backend.
- **Customer / Account / Profile** (`operator-console-parties-customer`); **Hold registry / sanctions** (`operator-console-parties-compliance`); **Supplier** create; authority-tier RBAC; the TanStack portals.
- Enforcing **Producer↔Club consistency** on the agreement's optional Club picker (the backend does not; the console may *assist* by filtering but adds no rule).

## Decisions

These are the landmines an autonomous one-task-per-iteration agent must not step on. The tasks reference them as **D1–D12**.

- **D1 — Reuse the trait, not the catalog View base.** Each View page (`ViewClub`, `ViewProducerAgreement`) `extends \Filament\Resources\Pages\ViewRecord` and `use SurfacesDomainActions`, assembling its **own** verb set in `getHeaderActions()`. Do **not** extend `OperatorConsoleViewRecord` (its five hard-coded catalog governance verbs do not fit). Per ADR 2026-06-20.
- **D2 — Own status column, no `lifecycleStateColumn()`.** Each Resource `extends OperatorConsoleResource` (reuse `getModelLabel`/`getPluralModelLabel` off `static i18nKey()` + `versionColumn()`) but defines its **own** `status` badge `TextColumn` rendering the **state enum** via `getStateUsing` + `instanceof BackedEnum ? (string) $state->value : ''` — **no `Parties\Enums` import for the status column** (`ClubStatus` / `ProducerAgreementStatus` are *state* enums, rendered via the cast, never constructed). Mirror `ProducerResource::statusColumn()`.
- **D3 — Form-less verbs, no SoD affordance.** All four lifecycle verbs (sunset, close, activate, terminate) are `$this->lifecycleAction(verb, successKey, invoke)` with **no `form`** and **no `confirmationKey`**. Admin Panel §5.2 lists neither Club nor ProducerAgreement among the multi-actor patterns — they are single-operator. (Scope-guard the absence of a `requiresConfirmation()` affordance in tests.)
- **D4 — Parties Actions take `int $id`.** The lifecycle Actions are `SunsetClub::handle(int $clubId)`, `CloseClub::handle(int $clubId)`, `ActivateProducerAgreement::handle(int $agreementId)`, `TerminateProducerAgreement::handle(int $agreementId)`. Each invoke closure is `fn (Model $record, string $notes) => app(<Action>::class)->handle($this->recordOf(<Model>::class, $record)->id)` (`$notes` unused). `SunsetClub`'s optional `?int $causationId`/`?string $correlationId` are for the Producer-retirement cascade only — the console passes neither.
- **D5 — Name domain exceptions in PROSE backticks, never `{@see}`.** `IllegalClubTransition`, `IllegalProducerAgreementTransition`, `MissingClubProducer`, `MissingAgreementProducer` are all `RuntimeException` subtypes, caught by the trait's base-type `surfaceLifecycleOutcome`. Referencing them as `{@see}` would make Pint's `fully_qualified_strict_types` re-add a forbidden `Parties\Exceptions` import. Prose only. (Pint **may** add a `{@see}` docblock import for same-module `OperatorPanel` classes — that is intra-module, allowed; let it stand.)
- **D6 — Page/Action name collision → alias the import.** The Create page `CreateClub` and the Action `CreateClub` collide; import `use App\Modules\Parties\Actions\CreateClub as CreateClubAction;` (likewise `CreateProducerAgreement as CreateProducerAgreementAction`). The create **FORM lives on the Resource** (`ClubResource::form()`), not the page — Filament's `CreateRecord` inherits the Resource form; the page supplies only `createViaAction`/`createRejectionField`.
- **D7 — The operand-enum carve-out (the one architectural seam).** `CreateClub::handle()` requires a `ClubRegistrationFlowType` (an **operand enum**). The Club create page constructs it from the form value (`ClubRegistrationFlowType::from($selected)`) — which imports `Parties\Enums\ClubRegistrationFlowType`. This is admitted **only after** task group 1 widens `tests/Architecture/ModuleBoundariesTest.php` per ADR 2026-06-21 (`{Models, Actions}` → `{Models, Actions, Enums}`). **Task group 1 must land before the Club create task** or `ModuleBoundariesTest` reds. The `Select` options may be a hardcoded `value => __(label)` map or driven off the enum; either way the **construction** in `createViaAction` is the import that needs the carve-out. `ProducerAgreement` create needs **no** operand enum (ids/dates/string only) and stays within `{Models, Actions}`.
- **D8 — No standalone Supersede verb.** `ProducerAgreementSuperseded` is emitted **inside** `ActivateProducerAgreement` when a prior `active` exists in the same `(producer_id, club_id)` scope. There is **no** `SupersedeProducerAgreement` Action and there must be **no** supersede button. `ViewProducerAgreement` assembles **activate + terminate** only. (Scope-guard `assertActionDoesNotExist('supersede')`.)
- **D9 — No activate verb for Club.** A Club is born `active` by `CreateClub`. `ViewClub` assembles **sunset + close** only. (Scope-guard `assertActionDoesNotExist('activate')`.) `close` is reachable only from `sunset` — `CloseClub` rejects a close on an `active` Club; assert that branch.
- **D10 — Rule-of-three: DEFER the verb-list generalization (conscious decision).** With Club + ProducerAgreement, three non-catalog consoles exist, reaching the count ADR 2026-06-20 set for revisiting a `OperatorConsoleViewRecord` verb-list base. **We deliberately do not generalize in this change.** Rationale (faithful to that ADR's own criterion): the three supply-side FSMs are structurally alike (form-less verbs, `app(Action)->handle($record->id)`, base-type catch, own status column), so a base extracted now would be evidenced only by similar shapes; the ADR explicitly wants "at least one demand-side console" (Customer — three orthogonal FSMs + co-provisioned Account + multi-Profile, the least kit-shaped surface) as the stress-test before committing the abstraction (per-verb forms? confirmation affordances? ordering? a shared column contract across differently-named state attributes?). Keep reusing the trait directly; record this observation so the next console (Customer) is the deliberate trigger. No superseding ADR — this follows the existing one. The retrofit, when it comes, is cheap (all consoles already route through the trait).
- **D11 — Money construction for the Club fee.** `Money::of(int $minorUnits, Currency $currency)` with `Currency::of(string $code)` — both `App\Platform\Money\*` (Platform types, freely importable, **not** cross-module). In `createViaAction`, assemble `Money::of($amount, Currency::of($code))` only when **both** the amount and currency are present, else pass `null` (the Action's `?Money $fee = null` default). Money is integer minor units + ISO 4217 — never a float (Invariant 6).
- **D12 — The Club close precondition is a vacuous backend seam.** PRD §4.3 says `closed` is reached "once all members have migrated or expired"; the shipped `CloseClub` does **not** enforce that yet (Profile state is a demand-side concern, deferred). The console surfaces whatever `CloseClub` enforces (the from-state `sunset` guard only). Do not invent a member-migration check in the console.

## Risks & trade-offs

- **The `getPages()` boot coupling** (the §1 slice-shape driver, inherited from Producer): a Filament Resource's `getPages()` eagerly references each page class, and the panel calls it at registration — so the Resource cannot boot until **all three** pages (`List…`, `Create…`, `View…`) exist. The read-surface task must therefore scaffold `Create…` (with its real `createViaAction`) and `View…` (bare `ViewRecord` stub) even though the lifecycle verbs land in a later task. `discoverResources` scans `Filament/Resources` recursively, so the new classes under the existing `Resources/Parties/` are auto-discovered with no panel-provider change.
- **Carve-out ordering.** Task group 1 (widen `ModuleBoundariesTest`) is a **prerequisite** of the Club create task — sequence it first. It is the change's only non-additive edit; it touches a test allow-list + assertions, governed by ADR 2026-06-21, and must keep the carve-out guard's "OperatorPanel-only / whole-module-still-forbidden" assertions green.
- **Test DB trait.** Lifecycle/create tests use `DatabaseMigrations` (the Action opens its own real transaction → the recorder commits for real; factories bypass the Actions so record **no** event — the only events are the console's), **not** `RefreshDatabase`. Read-only resource tests use `RefreshDatabase`.
- **PG17 uncast bigints.** `actor_id`/`causation_id` are `int|null` and uncast → PostgreSQL returns numeric **strings**; assert them with loose `toEqual`, never strict `toBe`. `entity_id`/`correlation_id` are strings → `toBe` is safe. The closing-chain tests must be green on SQLite **and** PG17.
- **i18n kit-key completeness.** The kit resolves keys by string concatenation (`actions.<Str::snake(verb)>`, `notifications.<successKey>`), invisible to a source scan — so each console needs its own `…ConsoleI18nTest` enumerating the kit-contract suffixes (EN baseline + IT-differs + label/plural EN-fallback + IT⊆EN + the reused `scanOperatorConsoleHardcodedSinks` sink scan). **Reuse, never redeclare**, that scanner (declared top-level in `…/Catalog/ProductMasterConsoleI18nTest.php`); run the i18n test via `--filter`/full suite, **not** a bare file path (a path load misses the helper → false red). For a folder-wide PG17 run, append that Catalog declaring file.
- **OR-branch coverage** (a deliberate improvement over the Producer console, which left two OR-branches untested): assert **both** ProducerAgreement terminal branches — terminate **and** supersede-on-activate — and the Club close-from-`active` rejection, not just the happy linear path.
- **Pickers stay within the carve-out.** The Club create Producer picker and the ProducerAgreement Producer/Club pickers query `Parties\Models\{Producer,Club}` for options — within `{Models}`. They are reads; the no-Eloquent-write rule is unaffected. Do not add cross-module joins.
