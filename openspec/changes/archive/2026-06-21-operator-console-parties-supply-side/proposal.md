# Proposal — operator-console-parties-supply-side

## Why

`operator-console-parties-producer` opened the **Parties (Module K)** console with the Producer — the root the rest of the supply-side depends on — and proved the **non-catalog trait-reuse** pattern (ADR `decisions/2026-06-20-operator-console-non-catalog-lifecycle-trait-reuse.md`): a Resource that reuses `OperatorConsoleResource`'s label/version helpers but supplies its own status column; a View page that extends Filament `ViewRecord`, `use`s `SurfacesDomainActions`, and assembles its **own** verb set; a create write-through; the i18n kit-key completeness guard; and the page-driven closing-chain test. Its own slice-boundary table names **this** change as the one that completes the trio.

This change builds the remaining two supply-side consoles — **Club** and **ProducerAgreement** — over their fully-shipped backends. Both have complete domain Actions, FSMs, and events but **no operator surface**: a human cannot create a Club, sunset/close it, draft a ProducerAgreement, or activate/terminate it through the panel. This makes them operable, reusing the predecessor's recipe verbatim (its `progress.md` Codebase Patterns self-identifies as "the trait-reuse TEMPLATE the rest of Parties reuses").

- It introduces **no new domain logic**: zero new Parties Actions, events, migrations, or dependencies. It is a pure operator-surface layer over shipped behaviour (Admin Panel PRD §1.2 — the surface *references* the backend, never re-specs it). Every Action it invokes — `CreateClub`, `SunsetClub`, `CloseClub`, `CreateProducerAgreement`, `ActivateProducerAgreement`, `TerminateProducerAgreement` — already exists in `app/Modules/Parties/Actions/` (verified).
- It crosses **one** architectural seam the Producer console did not. The Club create surface must **construct** an *operand enum* — `ClubRegistrationFlowType`, a required argument of `CreateClub::handle()` — to route the create through the domain Action; but the OperatorPanel cross-module carve-out is `{Models, Actions}` and its guard forbids `Parties\Enums` imports. Producer's creates took only scalars, so it never surfaced. Resolved by a new ADR — `decisions/2026-06-21-operator-console-operand-enum-carveout.md` (**extends** `2026-06-19-operator-console-read-binding-write-through-actions.md`) — widening the carve-out to `{Models, Actions, Enums}` (operand enums are part of the Action's write-through call contract). Decided in a grill-with-docs session (2026-06-21). Implemented as this change's **task group 1**.
- It steps through **no open ADR gate** (root CLAUDE.md "Open stack decisions"): the Club/ProducerAgreement Actions are synchronous, inline-recording — no `queued` consumer (queue-driver gate not crossed); no invoice/statement/document is stored (object-storage gate not crossed); operator (Filament) surface only (frontend-stack gate not crossed). Operator auth is shipped, the Filament stack is pinned, and the read/write boundary is decided.
- It reaches the **rule-of-three** the trait-reuse ADR flagged: with Club + ProducerAgreement, three non-catalog consoles exist. This change **consciously defers** the verb-list generalization of `OperatorConsoleViewRecord` (the decision and its rationale are recorded in `design.md`, per that ADR's own criterion of awaiting a demand-side console) rather than defaulting silently.

## What Changes

- **Operand-enum carve-out (task group 1; ADR `2026-06-21-operator-console-operand-enum-carveout.md`)** — `tests/Architecture/ModuleBoundariesTest.php` `moduleBoundaryAllowedImports()` gains `Enums` for the OperatorPanel source only, and the carve-out guard test's assertions are updated to admit `Enums` (whole-module + other internals still forbidden). This is the **only non-additive** code change; it unblocks the Club create surface and every future operand-enum console create.
- **Club read-only Resource** (`ClubResource extends OperatorConsoleResource`; Admin Panel §3.K) — read-binds `App\Modules\Parties\Models\Club`, reusing the kit's label + `versionColumn()` helpers; supplies its **own** `status` badge column (a `ClubStatus`, rendered via the cast `->value` with no `Parties\Enums` import — a *state* enum) instead of `lifecycleStateColumn()`. List = display_name / operating Producer / registration flow / status / version; infolist adds the fee + flags + the operated-Producer read. No default create/edit/delete mutating action.
- **Club create surface** (`CreateClub extends OperatorConsoleCreateRecord`; party-registry — Club, Spine Creation Events) — a manual create form collecting **display name**, an operating **Producer** (a required picker), and a **registration flow type** (required), plus an **optional fee** (minor-unit amount + ISO currency → the `Money` value object) and the **generates-credit** (default on) / **invite-only** (default off) toggles. `createViaAction()` constructs the `ClubRegistrationFlowType` *operand enum* (the carve-out seam) and the `Money`, invokes `CreateClub::handle(...)`, and returns the model (never `$model->save()`). The created Club is born **`active`** and records `ClubCreated` with `actor_role: newco_ops`; the form exposes **no** status field. Creating a Club with a Producer id that matches no Producer is rejected (`MissingClubProducer`) and surfaced on the picker.
- **Club status lifecycle — sunset + close** (`ViewClub`; party-registry — Club Lifecycle, Supply-Side Lifecycle Events) — the View page assembles **two** verbs via `lifecycleAction`: **sunset** (`SunsetClub`, `active → sunset`, records `ClubSunset`) and **close** (`CloseClub`, `sunset → closed`, records `ClubClosed`). There is **no activate verb** — a Club is born `active`. Both are form-less, single-operator (no SoD affordance — §5.2 mandates none for Club). The console exposes **no** submit/reject/reopen/activate action. An out-of-state transition (`IllegalClubTransition` — e.g. close-from-`active`, which must pass through `sunset`) is surfaced as a notification, state unchanged. (The PRD "all members migrated/expired" close precondition is a backend seam still vacuous at launch — Profile state is a demand-side concern; the console surfaces whatever the Action enforces.)
- **ProducerAgreement read-only Resource** (`ProducerAgreementResource extends OperatorConsoleResource`; Admin Panel §3.K) — read-binds `App\Modules\Parties\Models\ProducerAgreement`; own `status` badge column (a `ProducerAgreementStatus` *state* enum, via the cast). List = producer / club (nullable → "Producer-wide") / status / term dates / version; infolist adds settlement cadence + the Producer/Club reads.
- **ProducerAgreement create surface** (`CreateProducerAgreement extends OperatorConsoleCreateRecord`; party-registry — ProducerAgreement, Spine Creation Events) — a manual create form collecting a required **Producer** picker, an **optional Club** narrowing picker (null = Producer-wide), optional **term start / term end** dates, and an optional **settlement cadence** (free string — the D19 Module-E seam). It needs **no** operand enum (all inputs are ids / dates / string), so it stays within the existing `{Models, Actions}` surface. `createViaAction()` invokes `CreateProducerAgreement::handle(...)` and returns the model; the agreement is born **`draft`** and records `ProducerAgreementCreated`; the form exposes no status field. A missing Producer is rejected (`MissingAgreementProducer`). The single-active-per-scope rule is **not** checked at creation (drafts create freely — it is an activation-time invariant).
- **ProducerAgreement status lifecycle — activate + terminate** (`ViewProducerAgreement`; party-registry — ProducerAgreement Lifecycle, Supply-Side Lifecycle Events) — **two** verbs: **activate** (`ActivateProducerAgreement`, `draft → active`, records `ProducerAgreementActivated`) and **terminate** (`TerminateProducerAgreement`, `active → terminated`, records `ProducerAgreementTerminated`). Activation **auto-supersedes** any prior `active` agreement in the same `(producer_id, club_id)` scope **in-domain** — the prior transitions `active → superseded`, recording a `ProducerAgreementSuperseded` causally linked to the activation; this is the Action's own behaviour, so the console exposes **no standalone supersede verb** (`superseded` is never a direct operator transition). Termination does **not** cascade to the Producer. Both verbs are form-less, single-operator; the console exposes no submit/reject/reopen action. An out-of-state transition (`IllegalProducerAgreementTransition`) is surfaced, state unchanged.
- **Every console write carries the audit envelope** (Admin Panel §1.3) — because writes route through the domain Actions, each records `actor_role: newco_ops` + operator id + timestamp via the existing `ActorContext` seam. No console-specific envelope code.
- **i18n EN + IT** (Architecture §5.1; Invariant 12) — all Club and ProducerAgreement operator copy through Laravel localization under new `operator_console.club.*` and `operator_console.producer_agreement.*` key roots. Domain rejection messages (the danger-notification *body*) are reused from the shipped Parties domain lang (`parties.club.*` / `parties.producer_agreement.*`); the console owns only the `notifications.action_failed` *title*. EN baseline + per-key IT fallback (the canonical "Producer"/"Club" terms stay EN-invariant).

### Slice boundary — deliberately NOT in this change

| Deferred concern | Future change / status | Why not here |
|---|---|---|
| **Producer** console (the trio's root) | shipped — `operator-console-parties-producer` (archived 2026-06-20) | Already done; this change is its named follow-on. |
| **Customer / Account / Profile** admin (the demand-side party administration) | `operator-console-parties-customer` | Customer has three orthogonal FSMs (status + KYC + sanctions) + a co-provisioned Account + multi-Profile membership — the biggest, least kit-shaped surface. Its own arc. |
| **Hold registry** + **sanctions verdict view** | `operator-console-parties-compliance` | A different console archetype (action/registry, not create→lifecycle). If KYC/compliance **document** upload/storage lands there, it crosses the **object-storage** ADR gate first. |
| **Verb-list generalization** of `OperatorConsoleViewRecord` (the rule-of-three) | a future kit change, after a demand-side console | The rule-of-three is reached on the supply side, but the ADR wants a differently-shaped demand-side console (Customer) before committing the abstraction. **Deliberately deferred — see `design.md`.** |
| **Field-editing / updating** a Club or ProducerAgreement in place ("Club config" beyond creation) | pending a verified Parties update Action | The shipped Parties backend exposes create + lifecycle transitions, **no** field-update Action. The console offers lifecycle transitions, not field edits. **deferred-pending-backend.** |
| **Producer-Club consistency** on the agreement's optional Club picker | not enforced (matches the backend) | `CreateProducerAgreement` takes `producer_id` + `club_id` independently and does not enforce that the Club belongs to the Producer. The console may *assist* by filtering the picker, but adds no domain rule. |
| **Supplier** create surface | a later Parties / Module-D slice | Minimal, event-silent, lifecycle-less create-only entity — folds naturally elsewhere. |
| **Authority-tier RBAC / persona-gating** | `feedback_prd_rr_approval` | Role-agnostic at this layer (Admin Panel §1.4). Club and ProducerAgreement lifecycle enforce no SoD (the spec mandates none — §5.2 lists neither). |
| **Producer / Consumer portals** (TanStack SPA) | the Module S storefront gate | Operator (Filament) surface only; the SPA frontend ADR is unwritten. |

## Capabilities

### New Capabilities

_None._ This change adds requirements to the existing `operator-console` capability.

### Modified Capabilities

- `operator-console`: **ADDED** four requirements — (1) *Operator creates a Club through the console*; (2) *Operator advances a Club through its status lifecycle* (sunset → close; born active, no activate; no SoD, no submit/reject/reopen); (3) *Operator creates a ProducerAgreement through the console*; (4) *Operator advances a ProducerAgreement through its status lifecycle* (activate with in-domain supersession of the prior active in scope; terminate; no standalone supersede verb). The capability's existing general requirements (read-only / write-through binding, the `actor_role` audit envelope, EN+IT localization) already apply to every console built on the capability and are **not** restated — they are cited as sources. The operand-enum carve-out is an OperatorPanel-internal architecture concern governed by ADR `2026-06-21` + the `ModuleBoundariesTest` guard, **not** a console-behaviour requirement, so it is recorded in the ADR and `design.md`, not as a spec requirement.

## Impact

- **Code (new, all under the OperatorPanel module + lang + tests):** `app/Modules/OperatorPanel/Filament/Resources/Parties/{ClubResource,ProducerAgreementResource}.php` + their `Pages/{List…,Create…,View…}.php`; `lang/{en,it}/operator_console.php` (+`club` and +`producer_agreement` blocks); `tests/Feature/Modules/OperatorPanel/Parties/{Club*,ProducerAgreement*}.php`. New resources join the existing `Resources/Parties/` sub-namespace beside `ProducerResource`.
- **Code (modified — the one non-additive change):** `tests/Architecture/ModuleBoundariesTest.php` — widen the OperatorPanel carve-out to admit `Enums` (`moduleBoundaryAllowedImports()` + the carve-out guard assertions), per ADR `2026-06-21`.
- **No change to:** any `spec/**` or `openspec/specs/**` by hand; any Parties domain code (Models, Actions, Events, Enums, migrations); `composer.json`; the catalog consoles, the kit's base classes, or the shipped Producer console.
- **Decisions / docs:** one new ADR (`2026-06-21-operator-console-operand-enum-carveout`, extends `2026-06-19`); `decisions/INDEX.md` updated; `CONTEXT.md` → Identity & Access gains the **operand enum** / **state enum** terms.
- **Guards that must stay green:** `NoEloquentWriteInOperatorPanelRule` (PHPStan) and `ModuleBoundariesTest` — the two new consoles import only `Parties\Models\{Club,ProducerAgreement,Producer}`, the six `Parties\Actions\*`, and (Club only) the `Parties\Enums\ClubRegistrationFlowType` *operand* enum; rejections are caught by `RuntimeException` base type (no `Parties\Exceptions` import). State enums (`ClubStatus`, `ProducerAgreementStatus`) are rendered via the cast, never imported.
- **Verification engine:** PG17 full-suite gate at close (the production engine), per the catalog + Producer precedent.
