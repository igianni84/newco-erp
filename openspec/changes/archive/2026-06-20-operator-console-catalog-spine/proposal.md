# Proposal — operator-console-catalog-spine

## Why

`operator-console-catalog-master` founded the `operator-console` capability and proved its read-binding / write-through-actions pattern end-to-end on the **richest** catalog entity (Product Master: create-with-dedup, the full `draft → reviewed → active → retired` lifecycle, the Creator → Reviewer → Approver floor, the Producer-activation gate, cascade-retire, reopen). The other **six** Module 0 spine entities — **Product Variant, Format, Product Reference, Case Configuration, Sellable SKU, Composite SKU** — have a fully-shipped backend (the same FSM, the same Action set) but **no operator surface**: they cannot be created, advanced, or retired by a human. This change makes all six operable through the panel, completing the Catalog console.

It is also the change the predecessor explicitly deferred its one abstraction decision to (`operator-console-catalog-master` design **L9 — "No premature abstraction… the right abstraction emerges from the second entity (`operator-console-catalog-spine`)"**). With seven entities now repeating one console shape, this change **extracts the shared console kit** (a base read-only Resource + an operator-action wrapper) and **retrofits Product Master onto it**, so the kit is proven against the hardest entity and no console remains a bespoke special case. That kit is the template every later console (Parties, then A/D/S/B/C/E) reuses.

- It introduces **no new domain logic**: zero new Catalog Actions, events, migrations, or dependencies. It is a pure operator-surface layer over shipped behaviour (Admin Panel PRD §1.2 — the surface *references* the backend, never re-specs it). The six entities' Create / Submit·Reject·Activate·Retire·Reopen Actions already exist in `app/Modules/Catalog/Actions/` (verified).
- It steps through **no open ADR gate**: operator auth is shipped (`2026-06-15-identity-auth`), the Filament stack is pinned (`2026-06-11-stack-versions-and-filament-ai-tooling`), and the read/write boundary is decided (`2026-06-19-operator-console-read-binding-write-through-actions`). The consumer/producer frontend-stack gate is **not** crossed — this is the operator (Filament) surface only.
- The shared-kit extraction is a new, forward-binding architectural decision (it sets the structural template for the seven remaining module consoles), so it is recorded in its own ADR — `decisions/2026-06-20-operator-console-shared-resource-kit.md` — and referenced from `design.md`.

## What Changes

- **Shared console kit + Product Master retrofit** (ADR 2026-06-20; predecessor design L9) — extract a base read-only Resource and an **operator-action wrapper** (the `surfaceLifecycleOutcome` try/notify helper, today a private static method duplicated per console) that encode the capability's invariant discipline once: read-only binding (no default create/edit/delete mutating actions), the uniform submit·reject·activate·retire·reopen lifecycle header actions wired to `app(<Action>)->handle(...)`, and the catch-and-surface contract. Per-entity divergences stay as explicit extension seams (parent pickers, create-rejection mapping, and the **Master-only** producer picker + cascade-retire). **Retrofit `ProductMasterResource` and its pages onto the kit** — Master becomes "kit + producer-gate extension + cascade-retire extension", its full shipped test suite + the arch/boundary tests staying green as the regression net. No Master behaviour changes; no spec requirement about Master changes.
- **Six read-only Resources** (product-catalog *Product Variant / Format / Product Reference / Case Configuration / Sellable SKU / Composite SKU*; Admin Panel §3.0) — one Resource per entity (list + view) binding **read-only** to its `App\Modules\Catalog\Models\*` model, with the `lifecycle_state` badge column and the entity's structural fields/relations rendered for display. No default mutating actions.
- **Create surfaces per entity** (product-catalog entity requirements + Spine Creation Events; Admin Panel §3.0) — a Create page per entity whose `handleRecordCreation()` invokes the entity's `Create<Entity>` Action and returns the model (never `$model->save()`): scalar create for **Format** (`name`, `sizeLabel`, `volumeMl`) and **Case Configuration** (`name`, `unitsPerCase`, `packagingType`); parent-picker create for **Product Variant** (Master + wine attrs), **Product Reference** (Variant + Format), **Sellable SKU** (Product Reference + Case Configuration + commercial attrs); and the N≥2 ordered-constituent picker for **Composite SKU**. The new entity is `draft`; its `*Created` event records with `actor_role: newco_ops`.
- **Create-rejection surfacing** — the two shipped create guards are surfaced as **form validation errors**: **Composite SKU**'s `< 2 distinct constituents` (a localized domain `InsufficientCompositeConstituents`, reused via its message) and **Product Reference**'s duplicate `(variant, format)` (a structural DB `UniqueConstraintViolationException` with **no** localized domain message — so the console maps it to a console-owned localized message). Variant / Format / Case Configuration / Sellable SKU ship no create guard.
- **Review-and-approval lifecycle, generalized to all six** (product-catalog *Product Lifecycle State Machine* + *Approval Governance*; Admin Panel §5.2) — Filament Actions for submit-for-review (`Submit<Entity>ForReview`, audit-only), reject (`Reject<Entity>Review` with a notes form, stays `reviewed`), and activate (`Activate<Entity>`). The activate Action carries the **"second actor required"** affordance; the Creator → Reviewer → Approver floor + self-approval rejection are enforced by the domain (`ApprovalGovernance`) and surfaced as a notification — never reimplemented.
- **Activation-cascade gate surfaced** (product-catalog *Activation Cascade*) — activating a **child** whose parent(s) are not all `active` is rejected by the domain and the reason surfaced (Variant ← Master; Product Reference ← Variant **and** Format; Sellable SKU ← Product Reference **and** Case Configuration; Composite SKU ← **every** constituent Product Reference). The standalone **Format** and **Case Configuration** have no parent gate. This is the spine's within-catalog analogue of the Master-only Producer gate; the domain copy (`catalog.gate.parent_not_active`, parameterized by `:entity`/`:parent`) already ships.
- **Retire / reopen, with the reference-integrity block surfaced** (product-catalog *Retirement Cascade and Reference Integrity*) — Actions for `Retire<Entity>` (`active → retired`; a hierarchy parent's retire preserves its existing active children) and `Reopen<Entity>` (`retired → reviewed`, audit-only). A single-entity retire blocked by an active within-catalog reference is surfaced (**Product Reference** referenced by an active Sellable/Composite SKU; **Case Configuration** referenced by an active Sellable SKU — `catalog.retirement.blocked_by_active_references`). The other four retire without a within-catalog block. **No cascade-retire** exists for these six (Master-only; deferred-pending-backend).
- **Every console write carries the audit envelope** (Admin Panel §1.3; AC-AP-PWB-6) — because writes route through the domain Actions, each records `actor_role: newco_ops` + operator id + timestamp + action + entity ref via the existing `ActorContext` seam. No console-specific envelope code.
- **i18n EN + IT** (Architecture §5.1; Invariant 12) — all spine operator copy through Laravel localization in the `operator_console` group (per-entity sub-keys mirroring the Master shape); domain rejection messages reused from `lang/{en}/catalog.php` (`composite_sku.*`, `lifecycle.*`, `approval.*`, `gate.parent_not_active`, `retirement.blocked_by_active_references`); the console owns the **Product Reference duplicate** message (no domain copy exists for it). EN baseline + per-key IT fallback (DEC-127).

### Slice boundary — deliberately NOT in this change

| Deferred concern | Future change / status | Why not here |
|---|---|---|
| **Field-editing / updating** any spine entity in place | pending a verified Catalog update Action | The shipped Catalog backend has **no** update Action for any entity — only create/submit/reject/activate/retire/reopen (+ Master cascade). The console offers lifecycle transitions, not field edits. **deferred-pending-backend.** |
| **Cascade-retire** for the six spine entities | pending a Catalog backend change | `RetireProductMasterCascade` is the **only** cascade Action (Master-only, verified). No `Retire*Cascade` exists for the six. A spine-level cascade would need new backend. **deferred-pending-backend.** |
| **Bulk import** of catalog entities (§3.0 cap 4) | pending a Catalog backend change | No bulk-import Action exists; the `product-catalog` spec does not specify it. **deferred-pending-backend.** |
| **Enrichment-metadata update** (§3.0 cap 6) | pending a Catalog backend change | No update Action; `EnrichmentDataUpdated` is unemitted; the spec marks enrichment "observational, not in launch". **deferred-pending-backend.** |
| **Cross-module** retire reference-integrity (active Allocations, issued vouchers, in-flight orders, SKUs on live Offers) | the Phase-3 referencer changes | Those referencers do not exist yet (documented spec seam); this change surfaces only the **within-catalog** block the backend ships today. |
| **Authority-tier RBAC / persona-gating** (which role runs which capability) | `feedback_prd_rr_approval` | Role-agnostic at this layer (Admin Panel §1.4). The only floor preserved is the spec-mandated SoD, enforced in the domain. |
| **Parties console**, then the **A/D/S/B/C/E** consoles | future `operator-console-*` changes | Different modules; each reuses this change's shared kit. |
| **Producer / Consumer portals** (TanStack SPA) | the Module S storefront gate | Operator (Filament) surface only; the SPA frontend ADR is unwritten. |
| **UX / IA / layout polish** | roadmap | The PRD is capability-not-layout (DEC-073). |

## Capabilities

### New Capabilities

_None._ This change extends the existing `operator-console` capability.

### Modified Capabilities

- `operator-console`: **ADDED** requirements extending the operator console — founded on Product Master by `operator-console-catalog-master` — to the six remaining catalog spine entities (Variant, Format, Product Reference, Case Configuration, Sellable SKU, Composite SKU): their create surfaces (with parent pickers and the PR-dedup / Composite-constituent form-error surfacing), the generalized review-and-approval lifecycle with the SoD affordance, the within-catalog **activation-cascade** gate surfacing, and the **retire / reopen** with reference-integrity surfacing. The existing capability-wide requirements (read-projection discipline, the `actor_role: newco_ops` envelope, EN+IT i18n) already govern these surfaces by their own general wording and are **not** modified; the five Master-lifecycle requirements stay Master-scoped and are **not** modified. No requirement is removed.

## Impact

- **New code** — six Resources under `app/Modules/OperatorPanel/Filament/Resources/Catalog/` (`ProductVariantResource`, `FormatResource`, `ProductReferenceResource`, `CaseConfigurationResource`, `SellableSkuResource`, `CompositeSkuResource`) each with List/Create/View pages and their lifecycle Actions; the **shared console kit** (base Resource + operator-action wrapper, under `app/Modules/OperatorPanel/Filament/`); per-entity console i18n keys in `lang/{en,it}/operator_console.php`; Pest tests under `tests/Feature/Modules/OperatorPanel/Catalog/`.
- **Modified code** — `app/Modules/OperatorPanel/Filament/Resources/Catalog/ProductMasterResource.php` + its pages, **retrofitted** onto the shared kit (app/module code, not protected); the shipped Master tests act as the retrofit's regression net.
- **No migration, no new dependency** — `git diff main -- composer.json composer.lock` stays empty; no `database/migrations/*` added (nothing is written to the DB that does not route through a shipped Action).
- **Reuses, does not modify** — the Catalog domain Actions for all six entities; `LifecycleTransition` + its gates (`ApprovalGovernance`, the activation-cascade gate, the retirement reference-integrity guard); the six models (read-binding) and the `catalog_composite_sku_constituents` junction; `ActorContext` / `ActorRole`; the `operator` guard + `Operator` principal; the `NoEloquentWriteInOperatorPanelRule` PHPStan rule and the `ModuleBoundariesTest` carve-out (both already cover the OperatorPanel namespace — no amendment expected).
- **Boundary & arch** — every new Resource reads `App\Modules\Catalog\Models\*` read-only and writes only through Catalog Actions; the existing architecture test forbids any Eloquent write in the OperatorPanel namespace; module models carry no cross-module relations, so Filament eager-loads cannot cross a boundary. The console catches domain rejections via base types (and the PR case via the framework `UniqueConstraintViolationException`) — no `Catalog\Exceptions` import — so the `{Models, Actions}` cross-module surface stays exactly as carved.
- **DB engines** — every test that drives a domain Action touches the DB, so the capability-close task verifies the suite on **PostgreSQL 17** as well as SQLite (`knowledge/testing/rules.md`).
- **Deliberate traceability gaps** — field-edit/update, cascade-retire (for the six), bulk import, and enrichment-metadata update have **no shipped backend** and are recorded as named `deferred-pending-backend` rows above (not silent cuts); the cross-module retire leg is a Phase-3 seam; the Parties + A–E consoles are named future changes.
