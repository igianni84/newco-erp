# Progress — operator-auth-foundation

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **spatie / `laravel-package-tools` publish tags** — packages built on `spatie/laravel-package-tools` derive their publish tags from `Package::shortName()`, which strips the `laravel-` name prefix. So `spatie/laravel-permission` (named `laravel-permission`) publishes via `--tag=permission-config` and `--tag=permission-migrations` (NOT `laravel-permission-*`). Confirm any spatie tag by reading `->name(...)` in the package's ServiceProvider + `shortName()` in `vendor/.../Package.php` before publishing.
- **Vendor-published migration vs PHPStan-max** — a published third-party migration that resolves its names through Laravel's `config()` helper (typed `mixed`) trips `level: max` with ~dozens of `offsetAccess.nonOffsetAccessible` / `argument.type` errors (spatie's `create_permission_tables` did: 66 of them). Fix: add the **single file** to `excludePaths` in `phpstan.neon` (glob `database/migrations/*_create_permission_tables.php` survives a re-publish timestamp change) — we publish vendor code verbatim and never hand-edit it, and the errors are framework `config()`-is-mixed noise, not a real bug. Scope to the one file so our own migrations stay analysed. Do NOT add `@phpstan-ignore`/casts/`@var` to the vendor file.
- **Adding a new dependency → `docs/development.md` is self-checking** — `tests/Feature/DevelopmentDocsTest.php` cross-checks a fixed package list against `composer.lock`. The established pattern (pennant 2026-06-13, spatie 2026-06-15) when adding a dep: (1) add a row to the "Installed versions" table, (2) add a dated prose paragraph explaining the dep + its ADR authorisation, (3) add the package name to that test's `$packages` array so its pinned version can't drift unnoticed.
- **Cross-engine close recipe** (`knowledge/testing/rules.md`) — `docker run -d --name pg -e POSTGRES_DB=newco_test -e POSTGRES_USER=newco -e POSTGRES_PASSWORD=newco -p 55432:5432 postgres:17`, wait for `pg_isready` (busy-poll loop — foreground `sleep` is sandbox-blocked), then `DB_CONNECTION=pgsql DB_HOST=127.0.0.1 DB_PORT=55432 DB_DATABASE=newco_test DB_USERNAME=newco DB_PASSWORD=newco php artisan test`, then `docker rm -f pg`. phpunit.xml's non-`force` `<env>` lets the CLI env win.
- **Auth-principal models are EXEMPT from the module-table-prefix arch test** (`ModulePersistenceConventionsTest`; design D7; `decisions/2026-06-15-auth-principal-table-naming.md`) — a model implementing `Illuminate\Contracts\Auth\Authenticatable` keeps the framework's flat auth-table name (`Operator` → `operators`, never `operator_panel_operators`). The test skips them via `implementsInterface(Authenticatable::class)` right after the concrete-`Model` guard (the proven-empty `$scannedFiles[]` proof is unaffected — the skip is *after* the scan append). Invariant 10's substance (no cross-module DB access) stays enforced by `ModuleBoundariesTest` + guard-by-name access. Forward-binds the deferred `customer-identity`/`producer-identity` principals; **every non-principal module model still needs its module prefix.**
- **Framework/vendor getters are typed to a CONTRACT, not the concrete class — narrow with a real `instanceof`, never `assert()`/`@var`** (phpstan-max). Manifestations seen: `Auth::guard('operator')` is `Guard|StatefulGuard` and `Auth::createUserProvider('operators')` is `?UserProvider` (so `SessionGuard::getProvider()` / `EloquentUserProvider::getModel()` are off-contract); `Panel::getMultiFactorAuthenticationProviders()` is `array<string, MultiFactorAuthenticationProvider>` (so `AppAuthentication::isRecoverable()` is off-contract). Calling a concrete-only method on the contract trips `method.notFound` **and** breaks Pest's `->and()` template resolution (the off-contract method's return type is unresolved). `expect($x)->toBeInstanceOf(Concrete::class)` does **not** narrow `$x` for a following `->and(...)`. Fixes: (a) assert the runtime type with `toBeInstanceOf(Concrete::class)`; (b) for an extra method/property, pull it to a var and gate with a real `instanceof` — `expect($x instanceof Concrete && $x->method())->toBeTrue()`; (c) where possible prove bindings declaratively (`config('auth.providers.<name>.model')`) and functionally (`actingAs($model,'<guard>')` then `Auth::guard('<guard>')->user() instanceof <Model>` — a stronger proof than introspection). Forward-applies to the deferred customer/producer guard tests + task 4.1's `ActorContext` guard read. **4.1 resolution:** `Auth::guard('operator')->check()` (→`bool`) and `->id()` are BOTH on the `Guard` contract → **no narrowing needed** in production code; but `Guard::id()` is contract-typed `int|string|null`, so coerce a bigint key to `?int` with a real cast — `$id = $guard->id(); return $id === null ? null : (int) $id;` — never `@var`/`assert()`. This is the boundary-clean way to read an operator id; the deferred customer/producer guards copy it verbatim.
- **End-to-end actor-provenance proof** (4.2; the integration shape for any auth principal → recorder pipeline) — to prove a principal's emission carries the right envelope, feed the recorder the values **RESOLVED from `ActorContext`** (`$ctx->role()` / `$ctx->actorId()`), **never literals**, under `actingAs($principal, '<guard>')`; resolve the singleton context *after* `actingAs` so its lazy per-call resolution is exercised. Use **`DatabaseMigrations`, NOT `RefreshDatabase`** (same reason the recorder tests do): the recorder's `DB::transactionLevel() === 0` guard makes the explicit `DB::transaction()` wrapper load-bearing and the emission a REAL commit — under RefreshDatabase the wrapper transaction silently satisfies the guard, hiding a missing wrapper. Use a **synthetic** event name (no registered consumer → the afterCommit hook is a clean no-op; verbatim spec event names are reserved for module changes). Assert **both** the model enum (`->actor_role`) AND the raw stored token (`DB::table('domain_events')->where('id',$id)->value('actor_role') === ActorRole::X->value`) so the persisted value is pinned, not just the hydrated cast; `actor_id` is uncast bigint → `->toEqual()` (loose) spans SQLite/PG. The deferred customer/producer guard slices copy this verbatim (their resolution extends `ActorContext` step 2).
- **Spatie `assignRole('Name')` resolves on the MODEL's guard, not the app default** (5.2; every future principal's role-grant seeder copies this) — `$model->assignRole('Creator')` → `getStoredRole` → `Role::findByName('Creator', $model->getDefaultGuardName())`. For a model with **no** `guard_name` attribute/property/`guardName()` method (our `Operator`), `Guard::getDefaultName($model)` returns the app default (`web` until 6.1) **only if** that default is among `Guard::getNames($model)`; otherwise it returns the **first guard whose `config('auth.guards').*.provider` model === the model's class** (`getConfigAuthGuards`). So an `Operator` resolves to **`operator`** even while the app default is still `web` — the role grant lands on the right guard with no extra wiring. **`Role::findByName`→`findByParam` is a DIRECT `query()->where()->first()` (NOT the registrar cache)**, so role resolution is cache-independent — `WithoutModelEvents` on a seeder cannot break a chained `assignRole`, and no `forgetCachedPermissions()` is needed in the grant seeder (RoleSeeder's reset already covers the mechanism). `assignRole` is idempotent via `array_diff(roleKeys, currentRoles)` → attaches only the delta (re-seeding a fully-roled principal attaches nothing). `collectRoles` flattens, so pass the array directly: `assignRole(RoleSeeder::ROLES)`. Guard-sharing (`ensureModelSharesGuard`) passes because `getGuardNames(Operator)===['operator']` contains the role's `operator`.
- **Spatie role/permission seeding** (5.1; reused by every future guard's role/permission seeder) — seed bare roles idempotently with `Role::firstOrCreate(['name' => …, 'guard_name' => '<guard>'])`. Eloquent's `firstOrCreate` uses `newModelInstance() + save()`, so it **bypasses** spatie's static `Role::create()` `RoleAlreadyExists` guard (the *only* place that throws) → safely re-runnable, no duplicates. **Always pass `guard_name` explicitly:** spatie defaults it from `config('auth.defaults.guard')` (still `web` until 6.1), so an implicit guard would mis-scope the role. Reset the registrar cache at the top of any role/permission seeder — `app(\Spatie\Permission\PermissionRegistrar::class)->forgetCachedPermissions()` — spatie's documented seeder convention; the `Role`/`Permission` models also `use RefreshesPermissionCache` (auto-forgets on every `saved`/`deleted`), but the explicit reset forward-protects a chained assign-roles seeder (`DatabaseSeeder` → `RoleSeeder` then `OperatorSeeder`, D6/5.2). Role names are spec-named **data identifiers** (`Creator`/`Reviewer`/`Approver`), not UI copy → verbatim, no i18n. **Test note:** `$role->permissions` is an Eloquent `Collection` → assert emptiness with `->toHaveCount(0)` (unambiguous for a Collection). The `config('permission.models.role')` default IS `Spatie\Permission\Models\Role` — reference the FQCN directly.
- **Laravel deep-merges the framework's base `config/auth.php` UNDER the app's — `web`/`users` cannot be removed by editing the app file** (6.1; binds every future guard slice — customer/producer). `Foundation\Bootstrap\LoadConfiguration::mergeableOptions('auth')` returns `['guards','providers','passwords']`, and `loadConfigurationFile()` runs `array_merge($base['auth'][$opt], $config['auth'][$opt])` for each (base-first). So the framework's default `web` guard + `users` provider/broker (shipped in `vendor/laravel/framework/config/auth.php`) are **always present** in the merged `config('auth.*')` even when the app's `config/auth.php` omits them — `array_merge` adds app keys but **never deletes** a base key. Consequences: **(1)** `config('auth.guards.web')` / `config('auth.providers.users')` are **non-null at runtime** regardless of the app file → a test asserting their absence FAILS (this bit 6.1). **(2)** The non-mergeable `defaults` key **is** fully overridden by the app file (the top-level `array_merge($base['auth'],$config)` at line 118), so `defaults.guard => 'operator'` / `defaults.passwords => 'operators'` take effect cleanly. **(3)** The leftover `web`/`users` are **inert** as long as nothing resolves them — guarantee that by grep (`guard('web')`, `auth:web`, guard-less `actingAs`/`assertGuest` which use the *default* guard) before relying on it. Prove a principal-shell removal via the **empty old-model reference sweep + the dropped table**, NOT a config-key-absence assertion. Truly stripping them at runtime needs a provider mutating `config(['auth.guards' => …])` (out of scope; the inert defaults are accepted). **The 6.1 acceptance grep also forbids the old FQCN / factory tokens in ANY `app|config|database|tests` file *including comments*** → describe the sweep without writing the literal `App\Models\User` / `…Factory` tokens.
- **Doc-pinning tests share a `developerDoc()` reader across files** (defined in `tests/Feature/EventSubstrateDocsTest.php`; used by `FoundationsDocsTest`, `ModuleTemplateDocsTest`, …) — validate a docs change with the **full suite** (or include the helper-defining file), never a single-file `--filter`/path subset: a subset that excludes `EventSubstrateDocsTest` reds with `Call to undefined function developerDoc()`, an isolation artifact, NOT a real failure (`DevelopmentDocsTest` is self-contained via its own `developmentGuide()` and runs in isolation). The pins are byte-exact `toContain` checks (FoundationsDocsTest pins `_Avoid_: session, current user, auth guard` verbatim) → keep pinned tokens character-for-character when editing the doc, and reword the doc-change's own now-stale comments in the same pass.
