# Proposal — operator-auth-foundation

## Why

The Identity/auth gate closed on 2026-06-15 (`decisions/2026-06-15-identity-auth.md`) but **built nothing**: there is no `Operator` model, no `operator` guard, no Filament login wired to one, no `spatie/laravel-permission`, and `App\Platform\Events\ActorContext` still hard-defaults to `system` and reads no authentication state (its own requirement says so — `openspec/specs/event-substrate/spec.md` Requirement: *Actor Context Resolution*). The bootstrap `App\Models\User` shell literally documents this as the gate to revisit (*"Operator RBAC arrives with the Module K identity/auth ADR (open gate) — revisit this gate there"*).

This slice builds the **operator-side authentication foundation** the ADR decided — the reusable platform foundation (not Catalog-specific) that **unblocks `catalog-lifecycle-approval`** (the operator principal who approves) and **all of Module K** (operator-driven onboarding / KYC / sanctions / Hold, every action carrying a real `actor_role: newco_ops` + actor identity). It is built now because the Build Workplan Phase 1 floor requires authentication wired and the `actor_role` audit envelope populated on every operator action (`spec/05-release/Build_Workplan_v0.3-MVP.md` §2 Phase 1; `spec/02-prd/Admin_Panel_PRD_v0.3-MVP.md` §1.3), and the downstream slices that consume an authenticated actor are blocked until the seam is wired.

This change **decides nothing new** — it implements the operator slice of `decisions/2026-06-15-identity-auth.md` against the spec's `actor_role` / RBAC requirements. No new ADR is required (the identity/auth ADR governs; `spatie/laravel-permission` adoption is already authorised there per the no-new-dependency rule). The one founder choice taken this session: **opt-in TOTP 2FA is available at launch** (enrolment optional; MFA *enforcement* policy stays deferred to the Architectural Security Review gate, per the ADR's trade-offs).

## What Changes

- **Operator principal + `operators` table**, owned by the **OperatorPanel** module (`App\Modules\OperatorPanel\Models\Operator`). For an operator the **login principal IS the acting identity** — operators have no Module K party row and the principal references none. Email globally unique, password hashed, remember token. **BREAKING (internal, pre-launch):** the generic `App\Models\User` + the default `users` table are **removed** and replaced by `Operator` / `operators`; Laravel's auth defaults are repointed to the `operator` guard. (`sessions` and `password_reset_tokens` are retained — the session guard and the operator password broker use them.)
- **`operator` session guard** in `config/auth.php` (provider = the `Operator` model; an `operators` password-reset broker), and the Filament `/admin` panel **authenticated against it** using Filament 5's native session auth: **login + password reset + opt-in TOTP 2FA** (with recovery codes — `Filament\Auth\MultiFactor\App\AppAuthentication`). **No** operator self-registration (staff are provisioned); **no** email verification (deferred).
- **`spatie/laravel-permission` installed** (latest stable, pinned at install per the stack-ADR discipline) as the operator RBAC **mechanism** — guard-aware, **operator-scoped**; the `Operator` model uses `HasRoles`. A seeder seeds the three spec-named roles **Creator / Reviewer / Approver** on the `operator` guard as **bare roles (data/mechanism only)** — no permission grants, no authority-tier "which-role-may-do-what" policy (that is admin-configurable + downstream, `feedback_prd_rr_approval`).
- **`ActorContext` wired** to the operator guard: on an authenticated `operator`-guard request it resolves (`newco_ops`, `Operator.id`); an explicit scoped **run-as override still wins**; console / queue / unauthenticated contexts **and the not-yet-wired customer/producer guards** still resolve to (`system`, `null`). The "reads no authentication state / gate-safe" clause is **retired**.
- **Seeding repurposed** from `User` → `Operator`: `OperatorSeeder` provisions the initial operator from environment credentials (never committed), idempotently; a `RoleSeeder` seeds the three roles; `DatabaseSeeder` orchestrates them.
- **Docs**: `CONTEXT.md` → *Actor context* entry updated (the seam is now wired to the operator guard — it no longer "reads NO authentication state"); `docs/development.md` records the pinned `spatie` version + the operator auth wiring + the operator credential env vars; `decisions/INDEX.md` open-decisions line marks the Identity/auth gate **built**.

### Slice boundary — deliberately NOT in this change

| Deferred concern | Future change / gate | Why not here |
|---|---|---|
| **Customer + producer guards** (Sanctum SPA auth; cookie-vs-token session mechanics) | **Module S storefront gate + TanStack frontend ADR** | The ADR fixes the *shape* (Sanctum, principal-references-party) but defers SPA session mechanics; those surfaces don't exist yet. The `producer` `actor_role` exists in the contract from day one and "switches on" when the Producer Portal login ships. |
| **Fortify / Sanctum install** (registration/reset/verify/2FA backend for the SPA) | **Module S gate** | Operators use Filament's **native** session auth, not Fortify; Fortify + Sanctum serve the customer/producer SPA. |
| **Catalog / authority-tier RBAC policy** (which named role may run which capability; the approval-tier enum) | **`feedback_prd_rr_approval`** | `Admin_Panel` §1.4 / §9.2 mark it admin-configurable + downstream; this slice seeds the roles as **mechanism only**, asserts no role→capability mapping. |
| **Filament Shield** (admin RBAC UI) | deferred (its own change if ever) | ADR: adopting it now builds the explicitly-deferred admin RBAC UI prematurely; spatie core + Policies suffice at launch. |
| **Separation-of-duties FLOOR enforcement** (3-step Creator→Reviewer→Approver transition; no self-approval) | **`catalog-lifecycle-approval`** (+ Module K producer activation) | It is **per-instance module transition logic** (reads the prior actor from the entity's audit/FSM and rejects a colliding actor), never a static permission. This foundation supplies the actor identity (`ActorContext`) + the role rows it consumes. |
| **MFA *enforcement* policy + session-security hardening** | **Architectural Security Review gate** | ADR routes enforcement there; this slice ships 2FA as **opt-in availability** only. |
| **Email verification for operators** | deferred | Operators are provisioned (no self-registration); low value, not spec-required. |
| **Operator-admin UX** (Filament Resource to manage operators / assign roles; the dead-letter & Hold operator surfaces) | later OperatorPanel / Module K changes | The foundation seeds operators + roles; management UX is not the auth foundation. |
| **Laravel Policies gating real resources** | the slice that adds the first protected resource (`catalog-lifecycle-approval`) | No protected operator resource exists yet; the RBAC mechanism is wired so the next slice adds Policies that consult spatie. |

## Capabilities

### New Capabilities

- `operator-identity`: the **operator authentication & access foundation** — the `Operator` authenticatable principal (owned by OperatorPanel; the principal *is* the identity, no party row, email globally unique), the **`operator` session guard** + Filament panel authentication (login, password reset, opt-in TOTP 2FA), and the **operator RBAC mechanism** (`spatie/laravel-permission`, guard-aware/operator-scoped; the seeded Creator/Reviewer/Approver roles as data). The umbrella capability that `feedback_prd_rr_approval` (authority-tier policy) and `catalog-lifecycle-approval` (separation-of-duties transition) later build on; the sibling future capabilities `customer-identity` / `producer-identity` activate the other two guards.

### Modified Capabilities

- `event-substrate`: Requirement *Actor Context Resolution* — the resolver now **reads the operator guard** (authenticated operator → `newco_ops` / `Operator.id`), **retiring** the "SHALL NOT read authentication state / gate-safe" clause. The `system` default for console/queue/unauthenticated **and** the not-yet-wired customer/producer guards is preserved; the scoped run-as override still wins.
- `platform`: Requirement *Operator Panel Shell* — the `/admin` panel now authenticates against the **`operator` guard** backed by the **`Operator`** principal (the seeded operator is an `Operator` row), replacing the bootstrap "every user can access" shell.

## Impact

- **New code** — `app/Modules/OperatorPanel/Models/Operator.php`, `database/factories/OperatorFactory.php`, the `operators` migration (repurposing the default `users` migration) + the 2FA columns, `config/auth.php` (operator guard/provider/passwords), `app/Modules/OperatorPanel/Providers/AdminPanelProvider.php` (auth guard + 2FA + password reset), `app/Platform/Events/ActorContext.php` (operator-guard resolution), `database/seeders/{RoleSeeder,OperatorSeeder,DatabaseSeeder}.php`.
- **New dependency** — `spatie/laravel-permission` (authorised by `decisions/2026-06-15-identity-auth.md` per the no-new-dependency rule; version **pinned at install**, recorded in `docs/development.md` + `progress.md`). Unlike prior slices, `composer.json` / `composer.lock` **do** change here.
- **Removed** — `app/Models/User.php`, `database/factories/UserFactory.php` (→ `OperatorFactory`), the generic `users` table (→ `operators`). References swept in `config/auth.php`, `DatabaseSeeder`, `OperatorSeeder`, and three tests (`tests/Feature/OperatorSeederTest.php`, `tests/Feature/OperatorPanelTest.php`, `tests/Feature/Platform/Events/ActorContextTest.php` — the gate-safe scenario flips).
- **Module boundary** — `ActorContext` (platform code) reads the operator guard **by name** (`Auth::guard('operator')`), **never importing** the OperatorPanel `Operator` model — no cross-module Eloquent. `spatie` tables/traits are a platform-level dependency the module model consumes (a vendor import, not a cross-module import). `tests/Architecture/ModuleBoundariesTest.php` stays green with **no amendment**.
- **DB engines** — the `operators` migration + the published `spatie` migrations are Postgres-truthful / SQLite-compatible; the full suite must be green on SQLite **and** verified on a local PostgreSQL 17 (print `DRIVER=pgsql`, remove the container) before close.
- **Deliberate traceability gaps** — customer/producer guards, the authority-tier policy, the SoD-floor enforcement, MFA enforcement, email verification, the operator-admin UI, and resource Policies are each mapped to a named future change/gate in the slice-boundary table above.
