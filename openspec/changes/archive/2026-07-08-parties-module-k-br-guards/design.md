## Context

This change **batches eight Module K / `party-registry` business-rule guards** to amortize the §2.7 close ritual + review over the set (cadence rule, `lessons.md` 2026-07-07). Four are frozen-spec guards the code never enforced (RM-19/20/21 + the RM-22 concept); four are canon acceptance criteria added after our frozen `spec/` pin (MVP-DEC-007) — RM-22's closed set (MVP-DEC-010), Agreement-4 (MVP-DEC-009), and Club-6/Identity-6/Profile-5/Producer-5 (MVP-DEC-022). All live inside `app/Modules/Parties/`; none crosses a module boundary (invariant 10 preserved).

Grounding was per-item and code-verified (four subagents + targeted greps). It **corrected two tracker premises**: RM-20 has no "producer-wide flag" (scope is a single nullable `club_id`, so the real gap is the cross-shape *activation* invariant, and our truth-spec currently asserts the **opposite** of the frozen rule); RM-22's `settlement_cadence` is **free text**, not a dropdown. It also **re-scoped RM-23** from the tracker's loose "DEC-009/013/021/022" to the five named criteria's true lineage: Agreement-4 ← MVP-DEC-009, Club-6/Identity-6/Producer-5/Profile-5 ← MVP-DEC-022, **J-15a ← MVP-DEC-011 (capacity → RM-05, carved out)**, while MVP-DEC-013 (no BR/AC) and MVP-DEC-021 (= RM-25/RM-01/others) fall outside the named set.

## Decisions

- **D1 — RM-19 ships the orchestration, not a new event.** `RetireProducer` cascades the Profile leg (drive `CancelProfile` per `Active`/`Lapsed` Profile with a producer-initiated reason, after the `ClubSunset`), audit-only. _Rationale:_ frozen §15.2 names **no** `ProfileCancelled` and §15.7 declares the exact signal-event shape a "deferred downstream-consumer concern"; §15.7's stated Module-K contribution is exactly "the transition logic + the cancellation reason." Zero-invention, zero-divergence, no closed-event-set change. _Alternative (rejected):_ emit a dedicated per-Profile cancellation event now — diverges from a coherent frozen position (§15.2), needs its own mini-ADR + a *Demand-Side Status Events* closed-set MODIFIED, for a subscriber (Module S) that does not exist. The event stays the §15.7 seam; when Module S lands it defines what it consumes.
- **D2 — RM-20 rejects the cross-shape activation (does not supersede).** _Rationale:_ frozen `AC-K-BR-Agreement-1` verification is literally "attempt Producer-wide + per-Club both active … **assert rejection**"; and the existing supersession mechanic is scope-keyed on `(producer_id, club_id)`, so a cross-shape "supersede" would have to supersede *all* opposite-shape actives across different scopes — semantically wrong. The operator terminates/supersedes the existing shape first. _Inversion:_ this replaces the shipped *ProducerAgreement Lifecycle* claim "MAY both be `active`" and its "Scope isolation" scenario. **No ADR to supersede** — the coexistence claim traces to an AskUserQuestion 2026-06-15 on the *scope definition* (tuple + NULL = Producer-wide), which never addressed the cross-shape clause; this is a plain MODIFIED correction toward the frozen rule.
- **D3 — RM-21 guards at Profile creation (`CreateProfile`), Club must be `active`.** A `sunset`/`closed` target → `ClubNotAcceptingMemberships`. _Rationale:_ the frozen rule is "sunset blocks **new memberships**"; creation is the chokepoint (`AC-K-BR-Club-3`: "new-membership creation rejected when Club is `sunset`"). _Not gating `ApproveProfile`_ (approval into an already-Applied Profile whose Club later sunset) — creation-time is the faithful minimal reading; an approval-time gate is a later refinement.
- **D4 — RM-22 = a `SettlementCadence` enum + DB CHECK, server-enforced.** Closed set `{quarterly (default), monthly, semi-annual}` (MVP-DEC-010). _Rationale:_ canon enforces at API + DB (times Module-E settlement + Module-D PO issuance). The literal representation is the dev's call (DEC-073) → a backed enum + a migration CHECK regen (mirrors the `parties_holds` pattern). **`annual` is out of set** (see R2).
- **D5 — Agreement-4 = a Club-active scoping guard at agreement creation.** Producer-wide ungated; **supersession inherits scope, exempt** (a wind-down amendment on a since-`sunset` Club is unaffected — canon-specified). Guard lives in the `CreateProducerAgreement` path.
- **D6 — Club-6 = collapse `invite_only` + gate `open_registration` out of the selectable set + assert no-auto-approve.** The enum keeps four cases; `OpenRegistration` becomes non-selectable (latent). _Rationale:_ our model already has both `registration_flow_type` and the redundant `invite_only` boolean — canon says collapse (invite_only=true ⟺ invitation_only). No enum reshape.
- **D7 — Identity-6 = an age-gate in `CreateCustomer` + a configurable platform min-age constant (default 18).** A `date_of_birth` implying an age below the minimum — **or a missing DOB** (age attestation is mandatory, BMD §2.8) — is rejected pre-creation (`BelowMinimumRegistrationAge`), no Customer/Account/event. The constant mirrors the RM-02 enhanced-KYC threshold pattern (representation dev's call, DEC-073). Existing Customers are grandfathered (gate is creation-only).
- **D8 — Profile-5 (K-side) = `auto_renew` inherit-at-creation + operator override.** New columns `parties_profiles.auto_renew` + `parties_clubs.auto_renew_default`; `CreateProfile` inherits the Club default; an operator Action overrides (audit-only, no event). The customer self-toggle is the deferred Consumer-Portal seam. We ship only the `auto_renew` element of the deferred `renewal_policy` blob (MVP-DEC-013).
- **D9 — Producer-5 (interim) = a model-level content lock (the RM-24 pattern).** A `Producer` `updating` guard throws `ProducerReviewGovernedContentLocked` when `name`/`description`/`region`/`website` is dirty while `status = active` — the only path-complete chokepoint (no Action-layer content-edit writer exists). Codifies the safety core (unreviewed content never publishes) with **zero behaviour change today** (see R5). The full "edit re-arms Creator→Reviewer→Approver" UX is deferred (no Producer `reviewed` state / edit path — building it now = dead code, per RM-06/RM-14). _Chosen over carving Producer-5 out entirely (Giovanni, 2026-07-07: "includi un guard minimo ora")._

## Risks / Trade-offs

- **R1 — Blast radius of wiring guards into shared Actions (the RM-08 landmine, `lessons.md` 2026-07-06).** The migration blast radius is the grep, not the task list. Confirmed hotspots:
  - **RM-20 → invert shipped tests.** `tests/Feature/Modules/Parties/ProducerAgreementLifecycleTest.php` **L157 (Club-narrowed activation alongside an active Producer-wide) and L206 (Producer-wide alongside an active Club-narrowed) currently assert coexistence** — they MUST flip to assert `ProducerAgreementScopeConflict`. Also `ProducerAgreementLifecycleConsoleTest`, `ComplianceIndependenceTest:140` (Action-set list — a new exception path, not a new Action, so likely unaffected). → *Mitigation:* invert L157/L206, keep the same-scope supersession scenarios (L95/L98) green.
  - **Club-6 → drop `invite_only`: 29 refs / 16 files** (`Club` model, `CreateClub`, `ClubCreated` payload, `ClubResource` + `CreateClub` page + `ClubsRelationManager`, `ClubFactory`, `DemoSeeder`, `ClubResourceTest`/`ClubConsoleChainTest`/`ClubCreateConsoleTest`/`ClubConsoleI18nTest`/`ClubTest`, migration, EN/IT lang). → *Mitigation:* one migration drops the column; sweep all 16 files; the `ClubCreated` payload drops the field.
  - **RM-21 → safe:** every `CreateProfile` caller (app/tests/seeder/factory) targets an `active` Club (verified in grounding) → the new guard breaks nothing. Add tests for the reject path.
  - **Identity-6 → mostly safe:** `CustomerFactory` DOB = `1990-01-01` (adult). *Mitigation:* grep `CreateCustomer` callers for any explicit under-18 / null DOB and migrate; the console DOB field becomes effectively required.
  - *Discipline:* before each guard, `grep -rn '{Action}' app/ tests/ database/` **and** `grep -rn "callAction('{verb}')" tests/` (console callers are invisible to the Action grep — the RM-08 miss); run the **full** suite (`php -d memory_limit=-1 vendor/bin/pest`) — a `--filter` pass hides cross-chain breakage.
- **R2 — `settlement_cadence = 'annual'` in `DemoSeeder`** (row ~296) is out of the canon set → the CHECK/enum cast will reject it. `ProducerAgreementFactory` default `'monthly'` is valid. → *Mitigation:* migrate the seeder's `annual` to `semi-annual` (or `quarterly`); the mini-ADR records the value change.
- **R3 — RM-20 inverts shipped behaviour + a prior decision.** Beyond the two tests, grep for any other assertion of Producer-wide+per-Club coexistence and the truth-spec scenario (replaced in the delta). → *Mitigation:* the delta replaces the "Scope isolation … MAY both be active" scenario; the semantic-verify must confirm no residual coexistence claim.
- **R4 — Module boundary (invariant 10) + the Pint `{@see}` auto-import trap (`lessons.md` 2026-07-06).** RM-19's `Producer→Profile` walk is intra-Parties (query by `club_id`, no new relation needed). New exceptions/guards are Parties-local. → *Mitigation:* name any cross-module class in **backticked prose**, never a `{@see FQCN}` docblock; re-run Pint + `grep` for a stray cross-module `use` before the suite.
- **R5 — Producer-5 lock is zero-behaviour today but must not false-positive.** Every existing Producer `->update()` touches only `kyc_status` or `status` (verified) → the `isDirty({name,description,region,website})`-keyed lock fires on none of them. → *Mitigation:* the guard keys on the review-governed set, not on the transition; a regression test asserts activation/retirement/KYC updates pass while a content edit on an `active` Producer throws (the RM-24 no-false-positive test shape).
- **R6 — Age-gate on a missing DOB.** Decision D7 blocks a null DOB at registration (mandatory attestation). If any legitimate flow creates a Customer without a DOB, it breaks. → *Mitigation:* grep confirms the factory supplies an adult DOB; the console DOB field becomes required; document null→block in the mini-ADR.
- **R7 — Batch size.** ~14 delta requirements / ~20+ tasks is at the upper end of a single change. → *Trade-off accepted:* that is the point of the batch (amortize §2.7); the loop still runs one task/iteration, and the thematic coherence (all `party-registry` Module K) keeps the semantic-verify per-requirement matrix clean.
