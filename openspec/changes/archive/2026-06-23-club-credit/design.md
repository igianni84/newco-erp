# Design — club-credit

## Context

Club Credit is a **greenfield** Module K entity (no `ClubCredit` code today — grep confirms every existing mention is forward-pointing docblock prose). It is the per-Profile prepayment instrument the Hero-Package membership fee converts into (DEC-007; PRD §11). The shipped Parties slices named this seam explicitly: `parties-membership-activation` design **L5** ("Club Credit … is an independent consumer in `club-credit`; `ProfileActivated` provisions no credit inline, so the split breaks nothing") and `parties-membership-suspension` ("the Club Credit frozen-while-suspended guarantee is a deferred `club-credit` seam (the credit entity reads `state`)").

The substrate this builds on (verified in `app/Platform/Money` + `app/Modules/Parties`):

- **`Money`** (`app/Platform/Money/Money.php`): immutable, integer `public readonly int $minorUnits` + `Currency $currency`; `of()`, `plus`/`minus` (same-currency, throws on mismatch), `negate`, `equals`, `toPayload()`. Negative amounts are valid. **`MoneyCast`** maps `'amount' => MoneyCast::class` to backing columns `amount_minor` (int) + `amount_currency` (ISO 4217), nullable-safe — its docblock already names "Club Credit balance" as its first real consumer.
- **The transition-Action idiom** (`ActivateProducer.php` / `LapseProfile.php` are verbatim templates): `DB::transaction` → `Model::query()->whereKey($id)->lockForUpdate()->firstOrFail()` → from-state guard throwing a localized `Illegal*Transition` → optional extra gate → `update([...])` → (for an evented transition) `record(...)`. The model stays persistence-only (`$guarded = []`); the **Action is the sole writer**.
- **The audit-only-write precedent**: `RecordKycVerified` writes `kyc_status` and records **no** KYC event (the audit trail is the record — activation design **L2**); the Account family (`SuspendAccount`/`CloseAccount`) likewise writes `status` and records **no** event (§15 names none).
- **The partial-unique-index DDL** (`2026_06_15_000007_create_parties_profiles_table.php`): `(customer_id, club_id) WHERE state NOT IN (…)` created with **raw `DB::statement('CREATE UNIQUE INDEX … WHERE …')` on BOTH engines** (not driver-guarded — it is the structural uniqueness guard; identical syntax on SQLite + PG).
- **`Club`** carries `generates_credit` (boolean, default true) and a nullable `fee` (`Money` via `MoneyCast` → `fee_minor`/`fee_currency`). **`Profile`** is the membership (Netflix model), `state` a `ProfileState` (born `Applied`); `Active`/`Suspended` among its nine states.
- i18n: only `lang/en/parties.php` exists (English baseline + per-key fallback — DEC-127). New copy goes there.

## Goals / Non-Goals

**Goals:**
- The `parties_club_credits` entity + the **one-active-Club-Credit-per-Profile** structural invariant (partial unique index).
- The full within-module FSM as four sole-writer Actions: `IssueClubCredit`, `ApplyClubCredit` (K.17 carry-forward), `ForfeitClubCredit`, `RestoreClubCredit` — each from-state guarded, transaction-locked, **audit-only (no domain event)**, tested directly.
- The §11.4 boundary asserted by a guard test: Parties records no `ClubCredit*`/`MembershipFeePaid` event and fabricates no such class.
- Every DB-touching task verified on PostgreSQL 17.

**Non-Goals (named seams — proposal boundary table):**
- The **Module-E** `MembershipFeePaid` listener (issuance/renewal trigger, payment-provider-confirmed) + the `ClubCreditIssued/Applied/Restored/Forfeited` consumers → Module E (Phase 6).
- **Module S** checkout redemption mechanics: price resolution, coupon mutual-exclusion (DEC-110), auto-apply (DEC-111), Hold-gated price resolution, and the Module-S eligibility **read contract**.
- The **forfeiture triggers**: year-end-lapse scheduler; renewal forfeit-before-issue orchestration; Profile-cancellation cascade; Club-closure → Discovery store-credit conversion (DEC-043, **owned by Module S** — AC-K-XM-23).
- **K.18** welcome-window scaling + **K.19** operator manual issuance (retained seams). The Filament console over these Actions.

## Decisions

### L1 — New `parties_club_credits` table; one-active invariant = partial unique index `(profile_id) WHERE state = 'active'`
Columns: `id`, `profile_id` (within-module FK → `parties_profiles`, `RESTRICT` on delete — the Profile is a shared parent, not owned by the credit), `amount_minor`/`amount_currency` (MoneyCast `amount`), `remaining_minor`/`remaining_currency` (MoneyCast `remaining`), `valid_from`/`valid_to` (`immutable_datetime`), `state` (`ClubCreditState`), `timestamps`. The invariant is enforced **structurally** by a partial unique index `parties_club_credits_one_active_per_profile ON parties_club_credits (profile_id) WHERE state = 'active'`, created with raw `DB::statement` on both engines (the Profile non-terminal-index precedent). A `redeemed`/`forfeited` credit frees the slot, so the next issuance inserts cleanly — no application-level "find existing and update" race. _Alternative rejected:_ an application-level uniqueness check — loses the structural guarantee under concurrency; the partial index is the same discipline the spine already uses.

### L2 — `amount`/`currency` source = `Club.fee` verbatim; `remaining` starts = `amount`; `valid_to` = end-of-issuance-year (interview Q2)
At launch K.18 (welcome-window scaling) is deferred → **full-fee → full-credit**, so the credit amount *is* the fee. `IssueClubCredit` sets `amount = Club.fee` (both `minorUnits` AND `currency` from the fee), `remaining = amount`, `valid_from = CarbonImmutable::now()`, `valid_to = CarbonImmutable::now()->endOfYear()` (31 Dec of issuance year — the PRD default; the per-Club credit-policy override for a custom `valid_to`/grace is a deferred seam, no new Club column now). This sidesteps the fee-currency-vs-`Account.default_currency` question: `Account.default_currency` is a *preference* string (not a money amount), there is **no FX in Module K**, and the credit is the fee made spendable — same currency by construction. The spec's "Account's currency at the moment of fee payment" (§11) and the fee currency are identical at full-fee issuance. **Fee-null guard:** a Club with `generates_credit = true` but `fee = null` cannot define an amount — `IssueClubCredit` rejects (a localized issuance-precondition exception), it does not issue a zero credit. _Alternative rejected:_ amount-minor-from-fee + currency-from-`Account.default_currency` with a mismatch reject — couples two entities and needs a mismatch rule for a case that cannot arise at full-fee issuance.

### L3 — Audit-only writers: Module K records state, emits NO domain event (interview Q1 / §11.4) — the headline decision
§11.4 is unambiguous: "Module K does NOT emit Club Credit lifecycle events. **Module E emits** `ClubCreditIssued`, `ClubCreditApplied`, `ClubCreditRestored`, `ClubCreditForfeited`; Module K consumes those events and records the resulting state on its Club Credit entity." `MembershipFeePaid` is likewise a Module E event (§15.8). So — exactly as `RecordKycVerified` writes `kyc_status` and records no KYC event (activation L2) — the four writers `update()` the credit `state`/`remaining` and record **no** domain event; the append-only audit trail + the entity state are the launch record. **No `MembershipFeePaid` or `ClubCredit*` event class is fabricated** (zero-invention; mirrors L5). When Module E lands, its `MembershipFeePaid` listener and `ClubCredit*` consumers invoke **these same within-module writers** — the split breaks nothing. _Alternative rejected (interview B):_ emit Module-K-owned `ClubCredit*` events now to populate the event store — fabricates/duplicates Module E's §11.4 contract and creates a double-emission problem when E lands.

### L4 — The four writers follow the transition-Action idiom; Actions are the sole state writers
`IssueClubCredit` / `ApplyClubCredit` / `ForfeitClubCredit` / `RestoreClubCredit` each inject only what they need (no `DomainEventRecorder` for the state write — there is no event; an `ActorContext` is carried only where an audit actor is wanted), run one `DB::transaction`, `lockForUpdate()` re-read the credit (and, for `Issue`/`Apply`, the Profile/Club), assert the from-state, then `update([...])`. The `ClubCredit` model is persistence-only (`$guarded = []`, casts + `@property`). A wrong-from-state call throws a localized `IllegalClubCreditTransition` **before any write**; the transaction rolls back leaving state, `remaining` and the audit log unchanged. _Alternative rejected:_ an FSM library — over-engineering; four linear guards mirror the dozen shipped Parties transition Actions.

### L5 — Full within-module FSM shipped + tested directly; triggers are seams; forfeit-before-issue is provable via the invariant (interview Q3)
All four writers ship and are tested **directly** (operator/seam-invoked now — exactly as `ActivateProfile`/`LapseProfile` ship transitions whose production triggers do not yet exist). The shipped Profile Actions are **not** re-touched: the Profile-cancellation→forfeit cascade, the renewal forfeit-before-issue orchestration, and the year-end scheduler are documented seams completed by named future changes. The **forfeit-before-issue ordering** (§11.3 renewal-replacement) is nonetheless exercised now: the one-active partial index makes `IssueClubCredit` reject when an `active` credit already exists, so the only way to re-issue is `ForfeitClubCredit` then `IssueClubCredit` — a tested pair that proves the ordering the future renewal listener will perform. _Alternative rejected (interview B):_ issuance+forfeiture only — leaves a half-built FSM and forces a near-future re-touch. _(interview C):_ wire the cascades into green Profile Actions now — re-touches shipped code + pulls in the per-Club grace-period policy.

### L6 — `ApplyClubCredit`: minor-unit decrement, K.17 carry-forward, currency-match + freeze guards
Apply takes an `amount` to redeem. Guards (all before the write): the credit is `active`; the redeemed amount's `currency` equals the credit currency (explicit equality check → localized exception, so `Money::minus`'s `InvalidArgumentException` can never surface); the redeemed `minorUnits` does not exceed `remaining->minorUnits` (no negative balance). Then `remaining = remaining->minus($applied)`: if `remaining->minorUnits === 0` → `state = redeemed`; if `> 0` → stays `active` (**K.17 carry-forward** — the balance carries to future purchases). **Freeze (AC-K-FSM-2a):** Apply re-reads the owning Profile `lockForUpdate()` and rejects if `Profile.state === Suspended` (no redemption while suspended — the within-module realization of "frozen while suspended", both entities Module K). The issuing-Club eligibility (`credit.profile.club_id ∈ offer.club_ids`) and currency-match are the **eligibility data Module K exposes**; the checkout *decision* (mutual-exclusion, the Hold-gated price resolution, Offer matching) is the Module-S seam. Issuance is **not** Hold-gated (the entitlement is recorded; only redemption gates — §11.2).

### L7 — `RestoreClubCredit`: `redeemed → active`, one-active-respecting
Restore (the order-cancellation downstream effect, §11) re-activates a `redeemed` credit and restores its `remaining`. Guards: the credit is `redeemed`; **no other `active` credit exists for the Profile** (the one-active invariant — if a replacement was already issued, restore is rejected with a localized exception rather than violating the partial index). The order-cancellation-window trigger is a **Module-S seam**; the writer ships + is tested directly.

### L8 — K.18 / K.19 deferred with retained seams
`IssueClubCredit`'s `amount = Club.fee` path **is** the K.18 seam: when welcome-window scaling restores, the `policy_amount × (fee_paid / full_fee)` formula slots in at this hook (PRD §11.1; AC-K-MVP-3). No operator manual-create surface is built (K.19): launch goodwill routes through the Module S `REFUND_COMPENSATION` coupon (AC-K-MVP-4); the writer itself is the retained manual-create seam. Both are documented in the Action docblocks + the spec delta.

### L9 — No protected-test realignment (unlike activation L8); add a new §11.4 guard test
`grep` over `ComplianceIndependenceTest` / `SupplyLifecycleChainTest` confirms **no** shipped forbidden-name list pre-names `ClubCredit`, `IssueClubCredit`, or the `ClubCredit*` events — the new names collide with nothing, so no existing guard test is narrowed. Instead this change **adds** a guard test asserting the §11.4 audit-only contract: no `app/Modules/Parties` code records a `ClubCredit*` or `MembershipFeePaid` domain event, and no such event class exists under Parties (the invent-no-event discipline, mirroring the `SupplyLifecycleChainTest` event-non-existence loop).

## Risks / Trade-offs

- **No `domain_events` row for Club Credit at launch (L3).** → Faithful to §11.4 (Module E is the financial event recorder; the entity state + audit trail is the launch record). When Module E lands it emits the financial events and its consumers drive these writers. Surfaced at the gate.
- **Writers ship with no production caller (L5).** Only operator/seam + direct tests invoke them now. → Mirrors `ActivateProfile`/`LapseProfile`; the FSM is proven by direct Pest tests, the triggers wire in later. No live commercial surface consumes Club Credit yet (Module S unbuilt).
- **One-active invariant ↔ Restore interaction (L7).** A `redeemed` credit cannot be restored once a replacement `active` credit exists. → Guarded explicitly (reject, not index-violate) + tested; at launch order-cancellation precedes re-issuance in practice.
- **`Club.fee = null` with `generates_credit = true` (L2).** → `IssueClubCredit` rejects (issuance-precondition exception); no zero/undefined credit. Tested.
- **Currency comparison via `minorUnits` (L6).** `Money` exposes no `<`/`>`; comparisons use the public `->minorUnits` after an explicit same-currency guard. → The guard makes `minus` total; the minor-unit compare is exact-integer (no float). Documented in the Apply task.

## Open Questions

None blocking. The three gate decisions are recorded above: **Q1** event model → audit-only writers, no Module-K event (L3); **Q2** amount/currency → `Club.fee` verbatim (L2); **Q3** FSM scope → full FSM + seams, green Profile Actions untouched (L5).
