# Design — parties-membership-activation

## Context

The three prior Parties slices shipped the spine (parties-core), the supply-side lifecycle (parties-producer-lifecycle), the compliance FSMs (parties-compliance) and the Hold registry (parties-holds), and each deferred the **demand-side status** FSMs in the shipped `party-registry` spec's *Birth States Recorded, Lifecycle Transitions Deferred* requirement. This change closes the **activation** subset: Customer `pending → active`, Profile `Applied → Approved | Rejected → Active`, and the Originating-Club one-shot lock (the `originating_club_id` no-mutation seam parties-core left). It is **additive** — one migration adding three nullable columns to `parties_customers`, four new Actions, three new events, two new exceptions — plus the narrowing of two archived-change guard tests. **No new dependency**, **no open ADR gate** (events deliver inline; no queue/object-storage/payment-provider/frontend).

The substrate this builds on (verified in `app/Platform/Events` + `app/Modules/Parties`):

- `DomainEventRecorder::record(string $name, string $module, ActorRole $actorRole, ?int $actorId, string $entityType, string $entityId, array $payload, ?string $correlationId = null, ?int $causationId = null): DomainEvent` — appends inside the caller's open transaction (throws at `transactionLevel() === 0`); inline delivery (no `queued` consumer → no ADR gate). A root event omits `correlationId`/`causationId` (defaults: `correlation_id` = own `event_id`, `causation_id` = null).
- `ActorContext::role(): ActorRole` / `actorId(): ?int` — the actor seam (`ActorRole::System` default; an authenticated operator → `ActorRole::NewcoOps`). `Module::Parties->value === 'parties'`.
- The transition-Action idiom (`app/Modules/Parties/Actions/ActivateProducer.php` is the verbatim template): `DB::transaction` → `Model::query()->whereKey($id)->lockForUpdate()->firstOrFail()` → from-state guard throwing a localized `Illegal*Transition` → optional extra gate → `update([...])` → `record(...)`, all in the transaction. Models stay persistence-only (`$guarded = []`); the **Action is the sole writer**. `RetireProducer` is the cross-entity-cascade precedent (writes Producer + Club in one tx); `RecordKycVerified` is the **audit-only-status-write + same-tx side-effect** precedent (writes `kyc_status`, records no KYC event, lifts the coupled Hold).
- `Profile` (`app/Modules/Parties/Models/Profile.php`): the state column is **`state`** (a `ProfileState` cast — values lowercase: `applied`/`approved`/`rejected`/`active`/…), born `Applied`; FKs `customer_id`/`club_id`; a **partial unique index** `(customer_id, club_id) WHERE state NOT IN ('rejected','cancelled','inactive')` (already excludes the terminal set — so re-application after a decline inserts cleanly with no index migration). `Customer` (`Customer.php`): `status` (a `CustomerStatus` cast, born `Pending`); `originating_club_id` (nullable FK, born NULL, no current mutation surface); `kyc_status`/`sanctions_status` additive-nullable (parties-compliance). `KycStatus::clears()` (true for `verified`/`not_required`), `SanctionsStatus::Passed`.
- The additive-migration idiom (`database/migrations/2026_06_17_000001_add_compliance_to_parties.php`): nullable columns, no backfill, reversible `down()`; casts + `@property` on the model.
- i18n: only `lang/en/parties.php` exists (English baseline + per-key fallback — DEC-127). New copy goes there only, in the existing `profile`/`customer` groups.

## Goals / Non-Goals

**Goals:**
- `ApproveProfile` / `DeclineProfile` (`Applied → Approved | Rejected`) — the one retained producer write (L-PP), audit-only, from-state guarded; operator/console-invocable.
- The Originating-Club one-shot lock — an in-tx side-effect of `ApproveProfile`, first-ever-approval gated, immutable, may-stay-unset; records `OriginatingClubLocked`.
- `ActivateProfile` (`Approved → Active`) → `ProfileActivated`; the `MembershipFeePaid` trigger a documented Module-E seam.
- `ActivateCustomer` (`pending → active`) behind the composite onboarding gate → `CustomerActivated`; explicit, not auto-driven.
- Three additive nullable acceptance-timestamp columns on `parties_customers` (the gate inputs).
- Every DB-touching task verified on PostgreSQL 17.

**Non-Goals (named seams — proposal boundary table):**
- The **Hero Package Capacity Invariant** at approval/activation (reads Module A `qty`) → `parties-hero-package`; approval + activation ship **uncapped**.
- `Applied → WaitingList` + `WaitingListJoined` → `parties-hero-package`. The **`MembershipFeePaid` listener** (Module E). **Club Credit** → `club-credit`.
- The **Hold→`suspended`** coupling, `Active → Suspended | Lapsed | Cancelled | Inactive`, Account status transitions → `parties-membership-suspension`. **Customer segments** → `parties-customer-segments`.
- The producer-facing HTTP surface + the Filament console over these Actions → `parties-operator-console` / producer portal.

## Decisions

### L1 — One additive migration on `parties_customers`; NO change to `parties_profiles`
The only schema change is three **nullable timestamp** columns on `parties_customers` — `email_verified_at`, `tc_accepted_at`, `privacy_accepted_at` (the `ActivateCustomer` gate inputs; § 4.1 "tracked at Customer level with timestamps"; DEC-071 additive). No backfill, reversible `down()`, `immutable_datetime` casts + `@property` on `Customer`. **`parties_profiles` is untouched**: the `ProfileState` domain already carries `Approved`/`Rejected`/`Active`, and the partial unique index already excludes the terminal set — making `Rejected` reachable merely *exercises* the pre-existing index. _Alternative rejected:_ a combined `terms_accepted_at` — § 4.1 names T&C **and** privacy as two acceptances; "timestamps" (plural) and the two distinct documents → two columns (DEC-073 realization). _Alternative rejected:_ modelling email-verification on a customer-auth table — none exists (Operators are the only auth principal), and § 4.1 captures verification "at Customer level"; the column is the faithful realization (DEC-073), set by the deferred registration surface or an operator.

### L2 — Approve/decline are AUDIT-ONLY; the approve path's only event is the conditional `OriginatingClubLocked`
§ 15.2's Profile event family names **no `ProfileApproved` / `ProfileRejected`** (and AC-K-FSM-2 lists no event for `Applied → Approved | Rejected`). So — exactly as `RecordKycVerified` writes `kyc_status` and records **no** KYC event (the audit trail is the record) — `ApproveProfile` and `DeclineProfile` write `state` and record **no** Profile event. The **sole** domain event the approve path may record is `OriginatingClubLocked`, and **only** on the first-ever approval. Inventing a `ProfileApproved`/`ProfileRejected` event is forbidden (it would coin a name the PRD does not name — the codebase's invent-no-event discipline; the guard test in L8 pins `ProfileApproved` as never-existing). _Alternative rejected:_ a `ProfileApproved` event "for symmetry" — not in § 15.2; vacuous invention.

### L3 — The Originating-Club lock is an in-tx side-effect of `ApproveProfile`, NOT a standalone Action
The lock fires inside `ApproveProfile`: after the `state = Approved` write, re-read the **Customer** `lockForUpdate()`; if `originating_club_id === null`, set it to the Profile's `club_id` and `record(OriginatingClubLocked, entityType: 'Customer', entityId: (string) $customer->id, payload: {customer_id, club_id, profile_id, locked_at})` in the same transaction. The NULL-gate makes it **idempotent** (a second Club's approval finds it set → no write, no event) and **immutable** (no other Action writes the column — pinned by the guard in L8). There is **no** `LockOriginatingClub` / `SetOriginatingClub` Action (both are in the codebase's forbidden-Action list and stay there — L8). The lock is a Customer-state write inside a Profile-transition Action — a within-module cross-entity write, exactly the `RetireProducer` (Producer + Club) pattern; both rows are locked in the one transaction. `OriginatingClubLocked` is a **root** event (the approval records no parent). _Alternative rejected:_ a dedicated lock Action invoked by `ApproveProfile` — the forbidden-list names it as never-existing, and the lock has no caller other than approval; an in-tx side-effect is the zero-surface realization.

### L4 — Actions are the sole writers; mirror `ActivateProducer`; `update()` the locked row (never insert)
`ApproveProfile`/`DeclineProfile`/`ActivateProfile`/`ActivateCustomer` each follow the `ActivateProducer` shape (inject `DomainEventRecorder` + `ActorContext`; one `DB::transaction`; `lockForUpdate` re-read; from-state guard → localized exception; `update([...])`; `record(...)` for the evented transitions). Because `Approved` and `Active` are **non-terminal** (in the partial unique index), the transitions are `update()` on the **existing locked row** — never a new insert (the only new-row path is re-application after a *terminal* decline, which the index admits). The `Profile`/`Customer` models stay persistence-only. _Alternative rejected:_ an FSM-library — over-engineering; the four linear guards mirror the eight shipped transition Actions.

### L5 — `ActivateProfile`: the `MembershipFeePaid` listener is a deferred Module-E seam
§ 4.2.1 drives `Approved → Active` from Module E's `MembershipFeePaid` (or a free-club, no-fee activation); Module E does not exist. `ActivateProfile` ships as the within-module **writer** of the transition (records `ProfileActivated`), invoked by the free-club / operator path now and directly in tests. The cross-module listener that calls it on `MembershipFeePaid` is a **Module-E seam** — **no** `MembershipFeePaid` event class is fabricated (zero-invention; § 15.2: "Module K consumes Module E's `MembershipFeePaid`"). This mirrors `ActivateProducer` shipping the transition with its upstream KYC gate as a seam. Club Credit (also `MembershipFeePaid`-coupled) is an independent consumer in `club-credit`; `ProfileActivated` provisions no credit inline, so the split breaks nothing.

### L6 — `ActivateCustomer`: explicit composite gate; never auto-driven from another FSM
The gate is the conjunction (AC-K-J-1 four gates + AC-K-BR-Identity-3 KYC rider): `email_verified_at` set ∧ `tc_accepted_at` set ∧ `privacy_accepted_at` set ∧ `sanctions_status === Passed` ∧ (`kyc_required` ⇒ `kyc_status` cleared — reuse `KycStatus::clears()`, NULL treated as cleared per DEC-071). `ActivateCustomer` is an **explicit** Action: it is **not** auto-fired from `RecordCustomerScreening` or any KYC Action — the status FSM is independent of the KYC/sanctions FSMs (§ 9.4; AC-K-BR-Customer-1 "activation explicit, not auto-driven"), and auto-firing would force editing the shipped `RecordCustomerScreening` and break the shipped `ComplianceIndependenceTest` "screening performs no status transition" assertion. Customer activation performs **no Account transition** (Account is born `active` — AC-K-FSM-9). A gate-unmet or wrong-from-state call throws `IllegalCustomerTransition`. _Alternative rejected:_ auto-activate on sanctions-pass (a literal reading of § 7.1 step 5) — § 7.1 is the happy-path narrative of the § 4.1 **conjunctive** gate; auto-firing couples two deliberately-separate FSMs.

### L7 — The §13 Hero Package capacity gate is a deferred Module-A seam on BOTH approval and activation
§ 13 mandates the capacity invariant "at every membership approval" (§ 13 header) and "at every Profile transition into `Active`" (§ 13.1); § 13.2 / AC-K-XM-18 put the cap on **Module A's Hero-Package Allocation `qty`**. Module A is unbuilt, so the gate cannot read it without inventing A's contract. `ApproveProfile` **and** `ActivateProfile` therefore ship **uncapped**, each carrying an explicit deferred-seam docblock — the same discipline as `ActivateProducer` (ships ungated, "compliance tightens it later") and `CloseClub` (ships without the all-members-gone gate, "the demand-side change tightens it"). The seam is documented in the spec delta, both Action docblocks, and `CONTEXT.md`. Without this explicit framing the change reads as §13-non-compliant — this is the load-bearing wording item for the gate.

### L8 — Guard-test realignment: narrow two archived-change protected tests (the names are pre-fixed by them)
Two shipped guard tests assert the **absence** of precisely this slice's classes; the codebase **pre-named** the Actions in their forbidden lists (zero-invention naming):
- `tests/Feature/Modules/Parties/ComplianceIndependenceTest.php` — the forbidden-Action list (≈ L149-156) names `ActivateCustomer, SuspendCustomer, CloseCustomer, ActivateAccount, SuspendAccount, CloseAccount, ApproveProfile, ActivateProfile, DeclineProfile, LockOriginatingClub, SetOriginatingClub`. Remove **only** the four now-shipped (`ActivateCustomer`, `ApproveProfile`, `ActivateProfile`, `DeclineProfile`); **keep the rest** — `SuspendCustomer`/`CloseCustomer`/`ActivateAccount`/`SuspendAccount`/`CloseAccount` (suspension slice) and **`LockOriginatingClub`/`SetOriginatingClub`** (the lock stays an in-tx side-effect — L3). The `originating_club_id` write-scan (≈ L168-177) asserts only `CreateCustomer` writes the column — **admit `ApproveProfile`'s conditional write** (its own comment already anticipates "the one-shot OriginatingClubLocked write arrives with the deferred membership-approval change").
- `tests/Feature/Modules/Parties/SupplyLifecycleChainTest.php` — the exact-set transition whitelist (≈ L322-323) must **add** the four new Actions to the admitted set; the event-non-existence loop (≈ L331-333) names `CustomerActivated, AccountActivated, ProfileActivated, ProfileApproved, OriginatingClubLocked, CustomerSegmentChanged` — remove **only** the three now-shipped (`CustomerActivated`, `ProfileActivated`, `OriginatingClubLocked`); **keep** `AccountActivated` (suspension slice), **`ProfileApproved`** (the audit-only proof — L2) and `CustomerSegmentChanged` (segments slice).

The blast radius is grep-derived, not handoff-trusted (`grep -rn 'ApproveProfile\|ActivateProfile\|ActivateCustomer\|DeclineProfile\|OriginatingClubLocked\|ProfileActivated\|CustomerActivated' tests/` before editing). The other five guard files (`SpineCreationChainTest`, `ComplianceChainTest`, `CustomerTest`, `HoldChainTest`, `HoldRegistryTest`) assert demand-side events are not recorded in **their** flows — those flows trigger no approval/activation, so they stay green unamended; the full-suite step confirms.

### L9 — Three events, verbatim names, PII-free, root
`CustomerActivated` (`ENTITY_TYPE = 'Customer'`, payload `{customer_id, status}`), `ProfileActivated` (`'Profile'`, `{profile_id, state}`), `OriginatingClubLocked` (`'Customer'`, `{customer_id, club_id, profile_id, locked_at}` — § 6.1 verbatim). Each a `final` class with an untyped `const NAME` (verbatim § 15 name), `const ENTITY_TYPE`, and a static `payload(...): array` (enum→`->value`; ids + business values only; no name/email/phone/DOB) — the `ProducerActivated` shape. All three are **root** events. _Alternative rejected:_ threading `OriginatingClubLocked`'s causation to an approval event — approval records no event, so there is no parent; it is a clean root.

## Risks / Trade-offs

- **Email-verification column with no production setter (L1).** `email_verified_at` (and the two acceptance columns) have no writer in this slice — the consumer registration surface (which sets them) is deferred. They are operator/test-set now. This is identical to how T&C/privacy work and is the additive-seam pattern (the column exists; the flow that fills it lands later). Surfaced at the gate.
- **Shipping the activation transition uncapped (L7).** A Profile can be approved/activated with no Hero Package capacity ceiling until `parties-hero-package` lands. Mitigated: the seam is explicit in the spec + both Action docblocks + `CONTEXT.md`; it mirrors two shipped precedents (`ActivateProducer`, `CloseClub`). No live commercial surface consumes membership capacity yet (Module S storefront is unbuilt).
- **Editing two shipped, protected guard tests (L8).** The realignment is narrowing (remove the now-true negatives), not deletion — the still-deferred names stay asserted-absent. A reviewer must read precisely: the slice un-forbids exactly four Actions + three events and keeps every other demand-side name forbidden. The full Parties suite on SQLite + PG17 is the backstop.
- **Splitting the `MembershipFeePaid` consumer across slices (L5).** `ProfileActivated` (here) and Club Credit issuance (`club-credit`) both key off `MembershipFeePaid`. Because the orchestrating listener is a deferred Module-E seam, neither fires from a real signal yet; the eventual listener will invoke both. No ordering coupling is broken.
- **Cross-entity write in `ApproveProfile` (L3).** Writing `Customer.originating_club_id` from a Profile-transition Action locks two rows. Mitigated: both within Module K (no boundary breach), both `lockForUpdate` in one transaction, the NULL-gate idempotent — the `RetireProducer` cascade precedent.

## Open Questions

None blocking. Two items surfaced at the gate (both resolved from spec/AC, recorded here for the reviewer): (1) the email-verification **column shape** is a DEC-073 dev-team realization of the AC-K-J-1 / AC-K-BR-Identity-3 gate condition — modelled as `email_verified_at` on `parties_customers`; (2) the **§13 capacity gate** ships as a deferred Module-A seam (L7) — the only zero-invention option while Module A is unbuilt, framed exactly as the `ActivateProducer`/`CloseClub` precedents.
