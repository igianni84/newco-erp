# Proposal: catalog-module-0-completeness-sweep

## Why

The Module 0 validation verdict (`docs/validation/Module_0_Verdict_v0.3-MVP.md`) leaves four launch-scope build-completeness holes open — the "P3 sweep" of the Remediation Tracker (RM-12/13/14/15, batched per Giovanni's 2026-07-08 scope decision): the Layer-1 possible-case-configurations whitelist does not exist (`AC-0-J-13` Fail/Gap, `AC-0-XM-11` Partial — "PIM's only contribution to layered breakability … real gap, not a deferral"), `EnrichmentDataUpdated` is the 1 missing event of 22 (`AC-0-EVT-8` Fail/Gap), re-versioning on identity edit is unbuilt (`AC-0-BR-Audit-1` Partial — "`version` stays 1; no Update action / Edit page"), and Product Master creation accepts any `producer_id` (`AC-0-XM-2` Partial — "Literal divergence"). All four criteria are **frozen-spec** (none is canon-new — no mini-ADR gate). RM-13/14 are design-coupled: both introduce the catalog's first edit paths, so the RM-06 **S1 forward fix** (semantic-verify 2026-07-02) folds in mandatorily — `ApprovalGovernance::assertNotRejectionPending` reads the *raw* latest catalog audit action, safe only while `LifecycleTransition` is the sole catalog audit writer; the first edit-audit row would clear the review-freshness block **without a re-submit**.

## What Changes

- **RM-12 — Layer-1 whitelist (J-13, XM-11):** new per-(Product Variant, Format) whitelist of admissible Case Configurations (pivot table, empty = permissive per PRD §7.1); operator-editable at any non-retired Variant state **including `active`** (the J-13 core); enforcement at **Sellable SKU activation** (CC ∉ non-empty whitelist ⇒ activation rejected); reductions never invalidate existing SKUs/whitelist-independent state retroactively. Whitelist edits are audit-only (no version bump, no review re-arm, no domain event — the event set stays closed).
- **RM-13 — Enrichment edit + event (EVT-8):** `EnrichmentDataUpdated` event class + `UpdateProductVariantEnrichment` action over the Variant's observational enrichment (today: `tasting_notes`; field-agnostic for the future adapter columns). Mutable in `draft`/`reviewed`/`active` (blocked on `retired`); fires **only on a real change**; audited with before/after; **no** version bump, **no** review re-arm ("observational edits don't gate").
- **RM-14 — Identity edit + re-versioning (BR-Audit-1):** in-place edit + `version`++ + full before/after audit (old version retrievable from the append-only audit trail — the physical shape decision, DEC-073). Scope: **Product Master identity** (name, appellation, region, winery_story — all review-governed; dedup BR-Identity-1 re-checked when name/appellation change) and **Composite SKU constituent composition** (ordered set replace, N ≥ 2 re-checked; on an `active` Composite every new constituent PR must be `active` — cascade re-assert). Per-state: `draft` free, `reviewed` **re-arms review** (activation blocked until explicit re-submit — the RM-06-deferred DEC-019 leg), `active` stays active, `retired` rejected. No domain event (the 22-event set is closed).
- **S1 fold-in (mandatory):** `assertNotRejectionPending` filtered to review-freshness-relevant audit verbs — among actions ending `.submitted`/`.resubmitted`/`.rejected`/`.identity_updated`, activation is blocked iff the latest is `.rejected` **or** `.identity_updated`. Enrichment/whitelist edit rows never unblock (or block) review-freshness.
- **RM-15 — Producer existence at creation (XM-2):** the Catalog producer projection **widens**: `ProducerLifecycleProjector` also consumes `ProducerCreated` → new `ProducerProjectionStatus::Registered`; `CreateProductMaster` rejects a `producer_id` with **no projection row**. The activation gate is unchanged (still requires `active`, fail-closed).
- **Console:** modal header actions on the existing View pages (kit pattern, no Edit pages): `edit identity` on ViewProductMaster, `edit enrichment` + `manage whitelist` on ViewProductVariant, `edit composition` on ViewCompositeSku; producer-unknown surfaced as a create-form validation error; "deliberately NO Edit page" comment rationales updated (update Actions now exist; the no-Edit-PAGE stance stands).
- **Truth-spec sync fold-in (F4 precedent):** the operator-console spec never recorded the console re-submit RM-06 already shipped (grep: 12× in product-catalog, 0× in operator-console) — the two console lifecycle requirements gain the shipped re-submit affordance.

## Capabilities

### New Capabilities

*(none — both touched capabilities exist)*

### Modified Capabilities

- `product-catalog`: **MODIFIED** *Approval Governance* (review-freshness verb filter + identity-edit re-arm), *Product Master* (creation rejects a `producer_id` unknown to the projection), *Producer-State Projection and Event Consumption* (consumes `ProducerCreated`; three-status projection), *Producer Activation Gate* (stale "no projection row = never activated" parenthetical corrected; `registered` is not gated open); **ADDED** *Layer-1 Case-Configuration Whitelist*, *Identity Edit and Re-Versioning*, *Enrichment Data Update*.
- `operator-console`: **MODIFIED** *Operator creates a Product Master through the console* (producer-unknown validation error), *Operator advances a Product Master through the review-and-approval lifecycle* + *Operator advances each catalog spine entity through the review-and-approval lifecycle* (shipped re-submit affordance sync + stale-edit block surfacing); **ADDED** *Operator edits catalog identity content through the console*, *Operator maintains Variant enrichment and the Layer-1 whitelist through the console*.

## Impact

- **Code:** `app/Modules/Catalog/` — 1 new migration (whitelist pivot), 1 enum case (`ProducerProjectionStatus::Registered` — `EnumsTest` count 2→3), new Actions (`UpdateProductMasterIdentity`, `UpdateCompositeSkuComposition`, `UpdateProductVariantEnrichment`, `SetVariantCaseWhitelist`), 1 new event (`EnrichmentDataUpdated`), new exceptions + `lang/{en,it}` keys, `ApprovalGovernance` refactor (S1), `ActivateSellableSku` gate extension, `ProducerLifecycleProjector` + `CreateProductMaster` guards; `app/Modules/OperatorPanel/Filament/` — 4 modal actions on 3 View pages + create-Master validation surfacing.
- **Blast radius (verify at build):** every `CreateProductMaster` caller (tests, `DemoSeeder`, console) must reference a projected producer once the RM-15 guard lands — classify per the 2026-07-06 floor-wiring lesson; `EnumsTest` pins the projection enum count.
- **Docs:** `CONTEXT.md` glossary — *Producer-state projection* entry updated (three statuses, `ProducerCreated`), *Approval governance* entry updated (edit re-arm), new *Layer-1 whitelist* / *Enrichment data* / *Identity edit (re-versioning)* entries; console comment rationales.
- **Deliberately NOT in this change:** bulk import (RM-11, deferred-by-choice); `critic_scores`/`market_data` columns (land with the enrichment-adapter change; the edit path is field-agnostic); identity edits on Variant/Format/Case Configuration/Sellable SKU commercial fields (future change if needed); PR composition edit (immutable by design — BR-Identity-3/4: a new composition is a new PR); Module A Layer-2 upper-bound check (Module A seam, PRD §7.2); Module S `EnrichmentDataUpdated` consumer (§14.5 seam); version-row physical model (rejected — see design D1); customer/producer-facing surfaces.
