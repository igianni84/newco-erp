## Context

The compliance-screening columns already exist (`enhanced_kyc_flag`, `enhanced_kyc_at`, `sanctions_status`, `last_screening_at`, `next_rescreen_at`, `screening_trigger_source`) as additive-nullable seams (migration `2026_06_17_000001_add_compliance_to_parties`, DEC-071), and `RecordCustomerScreening` is the **sole writer** of the sanctions fields + the single writer of the four § 15.6 screening events. `ScreeningTriggerSource::AmlThreshold` and `CustomerRescreening{Passed,Failed}` are defined. What is missing — and what the truth spec explicitly parks as *"deferred; only the fields ship"* (`Customer KYC Lifecycle`) and *"the AML-threshold auto re-screen … deferred"* (`Customer Sanctions Screening Lifecycle`) — is the **detection** that sets the flag, the **Compliance review-queue**, and the **AML-threshold re-screen**. RM-02 lands them.

The load-bearing constraint is that both the €10k-single and the €50k-cumulative signals read **transaction/spend data that lives in Module S (Commerce)**, which is a stub (Build Workplan Phase 4). Module K is boundary-forbidden from reading Module S tables (invariant 10). So the detection *workflow* is 100% Module-K-buildable and testable now, but its *data source* is a deferred seam — exactly the shape RM-01 used for its unbuilt downstream consumer, and the `HoldType` DEC-008 finance-driven types (classification + guard ship now, the Module-E consumer unwired).

DEC-030's E6-10 clarification is the authoritative build sentence: an AML-threshold breach fires **(a)** the lightweight DEC-030 sanctions re-check **and (b)** the enhanced-KYC review-queue entry. `AC-K-J-7a` (facet b + the flag) and `AC-K-EVT-12a` (facet a) are two faces of **one** detection event.

## Goals / Non-Goals

**Goals:**
- Land the €10k-single / €50k-cumulative (independent OR) enhanced-KYC detection, idempotently, via one workflow driven by a periodic scan now and an at-order-completion seam later.
- Create a Compliance review-queue entry + a PII-free `CustomerEnhancedKycReviewRequired` event on a breach.
- Initiate the lightweight AML re-screen (`under_review` + `trigger_source = aml_threshold`) through the existing sole sanctions-writer, blocking the Customer until Compliance resolves it.
- Keep everything Postgres-truthful + SQLite-green, money as integer minor units, no hardcoded strings, no cross-module DB access.

**Non-Goals** (see the proposal slice-boundary table):
- The real Module-S totals source + the at-order-completion trigger wiring (seams); the 12-month cadence job; the screening-vendor adapter; the enhanced-KYC document FSM; country-change detection; the review-queue resolve action; the Module-S order-completion gate. The Compliance ad-hoc re-screen already ships (not touched).

## Decisions

### D1 — One idempotent workflow, latched on `enhanced_kyc_flag`; two trigger paths call it
`EvaluateEnhancedKycThreshold::handle(int $customerId)` is the single unit. It reads the Customer's totals through the port (D4), and — inside one `DB::transaction` against a `lockForUpdate()` re-read — if `enhanced_kyc_flag` is not already set AND (largest single ≥ €10k OR rolling-12-month cumulative ≥ €50k), it performs the full escalation (D2/D5/D6); otherwise it is a no-op. The flag is the idempotency latch: the escalation fires **at most once** per Customer, so a daily re-scan of a still-above-threshold Customer changes nothing, and a periodic/order-completion race resolves on the row lock. The periodic job (D7) and the future at-order-completion trigger both invoke this one method → "both paths, identical state" (`AC-K-J-7a`) holds *by construction*. _Alternative rejected:_ per-path duplicated detection logic — divergent state and double-escalation risk.

### D2 — AML re-screen = `under_review` via `RecordCustomerScreening` (sole writer), which blocks
On a breach the workflow calls `RecordCustomerScreening($id, SanctionsStatus::UnderReview, ScreeningTriggerSource::AmlThreshold)`. That action already supports the call unchanged: only the `onboarding` source is from-state guarded, and `under_review` is **not** a completion so it records **no** screening event. The effect is `sanctions_status = under_review` + `trigger_source = aml_threshold` stamped on the screening record, which makes `ComplianceStatus::isClear()` false — **the Customer is blocked from transacting until Compliance resolves the re-screen** (operator posture chosen 2026-07-02: enhanced due diligence blocks until cleared, over the softer flag-and-queue-only option). The resolution runs through the existing `ViewCustomer` re-screen action and records the matching `CustomerRescreening{Passed,Failed}` — the "same outcome events as the cadence path". The sole-writer invariant is preserved: the workflow never writes a sanctions field directly. _Alternative rejected:_ queue-only, sanctions untouched (no auto-block) — softer UX but does not enforce enhanced-diligence-before-transact; Giovanni chose the blocking posture.

### D3 — Cumulative window = rolling trailing 12 months
The €50k cumulative figure is a **rolling trailing-12-month** total from the evaluation instant (standard AML; operator call 2026-07-02), not calendar-YTD. This is a semantic term of the `CustomerTransactionTotalsReader` contract (D4) that the future Module-S implementation honours; the K-side detection only compares the returned figure to €50k. _Alternative rejected:_ calendar year-to-date (resets abruptly at year boundary — a customer at €49k on Dec-31 drops to €0 on Jan-1).

### D4 — Totals via a within-module read-port + a null adapter (the Module-S seam)
`CustomerTransactionTotalsReader::forCustomer(int): CustomerTransactionTotals` returns two `App\Platform\Money\Money` (EUR): `largestSingleTransaction` and `trailingTwelveMonthCumulative`. At launch the interface is bound in `PartiesServiceProvider` to `NullCustomerTransactionTotalsReader` returning `Money::of(0, Currency::of('EUR'))` for both — so the periodic scan runs and detects nothing until Module S provides the real adapter (reading completed-order EUR history, respecting the dual-currency EUR-side rule). The detection logic is fully tested by binding a **fake** reader in tests. This crosses the module boundary the sanctioned way — a small read contract, never a cross-module Eloquent query (invariant 10; the RM-01 export-manifest precedent). _Alternative rejected:_ a K-side projection table fed by Module-S events — heavier (a consumer + a table) for no launch benefit while Module S emits nothing; the port defers the same seam more cheaply.

### D5 — Add a PII-free `CustomerEnhancedKycReviewRequired` event over frozen-catalog silence
The frozen § 15.6 names no enhanced-KYC event, yet the tracker's RM-02 scope names an *"`enhanced-kyc` review event"*. We add one — payload = `customer_id`, `enhanced_kyc_at`, `threshold_kind`, and the tripping amount (minor units + currency), **no PII** — because (a) it is the module's emit-on-state-change pattern, (b) it is the audit anchor for a compliance-floor state change in the 10-year store, and (c) it is the seam a future Compliance dashboard / Module-E consumer reads. This mirrors RM-01's `CustomerAnonymised` decision (a PII-free event added over the frozen spec's event-free anonymisation). _Alternative rejected:_ event-free — breaks the emit-on-state-change pattern and leaves the breach with no durable audit/seam once the Customer's `trigger_source` column later moves off `aml_threshold` (D2).

### D6 — Review-queue = a flag+timestamp record, no FSM
`parties_compliance_reviews` carries `customer_id` (within-module FK), `reason` (`ComplianceReviewReason::EnhancedKycThreshold`), `threshold_kind` (`ThresholdKind::{SingleTransaction,CumulativeAnnual}`), the tripping amount (`tripped_amount_minor` bigint + `tripped_currency`), and `resolved_at` (nullable; NULL = open). Open/resolved is **boolean-derivable** (`resolved_at IS NOT NULL`), not an enum FSM — § 9.1 says enhanced-KYC is handled operationally with *no* state machine (mirrors the `anonymised_at` flag pattern). Writer = `CreateComplianceReview` (a `Create*` action, so auto-allowed by the arch whitelist). If both thresholds trip on the same scan, `single_transaction` is recorded (the more acute signal). _Alternative rejected:_ an `open/resolved` status enum + transition action — an FSM the spec explicitly refuses, and scope the resolve action defers.

### D7 — Periodic job = a scheduled command running inline (a scheduler tick, not queued)
`ScanEnhancedKycThresholds` (`parties:scan-enhanced-kyc-thresholds`) iterates Customers and calls `EvaluateEnhancedKycThreshold`, registered in `bootstrap/app.php` `withCommands()` and scheduled `->daily()` in `routes/console.php` (the `events:sweep` precedent). It runs **inline on the scheduler tick** — `decisions/2026-06-12-event-substrate-and-audit-store.md` line 66 explicitly classifies scheduler ticks as *not* a queued/async workflow, so the **queue-driver gate is not tripped** (confirming the tracker's ADR? = "—"). _Alternative rejected:_ a `queued` job — would approach the undecided queue-driver gate and force a premature ADR for no launch benefit (null totals ⇒ the scan is a no-op anyway).

### D8 — Thresholds as EUR minor-unit constants
€10,000 = `1_000_000` and €50,000 = `5_000_000` minor units, compared as `Money` in EUR (invariant 6; no floats). Boundary is inclusive (`≥`) — a compliance floor triggers *at* the threshold. Held as named constants beside the workflow.

## Risks / Trade-offs

- **Auto-block of high-value customers (D2).** Crossing €10k moves `sanctions_status → under_review`, blocking further transactions until Compliance clears them → *Mitigation:* the latch (D1) fires the block exactly once; resolution uses the existing console re-screen action; documented as the deliberately-chosen stricter AML posture (Giovanni, 2026-07-02).
- **The resolution's `trigger_source` reverts to `compliance_ad_hoc`.** The `ViewCustomer` re-screen offers only `compliance_ad_hoc` once `last_screening_at` is set (it never offers `aml_threshold` — § 9.5), so an operator resolving the AML-initiated `under_review` records `CustomerRescreening*` with `trigger_source = compliance_ad_hoc`, overwriting the Customer column → *Mitigation:* the AML-threshold origin is **durably** recorded on the review-queue entry + the `CustomerEnhancedKycReviewRequired` event; `AC-K-EVT-12a` asserts `trigger_source = aml_threshold` **at breach** (the `under_review` write satisfies it) and "outcome events identical across sources" (the resolution's `CustomerRescreening*` satisfies it). The implementer MUST NOT try to force `aml_threshold` onto the resolution.
- **Nested transaction.** `EvaluateEnhancedKycThreshold` opens a `DB::transaction`, records the `CustomerEnhancedKycReviewRequired` event (recorder requires level ≥ 1 — satisfied), and calls `RecordCustomerScreening` (which opens its own `DB::transaction` → a PG savepoint; its `under_review` path records no event) → *Mitigation:* whole escalation commits/rolls back atomically; prove with `DatabaseMigrations` + a rollback test.
- **Never-screened Customer flagged.** A Customer with `sanctions_status = NULL` crossing €10k gets `under_review` + `aml_threshold` (only the `onboarding` source is guarded) → *Mitigation:* admissible and correct (it flags them for review); an edge case documented in the workflow.
- **PG bigint-as-string.** The `tripped_amount_minor` / totals reads back as a numeric string on PostgreSQL → *Mitigation:* assert DB scalars with `->toEqual`, never `->toBe` (testing rule); cast the column to `integer`.

## Migration Plan

- One additive migration: `create_parties_compliance_reviews_table` — `id`, `customer_id` FK → `parties_customers`, `reason` (string + enum cast), `threshold_kind` (string + enum cast), `tripped_amount_minor` (bigint), `tripped_currency` (string, 3), `resolved_at` (nullable `timestampTz`), `timestampsTz`. Postgres-truthful + SQLite-compatible; PG-only value-set CHECKs for `reason` / `threshold_kind` derived from `Enum::cases()` (the `add_compliance_to_parties` idiom); explicit **short** index name for the `customer_id` index (PG 63-char limit). No backfill (additive). Dev-only `down()`.
- The `enhanced_kyc_flag` / `enhanced_kyc_at` columns already exist — no migration for them.
