- **Non-relation table with per-row actions on a `ViewRecord` page = a `Filament\Widgets\TableWidget` footer widget** (the console's first per-row-action vehicle, pinned against installed Filament 5.6.7). A `RelationManager` can't host it (the source has no Eloquent relation — `Hold.scope_id` is polymorphic, no FK). `TableWidget` (`vendor/filament/widgets/src/TableWidget.php`) extends `Widget` with `InteractsWithTable`; override `public function table(Table $table): Table` (the override point is `InteractsWithTable::table()`, l.169). Verified v5 API names: `Table::query(Builder|Closure|null)`, `->columns(array)`, **`->recordActions(array<Action|ActionGroup>)`** — `recordActions()` is the v5 rename of v3's `actions()`, and its `Action` is the unified **`Filament\Actions\Action`**. Lazy: `protected static bool $isLazy = false` overrides `CanBeLazy`'s `true` so the table renders inline (observable in the host page's initial render). `columnSpan` PHPDoc must mirror `Widget::$columnSpan` exactly (`int|string|array<string, int|null>`) for phpstan property covariance.
- **Hosting a widget on a resource `ViewRecord` = `getFooterWidgets()` returning `WidgetClass::make(['record' => $this->getRecord()])`.** `Widget::make()` returns a `Filament\Widgets\WidgetConfiguration`. A resource `ViewRecord` does NOT auto-inject its `record` into widgets (base `Page::getWidgetData()` is `[]`) — pass it explicitly; the widget declares a public model property (Filament's Eloquent synth rehydrates the concrete model across Livewire requests). **Type that property to the CONCRETE model (`public ?Customer $record`) once the widget reads the host's relations, not just its key** — the synth keys off the runtime value being a `Model`, not the declared type, so `?Customer` rehydrates identically while letting phpstan resolve `$record->account` / `$record->profiles` without an `instanceof` narrowing dance (1.2 used the generic `?Model` when only `getKey()` was needed). Placing the widget under `…/{Resource}/Widgets/` (NOT `Filament/Widgets/`) keeps it out of the panel's `discoverWidgets(in: app_path('Modules/OperatorPanel/Filament/Widgets'))` sweep — it renders only where the page mounts it, never as a dashboard widget.
- **Non-relation scope-set table query = the `DatabaseComplianceStatusReader` OR-of-scopes idiom.** A Hold "belongs to a Customer" across THREE polymorphic scopes with no FK: `Hold::query()->where(fn (Builder $q) => $q->orWhere(customer-scope)->orWhere(account-scope)->orWhere(profile-scope))`, each group a `function (Builder $inner)` closure matching `(scope_type, scope_id)` (`Illuminate\Database\Eloquent\Builder`). Account is a nullable `hasOne` → `$record?->account?->id` (correct `?->`, no `nullsafe.neverNull`); a profile-less Customer → empty `whereIn('scope_id', [])` matches nothing (Laravel emits `0 = 1`); `where('scope_id', null)` becomes `whereNull` (harmless for the always-present Account). Render cast enum columns import-free via `getStateUsing(fn (Hold $r) => $r->getAttribute($attr) instanceof BackedEnum ? (string) $state->value : '')` (the resource's `enumBadgeState` twin) — state enums (`HoldStatus`) stay cast-only. Composite actor columns (`role #id`) read the typed enum property directly (`$r->placed_actor_role->value`) — phpstan resolves the type via the model's `@property`, no import.

- **Bespoke FORM-bearing header action (multi-operand) = `Action::make(id)->label(…)->schema([...])->action(…)`, NOT the kit's `lifecycleAction()`.** `SurfacesDomainActions::lifecycleAction()` threads only a single `notes` string — any verb needing ≥2 operands (place-Hold's type / scope / profile) is built bespoke in the page, added to `getHeaderActions()` alongside the form-less verbs. Form components are `Filament\Forms\Components\{Select,Textarea}` (both extend `Field`); `Action::schema([...])` is the canonical Filament 5 form attach (`->form()` just delegates to `->schema()`; `getForm()` is deprecated for `getSchema()`). **Operand-enum Select options** = value→value map `collect(Enum::cases())->mapWithKeys(fn ($c) => [$c->value => $c->value])->all()` (the resource's currency/locale-code precedent — domain token, no per-value i18n key); the `Parties\Enums` import is the namespace-prefix carve-out (ADR 2026-06-21), `ModuleBoundariesTest` needs NO widening. **Conditional field visibility** (console's FIRST — no prior `->live()` app-wide): the controller Select is `->live()`, the dependent field `->visible(fn (Get $get): bool => $get('controller') === Enum::Case->value)` (`Get` = `Filament\Schemas\Components\Utilities\Get`). Record-dependent options (`profile_id` over `$record->profiles`) pass a Closure `->options(fn () => $this->…($this->getRecord()))` so they resolve at render time (record loaded), not at build.
- **Bespoke-action WRITE-THROUGH (multi-operand) = `->action(function (array $data): void { …narrow… $this->surfaceLifecycleOutcome(fn () => app(DomainAction::class)->handle(…), $successTitle); })`** — REUSES the kit's `surfaceLifecycleOutcome()` for the uniform success / base-`RuntimeException`→`action_failed` notification (do NOT fork the trait for one caller — design D3). Filament injects `array $data` (the form state) by parameter name into the action closure, exactly like `lifecycleAction`'s `(Model $record, array $data)`; the closure reads the page record via `$this->getRecord()` (no need to also inject `$record`). **Narrow EVERY operand with `is_string` before use** — Filament Select values serialize to strings; `$x = is_string($data['k'] ?? null) ? $data['k'] : '<floor>'` (the trait's `$notes` idiom). Construct domain enums INSIDE/around the closure with `Enum::from($narrowed)` (the `CreateClub`→`ClubRegistrationFlowType` precedent); parse a `->live()`-gated enum ONCE (`$holdScope = HoldScope::from($scope)`) and reuse it for both the scope-target `match` and the `->handle()` arg. **Optional text operand normalizes blank → NULL** (`is_string(...) && $x !== '' ? $x : null`) — '' is never a domain value. **phpstan trap:** an inline closure `/** @param array<string,mixed> $data */` is NOT honored when `$data` is passed onward to a method typed `array<string,mixed>` (`argument.type` "array given") — so narrow each value IN the closure and pass already-typed scalars (`?string $profileId`, cast to int in the helper) to any extracted resolver, never the raw `$data` array. Scope-target id resolution = `match ($scope) { Customer => $customer->id, Account => (int) $customer->account?->id, Profile => (int) $profileId }` (nullable `hasOne` → `?->`, no `nullsafe.neverNull`; the polymorphic scope carries no FK — design L1). The console invokes ONLY the place/lift Action; any status coupling is domain-owned + additive (design D7), so `NoEloquentWriteInOperatorPanelRule` stays green.
- **Per-row write-through action on a `TableWidget`** (the console's FIRST per-row action) = REUSE the page's `SurfacesDomainActions` kit ON THE WIDGET: `use SurfacesDomainActions` + implement `i18nKey()` (→ the host page's copy root, e.g. `'customer'`). The trait's `surfaceLifecycleOutcome()` works unchanged in the widget's Livewire context (`Notification::make()->send()` dispatches a Livewire event — a `TableWidget` IS a Livewire component), so DON'T fork it for one caller (design D3); the trait's page-oriented `lifecycleAction()`/`recordOf()` just go unused. The row action's callbacks inject **`(Hold $record, array $data)` by parameter NAME** — `$record` is the ROW model (the per-row id), NOT the page record (a header verb keys off the page Customer's id; a per-row action keys off its own row). **Key off the typed `$record->id`** (the model's `@property int $id`), NOT `$record->getKey()` — `getKey()` is `mixed`, so `(int) $record->getKey()` trips phpstan-max `cast.int` and `$record->getKey()` → an `int` param trips `argument.type`. **Visibility gate on a STATE enum WITHOUT importing it** (design D2 — state enums stay cast-only even in PREDICATES, not just rendering): `$record->status->value === 'active'` (the cast `->value`, the literal token — the pseudocode `=== HoldStatus::Active` is semantic shorthand); the type-predicate half is a method call on the cast enum (`$record->hold_type->autoLiftable()` — no import). Extract the predicate to a `private static function isXLiftable(Hold): bool` (mirrors `castValueState`) so the magic-string token + the D2 rationale live in one documented place; `->visible(fn (Hold $record): bool => self::isXLiftable($record))` (the closure param MUST be named `$record` for Filament's by-name injection). **Test per-row visibility** with Filament tables `TestsActions`: `assertTableActionVisible('lift', record: $hold)` / `assertTableActionHidden('lift', record: $hold)` (both take a `record:` arg and evaluate the `->visible()` closure per record) — distinct from `assertTableActionExists`/`DoesNotExist` (registry presence, ignore per-record visibility). Verified against installed Filament tables 5.6.7. The domain Action STILL enforces what visibility surfaces (e.g. `IllegalHoldLift` for an auto-managed/already-lifted Hold) → caught by base `RuntimeException`, surfaced as `action_failed`, never imported (design D6).
- **Testing a per-row write-through action + its defense-in-depth reject.** Drive a VISIBLE per-row action via `callTableAction(string $name, $record, array $data)` — record is the **2nd positional** (a model is fine, mirroring `assertTableActionVisible('x', record: $h)`), data 3rd (verified `vendor/filament/tables/src/Testing/TestsActions.php`); it mounts → fills → calls and the widget's reused `surfaceLifecycleOutcome` fires the SUCCESS notification (`assertNotified((string) __('…hold_lifted'))`). Arrange a coupled precondition through the REAL domain Action (`app(PlaceHold::class)->handle(...)`), NEVER the bare factory (the factory bypasses the status coupling, so the Customer wouldn't actually be `suspended`); re-read coupled state from the DB (`Customer::findOrFail($id)->status`), not the in-memory arrange model. **A HIDDEN per-row action's reject branch is UNREACHABLE through the test harness** — `callTableAction` asserts visibility up front, and even `mountTableAction`+`callMountedTableAction` won't run it (Filament re-resolves record-action visibility on EVERY resolution and drops a hidden action server-side — empirically `getMountedActions() === []`; runtime `resolveTableAction`/`mountAction` gate on this, lessons.md 2026-06-22). When a per-row action's `->visible()` predicate is the COMPLEMENT of the domain guard it surfaces (the standard D5/D6 "surface hides + domain enforces"), the widget's `action_failed` branch can NEVER fire — so test the reject as `assertTableActionHidden('x', record: $h)` (the surface) **+** `expect(fn () => app(DomainAction::class)->handle($h->id))->toThrow(DomainException::class)` (the enforcement, imported freely in tests), Hold unchanged + no event — NOT a notification the widget can't send. The kit's `RuntimeException→action_failed` surfacing is a shared guarantee (proven on the catalog/page consoles; its SUCCESS half exercises the same dispatch in the widget's Livewire context).
- **Closing-chain integration test (one `it()`, emergent event SET) with a HETEROGENEOUS envelope.** The predecessor idiom (`CustomerConsoleChainTest`, design D9): drive the WHOLE console slice end-to-end through the PAGES/WIDGETS (not raw Actions) in one `it()` under `DatabaseMigrations` (each console action opens its OWN `DB::transaction` → the in-tx event append must commit; `RefreshDatabase`'s never-committed outer wrapper would swallow it), then a three-part emergent proof: (a) `DomainEvent::query()->pluck('name')->all()` `->toEqualCanonicalizing([...])` (a **multiset** — repeat a name once per occurrence; order-free); (b) SET-WIDE loop `foreach ($events …) module=parties / actor_role=NewcoOps / actor_id not null` + `expect($events)->toHaveCount(N)`; (c) representative per-event envelope (strict `entity_type`/`entity_id`, loose `actor_id` `toEqual` — PG returns the uncast bigint as a numeric string). **Determinism guard:** factories record NO domain event and a rolled-back `toThrow` records none, so neither pollutes the set; keep cascade-bearing entities (e.g. profile-less Customers) free of co-provisioned children so the coupling's cascade events stay silent and the count is exact. **The Holds chain's envelope is HETEROGENEOUS** (unlike a single-FSM chain's uniform `entity_type`): the Hold verbs record `entity_type=Hold`/`entity_id=$hold->id` while the domain-owned status **coupling** records `entity_type=Customer`/`entity_id=$customer->id` in the SAME run — so the representative assertions must cover BOTH, and a repeated event (two `CustomerHoldLifted`) is disambiguated by `->where('entity_id', (string) $hold->id)->sole()`. Pin a coupling no-op (e.g. "lift one of two covering Holds → NO `CustomerReactivated` yet") at its point of cause with a scoped `count()`, before the whole-run set assertion, so a regression fails at the cause not the summary.
- **Testing a header-action form (mount + inspect, no call):** `Livewire::test(Page::class, ['record' => $id])->mountAction('actionId')` then `assertFormFieldExists('field', fn (Select $f): bool => array_keys($f->getOptions()) === array_map(fn ($c) => $c->value, Enum::cases()))` — when the component `instanceof HasActions` and an action is mounted, `assertFormField*` auto-targets the mounted-action schema (`getMountedActionSchemaName()`); pass the inspector Closure as the 2nd arg (it reassigns to `$checkFieldUsing`). `Select::getOptions()` returns the resolved flat array verbatim → `array_keys()` is the robust option-set assertion. **Conditional visibility:** `assertFormFieldHidden('f')` → `setActionData(['ctrl' => Enum::Case->value])` (delegates to `fillForm`, re-evaluates the `->visible()` Closure against current `$get` state) → `assertFormFieldVisible('f')`. `assertFormFieldExists` uses `getFlatFields(withHidden:true)` (finds hidden); `assertFormFieldHidden/Visible` use `withHidden:false` (the visibility cut).
