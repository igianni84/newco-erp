# Progress — operator-console-parties-holds

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Full suite OOMs at PHP's default 128 MB memory_limit** — `php artisan test` / bare `vendor/bin/pest` abort with "Allowed memory size of 134217728 bytes exhausted" in `filament/…/routes/web.php` during result collection. Run the full suite as `php -d memory_limit=-1 vendor/bin/pest` (the same flag the PG17 ritual already uses). `--filter` runs and `phpstan analyse` are fine at the default.

- **Non-relation table with per-row actions on a `ViewRecord` page = a `Filament\Widgets\TableWidget` footer widget** (the console's first per-row-action vehicle, pinned against installed Filament 5.6.7). A `RelationManager` can't host it (the source has no Eloquent relation — `Hold.scope_id` is polymorphic, no FK). `TableWidget` (`vendor/filament/widgets/src/TableWidget.php`) extends `Widget` with `InteractsWithTable`; override `public function table(Table $table): Table` (the override point is `InteractsWithTable::table()`, l.169). Verified v5 API names: `Table::query(Builder|Closure|null)`, `->columns(array)`, **`->recordActions(array<Action|ActionGroup>)`** — `recordActions()` is the v5 rename of v3's `actions()`, and its `Action` is the unified **`Filament\Actions\Action`**. Lazy: `protected static bool $isLazy = false` overrides `CanBeLazy`'s `true` so the table renders inline (observable in the host page's initial render). `columnSpan` PHPDoc must mirror `Widget::$columnSpan` exactly (`int|string|array<string, int|null>`) for phpstan property covariance.
- **Hosting a widget on a resource `ViewRecord` = `getFooterWidgets()` returning `WidgetClass::make(['record' => $this->getRecord()])`.** `Widget::make()` returns a `Filament\Widgets\WidgetConfiguration`. A resource `ViewRecord` does NOT auto-inject its `record` into widgets (base `Page::getWidgetData()` is `[]`) — pass it explicitly; the widget declares `public ?Model $record = null` (Filament's Eloquent synth rehydrates the concrete model across Livewire requests). Placing the widget under `…/{Resource}/Widgets/` (NOT `Filament/Widgets/`) keeps it out of the panel's `discoverWidgets(in: app_path('Modules/OperatorPanel/Filament/Widgets'))` sweep — it renders only where the page mounts it, never as a dashboard widget.
