# Progress — parties-hero-package

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **The K read-port seam, now shipped three times.** A Module-K read of a deferred module is three files + one line: `Contracts/<X>Reader.php` (interface, one method), `Reads/<Adapter>.php` (launch stand-in), and a plain `$this->app->bind(Interface::class, Adapter::class)` in `PartiesServiceProvider::register()` — never `singleton`. Precedents: `CustomerTransactionTotalsReader`→`Null…` (RM-02, Module S), `PartyComplianceStatusReader`→`Database…`, and now `HeroPackageCapacityReader`→`ConfigHeroPackageCapacityReader` (Module A). The adapter's docblock carries *why it is a stand-in* and *what replaces it*.
- **PHPStan max analyses `tests/`, and it audits your fakes.** An in-test anonymous-class fake of a `?int`-returning contract that never returns `null` on any path draws `return.unusedType` ("never returns null so it can be removed from the return type"). Do **not** narrow the fake's signature to silence it — the flag means the fake is too weak to prove the consumer handles the null arm. Make it answer both arms (`$clubId === 1 ? null : $clubId * 10`). **Tasks 2.2 / 2.4 / 3.2 / 6.1 all bind fake capacity readers — they will hit this.**
- **`env()` yields a string; cast at the adapter boundary, once.** `config('parties.hero_package.capacity.default')` is `mixed`. The house normalisation idiom is `is_numeric($x) ? (int) $x : <safe default>` (`Catalog\Lifecycle\ApprovalGovernance::roleCount()` is the precedent). For capacity the safe default is `null` ⇒ uncapped, so a malformed value dark-launches rather than garbage-casting.
- **A per-id config override must be keyed on `array_key_exists`, not truthiness / `??`.** `by_club_id[$clubId] ?? $default` would silently discard an explicit `null` override (pin one Club uncapped beneath a capped default) and fall through to the default. Presence wins.
- **The uncapped test default is the reason the suite stays green.** No `PARTIES_HERO_PACKAGE_CAPACITY` in `phpunit.xml`, and no `.env.testing` exists in this repo at all ⇒ `default = null` ⇒ every one of the 2221 pre-existing tests runs against unchanged behaviour. Any task tempted to set that env var globally breaks the whole suite; bind a fake reader or `config()->set()` per-test instead.
- **The seat set has ONE definition: `ClubSeatOccupancy::OCCUPYING_STATES` (`[Active, Suspended]`).** Never re-spell it in an Action, a query or a test. `Approved` is excluded, and that is sound *only* because `Approved` is transient (`ApproveProfile` gates before writing it and drives through to `active` in the same transaction). If any task ever makes a Profile rest durably in `approved`, the seat set is wrong — not the Action.
- **Two counting entry points, and choosing between them IS the invariant.** `lockAndCountOccupiedSeats()` (locks the `parties_clubs` row, *then* counts) is the ONLY one a **seat-consuming** transition may call, and only inside an already-open `DB::transaction` — tasks 2.2 / 2.4 / 3.2. `countOccupiedSeats()` is lock-free and exists for exactly one caller: `CreateProfile`'s birth-state routing gate (D6), whose two outcomes (`Applied`/`WaitingList`) hold no seat and therefore structurally cannot oversell. `wouldOversell($clubId, $occupied)` touches no database at all — the caller supplies the count it already made, and reuses that same number in the 1.3 rejection message rather than counting twice.
- **`{@see \Fully\Qualified\Name}` in a docblock is not prose — Pint turns it into a real `use` import.** The `fully_qualified_strict_types` fixer shortens docblock FQCNs *and adds the import to the file*. Pest's `not->toUse(...)` reads real imports, so an explanatory `{@see \App\Modules\Allocation\…}` inside a Module K docblock would manufacture the exact boundary violation task 4.1 asserts against. Name non-dependencies as plain backticked text; `git diff` the `use` block after the first `pint` run. (`knowledge/laravel/hypotheses.md`)
- **Inject the seat ledger; inject the capacity port ONLY if you read the number.** PHPStan max reports `property.onlyWritten` on a constructor-promoted dependency that is never read, so a `HeroPackageCapacityReader` injected "for symmetry" alongside `ClubSeatOccupancy` fails typecheck. `CreateProfile` routes on a bool ⇒ helper only. `ApproveProfile` / `RenewProfile` throw `IllegalProfileTransition::clubAtCapacity($from, $capacity, $occupied)` ⇒ they need **both** (the port for the number, the ledger for the locked count). The capacity still reaches every one of them through Module K's own contract — autowired into the helper.
- **A from-state guard must precede a capacity gate, and the delta spec's prose says otherwise.** `specs/party-registry/spec.md:212` orders it *"lock; count; read capacity; … then, only if a seat is free: assert the from-state"*. Taken literally, an `active` Profile approved into a full Club falls into the at-parity branch, is not `waiting_list`, and gets **diverted onto the waitlist**. D8's table is authoritative — its capacity column reads `—` for every non-approvable state. Guard first; it also stops a doomed call from locking a Club row. Same ordering obligation lands on 2.4 (`RenewProfile`: from-state → **grace** → lock → count → gate) and on any future seat-consuming caller.
- **PHPStan max will not let a `?int` capacity reach `clubAtCapacity(int $capacity, …)`, and that is the design working.** The narrowing that costs nothing is `if ($capacity !== null && $this->seats->wouldOversell(...))` — the conjunct is *redundant to the rule* (`wouldOversell()` already returns `false` when capacity is `null`) and *load-bearing to the types*. Do not `(int)`-cast (it prints `0` if the invariant ever breaks) and do not `?? $occupied` (a lie on an unreachable path). Read the port AFTER `lockAndCountOccupiedSeats()` so the SQL statement order stays `parties_clubs` → `count(*)`; a config read emits no statement, so the ordering pin is indifferent, but the code then reads in the spec's own sequence.
- **A Pest global helper is declared once per PROCESS, not per file.** PHPUnit's `TestSuiteLoader` `include`s every selected test file into one process while *building* the suite, so a second `function clubAtCapacity()` is a **fatal redeclare that kills the entire run before a single test executes** — not a shadow, and not confined to the offending file (verified empirically, 2.2). Name each file's fixture helper distinctly: `clubAtCapacity` in `ProfileBirthStateRoutingTest`, `seatLedgerQueries` in `ClubSeatOccupancyTest`, `seatClubTo` / `freeOneSeat` in `ProfileApprovalCapacityGateTest`. Tasks 3.1 / 3.2 / 6.1 all want an at-parity fixture — each needs its own name.
- **Widening a from-state guard falsifies the localized reason that names the old set.** `lang/en/parties.php`'s `cannot_*` reasons all end *"…only from `<state>`"* — operator-facing prose that is a *specification restatement*, so a guard change silently strips it of truth, and no test catches it (the exception tests render the template and compare against themselves; the assertions pick from-states absent from every template on purpose). 2.2 widened `ApproveProfile` and left `cannot_approve` reading *"approved only from applied"*; 2.3 corrected both keys. **§ 7.1's sweep grep (`UNCAPPED|uncapped|deferred Module-A seam|WaitingListJoined`) matches neither.** Rule: when a task widens or narrows a guard, `grep -n "only from" lang/en/*.php` in the same iteration. **Widening a guard falsifies copy; merely ADDING a gate does not** — 2.4 ran the grep and correctly changed nothing: `RenewProfile` keeps its `{lapsed}` from-state set, so `cannot_renew` stays literally true and its new capacity refusal speaks through the separate `club_at_capacity` key. § 7.1 need not touch `cannot_renew`.
- **A lock-before-count ordering is invisible to every behavioural test on SQLite** (its grammar compiles no lock clause at all), so pin it as **SQL statement order** via `DB::listen`, then driver-gate the `for update` needle — assert both halves, never skip (`ActorRoleConstraintTest:110`). That order-pin is the only cross-engine proof the D3 fix is still in place; task 3.2's two-connection race is its PG17-only complement. (`knowledge/testing/hypotheses.md`)
- **A guard-ORDER claim is pinned NEGATIVELY — by the trace the skipped gate did not leave, never by the reason it reported.** Asserting *"a past-grace renewal says grace"* is green under both orderings whenever the capacity gate would also have passed. The discriminator is that the doomed call emits **no `parties_clubs` statement at all** (`DB::listen` → `toBeEmpty()`), which is also the operational claim worth having: a doomed call must not serialise an unrelated healthy one. Drive the dataset across the *later* gate's outcomes ({at parity, free seat, **explicitly-uncapped** via a `null` `by_club_id` override}) so independence is shown, not merely coincidental agreement. 3.1's "these Actions are ungated" is the same negative shape. (`knowledge/testing/hypotheses.md`)
- **A NON-GATE is invisible to a diff, so its test is worthless until falsified — mutation-test it.** Nothing in a diff shows an absent `if`, and a test asserting "this transition succeeds" passes identically against a gated and an ungated implementation whenever the fixture is not at parity. Task 3.1's file went green on the first run and proved *nothing* until I injected the forbidden gate into both Actions and watched 9 of its 10 assertions turn red (the survivor drove neither Action). Procedure: `cp` the Action aside, add the gate, run **the new test file only**, confirm red, restore, re-run. The same obligation lands on 4.1 (*"no auto-promotion exists"*) and on every future *"we deliberately did NOT do X"* acceptance bullet. A deliberate omission that no test can distinguish from an oversight will be "fixed" by a later iteration.
- **The at-parity fixture must be built for the FROM-STATE's own seat arithmetic.** `suspended` occupies, so `capacity: 2, occupied: 1` + the suspended Profile = exact parity. `approved` does NOT occupy, so the same arrangement leaves the approved Profile's Club at `1 of 2` — below parity, and a gate would not have fired. **Capacity `0` sidesteps it entirely**: `wouldOversell()` is true at *every* occupancy, so one fixture serves every from-state — which is what lets a cross-Action dataset share a Club.
- **An explicit `null` in `by_club_id` is the ONLY honest uncapped fixture.** A huge cap (`PHP_INT_MAX`) exercises the *capped* branch (`$capacity !== null`) of a Club that merely isn't full — a different code path. And `0` is a real capacity, which makes it the cheapest at-parity fixture in the repo: a Club admitting nobody is full while still empty, so a from-state dataset needs no seated members at all.
- **A second DB session is three lines, and it needs no change to the Action.** Copy the default connection's config to a new name, `DB::setDefaultConnection()` for the duration of the call, restore in a `finally`. `DB::transaction`, Eloquent's `resolveConnection(null)` and `DomainEventRecorder`'s `DB::transactionLevel()` all read `config('database.default')` **live**, so the unmodified Action runs on an independent backend. Pair it with a hand-opened outer transaction on the default connection — the Action's own `DB::transaction` then nests as a SAVEPOINT and its row locks survive its return. **`DatabaseMigrations`, never `RefreshDatabase`:** a wrapper transaction hides the fixture from the second session and demotes the hand-opened transaction to a savepoint that holds no top-level lock. Always roll the holder back in a `finally` — an open transaction at teardown makes the trait's `migrate:rollback` block on its own locks and **hangs the suite**, which looks like an infinite loop, not a failure.
- **`grep -rn "markTestSkipped" tests/` now self-matches.** `ProfileApprovalSeatRaceTest`'s docblock names the API in backticked prose to state the house rule. The precise tripwire is the **call form** — `grep -rn "markTestSkipped(\|->skip(" tests/` ⇒ must be `0`. The same trap awaits § 7.1's sweep grep, whose four deferred-seam tokens (`UNCAPPED|uncapped|deferred Module-A seam|WaitingListJoined`) match this change's own *inverted* prose as readily as the stale claims it hunts: **a residual-claim grep must be read, not counted.**

Determinism in **both** directions is the criterion for a race test — a proof that only sometimes catches the bug is worse than useless, because a green run then licenses the bug. The `lock_timeout` interleave is deterministic both ways.
  - **The timeout is not a compromise; it is the observation.** A single PHP thread cannot let A commit while B waits, so the wait must be surfaced. `55P03` (`lock_not_available`) is the locale-independent needle — `ActorRoleConstraintTest`'s *"assert the constraint NAME, never an engine SQLSTATE"* rule does not transfer here, because no constraint name exists; the SQLSTATE **is** the stable identifier. B's re-invocation after A commits is not a "retry": it is literally what a granted backend does — re-execute the cancelled statement, then count afresh.
  - **The 750 ms `lock_timeout` is load-bearing in two directions.** Too short and the different-Club test (which must complete *inside* it) turns flaky; too long and a lock regression hangs the suite instead of failing it. The uncontended `SELECT … FOR UPDATE` on a one-row table it must clear is sub-millisecond, so the headroom is ~3 orders of magnitude.
  - **`DomainEventRecorder`'s `DB::transactionLevel() === 0` guard resolves against the *default* connection**, which is why the connection swap must wrap the whole Action call and not just its first query. Under the hand-opened holder the level reads `2` inside the Action; under B it reads `1`. Both satisfy the guard. Note also that `DatabaseMigrations` (no wrapper transaction) is the only trait under which `DB::afterCommit` → `InlineDeliveryExecutor` actually **fires** — harmless here (no consumer is registered for `ProfileActivated` / `WaitingListJoined`, so `consumersFor()` returns `[]`), but a live consumer would run for real in this file and in `DemoSeederTest`.
  - **`heroPackageCappedClub()` merges into `by_club_id` rather than overwriting it** — the different-Club test caps two Clubs, and `config()->set()` on that key is last-write-wins. `array_replace`, not assignment. Task 6.1 caps the DRC Club and may want the same helper shape (under its own name — the per-process global-function rule still holds).
