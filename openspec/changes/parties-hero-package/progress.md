# Progress — parties-hero-package

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **The K read-port seam, now shipped three times.** A Module-K read of a deferred module is three files + one line: `Contracts/<X>Reader.php` (interface, one method), `Reads/<Adapter>.php` (launch stand-in), and a plain `$this->app->bind(Interface::class, Adapter::class)` in `PartiesServiceProvider::register()` — never `singleton`. Precedents: `CustomerTransactionTotalsReader`→`Null…` (RM-02, Module S), `PartyComplianceStatusReader`→`Database…`, and now `HeroPackageCapacityReader`→`ConfigHeroPackageCapacityReader` (Module A). The adapter's docblock carries *why it is a stand-in* and *what replaces it*.
- **PHPStan max analyses `tests/`, and it audits your fakes.** An in-test anonymous-class fake of a `?int`-returning contract that never returns `null` on any path draws `return.unusedType` ("never returns null so it can be removed from the return type"). Do **not** narrow the fake's signature to silence it — the flag means the fake is too weak to prove the consumer handles the null arm. Make it answer both arms (`$clubId === 1 ? null : $clubId * 10`). **Tasks 2.2 / 2.4 / 3.2 / 6.1 all bind fake capacity readers — they will hit this.**
- **`env()` yields a string; cast at the adapter boundary, once.** `config('parties.hero_package.capacity.default')` is `mixed`. The house normalisation idiom is `is_numeric($x) ? (int) $x : <safe default>` (`Catalog\Lifecycle\ApprovalGovernance::roleCount()` is the precedent). For capacity the safe default is `null` ⇒ uncapped, so a malformed value dark-launches rather than garbage-casting.
- **A per-id config override must be keyed on `array_key_exists`, not truthiness / `??`.** `by_club_id[$clubId] ?? $default` would silently discard an explicit `null` override (pin one Club uncapped beneath a capped default) and fall through to the default. Presence wins.
- **The uncapped test default is the reason the suite stays green.** No `PARTIES_HERO_PACKAGE_CAPACITY` in `phpunit.xml`, and no `.env.testing` exists in this repo at all ⇒ `default = null` ⇒ every one of the 2221 pre-existing tests runs against unchanged behaviour. Any task tempted to set that env var globally breaks the whole suite; bind a fake reader or `config()->set()` per-test instead.
- **The seat set has ONE definition: `ClubSeatOccupancy::OCCUPYING_STATES` (`[Active, Suspended]`).** Never re-spell it in an Action, a query or a test. `Approved` is excluded, and that is sound *only* because `Approved` is transient (`ApproveProfile` gates before writing it and drives through to `active` in the same transaction). If any task ever makes a Profile rest durably in `approved`, the seat set is wrong — not the Action.
- **Two counting entry points, and choosing between them IS the invariant.** `lockAndCountOccupiedSeats()` (locks the `parties_clubs` row, *then* counts) is the ONLY one a **seat-consuming** transition may call, and only inside an already-open `DB::transaction` — tasks 2.2 / 2.4 / 3.2. `countOccupiedSeats()` is lock-free and exists for exactly one caller: `CreateProfile`'s birth-state routing gate (D6), whose two outcomes (`Applied`/`WaitingList`) hold no seat and therefore structurally cannot oversell. `wouldOversell($clubId, $occupied)` touches no database at all — the caller supplies the count it already made, and reuses that same number in the 1.3 rejection message rather than counting twice.
- **`{@see \Fully\Qualified\Name}` in a docblock is not prose — Pint turns it into a real `use` import.** The `fully_qualified_strict_types` fixer shortens docblock FQCNs *and adds the import to the file*. Pest's `not->toUse(...)` reads real imports, so an explanatory `{@see \App\Modules\Allocation\…}` inside a Module K docblock would manufacture the exact boundary violation task 4.1 asserts against. Name non-dependencies as plain backticked text; `git diff` the `use` block after the first `pint` run. (`knowledge/laravel/hypotheses.md`)
- **Inject the seat ledger; inject the capacity port ONLY if you read the number.** PHPStan max reports `property.onlyWritten` on a constructor-promoted dependency that is never read, so a `HeroPackageCapacityReader` injected "for symmetry" alongside `ClubSeatOccupancy` fails typecheck. `CreateProfile` routes on a bool ⇒ helper only. `ApproveProfile` / `RenewProfile` throw `IllegalProfileTransition::clubAtCapacity($from, $capacity, $occupied)` ⇒ they need **both** (the port for the number, the ledger for the locked count). The capacity still reaches every one of them through Module K's own contract — autowired into the helper.
- **A from-state guard must precede a capacity gate, and the delta spec's prose says otherwise.** `specs/party-registry/spec.md:212` orders it *"lock; count; read capacity; … then, only if a seat is free: assert the from-state"*. Taken literally, an `active` Profile approved into a full Club falls into the at-parity branch, is not `waiting_list`, and gets **diverted onto the waitlist**. D8's table is authoritative — its capacity column reads `—` for every non-approvable state. Guard first; it also stops a doomed call from locking a Club row. Same ordering obligation lands on 2.4 (`RenewProfile`: from-state → **grace** → lock → count → gate) and on any future seat-consuming caller.
- **PHPStan max will not let a `?int` capacity reach `clubAtCapacity(int $capacity, …)`, and that is the design working.** The narrowing that costs nothing is `if ($capacity !== null && $this->seats->wouldOversell(...))` — the conjunct is *redundant to the rule* (`wouldOversell()` already returns `false` when capacity is `null`) and *load-bearing to the types*. Do not `(int)`-cast (it prints `0` if the invariant ever breaks) and do not `?? $occupied` (a lie on an unreachable path). Read the port AFTER `lockAndCountOccupiedSeats()` so the SQL statement order stays `parties_clubs` → `count(*)`; a config read emits no statement, so the ordering pin is indifferent, but the code then reads in the spec's own sequence.
- **A Pest global helper is declared once per PROCESS, not per file.** PHPUnit's `TestSuiteLoader` `include`s every selected test file into one process while *building* the suite, so a second `function clubAtCapacity()` is a **fatal redeclare that kills the entire run before a single test executes** — not a shadow, and not confined to the offending file (verified empirically, 2.2). Name each file's fixture helper distinctly: `clubAtCapacity` in `ProfileBirthStateRoutingTest`, `seatLedgerQueries` in `ClubSeatOccupancyTest`, `seatClubTo` / `freeOneSeat` in `ProfileApprovalCapacityGateTest`. Tasks 3.1 / 3.2 / 6.1 all want an at-parity fixture — each needs its own name.
- **Widening a from-state guard falsifies the localized reason that names the old set.** `lang/en/parties.php`'s `cannot_*` reasons all end *"…only from `<state>`"* — operator-facing prose that is a *specification restatement*, so a guard change silently strips it of truth, and no test catches it (the exception tests render the template and compare against themselves; the assertions pick from-states absent from every template on purpose). 2.2 widened `ApproveProfile` and left `cannot_approve` reading *"approved only from applied"*; 2.3 corrected both keys. **§ 7.1's sweep grep (`UNCAPPED|uncapped|deferred Module-A seam|WaitingListJoined`) matches neither.** Rule: when a task widens or narrows a guard, `grep -n "only from" lang/en/*.php` in the same iteration. 2.4 (`RenewProfile`) changes no from-state set, so `cannot_renew` stays true — but it gains a *second* rejection reason, and its copy already names only the grace one.
- **A lock-before-count ordering is invisible to every behavioural test on SQLite** (its grammar compiles no lock clause at all), so pin it as **SQL statement order** via `DB::listen`, then driver-gate the `for update` needle — assert both halves, never skip (`ActorRoleConstraintTest:110`). That order-pin is the only cross-engine proof the D3 fix is still in place; task 3.2's two-connection race is its PG17-only complement. (`knowledge/testing/hypotheses.md`)

## [2026-07-09 14:08] — 2.2 `ApproveProfile`: Club-row lock → seat count → gate; at parity divert to `WaitingList`
