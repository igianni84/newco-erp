- **Parties state-enum idiom (sibling templates `KycStatus`/`HoldType`):** a `string`-backed `enum` in `app/Modules/Parties/Enums/`, **no `declare(strict_types=1)`** (no sibling has it), case names PascalCase, backing values the lowercase persisted token. Test in a domain-grouped `tests/Unit/Modules/Parties/Enums/{Domain}EnumsTest.php` (Pest `it()` style) asserting the case→value map **verbatim + order-sensitive via `toBe([...])`** + `toHaveCount(n)`, a `from()` round-trip, the predicate truth tables, and `from('bogus')->toThrow(ValueError::class)`.
- **Full-suite invocation — use `php -d memory_limit=-1 vendor/bin/pest`, NOT `php -d memory_limit=-1 artisan test`.** The `artisan test` wrapper (laravel/pao) spawns the real test process *without* inheriting the outer `-d memory_limit=-1`, so it OOMs at the 128 MB default (fatal in `filament/.../routes/web.php` during `setUp()`). Invoking `vendor/bin/pest` directly keeps the run in one unlimited-memory process. Quality cmd `php artisan test` (root CLAUDE.md) is nominal; this is the memory-safe equivalent.
- **Parties migration idiom (templates `2026_06_15_000003_create_parties_clubs_table.php` / `…000007_create_parties_profiles_table.php`):** rich class docblock citing the PRD §/design; `Schema::create` with within-module `foreignId(...)->constrained(table: '…', indexName: '<short>_fk')` (+ explicit `->restrictOnDelete()` for a shared parent); Money via two columns `{key}_minor` (`integer`) + `{key}_currency` (`string(3)`) — never a float; `timestampTz`/`timestampsTz` for datetimes; a value-set `state`/`status` column as `string` + the enum cast, with a **PG-only CHECK** (`if (DB::getDriverName() === 'pgsql')`) whose accepted set derives from `Enum::cases()` so it can't drift; any **partial unique index** via raw `DB::statement('CREATE UNIQUE INDEX <name> ON <t> (<cols>) WHERE <pred>')` run on **BOTH** engines (NOT driver-guarded — identical SQLite/PG syntax), with the predicate literal derived from the enum (`Enum::Case->value`). Index names < 63 chars (PG limit). `down()` = bare `Schema::dropIfExists`.
- **Raw-insert schema test for a migration whose model lands later (template `tests/Feature/Platform/EventDeliveriesSchemaTest.php`):** name it `{Entity}SchemaTest.php`, `uses(RefreshDatabase::class)`, a top-level `function {entity}Row(int $parentId, array $overrides = [])` helper (UNIQUELY named — top-level test funcs are global, collide across files). Prove: `Schema::hasColumns`; **partial unique index by NAME** via `Schema::hasIndex($t, '<index_name>')` (getIndexes() exposes name+cols but NOT the predicate on either engine, so name is the portable proof — a bad predicate would abort the migration and red the file); constraint rejection via `expect(fn () => DB::transaction(fn () => DB::table($t)->insert(...)))->toThrow(QueryException::class)` (the `DB::transaction` wrap = SAVEPOINT, so PG's transaction-abort stays isolated and a post-throw count check is valid); a NOT-NULL/no-default floor by `unset($row['col'])` then expecting a throw; FK orphan rejection with a non-existent parent id (SQLite enforces FKs here).
- **Don't down-test a Laravel migration.** The base `Illuminate\Database\Migrations\Migration` declares **no** `up()`/`down()` (the migrator calls them by reflection), so a `require`d migration is `mixed` → `instanceof Migration` narrows to a type that *still* lacks `up()`/`down()` → PHPStan max reds either way (`method.nonObject`, then `method.notFound`). No `parties_*` sibling down-tests; `down()` is a trivial `Schema::dropIfExists` (drops the table and, with it, every dependent index). Assert the forward/up state only; treat `down()` correctness as structural. Cost me two PHPStan cycles before reverting.
- **Scoped within-module `hasOne` relation + nullable-accessor test idiom (1.4):** a state-filtered single-child relation is `return $this->hasOne(Related::class)->where('col', Enum::Case->value);` with declared return `HasOne` + docblock `@return HasOne<Related, $this>` — larastan 3.x preserves the relation generic THROUGH the `->where()` `__call` forward (no precedent needed; PHPStan max stays 0). Because a scoped `hasOne` genuinely resolves to `null`, the class `@property-read` is `Related|null $rel` (contrast a required `belongsTo`, which larastan types non-null, so `ProfileTest` can chain `->customer->is(...)` directly). **The null type then bites the TEST:** Pest's `expect($rel)->not->toBeNull()` does NOT narrow `$rel` for PHPStan max, so a following `$rel->method()`/`$rel->prop` reds with `method.nonObject`/`property.nonObject`. Use the suite-wide `?->` idiom (`$rel?->is($x)`, `$rel?->state`) for the chained accesses — it's analyzer-safe AND stays strict (the explicit `->not->toBeNull()` is the behavioral check; a surprise null makes `?->…` yield null and the following `toBeTrue()`/`toBe()` fail).
- **A NEW non-`Create` Parties Action MUST be registered in `SupplyLifecycleChainTest`'s scope-guard whitelist (NEW, 2.1):** `tests/Feature/Modules/Parties/SupplyLifecycleChainTest.php` runs an EXHAUSTIVE `expect($transitions)->toEqualCanonicalizing([...])` over every Action whose class name does NOT start with `Create` (`! str_starts_with($name, 'Create')`, globbed from the filesystem). So any new Action named `Issue*`/`Apply*`/`Forfeit*`/`Restore*` (anything non-`Create*`) lands in that set and **reds the exact-match** until added to one of the `$…Transitions` arrays + spread into the assertion. Add ONLY the Actions that EXIST this iteration (the glob is filesystem-truth — pre-listing a not-yet-created sibling over-specifies and fails the other way). I added a `$clubCreditWriters = ['IssueClubCredit']` array; **the next Club Credit Action tasks each append their writer here** — 3.1 `ApplyClubCredit`, 4.1 `ForfeitClubCredit`, 4.2 `RestoreClubCredit`. (Design L9's grep checked only the FORBIDDEN-name lists and missed this allow-list, so this is routine guard-test maintenance, not a realignment — and it's the SAME maintenance every prior Action slice did, per that test's own comments.)
- **Exception split for a guarded transition Action: from-state vs. precondition (NEW, 3.1):** the Parties idiom has TWO exception families and a value-moving Action uses BOTH. (1) `Illegal{Entity}Transition` (e.g. `IllegalProfileTransition`, `IllegalClubCreditTransition`) holds the FSM **from-state** guards — `cannot{Edge}({Entity}State $from)`, `:state` = `$from->value`, one factory per edge, grown one edge per task. (2) `{Entity}{Operation}Precondition` (e.g. `ClubCreditIssuancePrecondition`, `ClubCreditRedemptionPrecondition`) holds the **value/context** guards that reject a credit that IS in the right from-state (currency mismatch, over-application, freeze-while-suspended). Both `extend RuntimeException`, both localize via `(string) __('parties.{group}.{key}', [...])` with the keys landing in the i18n task (tests assert by CLASS until then). Interpolate ONLY operator-facing identity/business tokens (ids, enum values, ISO currency codes) — NEVER money minor-units/names (PII/financial-data discipline, matching the `:club` sibling). Order the guards from-state → currency → over-application → freeze so each test isolates exactly one (currency BEFORE over-application so a wrong-currency redeem never reaches the meaningless cross-currency minor-unit compare).
- **`Club.fee` is `Money|null` → never pass it straight to `Money::equals()`/`minus()` in a test (NEW, 2.1):** `@property Money|null $fee` means `$club->fee` / `$profile->club->fee` is nullable, and PHPStan max reds `argument.type` when it reaches a `Money`-typed param (and `expect()->not->toBeNull()` does NOT narrow — the suite-wide trap). For issuance/redemption tests, build an explicit non-null `$fee = Money::of(…)` LOCAL, pass it to `Club::factory()->create(['fee' => $fee])`, and assert against that local — it both satisfies the analyzer AND makes "amount = THE Club's fee" provable (you gave the Club that exact fee). The `IssueClubCredit` Action narrows the same nullable via a control-flow `if ($fee === null) throw …` (the fee-null guard) before the create, so `$fee` is `Money` at the call site — no cast/assert needed.
- **A terse task step can under-specify vs the delta-spec SCENARIO it cites — the scenario is the acceptance authority (NEW, 4.2):** task 4.2's step said only "set `state = Active`", but the delta-spec scenario (`specs/party-registry/spec.md` → "Restore a redeemed credit, one-active-respecting") says the credit "returns to `active` with its **`remaining` restored**" and design **L7** says restore "re-activates a `redeemed` credit **and restores its `remaining`**". A `redeemed` credit ALWAYS has `remaining = 0` (`ApplyClubCredit` reaches `redeemed` ONLY when the balance hits zero — a partial redemption stays `active`), so "restored" can only mean `remaining = amount` (full face value — full redemption is the § 11.2 norm; the entity has no per-order ledger, and per-order partial restore is the Module-S seam). `RestoreClubCredit` therefore writes BOTH `state = Active` AND `remaining = amount`. **Rule: when a task step is terse on FIELD WRITES, read the delta-spec scenario's THEN clauses + the cited design decision before coding — the scenario is the acceptance contract, and money/balance-adjacent writes (invariant 6) especially must trace to it, not to the headline verb.**
- **i18n task for module exception copy = lang group + sibling `{Domain}ExceptionsTest.php`, with TWO PII registers (NEW, 5.2):** the i18n task (here 5.2) does two things: (1) add the `{group}` to `lang/en/parties.php` (English baseline only — DEC-127; ONLY the `en` file exists), one key per `__('parties.{group}.{key}', [...])` call the exception factories already make (verify each key + its placeholder NAME against the factory before writing — the factories ship first, asserting by CLASS, so the key strings are the contract); (2) add a unit test mirroring the siblings in `tests/Unit/Modules/Parties/Exceptions/` (`uses(TestCase::class)`, NO `RefreshDatabase` — booting the app makes the translator resolve copy instead of echoing the key). The money balance never leaks from (b) because those factories receive only an `int` id (structural — no `Money` arg exists). Plus a parametrized "resolves every key" test (interpolate a token ABSENT from every template — `'retired'` for `:state`, a distinctive id like `7777` for ids — so its presence proves interpolation, not coincidental spelling; a missing key makes Laravel echo the key back so `not->toBe($key)` catches it) and a "preserves pre-existing groups" guard.
