# Progress — parties-hero-package-residuals

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Two mutants, not one — and the second is the one that matters.** A mutant that reds the new pin *and* the pre-existing tests proves the pin **fires**, not that it is **needed** (delete the pin; the suite still catches that mutant). Necessity is proved by the weaker, realistic drift that the existing suite **passes** and only the new pin fails. Empirically here: moving `ApproveProfile`'s from-state guard below the whole gate block reds 13 tests (7 of them pre-existing) — loud and uninformative; moving it just past `lockAndCountOccupiedSeats()` but **before** the gate's `if` leaves all 13 pre-existing tests green and reds exactly the 6 new dataset rows, each on `expect($statements)->toBeEmpty()`. Design R5 asks for the first; ask for the second too, or "the positive assertion discriminates nothing" stays an argument rather than a measurement. (`knowledge/testing/hypotheses.md` — corollary added.)
- **A chained `expect(...)->and(...)` SHORT-CIRCUITS, so ONE MUTANT PER CONJUNCT — and the assertion COUNT proves which conjuncts ran.** Pest evaluates the chain left to right and throws on the first failure, so a mutant breaking conjunct 1 never evaluates conjunct 2, leaving it unfalsified — decoration a later iteration will delete. Count the independent ways the *claim* can break, not the assertions. Measured in 2.1 on the root-ness pair (`causation_id` null · `correlation_id === event_id`): green baseline **12384** assertions; mutant A (`causationId:`) → 1 failure at the `toBeNull()` line, **12383** assertions (the chain aborted); mutant B (a shared `correlationId:`) → 1 failure at the `toBe()` line, **12384** (conjunct 1 ran and passed). Two numbers, and the short-circuit stops being a guess. Design each mutant so the other conjuncts stay green. **Not** the two-mutants corollary above: that is loud-vs-realistic drift against *one* assertion; this is one mutant per *independent* assertion. Task 3.2's `title` + `status` + `body` notification pin is three conjuncts. (`knowledge/testing/hypotheses.md` — new hypothesis, 1/3.)
- **`domain_events` is immutable BY TRIGGER, and `causation_id` is an FK — so an event's causality can only be set at `record()` time, from a donor that already exists.** `2026_06_12_000004_add_immutability_triggers.php` rejects `UPDATE` at the engine (`domain_events is immutable: UPDATE is rejected`, both lanes) — note `2026_06_12_000001`'s docblock still says the triggers "arrive in a later migration"; they arrived. And `causation_id` is `foreignId()->nullable()->constrained('domain_events')`, so a fabricated id raises `FOREIGN KEY constraint failed` at insert. Consequence for mutation work: in a transaction that records exactly ONE event, `causation_id` cannot be driven non-null by any *quiet* drift, and the one drift that can (recording a donor event first) is caught by whatever `DomainEvent::query()->count()` assertion the test already carries. **Count the transaction's events before promising a `causationId` mutant.**
- **A conjunct can be DOMINATED by an earlier assertion in the same test — then enumerate the property's CAUSES instead of hunting for a mutant.** When every candidate mutant reds an integrity error or an earlier line rather than the line you aimed at, that assertion is unfalsifiable *today*. This is not grounds to delete it — the spec may require it, as `WaitingListJoined`'s root-ness does at *both* entry points — it is grounds to write the enumeration into the test comment: name the assertion that dominates it, and the condition under which it becomes the last guard standing. Skip that and a later iteration deletes it, correctly, as decoration. Measured in 2.2: the donor mutant reds `count()->toBe(1)` 27 lines above the pin (12373 assn, 2 failures, neither at the pin). (`knowledge/testing/hypotheses.md` — corollary to the short-circuit hypothesis, now 2/3.)
- **Pest's `toThrow($class, $message)` compares the message with `toBe`, i.e. EXACTLY** (`vendor/pestphp/pest/src/Mixins/Expectation.php:960`; the `assertStringContainsString` calls on :972/:978 serve the *other* arities). So *"raises reason X, never reason Y"* needs no second assertion — passing the localized `cannot_approve` string already excludes `club_at_capacity`. Do not add a defensive `->not->toContain(...)`; it is dead weight.
- **`Factory::count(0)->create()` is safe** — `Factory::make()` short-circuits on `$this->count < 1` and returns an empty collection (`vendor/laravel/framework/.../Factories/Factory.php:443`). It does **not** reach `range(1, 0)`, which in PHP counts *downwards* and would seed two rows. A capacity-`0` at-parity fixture therefore needs no `if ($occupied > 0)` guard.
- **An at-parity fixture must account for the SUBJECT's own seat, and capacity `0` is the only one that need not.** `wouldOversell()` is true at every occupancy when capacity is `0`, so one fixture serves every from-state. A *free-seat* fixture cannot be reused that way: `active` occupies a seat (`OCCUPYING_STATES = [Active, Suspended]`) and `lapsed` does not, so `capacity: 2, occupied: 1` is at parity for an `active` subject and below parity for a `lapsed` one. Cap well above the seeded occupancy (`5, 1`) when the row must read "a free seat" for both.
- **The only honest uncapped fixture is an explicit `null` in `by_club_id` beneath a CAPPED `default`.** The adapter resolves the override on `array_key_exists`, so presence wins and the null pins that one Club uncapped. A huge cap (`PHP_INT_MAX`) exercises the **capped** branch (`$capacity !== null`) of a Club that merely is not full — a different code path, and not the claim.
- **Inline `config()->set(...)` beats a new global helper.** Pest `include`s every selected test file into one process while building the suite, so a second `function fooBar()` is a **fatal redeclare that kills the whole run** before any test executes. This file already declares `seatClubTo()` / `freeOneSeat()`; the new pin needed a `?int` capacity and a capped default, so it sets config inline — the shape the file's own from-state test and `ProfileRenewalCapacityGateTest`'s dataset both already use. A helper would have bought nothing and risked everything.
- **An envelope pinned on event E recorded by Action X says NOTHING about event E recorded by Action Y — `grep`ping the event NAME will tell you it is covered, and it is not.** Design R3 states this for *causality* (two `record()` call sites ⇒ two pins); it holds identically for the **actor envelope**, and the repo proves it. `ProfileApprovalConsoleTest.php:330` asserts `WaitingListJoined.actor_role === newco_ops` — at `ApproveProfile`'s **divert**. `CreateProfile`'s **birth** call site was blind to both `actor_role` and `actor_id`, and `actor_id` was asserted at *neither*. Measured in 3.1: hardcoding `ActorRole::System` on the birth call reds exactly 1 test of 2388. Before promising a pin is redundant, locate the `record()` **call site**, not the event class.
- **A KIT-LEVEL field is never blind — it is dominated by every other console test on that kit. The pin's real content is the WIRING that reaches the kit.** A danger toast's `title`/`status`/`body` are produced by the one shared `SurfacesDomainActions::surfaceLifecycleOutcome()`; measured in 3.2, stripping `->body($exception->getMessage())` from it reds **6 tests, 5 pre-existing** across four unrelated console files. So a new console-rejection test adds nothing by asserting those three fields — what only it can hold is that the verb is **reachable** at all in that state: `assertActionVisible('renew')` against a Club at capacity reds **exactly 1 of 2389** when a console-side capacity gate is added to the visibility predicate. Before writing a console pin, ask which line survives the kit's own tests; that line is the pin, the rest is (kept, documented) restatement.
- **When the conjuncts are fields of ONE structured value, assert the structure EXACTLY instead of chaining.** `expect(toasts())->toBe([[ 'title' => …, 'status' => 'danger', 'body' => … ]])` is a *single* assertion: no short-circuit surface, falsifies on any field, and it pins what no `->and()` chain can — that **exactly one** toast was sent. The short-circuit rule above governs conjuncts on *different subjects*; this one governs fields of the *same* subject. (This corrects the standing prediction that 3.2's notification pin would be "three conjuncts, three mutants".)
- **A close-gate residual sweep is a CLASSIFICATION, not a count — and three of its four hit-classes are SUPPOSED to fire.** Grepping for the prose a change falsifies returns four kinds of hit, and only one is actionable: (1) **the truth spec** (`openspec/specs/**`) still carries the superseded wording — *expected*: invariant 11 forbids hand-editing it, and the delta folds in **on archive**; (2) **archived changes** quote it as history — immutable, never rewritten; (3) **this change's own** `proposal.md` / `design.md` / `_Source:_` line quote it deliberately, to state what is being corrected; (4) a **live doc, docblock or comment** outside those three — *the only defect*. `grep -c` cannot tell these apart, so a close gate that counts either panics or rubber-stamps. Measured in 4.1: 11 hits for `"only if a seat is free"`, **zero in class 4**. Corollary — report the near-miss explicitly: `party-registry:889` (`RenewProfile`) also reads *"only if a seat is free"*, but it never places the from-state assert inside that branch, and `RenewProfile.php:112`→`:124` guards before it locks, so it is **silent, not contradicted**. The discriminator was the *second* grep: `"assert the from-state"` hits `:670` and **not** `:889`. Two greps whose intersection is the defect beat one grep whose union is noise.
- **A status marker duplicated across sections goes stale in every copy but the one you remember to edit — and the self-contradiction is the tell.** `docs/validation/Remediation_Tracker.md` asserted *"**Semantic-verify has NOT run** — it is a §2.7 close step"* in §4 while §7's **F12** was already stamped *"found 2026-07-09 (RM-05 §2.7 semantic-verify)"*; six further copies of *"pending §2.7 close"* outlived the merge **and** archive of the change they described. Rule: **current-state sections (§1 Now/Next · §3 at-a-glance · §4 detail) get corrected; append-only chronologies (§6 Session Log, `log.md`) do NOT** — an entry that was true when written stays true as history. And before appending a bullet next to a claim, *re-read the claim*: writing the truth beside a falsehood you left standing is worse than either alone.
- **When a mutant reds a Livewire/Filament console test, trust the test NAME and the failure MESSAGE — the reported LINE can point outside the failing test entirely.** Measured in 3.1: mutant C (a `capacity` field added to `ProfileResource::form()`) reported `line: 247`, a **comment** inside a different test 114 lines past the failing test's closing brace; the failing test was the form-shape pin at :133 (*"Failed asserting that a field with the name [capacity] does not exist"*). Mechanism undetermined (Pest resolves the frame through Livewire's `Testable` `__call` chain, and every Pest closure shares the file). The near-miss is expensive: a line number that lands in an unrelated body reads as *"my pin never fired, the mutant is dominated"* — the exact conclusion that deletes a live pin. Cost one re-run with the full failure record; budget for it. Task 3.2's notification mutants run on this same surface.
