Guard is the INVERSE of the others: legal from ANY state (`pending`/`rejected`/`verified`/NULL) **except** already-`not_required` → `IllegalKycTransition::cannotWaive($from)`. `cannotWaive` stays **non-null** `KycStatus` — NULL is a *legal* from-state (never-screened → explicit `not_required`), so the guard `if ($from === KycStatus::NotRequired)` narrows `$from` to the non-null case before the throw (PHPStan-clean). - **Nullable from-state ⇒ nullable exception factory (task 2.1, shipped).** `kyc_status` is nullable (DEC-071 un-screened), so verify/reject are reachable from NULL → `IllegalKycTransition::cannotVerify`/`cannotReject` take **`?KycStatus`** and render NULL via **`$from->value ?? 'unset'`** (NOT `$from?->value ?? 'unset'` — PHPStan strict `nullsafe.neverNull` rejects the redundant `?->`; `??` already null-coalesces via isset-semantics). `cannotRequire`/`cannotWaive` stay **non-null** — NULL is a *legal* from-state for require (→pending) and waive (→not_required), so their guards never reach the throw with NULL. Widening a 1.3-shipped factory the moment the consuming Action reveals the NULL case is legitimate intra-change refinement (non-null callers are unaffected — contravariant). **Producer KYC (3.1) reuses the SAME exception; its verify/reject also face NULL.**

The class is the single source of truth for the event's contract facets so the recording Action stays thin/magic-string-free. **Reading a nullable enum `@property` in a payload uses bare `?->value`** (e.g. `$c->sanctions_status?->value`) — clean under PHPStan-max (the property genuinely is nullable, so no `nullsafe.neverNull`); do NOT combine `?->` with `?? fallback` (THAT trips the redundant-nullsafe rule — see the exception-factory note). Assert each `NAME` **byte-for-byte against a string literal via the concrete class ref** (`CustomerOnboardingScreeningPassed::NAME`), never via a `$class::NAME` on a class-string variable (PHPStan can't resolve a const on a generic string). `make(['id' => 7, ...])` works because the model is `$guarded = []` (id mass-assignable). Prove PII-free by seeding `email`/`name`/`phone` sentinels on the fixture and asserting `array_keys(...)->toBe([...3])` + `->not->toHaveKey(...)` + `array_values(...)->not->toContain('<sentinel>')`. A shared `function (array $payload, string $status, string $source): void` closure (typed `array<string, mixed>`) DRYs the per-event payload assertion; captured via `use (...)`.
- **Evented-screening-Action idiom (task 4.2, shipped — `RecordCustomerScreening`).** The first *evented* compliance Action: it **injects `DomainEventRecorder` + `ActorContext`** (the spine idiom — contrast the eventless KYC Actions 2.1/3.1 which inject nothing).
