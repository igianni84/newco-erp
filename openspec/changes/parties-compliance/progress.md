Tests pin **verbatim, order-sensitive** case→value maps + `toHaveCount` + a `from()` ValueError rejection — mirror `tests/Unit/Modules/Parties/Enums/EnumsTest.php`. New compliance enum test is a **separate** file `ComplianceEnumsTest.php` (don't amend `EnumsTest.php`).
- **Quality loop runners.** Full suite = `php -d memory_limit=512M vendor/bin/pest` (plain `php artisan test` OOMs at 128M). Filtered = pass the test file path. `vendor/bin/phpstan analyse --no-progress` (level max, 0 errors required); `vendor/bin/pint` to format, `vendor/bin/pint --test` to check. `openspec validate parties-compliance --strict` is a standing per-task gate (delta specs present).
- **Additive nullable-value-set migration idiom (task 1.2, shipped).** `Schema::table()` + `string('col')->nullable()` (no default — DEC-071) + the backed-enum cast on the model; the PG-only CHECK is **widened** to `CHECK (col IS NULL OR col IN (<Enum::cases()>))` (a private `addNullableValueSetCheck($table,$column,$cases)` helper on the anonymous migration keeps the 4 constraints DRY). Constraint name `<table>_<column>_check` (auto, ≤63 chars). `down()` drops the PG CHECKs (`DROP CONSTRAINT IF EXISTS`, pgsql-guarded) **then** the columns. Mirrors the create-table idiom (`2026_06_15_000005`), just nullable. **Pint rewrites `\BackedEnum` → `BackedEnum`** in the no-namespace migration file (`fully_qualified_strict_types`) — resolves to global, PHPStan-clean.
- **timestampTz columns cast `immutable_datetime`** (matches `domain_events.occurred_at`/`event_deliveries.available_at`); assign `CarbonImmutable::now()` in tests, **never `now()`** (mutable `Illuminate\Support\Carbon` trips PHPStan against a `CarbonImmutable` `@property`). Nullable `boolean` cast → null stays null (primitive casts skip null). Enum casts on a NULL column → null.
- **PG-only CHECK test, per-file-safe.** Don't reuse the cross-file `captureConstraintViolation` global (a filtered single-file `pest <path>` run doesn't load sibling files → undefined-function fatal). Inline the savepoint locally: `try { DB::transaction(fn () => DB::table($t)->where('id',$id)->update([$col=>$bogus])); } catch (QueryException $e) { $msg=$e->getMessage(); }`, then branch on `DB::getDriverName()` — pgsql asserts `toContain('<constraint>')` + row untouched; sqlite asserts `$msg===''` + raw write accepted (the cast is the floor there; non-vacuous, never a skip).
- **Compliance transition-guard exceptions (task 1.3, shipped).** `app/Modules/Parties/Exceptions/{IllegalKycTransition,IllegalSanctionsTransition}.php` — both `extends RuntimeException`, mirroring the `IllegalProducerTransition` house style: a static named factory per illegal move building `new self((string) __('parties.<group>.<key>', ['state' => $from->value]))`. `(string)` coerces the translator `mixed` return (PHPStan-max). Factories the downstream Actions call: `IllegalKycTransition::cannotRequire|cannotVerify|cannotReject` (Customer 2.1 + Producer 3.1 KYC), `::cannotWaive` (Producer 3.1 only — illegal when already `not_required`); `IllegalSanctionsTransition::onboardingAlreadyScreened` (4.2 — `last_screening_at` already set, **no `:state`**, the timestamp is PII) + `::cannotResolve(SanctionsStatus $from)` (the `under_review → terminal` resolve path; re-screens stay permissive per design L4). Copy: new `kyc` + `sanctions` groups in `lang/en/parties.php` with a `:state` placeholder (a business enum value, never PII) — preserve the 5 pre-existing groups.
- **Lang-key interpolation test idiom (mirror `TransitionExceptionsTest`).** No DB → `uses(TestCase::class)` (NOT `RefreshDatabase`); booting the app gives the translator. Per factory, pick a from-state token **absent from that key's literal template** so its presence in the message proves `:state` was interpolated (not the copy spelling a similar word). A `->with([...])` dataset asserts every state-bearing key resolves with a neutral probe token (`'suspended'` — a Customer-status token absent from all KYC/sanctions templates), `->not->toBe($key)` catches a missing key; a final case re-asserts the pre-existing groups still resolve.
- **Migration `down()` reversibility — verify via the real command, not an in-suite test.** `require <migration>` returns `mixed` and the anonymous `extends Migration` class has no nameable type carrying `up()`/`down()` (the base `Migration` is abstract, declares neither) — so an in-suite `$m->down()` can't be PHPStan-max-clean without reflection. Prove it on the PG17 gate instead: `php artisan migrate:rollback --step=1` (rolls back exactly the latest migration) → assert columns+CHECKs gone → `php artisan migrate` → assert restored.
- **Eventless transition Action idiom (KYC, tasks 2.1 + 3.1, shipped).** KYC records NO domain event (design L3), so the KYC Actions have **NO constructor / no dependencies** — pure `DB::transaction` → `Model::query()->whereKey($id)->lockForUpdate()->firstOrFail()` → from-state assert → `$model->update([...])` → `return $model`. Contrast the *evented* Actions (`ActivateProducer`/`SunsetClub`) which inject `DomainEventRecorder` + `ActorContext` to `record(...)`. `version` is NOT bumped (identity-attribute revisions only). Invoke via `app(RequireKyc::class)->handle($id)` (zero-arg container build). 2.1 = the 3 Customer-KYC actions (the require sets `kyc_required = true` too); **3.1 = the 4 Producer-KYC actions** — same idiom, but the Producer carries **only `kyc_status`** (NO `kyc_required` column — the 1.2 migration added just `kyc_status` to `parties_producers`), so `RequireProducerKyc` writes `kyc_status` alone. 4.2 (sanctions) DOES emit → it injects the recorder + actor like the spine Actions. The future `kyc` Hold coupling (`parties-holds`) will add a dependency then (within-module Action orchestration, not events — design L3).
- **Producer KYC `WaiveProducerKyc` — any-state → `not_required`, the single non-`pending`-gated transition (task 3.1, shipped).** The operator "deselect" (ADR 2026-06-17 — `not_required` ≡ `verified` at every gate). Guard is the INVERSE of the others: legal from ANY state (`pending`/`rejected`/`verified`/NULL) **except** already-`not_required` → `IllegalKycTransition::cannotWaive($from)`. `cannotWaive` stays **non-null** `KycStatus` — NULL is a *legal* from-state (never-screened → explicit `not_required`), so the guard `if ($from === KycStatus::NotRequired)` narrows `$from` to the non-null case before the throw (PHPStan-clean). Customer KYC has NO waive (Producer-only). Producer verify/reject reuse the SAME widened `?KycStatus` `cannotVerify`/`cannotReject` factories as Customer 2.1 (NULL is a reachable illegal from-state there too) — no further exception change needed.
- **Nullable from-state ⇒ nullable exception factory (task 2.1, shipped).** `kyc_status` is nullable (DEC-071 un-screened), so verify/reject are reachable from NULL → `IllegalKycTransition::cannotVerify`/`cannotReject` take **`?KycStatus`** and render NULL via **`$from->value ?? 'unset'`** (NOT `$from?->value ?? 'unset'` — PHPStan strict `nullsafe.neverNull` rejects the redundant `?->`; `??` already null-coalesces via isset-semantics). `cannotRequire`/`cannotWaive` stay **non-null** — NULL is a *legal* from-state for require (→pending) and waive (→not_required), so their guards never reach the throw with NULL. Widening a 1.3-shipped factory the moment the consuming Action reveals the NULL case is legitimate intra-change refinement (non-null callers are unaffected — contravariant). **Producer KYC (3.1) reuses the SAME exception; its verify/reject also face NULL.**
- **Scope-guard exact-set whitelist GROWS per compliance task.** `SupplyLifecycleChainTest`'s "only these non-Create transition Actions exist" `toEqualCanonicalizing([...$supplySideTransitions, ...$complianceTransitions])` is an exact set — **every task that adds a non-Create transition Action MUST add it to `$complianceTransitions` or the full suite goes RED** (2.1 added the 3 Customer-KYC actions; 3.1 adds the 4 Producer-KYC; 4.2 adds `RecordCustomerScreening`). The guard's intent (no demand-side **status** transition: no `ActivateCustomer`/`SuspendAccount`/`ApproveProfile`/`LockOriginatingClub`) is preserved; task 6.1's `ComplianceIndependenceTest` adds the independence-matrix angle.
- **PHPStan: never call an enum method on a nullable `@property` in tests.** `$model->kyc_status->clears()` trips `method.nonObject` (the property is `?KycStatus`). Assert the reached state with `->toBe(KycStatus::X)` and leave the enum-predicate truth (`clears()` truth table) to `ComplianceEnumsTest` (1.1) — don't re-test the enum from the lifecycle test.
- **Domain-event class idiom (task 4.1, shipped — the four sanctions screening events).** Each event is a standalone `final class` (NO base class/interface — mirrors `ProducerActivated`/`CustomerCreated`) under `app/Modules/Parties/Events/` with three facets: untyped `public const NAME = '<verbatim § name>'`, `public const ENTITY_TYPE = 'Customer'`, and `public static function payload(Customer $c): array` (`@return array<string, mixed>`) returning the **PII-free** snapshot only (ids + enum `->value`s — never email/name/phone/DOB; the 10-year audit store holds no PII). The class is the single source of truth for the event's contract facets so the recording Action stays thin/magic-string-free. **Reading a nullable enum `@property` in a payload uses bare `?->value`** (e.g. `$c->sanctions_status?->value`) — clean under PHPStan-max (the property genuinely is nullable, so no `nullsafe.neverNull`); do NOT combine `?->` with `?? fallback` (THAT trips the redundant-nullsafe rule — see the exception-factory note). Payload key `trigger_source` maps to the model column `screening_trigger_source`.
- **Event-class unit test idiom (task 4.1, shipped — `tests/Unit/Modules/Parties/Events/ScreeningEventsTest.php`, the first Parties event test; mirror Catalog's `*LifecycleEventsTest`).** `uses(TestCase::class)` (NOT RefreshDatabase) — boots the app for casts + `factory()` while the `factory()->make([...])` fixture persists/queries nothing (the absent migrated schema is itself the guard a stray query fails loudly). Assert each `NAME` **byte-for-byte against a string literal via the concrete class ref** (`CustomerOnboardingScreeningPassed::NAME`), never via a `$class::NAME` on a class-string variable (PHPStan can't resolve a const on a generic string). `make(['id' => 7, ...])` works because the model is `$guarded = []` (id mass-assignable). Prove PII-free by seeding `email`/`name`/`phone` sentinels on the fixture and asserting `array_keys(...)->toBe([...3])` + `->not->toHaveKey(...)` + `array_values(...)->not->toContain('<sentinel>')`. A shared `function (array $payload, string $status, string $source): void` closure (typed `array<string, mixed>`) DRYs the per-event payload assertion; captured via `use (...)`.
- **Event CLASSES ≠ recorded event ROWS — adding event classes does NOT touch `SupplyLifecycleChainTest`.** Its `toEqualCanonicalizing([...10 names])`/`count → 14` assertions count rows the *supply chain records at runtime*; new event classes are recorded only by their writer Action (4.2), never by that chain. Its Events-namespace glob (lines 314-322) only forbids the **demand-side** names (`CustomerActivated`/`AccountActivated`/`ProfileActivated`/`ProfileApproved`/`OriginatingClubLocked`/`CustomerSegmentChanged`) — the screening event names aren't in that set. So **4.1 (events only) edits no arch/chain test**; only **4.2** (the Action) extends the Actions whitelist with `RecordCustomerScreening`.
