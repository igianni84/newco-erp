# Proposal — parties-compliance

## Why

This is the **compliance-floor slice of Module K (Parties)** — the KYC and sanctions-screening lifecycles that Invariant #7 and Phase-C floor chain 2 require (`Build_Workplan_v0.3-MVP.md` § Phase 2; Module_K_PRD § 9 "FLOOR"). Two prior slices left it as a **named seam**:

- `parties-core` created Customer and Producer with **no** KYC/sanctions fields (DEC-071 — "sanctions/KYC fields nullable, added additively" in this change; the spine ships Customers/Producers creatable un-screened).
- `parties-producer-lifecycle` shipped **`ActivateProducer` deliberately ungated** — the shipped `party-registry` spec states the § 4.4 KYC precondition is a *"deferred seam … implement the transition WITHOUT enforcing a KYC gate; `parties-compliance` SHALL tighten it."*

This change closes that seam. It adds the **KYC four-state** and **sanctions four-state** lifecycles (independent — § 9.4) additively on Customer/Producer, the four sanctions screening events, and **retro-tightens `ActivateProducer`** to the cleared-state rule `cleared = verified ∨ not_required` (ADR `2026-06-17-producer-kyc-gate-not-required-clears.md`; PRD § 4.4 / § 14.5 BR-K-Producer-2). The producer gate has a **real consumer today** — Module 0's already-shipped producer-state projection (`catalog-lifecycle-approval`) gates Product Master activation on producer-`active`, inheriting the KYC tightening transitively with no Module-0 change.

It steps through **no open ADR gate**: domain events deliver **inline** at launch (`decisions/2026-06-12-event-substrate-and-audit-store.md`), so the four screening events record like the spine's `*Created` events — no `queued` consumer. Screening is **manual-first** (§ 9.5 — "the screen is non-negotiable; only the integration is deferrable"; the acceptance criteria drive the lifecycles by setting state, not by calling a live vendor), so **no screening-vendor ADR** is needed. KYC document handling/storage is **out of scope** (below), so the open object-storage gate is not touched. No new architectural decision is required.

## What Changes

- **Customer KYC FSM** (PRD § 4.1 / § 9.1; AC-K-FSM-3) — a four-state lifecycle `not_required → pending → verified | rejected`, **separate from the Customer status FSM**, added as **additive nullable** fields (DEC-071): `kyc_status`, the administratively-set `kyc_required` flag (setting it transitions `not_required → pending`), and the **enhanced-KYC trigger** flag + timestamp (§ 4.1 / § 9.1; DEC-035). Operator Actions are the sole writers (`RequireKyc`, `RecordKycVerified`, `RecordKycRejected`), each from-state guarded.
- **Customer Sanctions FSM** (PRD § 4.1 / § 9.2; AC-K-FSM-4) — a four-state lifecycle `pending → passed | failed | under_review` and `under_review → passed | failed`, **separate from both the Customer FSM and the KYC FSM** and **independent of KYC** (§ 9.4; AC-K-FSM-5), added as additive nullable fields: `sanctions_status`, `last_screening_at`, `next_rescreen_at`, and the screening `trigger_source` (`onboarding | cadence | aml_threshold | compliance_ad_hoc` — § 9.2 / DEC-030; AC-K-EVT-12a). An operator Action records each screening verdict (onboarding screen + ad-hoc re-screen), firing the matching event.
- **Producer KYC FSM** (PRD § 4.4 / § 14.5 BR-K-Producer-2; AC-K-FSM-7) — the same four-state lifecycle at the Producer level (provenance KYC, distinct from Customer KYC), additive nullable `kyc_status`. Operator Actions `RequireProducerKyc`, `RecordProducerKycVerified`, `RecordProducerKycRejected`, and **`WaiveProducerKyc`** (the operator "deselect" → `not_required`, per the ADR).
- **Producer activation gate — retro-tightened** (MODIFY the shipped *Producer Lifecycle* requirement). `ActivateProducer` now enforces **`cleared = verified ∨ not_required`** and blocks on `pending` / `rejected` (PRD § 4.4; BR-K-Producer-2; AC-K-FSM-7). A **NULL** `kyc_status` (an existing or never-touched Producer) is **treated as cleared** — additivity preserves the shipped activation behaviour; only an explicit `pending`/`rejected` blocks. The `ProducerActivated` event prose (§ 15.4 "KYC cleared — `verified` or `not_required`") is now realized.
- **Four sanctions screening events** (verbatim PRD § 15.6) — `CustomerOnboardingScreeningPassed`, `CustomerOnboardingScreeningFailed`, `CustomerRescreeningPassed`, `CustomerRescreeningFailed`, each recorded transactionally with a PII-free payload. **KYC records no domain event** — the PRD event catalog (§ 15.1) names none; KYC state changes are audit-only (the same "invent no event" discipline `parties-producer-lifecycle` applied).
- **One additive migration** — nullable KYC/sanctions columns on `parties_customers` and `parties_producers` with driver-guarded `CHECK` constraints (Postgres-truthful, SQLite-compatible; no PG-only features). New `KycStatus`, `SanctionsStatus`, `ScreeningTriggerSource` enums. New transition-guard exceptions + localized copy in `lang/en/parties.php`. **No** change to creation Actions (the fields stay nullable / un-set at creation — DEC-071).
- **Docs** — extend `CONTEXT.md` with the resolved compliance terms (KYC cleared-state, sanctions independence, screening trigger-source, the NULL-cleared additive rule) and a Parties compliance contract note documenting the four screening event payloads and the deferred seams below.

## Capabilities

### New Capabilities

_None._ This change extends the existing `party-registry` capability with compliance-screening behaviour; it introduces no new capability (the Module-K-is-one-capability pattern the prior slices established).

### Modified Capabilities

- `party-registry`: the Customer and Producer entities gain their compliance-screening lifecycles, and the Producer activation gate is tightened. **ADDED** — *Customer KYC Lifecycle*, *Customer Sanctions Screening Lifecycle*, *Producer KYC Lifecycle*, *Sanctions Screening Events*. **MODIFIED** — *Producer Lifecycle* (the KYC-gate seam is closed: `ActivateProducer` enforces `cleared = verified ∨ not_required`, NULL treated as cleared); *Birth States Recorded, Lifecycle Transitions Deferred* (narrowed — the KYC/sanctions fields are now added and their screening FSMs implemented, while the `kyc`-Hold coupling, the demand-side Customer-activation gate, and the enhanced-KYC/cadence automation remain documented seams).

## Impact

- **New code** — `app/Modules/Parties/Enums/{KycStatus,SanctionsStatus,ScreeningTriggerSource}.php`; `app/Modules/Parties/Actions/{RequireKyc,RecordKycVerified,RecordKycRejected,RequireProducerKyc,RecordProducerKycVerified,RecordProducerKycRejected,WaiveProducerKyc,RecordCustomerScreening}.php` (final Action names settled in tasks.md); `app/Modules/Parties/Events/{CustomerOnboardingScreeningPassed,CustomerOnboardingScreeningFailed,CustomerRescreeningPassed,CustomerRescreeningFailed}.php`; `app/Modules/Parties/Exceptions/{IllegalKycTransition,IllegalSanctionsTransition}.php` + localized copy in `lang/en/parties.php`; one additive migration on `parties_customers` + `parties_producers`; tests under `tests/Feature/Modules/Parties/` and `tests/Unit/Modules/Parties/`.
- **One migration, no new dependency** — `git diff main -- composer.json composer.lock` stays empty; the only schema change is the **additive nullable** KYC/sanctions columns + their `CHECK`s (no backfill — NULL is the un-screened/cleared default; existing rows keep activating).
- **Reuses, does not modify** — `DomainEventRecorder` (incl. `correlationId`/`causationId` + the `DomainEvent` return), `ActorContext`/`ActorRole`, the `Module` enum (`Module::Parties->value`), and the spine Models (which stay persistence-only — the transition Action is the sole writer of the new state columns, preserving the immutability discipline for identity columns).
- **The one MODIFIED behaviour** — `ActivateProducer` flips from ungated to gated. The shipped scenario *"Activation enforces no KYC gate in this slice"* is **replaced** by *"Activation requires KYC cleared"* (NULL/`not_required`/`verified` pass; `pending`/`rejected` block); the existing `ProducerLifecycleTest` activation assertions are updated, and a NULL-cleared regression test is added so existing producers keep activating.
- **Arch tests need no amendment** — every reference is within Module K (`ModuleBoundariesTest`); the new Customer/Producer columns add no new model, so `ModulePersistenceConventionsTest` is unaffected.
- **DB engines** — the columns, the FSM transitions and the screening events touch the database, so every DB-touching task is verified on **PostgreSQL 17** before close, in addition to SQLite (`knowledge/testing/rules.md`); the nullable enum `CHECK` constraints are written driver-aware (Postgres-truthful, SQLite-compatible).
- **Deliberate traceability gaps** — each deferred concern above maps to a named future change. Three spec-faithful seams are recorded explicitly: the **`kyc` Hold coupling** (→ `parties-holds`), the **enhanced-KYC + cadence detection automation** (fields ship; jobs deferred), and the **sanctions order-completion enforcement** (→ Module S). The producer-KYC gate is the one behaviour this slice tightens.
