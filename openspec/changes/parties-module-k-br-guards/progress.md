- **Additive NOT-NULL column on an existing table ‚Üí a DB `default` is MANDATORY, not optional (first hit in 2.2, `auto_renew`/`auto_renew_default`).** Two independent forces each require it: (a) SQLite `ALTER TABLE ADD COLUMN ‚Ä¶ NOT NULL` is REJECTED without a non-NULL default ‚Äî a DDL rule, independent of row count (even an empty table); (b) every un-migrated writer that omits the new column (factory, `DemoSeeder`, the `Create*` Action until its later wiring task) would violate NOT-NULL at insert ‚Üí the whole suite reds. So a not-null additive column gets a DB default **even when the design frames it as "no default / set explicitly at creation"** ‚Äî the create-table classifier idiom (`registration_flow_type`/`state` carry no default) does NOT transfer to `ALTER TABLE`. - **Column DROP (pre-launch) = EDIT the create-table migration in place + it is an ATOMIC sweep (first hit 2.3, `invite_only`; the inverse of 2.2's additive rule).** TWO facets. (1) **Where the drop lives:** an acceptance that greps `database/` empty for the column is ONLY satisfiable by removing the column's line from the CREATE-TABLE file ‚Äî a separate `dropColumn` migration leaves the token in the create-table text, and a drop-migration + edited-create-table can't coexist (a fresh `migrate` would drop a never-created column).
