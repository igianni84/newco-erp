Mechanically: resolve a `?string` operand to its enum via `Enum::tryFrom($x) ?? throw Invalid*::forX($x)` (PHPStan-max clean on 8.5, narrows `?Enum`→`Enum`; parenthesize the `??` inside a ternary), then pass the resolved enum instance to `create()` — the enum-cast `is_object`/`is_null` branches store `->value`/null, so no `->value` unwrap is needed. Grep every Action caller for out-of-set operands first (RM-08): a new value-set guard reds any caller/seeder/console-fill passing a now-invalid literal.

- **Audit-only operator preference writer + Larastan-non-null-`first()` → ternary-not-nullsafe (4.2, Profile-5).** FOUR reusable calls. **(1) The audit-only writer shape** = the `CancelProfile`/`ApproveProfile` precedent: an operator Action that writes a column and records NO domain event (§ 15.2 names none) is `DB::transaction` → `Model::query()->whereKey($id)->lockForUpdate()->firstOrFail()` → `$m->update([col => $val])`, with NO constructor deps (no `ActorContext`/recorder — being event-silent it has no envelope; Module K writes no `audit_records` rows today, so the state write IS the record, the AnonymiseCustomer-documented reality). A PREFERENCE (not an FSM state) takes NO from-state guard — settable in any state, last-writer-wins, idempotent. **(2) Inherit-at-creation reusing a guard-fetched entity:** when a `Create*` action already fetched the parent for a guard, reuse that variable in the `create([...])` (don't re-query) — BUT if a business-rule guard fetched it via `->first()` (nullable), the inherited-attribute read collides with Larastan: **Larastan infers `Model::query()->whereKey()->first()` as NON-NULL**, so a defensive `$parent?->col ?? default` trips `nullsafe.neverNull` at PHPStan max. Persistence-only (D7) makes Actions the complete write surface, so the Action-set IS the proof no other (frontend/cross-module) writer exists.
- **Required-reference guard = UNCONDITIONAL (no null-wrapper); a blanket target-entity gate precedes a specific per-pair guard (4.1, RM-21 — the 3.2 pattern applied to a REQUIRED reference).** 3.2's Club-active guard wraps in `if ($clubId !== null)` because the ProducerAgreement Club scope is OPTIONAL (nullable). - **Model `saving` guard for a latent/non-selectable value on EVERY write path (4.3, Club-6; the write-path-complete complement to 3.1's action-boundary value-domain reject).** When a value must be rejected on CREATE **and** UPDATE (spec: "a create or update setting X → rejected") AND the entity's primary writers BYPASS the `Create*` action (the factory + DemoSeeder both do a direct `Model::create()`), enforce in the MODEL — `protected static function booted()` → `static::saving(fn ($m) => throw if $m->col === Enum::LatentCase)` — NOT at the action boundary. The differentiator vs 3.1's cadence reject: 3.1's operand arrived ONLY through the action's `?string` param, so an action-boundary guard was complete; here the latent value can arrive via factory/seeder/any-future-update, so ONLY a model guard is the complete floor (and it satisfies the literal "or update" without inventing an `Update*` action). It throws a value-domain-reject exception (`ClubRegistrationFlowNotSelectable::forFlow`, the `InvalidSettlementCadence` shape — NOT a 2.4 business-rule guard; the token is non-PII, interpolated), reading the CAST attribute (`$m->col === Enum::Case`) so a value set as either the enum or its backing string is caught. **Placement vs an action's other guards:** the `saving` guard fires INSIDE `Model::create()`, AFTER an action's pre-`create` existence checks (CreateClub's Producer-exists) — so an incidental use of the latent value in a DIFFERENT reject test (MissingClubProducer) stays green (the earlier check wins); only CREATE-with-the-value-expecting-success tests migrate. **MANDATORY ripple — migrate the factory DEFAULT + every seeder/fixture row that used the now-rejected value** (else the model guard reds the WHOLE factory-using suite at once — broader than 2.1's `annual→semi_annual` because it hits the factory default = every `Model::factory()`). Pick the spec's named default (here `application_with_approval`). Leave the enum case (the value is carried LATENT — DB/domain-valid, app-non-selectable; the `Enum::cases()` completeness test is untouched). **i18n dataset-count ripple:** removing a key from a list that feeds Pest datasets (`clubConsoleKitKeys()` → BOTH the en + it i18n datasets, per its "flows into both datasets" docblock) drops N cases per locale — reconcile the suite count as `baseline + new − (removed × locales)` (2029 +8 −2 = 2035; measure the true baseline via `git stash -u` when the arithmetic is off, rather than assuming a note's number). **"No auto-approve" = characterization, not enforcement:** the born-state is already correct (`CreateProfile` writes `Applied` regardless of the Club's `registration_flow_type`), so that half is a characterization test (create under each launch value → assert `Applied` → `ApproveProfile` → `Active`), zero code.
