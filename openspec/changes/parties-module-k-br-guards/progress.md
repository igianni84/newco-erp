- **Value-domain reject ≠ business-rule guard — two distinct exception shapes; pick by what failed (3.1, RM-22).** A **value-domain reject** = an operand outside a closed set/format (a bad cadence token, a malformed country code) → the `InvalidAddressCountryCode` shape: a `RuntimeException` subclass, `forX(string $value): self` → `(string) __('parties.{group}.invalid_x', ['x' => $value])`, thrown at the ACTION boundary as pure input validation (BEFORE the transaction — fail-fast, no DB read needed), the offending value INTERPOLATED (a token/code is not PII). The tell in the delta spec: a scenario that NAMES an exception (`ProducerAgreementClubNotActive`) is a business-rule guard; one that just says "rejected server-side" with no named exception is a value-domain reject (author a fresh `Invalid*` exception THIS task — it wasn't in the 2.4 guard batch precisely because it's a different shape). Mechanically: resolve a `?string` operand to its enum via `Enum::tryFrom($x) ?? throw Invalid*::forX($x)` (PHPStan-max clean on 8.5, narrows `?Enum`→`Enum`; parenthesize the `??` inside a ternary), then pass the resolved enum instance to `create()` — the enum-cast `is_object`/`is_null` branches store `->value`/null, so no `->value` unwrap is needed. - **Audit-only operator preference writer + Larastan-non-null-`first()` → ternary-not-nullsafe (4.2, Profile-5).** FOUR reusable calls. **(1) The audit-only writer shape** = the `CancelProfile`/`ApproveProfile` precedent: an operator Action that writes a column and records NO domain event (§ 15.2 names none) is `DB::transaction` → `Model::query()->whereKey($id)->lockForUpdate()->firstOrFail()` → `$m->update([col => $val])`, with NO constructor deps (no `ActorContext`/recorder — being event-silent it has no envelope; Module K writes no `audit_records` rows today, so the state write IS the record, the AnonymiseCustomer-documented reality). - **Fail-fast boundary input-gate on a required-then-nullable operand + migrate-every-omitting-caller (5.1, Identity-6; the 3.1 value-domain-reject applied to an INPUT that used to default null).** FIVE reusable calls. **(1) Placement = BEFORE `DB::transaction()`.** A gate on an INPUT operand (the registrant's age, derived from `date_of_birth`) is pure input validation — no DB read — so it's a fail-fast value-domain reject at the very top of `handle()`, ahead of the transaction (the 3.1 cadence-reject placement; contrast the in-txn `DuplicateCustomerEmail` pre-check, which reads). Consequence: "no Customer / Account / event created" is STRUCTURAL (the transaction never opens), not rollback-dependent — the "nothing created" test asserts count 0 on a clean DB. **(2) PHPStan-clean date math = arithmetic on the immutable `CarbonImmutable::now()`, the input only compared.** `$cutoff = CarbonImmutable::now()->subYears(self::MIN); if ($dob->greaterThan($cutoff)) throw` — the `RenewProfile.php:70` idiom (`now()->...->greaterThan()`), so the `?CarbonInterface` input is NEVER mutated (no `->copy()`/`->addYears()` on it — avoids mutating a caller's mutable Carbon) and `subYears`/`greaterThan` are the proven-green Carbon methods. Boundary is INCLUSIVE by construction: `dob == cutoff` ⇒ exactly-MIN-years ⇒ `greaterThan` false ⇒ admitted (spec "at the minimum admitted").
