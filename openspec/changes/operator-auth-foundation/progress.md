# Progress ‚Äî operator-auth-foundation

## Codebase Patterns
(consolidated reusable patterns ‚Äî read first each iteration)

- **spatie / `laravel-package-tools` publish tags** ‚Äî packages built on `spatie/laravel-package-tools` derive their publish tags from `Package::shortName()`, which strips the `laravel-` name prefix. So `spatie/laravel-permission` (named `laravel-permission`) publishes via `--tag=permission-config` and `--tag=permission-migrations` (NOT `laravel-permission-*`). Confirm any spatie tag by reading `->name(...)` in the package's ServiceProvider + `shortName()` in `vendor/.../Package.php` before publishing.
- **Vendor-published migration vs PHPStan-max** ‚Äî a published third-party migration that resolves its names through Laravel's `config()` helper (typed `mixed`) trips `level: max` with ~dozens of `offsetAccess.nonOffsetAccessible` / `argument.type` errors (spatie's `create_permission_tables` did: 66 of them). Fix: add the **single file** to `excludePaths` in `phpstan.neon` (glob `database/migrations/*_create_permission_tables.php` survives a re-publish timestamp change) ‚Äî we publish vendor code verbatim and never hand-edit it, and the errors are framework `config()`-is-mixed noise, not a real bug. Scope to the one file so our own migrations stay analysed. Do NOT add `@phpstan-ignore`/casts/`@var` to the vendor file.
- **Adding a new dependency ‚Üí `docs/development.md` is self-checking** ‚Äî `tests/Feature/DevelopmentDocsTest.php` cross-checks a fixed package list against `composer.lock`. The established pattern (pennant 2026-06-13, spatie 2026-06-15) when adding a dep: (1) add a row to the "Installed versions" table, (2) add a dated prose paragraph explaining the dep + its ADR authorisation, (3) add the package name to that test's `$packages` array so its pinned version can't drift unnoticed.
- **Cross-engine close recipe** (`knowledge/testing/rules.md`) ‚Äî `docker run -d --name pg -e POSTGRES_DB=newco_test -e POSTGRES_USER=newco -e POSTGRES_PASSWORD=newco -p 55432:5432 postgres:17`, wait for `pg_isready` (busy-poll loop ‚Äî foreground `sleep` is sandbox-blocked), then `DB_CONNECTION=pgsql DB_HOST=127.0.0.1 DB_PORT=55432 DB_DATABASE=newco_test DB_USERNAME=newco DB_PASSWORD=newco php artisan test`, then `docker rm -f pg`. phpunit.xml's non-`force` `<env>` lets the CLI env win.
- **Auth-principal models are EXEMPT from the module-table-prefix arch test** (`ModulePersistenceConventionsTest`; design D7; `decisions/2026-06-15-auth-principal-table-naming.md`) ‚Äî a model implementing `Illuminate\Contracts\Auth\Authenticatable` keeps the framework's flat auth-table name (`Operator` ‚Üí `operators`, never `operator_panel_operators`). The test skips them via `implementsInterface(Authenticatable::class)` right after the concrete-`Model` guard (the proven-empty `$scannedFiles[]` proof is unaffected ‚Äî the skip is *after* the scan append). Invariant 10's substance (no cross-module DB access) stays enforced by `ModuleBoundariesTest` + guard-by-name access. Forward-binds the deferred `customer-identity`/`producer-identity` principals; **every non-principal module model still needs its module prefix.**
- **`Auth::guard($name)` / `Auth::createUserProvider($name)` are typed to the CONTRACT, not the concrete class** ‚Äî at phpstan max `Auth::guard('operator')` is `Guard|StatefulGuard` and `createUserProvider('operators')` is `?UserProvider`; the concrete-only methods `SessionGuard::getProvider()` / `EloquentUserProvider::getModel()` are NOT on those contracts ‚Üí calling them trips `method.notFound` and breaks Pest's `->and()` template resolution. Do NOT narrow with `assert()`/`@var` (RALPH forbids). Instead assert the runtime object with `expect(...)->toBeInstanceOf(SessionGuard::class / EloquentUserProvider::class)`, and prove the provider‚Üímodel binding two other ways: declaratively via `config('auth.providers.<name>.model')` and functionally via `actingAs($model, '<guard>')` then `Auth::guard('<guard>')->user() instanceof <Model>` (the actual retrieval path ‚Äî a stronger proof than introspection). Forward-applies to the deferred customer/producer guard tests + task 4.1's `ActorContext` guard read.
