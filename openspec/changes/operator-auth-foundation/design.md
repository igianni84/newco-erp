## Context

The Identity/auth ADR (`decisions/2026-06-15-identity-auth.md`) is decided; this change is its **operator slice**. The substrate it wires into already exists and is green on both CI engines:

- `App\Platform\Events\ActorContext` — a container **singleton** holding `role`/`actorId` (defaulting to `ActorRole::System` / `null`) with a save/restore `runAs()`. It reads **no** auth state today (the gate-safe seam).
- `App\Platform\Events\ActorRole` — backed enum `NewcoOps='newco_ops' | Producer='producer' | Customer='customer' | System='system'`.
- `App\Platform\Events\DomainEventRecorder::record(name, module, ActorRole, ?int actorId, entityType, entityId, payload, …)` — callers pass the `ActorContext`-resolved role/id; `domain_events.actor_role` is NOT NULL with a driver-guarded `CHECK` over the four values, `actor_id` is a nullable bigint.
- The Filament panel `App\Modules\OperatorPanel\Providers\AdminPanelProvider` is configured at `/admin` with `->login()` and the `Authenticate` middleware, but **no** explicit auth guard — it rides Laravel's default `web`/`users`/`User` shell.
- The bootstrap principal is `App\Models\User implements FilamentUser` with `canAccessPanel()` hardwired to `true` ("revisit this gate") and a default `users` migration (+ `password_reset_tokens`, `sessions`). An `OperatorSeeder` already provisions `operator@newco.test` from env onto `User`.

Verified Filament 5 surface (in `vendor/filament`): panel methods `authGuard()`, `login()`, `passwordReset()`, `multiFactorAuthentication()`, `registration()`, `emailVerification()`; MFA provider `Filament\Auth\MultiFactor\App\AppAuthentication`; contracts `HasAppAuthentication` (`getAppAuthenticationSecret`/`saveAppAuthenticationSecret`/`getAppAuthenticationHolderName`) and `HasAppAuthenticationRecovery` (`getAppAuthenticationRecoveryCodes`/`saveAppAuthenticationRecoveryCodes`).

The boundary law (`tests/Architecture/ModuleBoundariesTest.php`) forbids `app/Modules/*` code from importing another module's namespace (except its `Contracts\*`/`Events\*` surface). It does **not** govern `app/Platform/*` or `config/*` or vendor namespaces.

## Goals / Non-Goals

**Goals:**
- An `Operator` authenticatable as the **sole** operator login principal (owned by OperatorPanel), replacing the bootstrap `User`; operators have no party row.
- An `operator` **session guard**; the Filament `/admin` panel authenticates against it with **login + password reset + opt-in TOTP 2FA** (Filament-native).
- `spatie/laravel-permission` as the **operator-scoped** RBAC mechanism; Creator/Reviewer/Approver seeded as **bare roles (data)**.
- `ActorContext` resolving an authenticated operator → (`newco_ops`, `Operator.id`); run-as override wins; everything else → (`system`, `null`).
- Green on SQLite **and** verified on local PostgreSQL 17 before close; phpstan 0 @ max; pint clean.

**Non-Goals (deferred — see the proposal slice-boundary table):**
- Customer/producer guards, Fortify/Sanctum, SPA session mechanics → Module S / TanStack gate.
- Authority-tier RBAC policy (role→capability), Laravel Policies on real resources → `feedback_prd_rr_approval` / `catalog-lifecycle-approval`.
- Filament Shield; the separation-of-duties **transition** (no-self-approval) → `catalog-lifecycle-approval`.
- MFA **enforcement** + session hardening → Architectural Security Review gate. Email verification; operator-admin UX.

## Decisions

**D1 — `Operator` is the operator principal; the bootstrap `User` is removed.** End state: `App\Modules\OperatorPanel\Models\Operator` + an `operators` table are the **only** authenticatable, and `App\Models\User` + `UserFactory` + the generic `users` table are gone. Rationale (ADR): operators have no party row, the principal **is** the identity, owned by OperatorPanel; customer/producer principals will be **separate** party-referencing tables (deferred), so the generic `users` table has no future home — the `User` shell was always a placeholder for this gate. `sessions` is retained (session guard); `password_reset_tokens` is retained generic (single authenticatable at launch — the operator broker uses it). _Alternative — keep `User` dormant alongside `Operator`:_ rejected as an end state (a dead table, two authenticatables, contradicts the single-operator-principal model and the elegance rule). _Incremental cutover to stay green:_ because renaming `users`→`operators` would break every `User`-based test at once, the tasks **build `Operator` alongside `User`**, cut the panel / `ActorContext` / seeders over to it, then **remove the orphaned `User` / `users` at a final cleanup step** (editing the default migration to drop only its `users` block, keeping `sessions` + `password_reset_tokens`) — so every iteration ends with the suite green; the two-authenticatable state is transient scaffolding, never the end state. _Boundary note:_ `config/auth.php` (platform glue) may reference `App\Modules\OperatorPanel\Models\Operator` as the provider model — framework config pointing at a class is not a cross-module Eloquent relation, and config/ is outside the arch-test scope.

**D2 — `operator` session guard + Filament-native auth.** `config/auth.php`: `guards.operator = {driver: session, provider: operators}`; `providers.operators = {driver: eloquent, model: env('AUTH_MODEL', Operator::class)}`; `passwords.operators = {provider: operators, table: password_reset_tokens, expire: 60, throttle: 60}`. The application default guard is repointed to `operator`. `AdminPanelProvider` sets `->authGuard('operator')`, keeps `->login()`, adds `->passwordReset()`; **omits** `->registration()` and `->emailVerification()` (operators are provisioned; verification deferred). `Operator implements FilamentUser`; `canAccessPanel()` returns `true` for **any** `Operator` at launch — since only operators authenticate on this guard, *"is an Operator"* **is** the access rule (the authority-tier refinement of `canAccessPanel` is deferred to `feedback_prd_rr_approval`). _Alternative — Fortify for operators:_ rejected per ADR (operators ride Filament's native session guard; Fortify+Sanctum are the customer/producer SPA's).

**D3 — Opt-in TOTP 2FA; enforcement deferred.** `AdminPanelProvider->multiFactorAuthentication([AppAuthentication::make()…], isRequired: false)` (confirm the exact recovery-codes builder, e.g. `->recoverable()`, and the `isRequired` flag in `vendor` before writing). `Operator implements HasAppAuthentication, HasAppAuthenticationRecovery`; the `operators` table carries `app_authentication_secret` (text, nullable, **encrypted** cast) and `app_authentication_recovery_codes` (text, nullable, **encrypted: array** cast), backing the four contract accessors. 2FA is **per-operator opt-in** via the panel profile; the platform enforces no enrolment at launch (ADR routes MFA-enforcement to the security-review gate). _Founder decision 2026-06-15: include opt-in 2FA now._

**D4 — `spatie/laravel-permission` as the operator-scoped RBAC mechanism.** Install **latest stable** (`composer require spatie/laravel-permission`; record the exact version in `docs/development.md` + `progress.md`). Publish its config + migration and run it. Config: **teams off**; the permission `models.role`/`models.permission` defaults stand; operator-scoping is achieved by **seeding roles with explicit `guard_name = 'operator'`** (robust regardless of the app default guard). `Operator` uses `Spatie\Permission\Traits\HasRoles`. A `RoleSeeder` seeds **Creator / Reviewer / Approver** as bare roles on guard `operator` with **no permissions** and **no** role→capability mapping (mechanism, not policy — `Admin_Panel` §1.4/§9.2). No `Permission` rows, no Policy is registered in this change (no protected resource yet) — the mechanism is wired so the next slice adds Policies consulting spatie. _Alternative — hand-rolled enum/table RBAC:_ rejected per ADR (the spec wants DB-backed/runtime-configurable "admin-configurable"; spatie is the Laravel standard and would otherwise be re-done later). _The SoD floor is **not** a permission (ADR)_ — it is per-instance transition logic, deferred to `catalog-lifecycle-approval`; this change only makes the actor identity + role rows available.

**D5 — `ActorContext` wiring + precedence.** Refactor so resolution is **lazy, per-call**, in this precedence: (1) an explicit **run-as override** (scoped) wins; else (2) `Auth::guard('operator')->check()` → (`ActorRole::NewcoOps`, `Auth::guard('operator')->id()`); else (3) (`ActorRole::System`, `null`). The customer/producer guards are **not** consulted (not wired — their future changes extend step 2). `ActorContext` reads the guard **by name** and **never imports** `Operator` (boundary-clean platform code). The container binding stays a singleton; the override becomes a **nullable** field distinct from the default so step 2/3 are evaluated at query time, not cached at construction. `runAs()` save/restore semantics are preserved for console/jobs/tests. _Landmine — singleton caching:_ resolve at `role()`/`actorId()` call time, never memoise the guard result across a request/worker (matters under Octane; lazy resolution is correct regardless). _Landmine — the gate-safe test flips:_ `tests/Feature/Platform/Events/ActorContextTest.php` currently asserts an authenticated user → `system`; it must be rewritten to: operator-guard authenticated → `newco_ops`/id, no-auth → `system`, run-as overrides an operator session.

**D6 — Seeding & the bootstrap operator.** `OperatorSeeder` `updateOrCreate`s an `Operator` from env credentials (e.g. `OPERATOR_EMAIL` / `OPERATOR_PASSWORD` — read the names already used and keep/ë document them; never committed), idempotently. `DatabaseSeeder` runs `RoleSeeder` **then** `OperatorSeeder` and drops the `Test User` factory line. The bootstrap operator **is assigned all three seeded roles** so a single launch operator is fully functional — role *possession* does **not** bypass the SoD floor, which (next slice) checks **distinct actor identity** at the transition, not role count. _Alternative — assign no roles:_ leaves the launch operator unable to be referenced by the next slice's role checks; assigning all three is the pragmatic default, revisited if `feedback_prd_rr_approval` prefers a named-tier default.

**D7 — Auth-principal models are exempt from the module-table-prefix convention.** _Added 2026-06-15 (HUMAN_NEEDED resolution, task 2.2)._ Introducing the **first** module Eloquent model (`Operator`) surfaced a collision the proposal's arch-test impact analysis missed: `tests/Architecture/ModulePersistenceConventionsTest` (the SQL-visible edge of invariant 10, from `foundations-modules-skeleton`) requires every concrete Eloquent model under `app/Modules/**` to declare a `$table` prefixed with its module value (`operator_panel_`), while the `operator-identity` delta spec, the **committed** 2.1 migration (`Schema::create('operators')`), and task 6.1 all fix the table as the flat `operators`. Resolution: an **authentication principal** (a model implementing `Illuminate\Contracts\Auth\Authenticatable`) is a **platform-foundation login shell**, not a domain aggregate — it mirrors the framework's flat auth-table naming (`users` → `operators`) and is reached only through its named guard (`Auth::guard('operator')`), never by a cross-module Eloquent query/join/import. Invariant 10's *substance* (no cross-module DB access) is therefore preserved by `ModuleBoundariesTest` + the guard-by-name access pattern, not by the cosmetic table prefix; the persistence test is amended to **skip principals** and the `module-architecture` truth spec is refined accordingly (a MODIFIED delta in this change). Recorded as a standalone ADR (`decisions/2026-06-15-auth-principal-table-naming.md`) because it **forward-binds** the deferred `customer-identity` / `producer-identity` principals. _Alternatives rejected:_ **(B)** rename `operator_panel_operators` — violates the delta spec + the committed 2.1 migration/test + task 6.1, and is non-idiomatic stutter; **(C)** relocate `Operator` to a platform namespace — violates "owned by the OperatorPanel module" (D1 + the identity-auth ADR + the `operator-identity` delta). This is the persistence-dimension analogue of the boundary law's own amendment rule (`module-architecture` "Boundary Enforcement": amend the law in the change that needs it, justify in design).

## Risks / Trade-offs

- **New dependency (spatie)** → authorised by the ADR; `composer.lock` changes (unlike prior slices). Mitigation: pin latest stable, record the version; the migration-smoke task proves the published migration runs on both engines.
- **Removing `User` / `users`** touches Laravel defaults → Mitigation: the reference sweep is an enumerated task (`grep -r 'App\Models\User'` empty at close); the full suite green proves it. `AUTH_MODEL` env indirection keeps the provider model swappable.
- **Singleton `ActorContext` memoising auth across requests** → resolve lazily per call (D5); a test asserts both an operator-session request and a console default in the same process.
- **Filament 5 API drift** → verify the exact builder/flag names (`authGuard`, `passwordReset`, `multiFactorAuthentication`, `AppAuthentication` recovery/`isRequired`, the encrypted casts) in `vendor/filament` **before** writing (lessons.md: never invent vendor APIs). Laravel Boost guidelines + `filamentphp.com/docs/llms.txt` are the references.
- **Cross-engine** → the `operators` migration + the published spatie migrations (unique indexes, the `model_morph_key` morph columns) verified on PostgreSQL 17; no jsonb/uuid here, but `migrate:fresh` must be clean on both. The encrypted 2FA casts store ciphertext as `text` — engine-neutral.
- **Module boundary** → `ActorContext` imports nothing from `App\Modules\OperatorPanel`; the `Operator` model importing `Spatie\…\HasRoles` and the Filament contracts are **vendor** imports (arch-test-irrelevant). Re-run `ModuleBoundariesTest` unchanged.
- **i18n** → operator-facing copy goes through localization (Filament ships its own translations); the role names `Creator`/`Reviewer`/`Approver` are **data identifiers** (spec-named role keys), not UI copy, and stay verbatim.

## Migration Plan

Additive-first, then a clean cutover; no production data. Order = the task order: install spatie (composer) → publish + run spatie migrations → **add** the `operators` table (+ 2FA columns) **alongside** `users` → add the `operator` guard/provider/password-broker in `config/auth.php` **alongside** `web`/`users` → `Operator` model (+ `FilamentUser`, `HasRoles`, the two MFA contracts) + `OperatorFactory` → cut `AdminPanelProvider` (authGuard/passwordReset/2FA), `ActorContext`, and the seeders (`RoleSeeder` + `OperatorSeeder` + `DatabaseSeeder`) over to `Operator`, each updating its own tests in the same task → **final cleanup**: remove `App\Models\User` + `UserFactory`, drop the `users` block from the default migration, finish `config/auth.php` (remove the `users` provider, repoint the app default guard to `operator`), and assert no `App\Models\User` reference remains → docs → full cross-engine close. Each step leaves the suite green. Rollback = revert the branch and drop the `operators` + spatie tables.
