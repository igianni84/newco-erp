# Progress — parties-anonymisation

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Anonymisation Hold-precedence gate (tasks 3.2 / 4.1) = `compliance`-only, count-independent.** The gate blocks iff an active `compliance` Hold covers the Customer; **no other type blocks**. Key on `compliance` **only** — do NOT enumerate the whole Hold set, so the gate is immune to the RM-04 6→8 Hold-count debt. Read coverage via `PartyComplianceStatusReader` (contract) / `DatabaseComplianceStatusReader` — **never** touch the `Hold` Eloquent model directly (no-model-leak boundary). Canon MVP-DEC-015 (ADR `2026-07-02-adopt-dec-015-…`).
- **There is NO `sanctions` Hold type.** `HoldType` = 8 cases (`admin|kyc|payment|fraud|compliance|credit|chargeback_review|storage_payment_failed`), none `sanctions`. Sanctions is a **separate** `sanctions_status` FSM (`SanctionsStatus`: `pending|passed|failed|under_review`) on the Customer — do NOT wire it into the anonymisation gate. A sanctioned-customer retention case is a `compliance` Hold placed by Compliance.
- **Frozen-spec anonymisation precedence is stated 3 disagreeing ways** — `DEC-027` (compliance non-blocking) vs `§8.2` / `AC-K-J-9a` (compliance blocks). Canon collapses to compliance-only; the ADR is the reconciliation. Cite the ADR, not the raw spec, when implementing the gate.
- **Owned-child FK = CASCADE; referenced-shared-parent FK = RESTRICT (module rule).** The sibling parties_* FKs RESTRICT with the explicit rationale "a shared parent it references, NOT a row it owns" (profiles→customers, club_credits→profiles). `parties_addresses.customer_id` is the module's first genuinely-OWNED child (Customer hasMany Address, overwritten in place on anonymisation) → the same ownership rationale inverts to `cascadeOnDelete()`. Inert in practice (Customers are never hard-deleted; anonymisation preserves the row). Use this test when adding any future FK.
- **ISO codes are fixed-width `string(col, N)` columns, validated at the action boundary, NOT a DB enum/CHECK.** `amount_currency` = `string(_,3)` (ISO 4217); `parties_addresses.country_code` = `string(_,2)` (ISO 3166-1 alpha-2). No value-set CHECK for open code-sets (contrast the enum columns, which get the PG-only CHECK) — the guard lives in the Create* action. **Two boundary shapes by whether a launch-set exists:** a CLOSED launch-set (currency = 5, locale = 6) → a **typed enum param** (`Currency`/`SupportedLocale`), fail-closed by type (can't pass an invalid value). An OPEN set with no launch-enum (country — collectors are international) → a **format regex** in the action (`preg_match('/^[A-Z]{2}$/', …) !== 1`) throwing a **localized exception** (`InvalidAddressCountryCode`, task 2.1); the offending value IS interpolated (a country code is not PII, unlike an email — the `:producer`/`:club` id discipline).
- **A new non-`Create*` Action MUST be registered in `SupplyLifecycleChainTest` the SAME iteration it lands — never deferred.** The test globs `Actions/*.php`, `array_filter`s out `Create*`, and asserts an EXACT `toEqualCanonicalizing` set (line ~389). So the instant a non-`Create*` Action file exists, the glob picks it up and the suite reds unless it is in the expected set. The Quality-Loop "suite stays green before commit" rule therefore OVERRIDES any task-plan note that assigns "registration" to a later task (RM-01 task 3.4's note is the latest-possible reminder + covers the still-unbuilt `ExportCustomerData`, but `AnonymiseCustomer` had to be registered in 3.2 where it was created). Add a labelled group + spread it into the assertion; a same-iteration edit of a test file to accommodate the code you just wrote is NOT scope creep. An event-free Action (records no domain event) is fine here — the whitelist already holds audit-only entries (`CancelProfile`, the Account family, the ClubCredit writers).
- **Thin `Create*` action = no event, no transaction, filtered out of the exhaustive non-`Create*` allow-list.** A creation Action that records NO domain event and does a single `Model::create()` needs neither a `DB::transaction` (one Eloquent insert is atomic) nor the `DomainEventRecorder`/`ActorContext` ctor deps — contrast the spine `Create*` actions, which wrap a transaction ONLY because they emit an event (write+emit atomicity). `CreateCustomerAddress` is the reference. Naming it `Create*` keeps it out of `SupplyLifecycleChainTest`'s non-`Create*` whitelist (`array_filter(..., fn => ! str_starts_with($name, 'Create'))`, line ~389) — so a pure creation seam never needs registering there; only NON-`Create*` writers (`AnonymiseCustomer`/`ExportCustomerData`, task 3.4) do.
- **Address is a persistence-only within-module child (no `casts()`).** `Address belongsTo Customer` / `Customer hasMany Address` are WITHIN Module K (invariant 10 — `ModuleBoundariesTest` scans the whole `App\Modules\Parties` namespace, so a new model/action is auto-covered; boundary law is NOT re-asserted in feature tests). All-`string` columns + `timestampsTz` need NO `casts()` method (contrast `ClubCredit`/`Customer`, which cast Money/enum/date). AC-K-XM-25 (no B2C/B2B discriminator on Customer; company-billing on Address) is a **schema-shape** assertion (`Schema::hasColumn` present on `parties_addresses`, absent on `parties_customers`).
- **Schema-test idiom: read columns via `->value('col')`, never `->first()->prop`.** `DB::table()->first()` is `stdClass|null` → PHPStan-max rejects a property access on the null branch. `->value('col')` returns the scalar directly (the ClubCreditSchemaTest pattern). Raw `DB::table()->insert()` + a local `xRow()` array-builder helper is the pre-model schema-layer proof (the model lands in a later task).
- **An internal within-module value object goes in `app/Modules/{Module}/Support/` — pure, import-free, `::for()`/`::of()` static factory.** A class that is neither a cross-module contract, an Action, nor an Eloquent model (e.g. `AnonymisedPlaceholders`) has no type-folder, so it lands in a new `Support/` sub-namespace. Verified safe against all four arch gates: `ModuleConformanceTest` gates only the TOP-LEVEL `app/Modules/` dir (9 module dirs + `Module.php`), NOT sub-namespaces; `ModulePersistenceConventionsTest` skips non-`Model` classes; `SupplyLifecycleChainTest` globs `Actions/*.php` ONLY (a `Support/` class is never caught — so it needs no allow-list registration, unlike the non-`Create*` Actions in 3.4); `ModuleBoundariesTest` only forbids cross-module imports (a pure value object has none). **Do NOT use `Contracts/`** — that folder is the module's cross-module PUBLIC surface (other modules may import `Parties\Contracts`, e.g. `ComplianceStatus`); an internal helper there would (mis)advertise as cross-module API. House style: plain `class` (not `final`), promoted `public readonly` props, private ctor + static factory (`Money::of` / `ComplianceStatus`). Mirror value-object tests on `Platform/Money/MoneyTest` — a pure unit test with NO `uses(TestCase::class)` and NO `RefreshDatabase` (`tests/Pest.php` binds `TestCase` to `Feature` only, so `tests/Unit` is boot-free).
- **A constant written into a fixed-width `string(N)` column must be ≤ N chars or it's a cross-engine trap** (extends the ISO-code pattern above). SQLite ignores `varchar(N)` length; Postgres ENFORCES it. So a placeholder/sentinel destined for a `string(N)` column (e.g. the `country_code` `string(2)` → the `ZZ` "unknown" ISO-3166 sentinel, 2 chars) MUST fit N, else it passes the `:memory:` suite and breaks at the PG17 close. Pin it with a `mb_strlen(...) <= N` + format-regex test on the placeholder — the module's Postgres-truthful-by-construction discipline made a unit assertion.
- **GDPR audit redaction = Platform MECHANISM + module POLICY.** The before/after→NULL redaction is the ONE DML the audit immutability triggers permit besides append (migration `2026_06_12_000004`), so it belongs to the substrate's single audit-DML class: `AuditRecorder::redactEntity(string $entityType, string $entityId): int`. It uses the BASE query builder (`DB::table('audit_records')->where(...)->update(['before'=>null,'after'=>null])`, the `ImmutabilityTest` idiom) so nulls bind as true SQL NULL — an Eloquent `->update()` would risk the `array` cast turning `null` into the string `"null"` and column-qualification SQLite's SET clause rejects. It reuses the `NotInTransactionException::forRecording` guard (atomic with the caller's erasure) and returns the count of rows ACTUALLY redacted (filtered to snapshot-bearing rows → 0 when the trail is PII-free). A module's erasure action injects `AuditRecorder` and calls `redactEntity(<EnvelopeEntityType>, (string) $id)` inside its transaction. **The audit `entity_type` == the domain-event `ENTITY_TYPE` for the same entity** (shared envelope core — e.g. `'Customer'` across every Customer event + any future Customer audit-writer + the redaction scope). Platform stays module-agnostic: string `entity_type`, NEVER an `App\Modules` import/`{@see}` in a Platform docblock (the arch-boundary law — same reason `AuditRecorder::$module` is a `string`, not the `Module` enum).
- **Investigation-first "documented no-op" seam (task-3.3 shape).** When a task says "wire the capability; if PII-free today, document a no-op": grep the module for the WRITE path first (`grep -rln 'AuditRecorder\|audit_records' app/Modules/{M}/`). Parties writes ZERO audit snapshots (it records only PII-free domain events via `DomainEventRecorder`; `audit_records` is written solely by Catalog + Platform) → the Customer audit trail is PII-free → redaction redacts 0 rows in production. Wire the mechanism anyway + PROVE it with a CONSTRUCTED PII-bearing row in the test (as a future writer would record it), so it is correct the day a PII snapshot lands. The finding ("no PII present today") IS the documented deliverable.
- **A PII-free event for an erasure/timestamp state reads the PERSISTED `*_at` column, not a fresh `now()` (task 3.4).** When the event marks a state that HAS a column (`CustomerAnonymised` ← `anonymised_at`), the payload renders that column: `'anonymised_at' => $customer->anonymised_at?->toIso8601String()` — so the event's moment == the row's (single source of truth). Contrast `OriginatingClubLocked`, whose "moment" has NO column, so it stamps the transaction clock (`CarbonImmutable::now()->toIso8601String()`). The `?->` is required (the column is `CarbonImmutable|null`; the getter is safe post-`update()` because Eloquent re-parses the cast). Event class is the 3-facet single source (`NAME`/`ENTITY_TYPE`/`payload()`), `final class`, mirroring `CustomerClosed`. **Reuse `ENTITY_TYPE` as the audit-redaction scope in the same Action** (`redactEntity(CustomerAnonymised::ENTITY_TYPE, …)`) so the event entity_type and the redaction scope are one literal. A transition Action that records an event carries the `DomainEventRecorder`+`ActorContext` ctor pair (System actor default in tests). **Test the payload with an EXACT-keys assertion** (`array_keys($e->payload)->toEqualCanonicalizing([...])`) — it structurally forbids any extra (PII) key — plus a `json_encode(...)` substring guard (no `@`, no captured-original name/email); assert `anonymised_at`==the re-fetched row's `?->toIso8601String()` (same source → cross-engine-safe, no stored-jsonb byte-compare). Flip any prior `DomainEvent::count()===0` assertions to `->sole()`/"exactly one"/"no second".
