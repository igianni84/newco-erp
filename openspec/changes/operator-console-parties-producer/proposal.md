# Proposal ‚Äî operator-console-parties-producer

## Why

`operator-console-catalog-spine` founded the operator-console **kit** and proved it across all seven Catalog (Module 0) consoles. The kit is "the template every later console (Parties, then A/D/S/B/C/E) reuses." This change opens the **Parties (Module K)** console with its first and most foundational surface ‚Äî the **Producer**: the winery-identity registry that Module 0's Product Master keys off. Producer has a fully-shipped backend (the `draft ‚Üí active ‚Üí retired` status FSM, the separate provenance-KYC FSM, the retirement‚ÜíClub-sunset cascade, the create/lifecycle/KYC Actions across `parties-core` / `parties-producer-lifecycle` / `parties-compliance`) but **no operator surface**: a human cannot create, activate, retire, or manage the KYC of a Producer. This change makes the Producer operable through the panel.

It also resolves the kit-fit question the catalog kit's own trade-off line anticipated. The kit's View base (`OperatorConsoleViewRecord`) hard-codes the **catalog** five-verb governance (`submit ¬∑ reject ¬∑ activate[SoD] ¬∑ retire ¬∑ reopen` + a `lifecycle_state` column), and **no** Parties entity shares that shape. Rather than bend Producer into a foreign FSM or refactor shipped catalog code on a sample size of one, this change **reuses the kit at the trait level** ‚Äî `SurfacesDomainActions` (the `lifecycleAction` factory + the surfacing wrapper + `recordOf`), `OperatorConsoleCreateRecord` verbatim, and `OperatorConsoleResource`'s label/version helpers ‚Äî and the Producer View page assembles its **own** verb set. That seam is recorded in its own ADR, `decisions/2026-06-20-operator-console-non-catalog-lifecycle-trait-reuse.md`, and is the template the rest of the Parties consoles reuse.

- It introduces **no new domain logic**: zero new Parties Actions, events, migrations, or dependencies. It is a pure operator-surface layer over shipped behaviour (Admin Panel PRD ¬ß1.2 ‚Äî the surface *references* the backend, never re-specs it). Every Action it invokes ‚Äî `CreateProducer`, `ActivateProducer`, `RetireProducer`, `RequireProducerKyc`, `WaiveProducerKyc`, `RecordProducerKycVerified`, `RecordProducerKycRejected` ‚Äî already exists in `app/Modules/Parties/Actions/` (verified).
- It steps through **no open ADR gate** (root CLAUDE.md "Open stack decisions", per GUIDE ¬ß2.2): operator auth is shipped (`2026-06-15-identity-auth`), the Filament stack is pinned (`2026-06-11-stack-versions-and-filament-ai-tooling`), and the read/write boundary is decided (`2026-06-19-operator-console-read-binding-write-through-actions`). No `queued` consumer is introduced (Producer Actions are synchronous, inline-recording), so the **queue-driver** gate is not crossed. No invoice/statement/KYC **document** is stored, so the **object-storage** gate is not crossed (KYC here is the verdict FSM only ‚Äî Producer carries no `kyc_required` flag and no document handling). The consumer/producer **frontend-stack** gate is not crossed ‚Äî this is the operator (Filament) surface only.
- The kit-reuse seam for non-catalog lifecycles is a new, forward-binding architectural decision (it sets the structural template for every non-catalog module console), so it is recorded in its own ADR ‚Äî `decisions/2026-06-20-operator-console-non-catalog-lifecycle-trait-reuse.md` ‚Äî and referenced from `design.md`.

## What Changes

- **Producer read-only Resource** (`ProducerResource extends OperatorConsoleResource`; Admin Panel ¬ß3.0) ‚Äî binds **read-only** to `App\Modules\Parties\Models\Producer`, reusing the kit's `getModelLabel()/getPluralModelLabel()` + `versionColumn()`. Because Producer's state attribute is `status` (a `ProducerStatus`), not `lifecycle_state`, the Resource supplies its **own** `status` badge column + a `kyc_status` badge column (nullable ‚Üí blank when never-screened) instead of the kit's `lifecycleStateColumn()`. List = name / region / country / status / kyc_status / version; infolist adds the translatable description + the operated-Clubs read (`Producer::clubs()`). No default create/edit/delete mutating action.
- **Producer create surface** (`CreateProducer extends OperatorConsoleCreateRecord`; party-registry ‚Äî Producer Registry, Spine Creation Events) ‚Äî a manual create form collecting **name, region, country** (required) and **appellation, description (translatable), website** (optional), whose `createViaAction()` invokes `CreateProducer::handle(...)` and returns the model (never `$model->save()`). The created Producer is `draft` and records `ProducerCreated` with `actor_role: newco_ops`. Producer ships **no** create-time uniqueness guard (`createRejectionField()='name'` is the inert safety net); the form exposes no status/KYC picker (both are lifecycle-managed, not set at creation).
- **Producer status lifecycle ‚Äî activate + retire** (`ViewProducer`; party-registry ‚Äî Producer Lifecycle, Supply-Side Lifecycle Events) ‚Äî the View page extends Filament `ViewRecord`, `use`s `SurfacesDomainActions`, and assembles **two** status verbs via `lifecycleAction`: **activate** (`ActivateProducer`, `draft ‚Üí active`, records `ProducerActivated`) and **retire** (`RetireProducer`, `active ‚Üí retired`, records `ProducerRetired`). Activation is **KYC-gated** ‚Äî the domain rejects it (`IllegalProducerTransition::kycNotCleared`) when `kyc_status` is `pending`/`rejected`; the console surfaces the rejection, leaving `draft` with no event. Retirement **cascades in-domain** to sunset every operated `active` Club (each recording a causally-linked `ClubSunset`); there is **no** separate console cascade affordance ‚Äî the cascade is the Action's own behaviour. The activate action carries **no "second actor required" affordance** ‚Äî Producer activation is a single-operator, KYC-gated transition, not the catalog SoD governance. The console exposes **no** submit / reject / reopen action ‚Äî the Producer FSM is linear, with no review-governance step. An out-of-state transition (`IllegalProducerTransition`) is surfaced as a notification, state unchanged.
- **Producer KYC management ‚Äî require / waive / verify / reject** (`ViewProducer`; party-registry ‚Äî Producer KYC Lifecycle) ‚Äî four further `lifecycleAction`s: **require** (`RequireProducerKyc`, `not_required`|NULL ‚Üí `pending`), **waive** (`WaiveProducerKyc`, any outstanding ‚Üí `not_required`), **verify** (`RecordProducerKycVerified`, `pending ‚Üí verified`), **reject** (`RecordProducerKycRejected`, `pending ‚Üí rejected`). These are **audit-only** (¬ß 15 names no KYC event) ‚Äî they record the `actor_role` envelope on the audit record but no domain event ‚Äî and the KYC FSM is **separate** from the status FSM (a KYC transition never moves `status`). An illegal KYC transition (`IllegalKycTransition`) is surfaced as a notification, state unchanged. No `kyc` Hold is placed/lifted (Producer KYC carries no Hold coupling ‚Äî the Customer `kyc` Hold is a separate, deferred concern). Surfacing KYC makes a KYC-blocked activation explainable to the operator end-to-end.
- **Every console write carries the audit envelope** (Admin Panel ¬ß1.3; AC-AP-PWB-6) ‚Äî because writes route through the domain Actions, each records `actor_role: newco_ops` + operator id + timestamp + action + entity ref via the existing `ActorContext` seam. Lifecycle writes record the verbatim domain event; the audit-only transitions (the four KYC verbs) record the audit envelope without an event. No console-specific envelope code.
- **i18n EN + IT** (Architecture ¬ß5.1; Invariant 12) ‚Äî all Producer operator copy through Laravel localization under a new `operator_console.producer.*` key root (labels, columns, the six action labels + their notifications, the create-form field labels). Domain rejection messages (the danger-notification *body*) are reused from the Parties domain lang (`parties.*`, already shipped ‚Äî the console owns only the `notifications.action_failed` *title*, as the catalog consoles do). EN baseline + per-key IT fallback (DEC-127).
- **ADR ‚Äî non-catalog-lifecycle trait reuse** (`decisions/2026-06-20-operator-console-non-catalog-lifecycle-trait-reuse.md`) ‚Äî records the seam: reuse the trait + create-base + resource label/version helpers; do not extend the catalog five-verb View base nor use `lifecycleStateColumn()`; revisit a verb-list generalization after ‚â•3 module consoles (rule-of-three).

### Slice boundary ‚Äî deliberately NOT in this change

| Deferred concern | Future change / status | Why not here |
|---|---|---|
| **Club** and **ProducerAgreement** consoles (the rest of the supply-side trio) | `operator-console-parties-supply-side` | Producer is the root the others depend on (Club's operating Producer; the agreement's Producer). This change proves the non-catalog trait-reuse pattern on the simplest divergent FSM first; the trio's other two reuse it. |
| **Customer / Account / Profile** admin (the demand-side "party administration") | `operator-console-parties-customer` | Customer has three orthogonal FSMs (status + KYC + sanctions) + a co-provisioned Account + multi-Profile membership ‚Äî the biggest, least kit-shaped surface. Its own arc. |
| **Hold registry** + **sanctions verdict view** | `operator-console-parties-compliance` | A different console archetype (action/registry, not create‚Üílifecycle). The Producer KYC actions here are the verdict FSM only and place/lift **no** Hold; the unified Hold surface is its own slice. If KYC/compliance **document** upload/storage lands there, it crosses the **object-storage** ADR gate first. |
| **Field-editing / updating** a Producer in place | pending a verified Parties update Action | The shipped Parties backend exposes create + lifecycle/KYC transitions, **no** field-update Action for Producer. The console offers lifecycle transitions, not field edits. **deferred-pending-backend.** |
| **Supplier** create surface | a later Parties console slice | Supplier is a minimal, event-silent, lifecycle-less create-only entity (no `version`, no status) ‚Äî low standalone value; folds naturally into the demand-side or a Module-D Supplier slice. |
| **The `kyc` Hold coupling** for Producer | not modelled (Customer-only at launch) | Producer KYC carries no Hold (the shipped actions place/lift none); the Customer-scope `kyc` Hold lives in `parties-holds`/the compliance console. |
| **Authority-tier RBAC / persona-gating** | `feedback_prd_rr_approval` | Role-agnostic at this layer (Admin Panel ¬ß1.4). Producer activation enforces no SoD (the spec mandates none ‚Äî unlike catalog), only the KYC-cleared gate. |
| **Producer / Consumer portals** (TanStack SPA) | the Module S storefront gate | Operator (Filament) surface only; the SPA frontend ADR is unwritten. |

## Capabilities

### New Capabilities

_None._ This change adds requirements to the existing `operator-console` capability.

### Modified Capabilities

- `operator-console`: **ADDED** three Producer-specific requirements ‚Äî (1) *Operator creates a Producer through the console*; (2) *Operator advances a Producer through its supply-side status lifecycle* (KYC-gated activate, cascade retire; no SoD, no submit/reject/reopen); (3) *Operator manages a Producer's KYC through the console* (require/waive/verify/reject, audit-only). The capability's existing general requirements (read-only/write-through binding, the `actor_role` audit envelope, EN+IT localization) already apply "to every console built on the capability" and are **not** restated ‚Äî they are cited as sources.
