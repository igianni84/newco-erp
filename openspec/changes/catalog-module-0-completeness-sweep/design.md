# Design — catalog-module-0-completeness-sweep

## Context

Module 0's verdict left four launch-scope completeness holes (RM-12/13/14/15 — the tracker's P3 set, batched by Giovanni's 2026-07-08 decision: same module, RM-13/14 design-coupled on the first catalog edit path). The catalog spine (7 entities), the 4-state FSM (`LifecycleTransition` — today the **sole** writer of `catalog.*` audit actions), `ApprovalGovernance` (SoD + the RM-06 rejection block-gate, derive-from-audit via a **raw latest-action read**), the producer projection (`catalog_producer_states`, 2 statuses, fed by `ProducerActivated`/`ProducerRetired`), and read-only consoles (List/Create/View, **no** Edit pages, kit modal actions) are all shipped and green (2080/2080). Every requirement here traces to the frozen spec: `AC-0-J-13`, `AC-0-XM-11`, `AC-0-EVT-8`, `AC-0-BR-Audit-1`, `AC-0-XM-2` — **all frozen-spec criteria, none canon-new → no mini-ADR gate** (the 2026-07-02 canon-adoption rule does not trigger; this design.md is the decision record for the intra-slice choices).

All structural decisions below were settled in the 2026-07-08 spec-to-change interview with Giovanni (3 rounds, 11 decisions — every recommendation accepted).

## Goals / Non-Goals

**Goals:**
- Layer-1 possible-case-configurations whitelist per (Variant, Format), operator-maintained, enforced at Sellable SKU activation (J-13, XM-11).
- `EnrichmentDataUpdated` — the 22nd event — plus the post-active enrichment edit path on the Variant (EVT-8).
- In-place re-versioning identity edits: Product Master identity + Composite SKU composition (BR-Audit-1), with the review-freshness re-arm (the deferred DEC-019 leg) and the **S1 verb-filter fix** landing together.
- Producer-existence validation at Product Master creation via the widened projection (XM-2).
- Console surfaces for all of the above + the F4-style truth-spec sync of the shipped console re-submit.

**Non-Goals:**
- Bulk import (RM-11, deferred-by-choice). `critic_scores`/`market_data` columns (enrichment-adapter change). Identity edits on Variant/Format/Case Configuration/Sellable SKU commercial fields. Any PR composition edit (new PR by design, BR-Identity-3/4). Module A Layer-2 upper-bound check + Layer-2 non-invalidation (documented seams). Module S `EnrichmentDataUpdated` consumer (seam). Version-row storage model. Filament Edit pages. Membership/Parties-side anything.

## Decisions

### D1 — "New version" = in-place edit + `version`++ + full before/after audit
The `version` column (default 1) already exists on all 7 spine tables; DEC-073 leaves physical shape to the dev team. An edit updates the same row (stable PK/FKs), increments `version` by exactly one in the same transaction, and writes an audit record whose `before`/`after` carry the changed fields — the append-only, 10-year audit trail is what keeps every old version retrievable ("deprecated, never deleted"). **Alternative rejected:** separate version rows (new row active / old row deprecated) — churns every FK and downstream reference for zero AC gain (`AC-0-BR-Audit-1` asks: old version retrievable + new version active + audit row with before+after — all satisfied by the audit trail).

### D2 — Edit scope = exactly what the ACs drive
`UpdateProductMasterIdentity` (name, appellation, region, winery_story — **all four review-governed identity content**; name/appellation changes re-check the BR-Identity-1 dedup against every *other* non-retired Master), `UpdateCompositeSkuComposition` (ordered constituent PR set; N ≥ 2 distinct re-checked), `UpdateProductVariantEnrichment` (tasting_notes), `SetVariantCaseWhitelist`. Nothing else gets an edit path. Per-state matrix (uniform for identity edits): `draft` free · `reviewed` re-arms review · `active` stays `active` (the FSM has no `active → reviewed`; the verdict's correction case) · `retired` rejected (reopen first). Every edit: operator-principal floor (reuse `ApprovalGovernance::requireOperator`), transaction + `lockForUpdate`, **no domain event** (the event surface stays closed at 21 lifecycle events + `EnrichmentDataUpdated`).

### D3 — A new `CatalogContentEdit` mechanic beside `LifecycleTransition`, not inside it
Edits are not lifecycle transitions: a sibling service in `app/Modules/Catalog/Lifecycle/` owns the shared mechanics (txn + locked re-read, state guard, operator floor, `version`++, audit write with before/after, verb parameter), mirroring `LifecycleTransition`'s audit-envelope conventions (`catalog.<segment>.<verb>`, `Str::singular(Str::after(table, 'catalog_'))`). This deliberately ends the "LifecycleTransition is the sole catalog audit writer" era — which is exactly why the S1 filter (D4) must land in the same change. **Alternative rejected:** folding `edit()` into `LifecycleTransition` (keeps single-writer true but muddies a transition FSM with non-transitions; the single-writer claim is obsolete either way).

### D4 — Review-freshness: verb-filtered derivation + identity-edit re-arm (S1 fold-in)
`ApprovalGovernance::assertNotRejectionPending` today reads the **raw** latest audit action (`orderByDesc('id')->value('action')`, blocked iff `str_ends_with('.rejected')`) — safe only while governance verbs are the only catalog audit rows. New derivation: among the entity's audit actions ending in one of the four **review-freshness-relevant suffixes** — `.submitted`, `.resubmitted`, `.rejected`, `.identity_updated` — take the latest; **blocked iff it ends `.rejected` or `.identity_updated`**. Consequences, all deliberate: a post-rejection enrichment/whitelist row no longer clears the block (the S1 hole); an identity edit in `reviewed` blocks activation until an explicit re-submit (the DEC-019 re-arm leg); a `draft` edit is cleared by the subsequent `submit` (`.submitted` **must** be in the relevant set or a draft-edit row would block forever); an `active`-entity edit followed by retire → reopen requires a re-submit before re-activation (content changed since last review — coherent with DEC-019, derived for free). `reviewerOf`'s `LIKE '%.submitted'` needs no change (`.resubmitted` and `.identity_updated` don't match that suffix; verified against the string forms).

### D5 — Audit-verb vocabulary + collision discipline
New verbs: `identity_updated` (Master + Composite — a composition change IS the Composite's identity change; one uniform suffix keeps D4's filter a 4-suffix set), `enrichment_updated`, `whitelist_updated` (Variant-scoped). **Discipline:** no future catalog audit verb may end with one of D4's four relevant suffixes unless it is meant to participate in review-freshness — recorded in CONTEXT.md (task 7.1) so the next writer inherits the rule.

### D6 — Whitelist = FK-integral pivot per (Variant, Format), empty pair ⇒ permissive
Table `catalog_variant_case_whitelists`: `id`, `product_variant_id` (FK → variants, cascade), `format_id` (FK → formats, restrict), `case_configuration_id` (FK → case_configurations, restrict), timestamps; **unique on the triple** + supporting index on the pair — every index/FK explicitly short-named with a `catalog_vcw_*` prefix (63-byte PG rule, `knowledge/data-model/rules.md`). Within-module relations are legal (architecture rule). The literal PRD reading won ("this product, **in this format**" — § 3.3 + § 7.1 twice); the AC's flat example is one format's view. Maintenance = **replace the pair's set** (`SetVariantCaseWhitelist`): audit-only (`whitelist_updated`, before/after CC-id sets), no `version`, no re-arm, no event, allowed on `draft`/`reviewed`/`active` (rejected `retired`). Enforcement joins `ActivateSellableSku`'s gate closure: resolve SKU → PR → (variant_id, format_id); if the pair has ≥1 whitelist row and the SKU's CC is not among them → new localized exception; zero rows ⇒ permissive (PRD default). **Alternatives rejected:** flat per-Variant pivot (loses the PRD's "per Format", would need a documented deviation); JSON column (no FK integrity, hand-rolled default). Creation stays free (draft-anything — no other `Create*` validates cross-entity state); only activation gates.

### D7 — RM-15 = widen the existing projection, not a second knowledge source
`ProducerLifecycleProjector` also consumes **`ProducerCreated`** → row with new `ProducerProjectionStatus::Registered` (enum case appended; PG CHECK derives from `cases()` so a fresh migrate emits 3 tokens — no ALTER needed pre-prod, same as RM-04). The watermark already guarantees no downgrade (a stale `ProducerCreated` after `ProducerActivated` is ignored). `CreateProductMaster` gains the **existence** guard: no projection row for `producer_id` → new localized exception; `registered`/`retired` rows admit creation (existence ≠ activeness — activation stays gated). Inline delivery makes the projection synchronous with `CreateProducer` today; when the queue ADR lands this becomes eventually-consistent — acceptable, the guard is fail-closed. **Alternatives rejected:** a Parties read-contract (`Parties\Contracts` precedent exists, but it would make Catalog hold **two** producer-knowledge sources — gate reads the projection, create reads the contract — and would deserve its own mini-ADR); accepting the divergence with an ADR note (Giovanni chose the fix). The `ProducerActivationGate` is **unchanged** (still `status === Active`, fail-closed on `registered`/`retired`/absent).

### D8 — Console = modal header actions on the existing View pages
`edit identity` on `ViewProductMaster`, `edit enrichment` + `manage whitelist` on `ViewProductVariant`, `edit composition` on `ViewCompositeSku` — via the shared `lifecycleAction()` kit factory (forms + `surfaceLifecycleOutcome`), no Edit pages (the read-projection discipline stands; the 7 resources' "deliberately NO Edit page" comments get their **rationale** updated — "the Catalog backend ships no update Action" is no longer the reason; the no-Edit-PAGE stance is). Dedup collision and unknown-producer surface as **form validation errors** (precedent: `CreateProductReference`'s unique-violation mapping); state guards and composition rejections surface as notifications through the kit's `RuntimeException` path. The Master create form's producer Select stays projection-fed (now listing `registered` too — coherent: creatable ≠ activatable).

### D9 — Console review-stale visibility mirrors the D4 filter
`OperatorConsoleViewRecord::isRejectionPending()` today duplicates the **raw** latest-audit read for the re-submit button's visibility — it has the same S1 hole and must move to the same 4-suffix filtered predicate (renamed `isReviewStale()`), in the **same task** as D4. It cannot import `ApprovalGovernance` (console may import only the operated module's `Models\*`/`Actions\*` + operand enums), so the predicate is **mirrored** over the platform `AuditRecord` — with a uniformity test asserting console visibility and domain block agree on the same audit histories (rejection-pending, identity-edited, enrichment-noise, resubmitted).

### D10 — No mini-ADR anywhere in this change
All five criteria are frozen-spec (verdict Part-4 check); no canon `MVP-DEC` is adopted; RM-15 is a within-module read-model extension (this design is its record). The verdict's "flag with an ADR/overlay note" applied only to the accept-the-divergence option we did not take.

### D11 — `EnrichmentDataUpdated` payload = minimal, PII-free
`NAME = 'EnrichmentDataUpdated'` (verbatim — generalisation keeps it), `ENTITY_TYPE = 'ProductVariant'`, payload `{product_variant_id}` (+ nothing else at launch: values are prose/JSON and belong to the audit before/after; the Module S consumer is a deferred seam and dispatches on the event name). Fired only when the stored value actually changes; a no-change update is a silent no-op (no event, no audit row, no write — idempotent). Event class follows the existing `final class` + `const NAME/ENTITY_TYPE` + static `payload()` shape.

## Risks / Trade-offs

- **[R1 — RM-15 blast radius]** The `CreateProductMaster` guard breaks every caller that invents a `producer_id`. → Before wiring (task 5.2): `grep -rn 'CreateProductMaster' app/ tests/ database/` **and** the console create tests; classify each caller (real `CreateProducer` lineage → already projected once the projector widens; bare int → seed a `ProducerState` row or create the producer properly). The 2026-07-06 floor-wiring lesson applies verbatim: the task list is NOT the blast radius, the grep is; only the FULL suite proves it.
- **[R2 — enum count pin]** `EnumsTest` pins `ProducerProjectionStatus::cases()` at 2. → Same task as the enum case: 2 → 3 (+ map/coverage assertions), mirroring the RM-04 pattern.
- **[R3 — console visibility drift]** If `isRejectionPending()` keeps the raw read while the domain filters (D4/D9), the re-submit button and the domain block disagree the moment the first edit-audit row lands. → One task carries both reads + the uniformity test; grep for any other raw `orderByDesc('id')->value('action')` over `audit_records` before closing it.
- **[R4 — freshness tests before edit paths exist]** Task 1.2 (governance filter) precedes the edit Actions. → Its tests simulate `identity_updated`/`enrichment_updated` rows by writing audit records directly through the platform `AuditRecorder` inside a transaction (tests may do what domain code does); the real-Action re-arm scenarios re-prove it in tasks 2.x/4.x.
- **[R5 — i18n scanners]** `SpineConsoleI18nTest` + `ProductMasterConsoleI18nTest` pin kit keys and scan for hardcoded literals; every new console label/notification/validation key must be authored in **EN and IT** in the same task as the surface, or the suite reds.
- **[R6 — Pint docblock trap]** Console/action classes must reference cross-module or Lifecycle classes in **backticked prose only** — a `{@see FQCN}` re-imports and reds `ModuleBoundariesTest` (standing 2026-06-13/20/07-06 lessons). After adding docblocks: re-run Pint, then grep the file's `use` block.
- **[R7 — naming cascade]** New classes/events must not start `Wine`/`BottleReference` (`CatalogNamingCascadeTest` negative walk) — all proposed names comply; keep it that way through the loop.
- **[R8 — TranslatableText form fields]** `winery_story`/`tasting_notes` are i18n-keyed JSON (`TranslatableText`). The edit modals must reuse the create-form field pattern for translatable prose (see `CreateProductMaster`/`CreateProductVariant` console pages) — do not invent a new input shape, and narrow `->numeric()`/array payloads per the standing console-payload lessons.
- **[R9 — audit `before`/`after` shape]** `AuditRecorder::record` expects arrays; identity edits write only the **changed** fields' before/after (plus `version`), not whole-row dumps — keeps redaction/PII posture and mirrors the transition rows' minimal shape. Whitelist rows write the pair + before/after CC-id lists; composition rows the ordered PR-id lists.
- **[R10 — J-13 non-retroactivity]** The whitelist gate must consult the whitelist **only at activation** — no sweep, no invalidation of `active` SKUs on reduction. The test fixes an `active` SKU on the removed CC and asserts it is untouched (state, audit, events).
- **[R11 — suite memory]** Full runs via `php -d memory_limit=-1 vendor/bin/pest` (never bare `artisan test`); PG17 recipe in `hot.md`. Shared-helper i18n tests need `--filter`/full-suite runs, not file paths.
- **[R12 — residual-claim sweep]** This change inverts two shipped claims: "LifecycleTransition is the sole catalog audit writer" and "the projection records `active`/`retired` only, fed by two events". → Task 7.1 greps `CONTEXT.md`, `decisions/`, `docs/`, docblocks (`LifecycleTransition.php` line ~229 "re-versioning deferred (RM-14)", `ApprovalGovernance` file docblock, the 7 resources' "ships no update Action" comments, `catalog_producer_states` migration docblock) for the stale forms — the 2026-07-08 inversion lesson: CONTEXT.md and ADR prose are first-class claim-holders.

## Migration Plan

Pre-production: one additive migration (`catalog_variant_case_whitelists`); the `registered` enum case needs no ALTER (the PG CHECK derives from `cases()` on fresh migrate — RM-04 precedent); every env is `migrate:fresh` + reseed. `DemoSeeder` needs no data change (it creates producers through the real actions, so `ProducerCreated` rows project inline), but its truncation list gains the whitelist table if it sweeps catalog tables — verify at task 5.2. Rollback = drop the change branch (no deployed DB).

## Build-time deviations (recorded 2026-07-08 at semantic-verify, before archive)

The decisions above are the pre-build record and are left as written. Three of them did not survive contact with the code exactly as stated; the shipped behaviour is correct and spec-compliant, but a reader of this file must not be misled by the prose.

- **D8 — factory name and surfacing rule.** D8 says the edit modals ship "via the shared `lifecycleAction()` kit factory" and that "state guards and composition rejections surface as notifications". Shipped: a *new* `contentEditAction()` factory, and **every** content-edit rejection lands as a validation error on one designated field. The delta spec permits either ("validation error or notification"); the split is by action SHAPE — verb-shaped actions notify, form-shaped actions validate — and is reasoned in `OperatorConsoleViewRecord`'s kit docblock. Lifecycle-verb rejections (activation on a review-stale entity, the SKU whitelist gate) do still notify.
- **D5 — from prose to guard.** D5 recorded the verb-collision discipline as a rule "the next writer inherits" via CONTEXT.md. Documentation cannot hold an invariant whose breach leaves the suite green: `CatalogContentEdit::maintain()` now **enforces** it, reading `ApprovalGovernance::REVIEW_FRESHNESS_VERBS` (promoted to `public` so the rule and its derivation name one list) and throwing a `LogicException` at the call site.
- **Migration Plan — `DemoSeeder` does *not* create its demo producers through the real actions.** `seedProducers()` uses a bare `Producer::create` (no `ProducerCreated`), and `seedProducerStates()` hand-writes `active` projection rows with a local watermark of `1..7`. The seeder is correct today and needs no data change, but the stated *reason* is wrong — and it hides a trap: those hand-seeded watermarks sit **below** real `domain_events` ids, so the day `seedProducers()` is switched to `CreateProducer`, the emitted `ProducerCreated` would strictly advance them and rewrite every `active` row back to `registered`. The one producer that *is* built through the real action (`seedProducerApprovalScenario`) now correctly projects as `registered`.
