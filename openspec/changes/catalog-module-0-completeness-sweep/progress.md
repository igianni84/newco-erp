# Progress — catalog-module-0-completeness-sweep

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Cross-engine constraint proof = nested `DB::transaction()` savepoint + the RIGHT exception class.** Wrap the violating DML in `DB::transaction(fn () => …)` so PostgreSQL's aborted-transaction state rolls back to the savepoint and the follow-on row-state assertions still run (SQLite needs no such care — PG does). Assert `UniqueConstraintViolationException` for a unique violation, plain `QueryException` for an FK `RESTRICT` (Laravel narrows only the unique case). Precedents: `ProductReferenceTest` (unique), `ProducerStateProjectionTest` (raw-insert CHECK).
- **Verify the 63-byte identifier rule empirically, don't eyeball it.** After a PG run the schema persists in `newco_test`: `docker exec pg psql -U newco -d newco_test -c "SELECT conname, LENGTH(conname), contype FROM pg_constraint WHERE conrelid = '<table>'::regclass ORDER BY conname;"` shows every FK/unique/CHECK name and its real byte length. PG truncates *silently*, so an assertion-by-name test can pass for the wrong reason if two names collide after truncation — this query is the cheap proof they didn't.
- **A composite unique's leftmost prefix IS the supporting index for the narrower lookup.** Both engines answer `WHERE a = ? AND b = ?` from a `UNIQUE (a, b, c)` index. Adding a separate `(a, b)` index alongside it is pure duplication. Order the unique's columns so the hot lookup is its prefix, then say so in the migration docblock (this is why `catalog_variant_case_whitelists` has no `catalog_vcw_variant_format_index`, despite design D6's prose).
- **`Schema::getColumnListing()` + `toEqualCanonicalizing()` pins a table's full shape cross-engine.** PG and SQLite return the columns in different orders; canonicalizing compares as sets. Pairs well with the `foreach ($columns as $column) expect($column)->not->toContain('break')` guard idiom (from `CaseConfigurationTest`) when a spec's contract is the ABSENCE of a field.
- **SQL `LIKE` is an over-approximating PREFILTER, never the predicate, when the pattern contains `_`.** `_` is a single-character wildcard on both engines, so `LIKE '%.identity_updated'` also matches a hypothetical `.identityXupdated`; `ESCAPE` semantics differ between SQLite and PostgreSQL, so escaping is not engine-neutral. Shape: `LIKE`-prefilter in SQL → `orderByDesc('id')->pluck(…)` → first EXACT `str_ends_with()` match in PHP. Identical result to an exact filter (a superset, narrowed exactly), engine-free. Both review-freshness readers use it.
- **A derive-from-audit predicate MUST name the verb set it derives from.** A raw "latest audit action" read is correct only while a single writer owns the trail; the second writer silently inverts it (here: a post-rejection `enrichment_updated` row would have unblocked a rejected entity — the S1 hole). Filter to the participating verbs, and record the collision rule — *no future verb may end with a participating suffix unless it is meant to participate* — where the next writer will read it (`ApprovalGovernance` docblock; CONTEXT.md at task 7.1).
- **When the console duplicates a domain predicate (it may not import `Catalog\Lifecycle`), pin the duplication with a lock-step test.** One dataset, one fixture, each row an audit history: assert the console's `->visible()` and the domain's activate verdict TOGETHER (visible ⇔ blocked, hidden ⇔ activates). Drift then reds a test instead of shipping a button that lies about what the domain will do. `ReviewFreshnessVerbFilterTest`'s last case is the template.
- **A method rename reaches into test PROSE.** Always `grep -rn '<oldName>' tests/` after renaming anything in `app/`: comments naming a method that no longer exists are stale documentation no compiler, linter or type-checker will catch (7 console test files here).
- **NEVER `->toBe()` a decoded jsonb snapshot map — use `->toEqual()`.** PG `jsonb` reorders keys by key LENGTH then bytewise, and Eloquent's `array` cast decodes what PG stored, so the PHP array's key order is the ENGINE's, not the writer's. An identical-comparison of an audit `before`/`after` (or any JSON payload) map passes or fails on PG according to how long its keys happen to be — `['name' => …, 'version' => 1]` passes, `['constituents' => […], 'version' => 1]` fails, in the same file. `==` (`toEqual`) ignores key order while still comparing nested LISTS element-wise by index, so ordered id-lists stay pinned; `toHaveKey('k', $v)` covers a single field. **Every remaining task (2.1, 2.2, 3.1, 4.1, 6.x) asserts an audit snapshot map.** (`knowledge/testing/rules.md` trap 3, refined 2026-07-08.)
- **A guard-then-`$apply`-closure mechanism must prove the closure is NOT invoked on rejection.** Flip a `$applied = true` flag inside the test's closure and assert it stayed `false`, alongside the row/audit/event-unchanged assertions. Otherwise "the transaction rolled back" is proven but "the Action's own re-checks never ran" is not — and that is exactly the property the per-entity Actions rely on when they put a dedup / N ≥ 2 / constituent-state check inside the closure.
- **When two writers share an audit trail that a THIRD party reads by string-matching, the derivation must be one shared function, not a copy.** `ApprovalGovernance` suffix-matches `catalog.<segment>.<verb>`; `LifecycleTransition` and `CatalogContentEdit` both produce it. `CatalogAuditEnvelope::action()` makes divergence structurally impossible instead of merely unlikely.
- **"This query was skipped" is a real, cheap assertion — `DB::listen` + a needle only that query emits, pinned in LOCK-STEP with a positive.** A conditional query (a dedup re-check that runs only when the identity key moves) is otherwise UNOBSERVABLE from outside: the skipped path and the ran-but-found-nothing path return the same verdict. Capture `QueryExecuted::$sql` and match the join's table **alias** (`as "wine"` — both engines' grammars wrap an alias in double quotes; the unaliased read of the same table does not match). Then assert the SAME needle fires on the key-moving edit, so a broken needle reds instead of silently "proving" a skip. Precedent: `UpdateProductMasterIdentityTest::it('runs the dedup join only when the identity key moves')`.
- **A `TranslatableText` diffs by its i18n-keyed MAP, never by object identity.** `$old?->jsonSerialize() != $new?->jsonSerialize()` — loose `!=` because `==` on string-keyed arrays ignores key order, and two prose values carrying the same locale ⇒ text pairs are the same content whatever order they were built in. `null` is a legitimate value on either side (untranslated, or cleared) and must survive verbatim into the audit snapshot. Applies to `winery_story` (2.1) and `tasting_notes` (4.1 — whose no-op rule makes the same comparison LOAD-BEARING, not merely cosmetic).
- **A spec that says "every OTHER row" may still be unreachable-by-construction — write the guard, and say so.** The BR-Identity-1 edit dedup runs only when the key MOVED, and a moved key can never equal the row's own current key, so `whereKeyNot` excludes nothing today. It ships because the delta spec states the rule that way, and because an unconditional future caller (a surface that re-checks on every save) would self-collide without it. Mark the branch as defence-in-depth in the docblock rather than letting a reviewer read it as live logic — and don't let a test claim to exercise it.
- **An Action's own re-checks belong INSIDE `$apply`, never before `edit()` — even when the sibling Create action validates the same rule up front.** `CreateCompositeSku` runs the N ≥ 2 floor as pre-transaction input validation; `UpdateCompositeSkuComposition` cannot, because the mechanic's state guard and operator floor must WIN over it: a one-element edit of a `retired` entity has to report the `retired` state, not the count. Pin the precedence directly — submit a *deliberately invalid* payload against a `retired` entity and assert the state rejection (2.2). Every Action on the mechanic inherits this (3.1's unknown-CC-id, 4.1's no-op).
- **One invariant ⇒ one exception CLASS; one surface ⇒ one reason.** The edit path re-asserts the activation cascade (an `active` Composite must never come to reference a non-`active` constituent — the gate never runs again on an already-`active` entity), so it throws `ActivationCascadeViolation` — but under a SECOND factory + lang key, because the operator pressed *save composition*, not *activate*. A catcher learns the cascade broke whichever door it came through; the operator reads copy naming what they actually did. Never reuse a gate primitive whose copy names the wrong verb.
- **PHPStan max: `Collection::map()->all()` is `array<int,int>`, never `list<int>`** — Eloquent collection keys are not provably contiguous. Wrap in `array_values(...)`; do NOT relax the `@return list<int>`, because a `list` is exactly what an ORDERED, index-compared audit snapshot (constituent order, CC-id sets) needs. Bit the Action and its test helper alike (2.2); will bite 3.1's CC-id sets and 6.3's multi-select the same way.
- **`sync()` on a relation carrying `orderByPivot` is safe** (`newPivotQuery()` does not inherit the relation's ORDER BY): it detaches, attaches and rewrites the pivot column on survivors in ONE call. Follow it with `unsetRelation(...)` so the returned model does not serve the stale cache the `before` read materialised. And a pure REORDER is a real content change — `!==` on two `list<int>` compares element-wise by index, the correct diff for an ordered set.
- **PHPStan max rejects a `void` call as a `match` arm** ("Result of function … is used"). Bind the arms to `function (): void {}` closures and invoke the result (`$applyHistory();`) — also keeps `actingAs()` re-binding inside the arm where it belongs.
- **Assert an append-only trail as ONE ordered list of action strings, not as N per-verb counts.** `expect(auditActions($entity))->toBe(['…submitted', '…rejected', '…identity_updated', …])` pins order, per-verb counts AND the absence of any unexpected extra row in a single expectation — the count-per-verb style silently tolerates a spurious row of a verb nobody counted. Order is the point on an append-only trail: it is what makes "latest wins" observable. Precedent: `IdentityEditRearmsReviewTest::rearmAuditActions()` (2.3).
- **Under PHPStan max, hydrate models rather than `pluck()` a scalar column you will type.** `pluck('action')` yields `mixed`, and `(string) $mixed` is `cast.string` — forbidden, and a cast added only to silence the analyser is the wrong fix. `->get()->map(fn (AuditRecord $r): string => $r->action)->all()` reads the column through its `@property string` declaration, so the `list<string>` return type is *earned*. `pluck()` stays right where the value is immediately narrowed (`is_string(…)` / `is_numeric(…)`, as `ApprovalGovernance` does) or compared loosely (`->toEqual()` on an uncast bigint `actor_id`). Still wrap in `array_values()` per the `list` pattern above.
- **A derived "latest wins" predicate needs a test where TWO arming causes coexist.** A trail holding a `.rejected` *and* a later `.identity_updated` is the only shape that discriminates latest-wins from any-cause-pending: both implementations block, both pass every count assertion, and they differ ONLY in the localized reason the operator reads. Assert the discriminating token (`edited` vs `un-remediated`), never merely that *an* exception was thrown. Generalises to any audit-derived condition with more than one arming verb (2.3).
- **`CatalogContentEdit` has TWO entry points over ONE private `perform()`: `edit()` (re-versioning) and `maintain()` (audit-only).** Pick by asking *"is this the entity's identity?"*, never *"should `version` bump?"* — the two questions have the same answer and only the first is a domain question. `version` is the IDENTITY version, so a whitelist / enrichment write must not move it; correspondingly its verb may not end with a review-freshness suffix (D5) — which is the SAME statement read from the governance side ("a reviewed-then-maintained entity still activates"). The three facts — no `version`, no re-arm, no event — are one fact. A boolean `$reVersion` parameter on `edit()` was rejected: it would make every call site restate a decision the verb already encodes. `maintain()` skips the `UPDATE` entirely when `$apply` reports no own columns, so a pivot-only write does not even touch `updated_at` (3.1).
- **Replace a pivot SET as a DELTA (`array_diff` both ways), not delete-then-reinsert.** Survivors keep their row identity and their `created_at` — which on an admission pivot is real information ("since when has this packaging been admitted?"). Scope BOTH legs to the owning key(s) explicitly (`relation()->where('format_id', …)`): the unique triple is what makes each pair independent, and a `delete()` that forgets the second predicate silently empties the sibling pairs. Canonicalise the incoming set (`array_unique` → `sort`) before diffing, so a given set always audits as the same list whatever order the console sent (3.1).
- **When an operator supplies arbitrary FK ids, pre-check existence INSIDE the write's transaction and reject with a domain exception naming only the OFFENDING subset.** The foreign key would refuse the write anyway — as a driver error carrying a constraint name and no operator-facing meaning. The pre-check owns the message, the constraint owns the integrity; neither is redundant. Contrast `CreateCompositeSku`, which lets its FK speak alone because no operator ever types a constituent id there (the console prefills a select). One parameterized `UnknownCatalogReference::forIds($entity, $missingIds)` serves every reference kind, mirroring `IllegalContentEdit`/`IllegalLifecycleTransition` — 5.2's producer-existence guard should reuse it rather than mint `UnknownProducerReference` (3.1).
- **A throwing assert whose caller must ALSO read the value should RETURN the proven value — never carry `@phpstan-assert`.** Both express "past this call it is non-null", but `@phpstan-assert !null $p` is reported as `method.alreadyNarrowedType` at every call site that already passes a non-nullable argument (`ActivateCompositeSku` loops relation-loaded constituents), so it cannot coexist with mixed call sites. `@template TParent of X` + `@param TParent|null` + `@return TParent` narrows for the one caller that needs it and is invisible to the ones that don't. **Cost to budget:** a `void`→value return breaks any `gate: fn (…) => $this->gate->assert(…)` arrow function — `Closure(T): void` contract meets PHP's implicit arrow-fn return, and an arrow fn *cannot* declare `: void` (`Fatal error: A void function must not return a value`). Convert that caller to a block closure (3.2).
- **A gate whose empty read means PASS must say so loudly in its docblock.** Module 0's cascade/producer gates are fail-closed (absent parent ⇒ reject); the Layer-1 whitelist is the sole inversion — zero rows for a pair is § 7.1's PERMISSIVE default ("presence narrows, absence admits"), because a whitelist states something about *packaging possibility*, not about a *reference resolving*. The two shapes are one character of intent apart (`if ($rows === []) return;` vs `if ($row === null) throw;`), so a new gate should name which family it joins (3.2).
- **`ApprovalGovernance::creatorOf` is a derive-from-EVENT predicate — the audit-trail derivation's twin, and the one the S1 lesson has NOT been applied to.** It reads the actor of the entity's EARLIEST `domain_events` row, unfiltered by event name. In production the claim holds *structurally* — an entity has no event before its `*Created`, so the earliest row IS the creation — which is why `EnrichmentDataUpdated` (the first non-lifecycle event to land on an entity's event stream) needed no filter. In TESTS the claim inverts: a **factory-built** entity has no creation event, so the first event any Action records takes the creator's seat in the separation-of-duties triple. **Rule:** once the Action under test records *any* domain event, a test whose subject is SoD must build its entity through the real `Create*` action. `SetVariantCaseWhitelistTest` may keep the factory only because whitelist maintenance records no event; `UpdateProductVariantEnrichmentTest` cannot (4.1 — two tests failed exactly this way, `insufficientSeparation` with the enricher mistaken for the creator).
- **A no-op is the `$apply` closure's DECLARATION, never the mechanism's inference.** `perform()` treats a `null` return as *nothing to record*: no `UPDATE`, no `version`, no audit row. It is a per-Action semantic, and only enrichment has an observable idempotence contract to justify it — `EnrichmentDataUpdated` must fire only on a real change, so a consumer can trust every event it receives. The contrast is `UpdateCompositeSkuComposition`: an identical ordered set still versions and still audits, because an operator re-affirming reviewed content IS the audited fact. Ask "does this Action's OUTPUT make idempotence observable?", not "did the values change?".
- **Prove a no-op branch and its comparator are load-bearing by MUTATING them, not by reading them.** A silent no-op is the one behaviour whose test passes for free if the branch is missing something (the counts are zero either way when the fixture is wrong). Flip `!=` → `!==` on the i18n-map diff and delete the early `return null;`, one at a time, and watch the same test red for two different counts (1 and 2 events, 4.1). Cheap, decisive, and it is the only check that distinguishes "the no-op works" from "the test never reached the write path".
- **"Untouched" is asserted against a SNAPSHOT taken before the disturbance, not against a re-derived expectation.** For non-retroactivity claims (R10: a whitelist reduction must not reach an already-`active` SKU), capture the entity's audit-row ids in order *before* the disturbing write, then `toBe()` the same query after — that catches a spurious *added* row, which per-verb counts and a state re-check both miss. Pair it with state + `version` + the named event's count (3.2).
- **Widening an enum whose PG CHECK derives from `cases()` needs an ADMISSION test, not the existing rejection test — and the SQLite lane structurally cannot fail.** The migration's driver guard means SQLite has *no* CHECK, so a stale `IN ('active','retired')` list is invisible there; the shipped "rejects an out-of-domain literal" test still passes on PG too (it probes a token that is out of domain either way). What reds is a RAW query-builder insert of **every** `cases()` token (raw bypasses the enum cast, so on PG only the CHECK can stop it) plus a `pg_get_constraintdef` token count. Proven by mutation (5.1): hardcoding the 2-token list reds PG with `23514` on the `registered` row and leaves SQLite green. Any future enum-case append inherits this obligation.
- **`DB::selectOne()` returns `mixed` — at PHPStan max, every `$row->col` on it is an error.** Use the repo's `DB::scalar()` idiom (`EnvironmentTest`) and narrow honestly at the boundary: `$raw = DB::scalar($sql); return is_string($raw) ? $raw : '';`. Then assert `not->toBe('')` first — that turns a *missing* row (constraint absent, table absent) into a failure instead of a vacuous pass through the string assertions below it. No cast, no `assert()`, no `@var`.
- **Adding an EARLIER event to a latest-wins watermark projection is free; adding a later one would not be.** The `last_event_id` guard (`$event->id <= $state->last_event_id` ⇒ no-op) already encodes "a lower id can never overwrite a higher". `ProducerCreated` is by construction the *lowest*-id event of its producer, so a redelivery after `ProducerActivated` cannot downgrade `active → registered` — no new mechanism, no ordering flag, no per-status precedence table (5.1, D7). Before adding an event to a projection, ask where its id falls in the entity's stream: that answers whether the watermark alone suffices.
- **A new guard that makes an existing test's PREMISE unreachable does not invalidate that test — it relocates the premise.** A creation-time guard (5.2's producer existence) can strand a downstream test that reached its state by *not* satisfying the new precondition (the gate's absent-row branch: no Master can now be BORN under an unprojected producer). Three wrong moves: delete the test, weaken the guard, or rebuild the entity by factory (which silently breaks any `creatorOf` / SoD read — no `*Created` event). The right move is to ask **which real states still reach that branch** (here: a Master predating the guard, or a purged read model) and construct one — create through the real lineage, then remove the row. The test survives, and its comment now documents a production reality the code depends on. Applies to every fail-closed check added upstream of an existing gate.
- **Choose a read-model test fixture by what the test MEASURES: seed the row when it counts events, drive the event when the projection IS the subject.** `ProducerProjectionFixture::known()` writes `catalog_producer_states` directly and returns the id, so `producerId: ProducerProjectionFixture::known(42)` states the precondition inline. Event-free is not laziness: `SpineCreationChainTest` pins exactly seven `*Created` families and `ProductMasterTest` asserts a lone `ProductMasterCreated` — a fixture that appends to the ledger it is measured against is a worse fixture (the file-header comment on `ProductMasterConsoleChainTest` had already reasoned this out for the gate). Seed `last_event_id: 0` so the projector's strictly-advances watermark still applies any REAL producer event recorded afterwards; make it `firstOrCreate` so it composes with the event path in either order. Default it to the WEAKEST admitting status (`registered`), so no test opens the activation gate by accident. The projector's "sole writer" claim is about `app/`; a `tests/Support/` fixture standing outside it is precedented (`tests/Support/Platform/*`).
- **A domain rejection's LANDING PLACE is decided by the console action's SHAPE, not by the exception's type.** The console may not import a module's `Exceptions` namespace, so it catches `RuntimeException` by base type and cannot discriminate. Therefore: a VERB-shaped write (submit / activate / retire — no form) surfaces every rejection as a danger notification (`lifecycleAction`); a FORM-shaped write (create page, content-edit modal) surfaces every rejection as a validation error on ONE designated field (`OperatorConsoleCreateRecord::createRejectionField()`, `contentEditAction($rejectionField)`). The spec asks for a validation error where a form exists and permits either where it does not — one rule covers all three surfaces, and no surface reimplements a gate (6.1).
- **A `ValidationException` thrown from inside a MOUNTED Filament action must be keyed at that action's state path (`mountedActions.<n>.data.<field>`), not at the bare field name** — otherwise the error exists in the bag and renders nowhere. Derive the path from the mounted schema (`getMountedActionSchemaName()` → `getSchema(…)->getStatePath()`), never by reconstructing the index: that is the exact derivation Filament's own `assertHasFormErrors()` performs, so the production key and the test assertion cannot drift. `callMountedAction()` catches the exception, keeps the modal open and rethrows (6.1).
- **A Filament `Select` silently contributes `Rule::in(<its options>)`, so a domain guard behind an operator-chosen FK is a BACKSTOP the console can never reach.** `Select::getInValidationRuleValues()` returns `[]` for a value with no option label, which reds the field before any Action runs. When the option list and the guard read the same source (both read `catalog_producer_states`), the guard is structurally unreachable from that surface. Assert the observable contract (validation error · nothing written) at the console and prove the guard at the domain (6.1 — the delta requirement is worded for exactly this layering).
- **The console needs no `$record->refresh()` after a content edit, because `recordOf()` hands the Action the page's OWN model instance and `CatalogContentEdit::perform()` locks, re-reads and updates THAT instance.** The infolist therefore renders the post-edit `version` in the same response. I shipped the defensive re-read first and mutation-tested it: no test red. Delete a "safety" re-read before believing in it — and pin the property it leaned on (read `version` back off the page's record, which reds if a future Action ever writes a copy) (6.1).
- **PHPStan max does not read a `@param` docblock on a closure in ARGUMENT position** (`->action(/** @param … */ function (…) {})`), nor reliably on one bound to a variable. Do not answer that with a wider docblock or an inline `@var`; answer it with a TRUER contract. Filament hands an action closure a native `array`, so `array<string, mixed>` is an unprovable claim at that boundary — and a worthless one, since every caller indexes by string literal (valid on `array<array-key, mixed>`) and must narrow each `mixed` value anyway (6.1).
- **`git-guardrails.sh` blocks a bash heredoc whose PROSE contains `rm` inside a word.** Its `(rm|mv)[^|;&]*[[:space:]](spec|openspec/specs)/` pattern matches the `rm` in `form`/`perform`/`transform` when ` spec/` appears later in the same sentence. Write long `progress.md` entries to a temp file and `cat` them in (6.1).
- **If PHPStan max can prove a test assertion, the assertion is dead code — delete it, don't restate it.** `is_a(A::class, B::class)` on two literals draws `function.impossibleType` ("always false") exactly as `class_exists('<literal>')` does: the analyser resolves class-name literals, so a runtime check of a *static* class relation asserts nothing at runtime and everything at analysis time. Where the relation genuinely matters (5.2's "the unknown-producer rejection is not the dedup speaking"), let the `toThrow(Class::class, 'message fragment')` pair carry it — the class AND its reason both have to match — and prove the assertion bites by MUTATION (swap the two guards; watch that one test red). Same family as `knowledge/testing/hypotheses.md` § "Prove a class is ABSENT by listing the directory".
- **Filament's `assertNotified()` cannot assert a notification's BODY, and it EATS the evidence.** It matches a `string` argument against the title only; its `Notification` overload matches by the notification's random `id`. And `Notification::assertNotified()` mounts the notifications Livewire component, whose `mount()` **pulls** the session — so any snapshot taken after it is empty. When the fact under test lives in the body (every domain rejection surfaced by `surfaceLifecycleOutcome` shares the `action_failed` title; the DOMAIN's localized message is the body), read the session yourself, FIRST. The key is not the obvious one: `Notification::send()` pushes onto `filament.notifications`, but on every Livewire `dehydrate` `NotificationsServiceProvider` **moves** that array onto `filament.claimed_notifications` with a `put` — overwriting, so only the LAST request's notifications survive. Read `claimed_notifications` first, fall back to `notifications` (a send outside a Livewire request). This is the only way to prove "the console surfaces the domain's own message" rather than merely "a notification appeared" (6.2).
- **A Filament `Textarea`/`TextInput` dehydrates a blank field to `null`, never `''`.** So a page's `$x === null || $x === ''` narrowing has a dead arm — deleting the `=== ''` half reds nothing, because the empty string never arrives. Keep the arm anyway (`TranslatableText::of(['en' => ''])` is a value the domain must never be handed, and the page is the one place that can promise it never will), but MARK it as defence-in-depth and let no test claim to exercise it: assert the observable clearing contract (stored value is `null`, a real change, one event, `after: {field: null}`) instead. Same family as the BR-Identity-1 `whereKeyNot` branch. `ViewProductMaster::editIdentityAction()` (6.1) carries the identical arm on `winery_story`.
- **Drive a modal's `live()` field in a Livewire test with `->set('mountedActions.<n>.data.<field>', $v)`, not `->setActionData()`.** `setActionData()` writes state without hooks, so an `afterStateUpdated` re-prefill is invisible to it; `set()` on the mounted action's state path takes the same round-trip the operator's select does and fires the hook. The state path is the one `mountedActionErrorKey()` derives in production — the test and the surface therefore agree by construction. Pair the assertion with a SECOND `set()` to a different operand and assert the shown set CHANGES: a re-prefill that fires once but never re-fires would otherwise pass (6.2).
- **When a modal has TWO operands and one SELECTS what the other replaces, the selector must be `live()` and re-prefill the payload — and the pair-scoped read is what makes it correct.** `SetVariantCaseWhitelist` takes a whole admitted set per (Variant, Format) pair, never a patch, so an operator who switches Format without a re-prefill would submit the previous pair's set and silently rewrite the new one. Two mutations pin it: delete the hook, and drop the `->where('format_id', …)` from the prefill read — each reds exactly the re-prefill test. Generalises to any replace-a-set-under-a-key modal (6.2; 6.3's ordered-constituent multi-select has ONE operand and needs neither).
- **`callAction($name, $data)` cannot SHRINK a list-valued modal field — the tail survives and the test passes for the wrong reason.** `fillFormDataForTesting()` writes state leaf-by-leaf (`Arr::dot` + `data_set`), and its `unsetMissingNumericArrayKeys()` prune accumulates `$currentStatePath .= ".{$key}"` *inside* the loop, so by the `data` key the path reads `mountedActions.0.name.data.…` and never `startsWith()` the schema state path — nothing is pruned. Submitting `[a]` against a `[a, b]` prefill therefore submits `[a, b]`: 6.3's `N ≥ 2` floor test saw zero errors and a real no-op edit that bumped `version`. Shrinking to `[]` DOES work (`Arr::dot` keeps an empty array as a leaf). The reliable shrink — and the operator's actual deselection gesture — is `->mountAction($name)->set('mountedActions.0.data.<field>', [a])->callMountedAction()`. Equal-length or longer submissions are safe through `callAction()`. See `knowledge/filament/hypotheses.md`.
- **`assertHasActionErrors(['field' => $message])` truncates the expected message at the first `:`.** Livewire's `assertErrorMatchesRuleOrMessage()` does `Str::before($expected, ':')` so `min:3` can be written as a rule — and applies it to messages too. A colon-bearing domain message (`'Cannot change the composition of this CompositeSku: it is active, …'`) then never matches, and the failure reads *"Component has no matching failed rule or error message"* — indistinguishable from a missing error. Colon-free messages match fine, so the trap bites only some copy, and a colon added to `lang/*/catalog.php` later reds a passing test.
