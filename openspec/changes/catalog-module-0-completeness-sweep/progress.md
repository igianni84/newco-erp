# Progress — catalog-module-0-completeness-sweep

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Cross-engine constraint proof = nested `DB::transaction()` savepoint + the RIGHT exception class.** Wrap the violating DML in `DB::transaction(fn () => …)` so PostgreSQL's aborted-transaction state rolls back to the savepoint and the follow-on row-state assertions still run (SQLite needs no such care — PG does). Assert `UniqueConstraintViolationException` for a unique violation, plain `QueryException` for an FK `RESTRICT` (Laravel narrows only the unique case). Precedents: `ProductReferenceTest` (unique), `ProducerStateProjectionTest` (raw-insert CHECK).
- **Verify the 63-byte identifier rule empirically, don't eyeball it.** After a PG run the schema persists in `newco_test`: `docker exec pg psql -U newco -d newco_test -c "SELECT conname, LENGTH(conname), contype FROM pg_constraint WHERE conrelid = '<table>'::regclass ORDER BY conname;"` shows every FK/unique/CHECK name and its real byte length. PG truncates *silently*, so an assertion-by-name test can pass for the wrong reason if two names collide after truncation — this query is the cheap proof they didn't.
- **A composite unique's leftmost prefix IS the supporting index for the narrower lookup.** Both engines answer `WHERE a = ? AND b = ?` from a `UNIQUE (a, b, c)` index. Adding a separate `(a, b)` index alongside it is pure duplication. Order the unique's columns so the hot lookup is its prefix, then say so in the migration docblock (this is why `catalog_variant_case_whitelists` has no `catalog_vcw_variant_format_index`, despite design D6's prose).
- **`Schema::getColumnListing()` + `toEqualCanonicalizing()` pins a table's full shape cross-engine.** PG and SQLite return the columns in different orders; canonicalizing compares as sets. Pairs well with the `foreach ($columns as $column) expect($column)->not->toContain('break')` guard idiom (from `CaseConfigurationTest`) when a spec's contract is the ABSENCE of a field.
- **SQL `LIKE` is an over-approximating PREFILTER, never the predicate, when the pattern contains `_`.** `_` is a single-character wildcard on both engines, so `LIKE '%.identity_updated'` also matches a hypothetical `.identityXupdated`; `ESCAPE` semantics differ between SQLite and PostgreSQL, so escaping is not engine-neutral. Shape: `LIKE`-prefilter in SQL → `orderByDesc('id')->pluck(…)` → first EXACT `str_ends_with()` match in PHP. Identical result to an exact filter (a superset, narrowed exactly), engine-free. Both review-freshness readers use it.
- **A derive-from-audit predicate MUST name the verb set it derives from.** A raw "latest audit action" read is correct only while a single writer owns the trail; the second writer silently inverts it (here: a post-rejection `enrichment_updated` row would have unblocked a rejected entity — the S1 hole). Filter to the participating verbs, and record the collision rule — *no future verb may end with a participating suffix unless it is meant to participate* — where the next writer will read it (`ApprovalGovernance` docblock; CONTEXT.md at task 7.1).
- **When the console duplicates a domain predicate (it may not import `Catalog\Lifecycle`), pin the duplication with a lock-step test.** One dataset, one fixture, each row an audit history: assert the console's `->visible()` and the domain's activate verdict TOGETHER (visible ⇔ blocked, hidden ⇔ activates). Drift then reds a test instead of shipping a button that lies about what the domain will do. `ReviewFreshnessVerbFilterTest`'s last case is the template.
- **A method rename reaches into test PROSE.** Always `grep -rn '<oldName>' tests/` after renaming anything in `app/`: comments naming a method that no longer exists are stale documentation no compiler, linter or type-checker will catch (7 console test files here).
- **NEVER `->toBe()` a decoded jsonb snapshot map — use `->toEqual()`.** PG `jsonb` reorders keys by key LENGTH then bytewise, and Eloquent's `array` cast decodes what PG stored, so the PHP array's key order is the ENGINE's, not the writer's. An identical-comparison of an audit `before`/`after` (or any JSON payload) map passes or fails on PG according to how long its keys happen to be — `['name' => …, 'version' => 1]` passes, `['constituents' => […], 'version' => 1]` fails, in the same file. `==` (`toEqual`) ignores key order while still comparing nested LISTS element-wise by index, so ordered id-lists stay pinned; `toHaveKey('k', $v)` covers a single field. **Every remaining task (2.1, 2.2, 3.1, 4.1, 6.x) asserts an audit snapshot map.** (`knowledge/testing/rules.md` trap 3, refined 2026-07-08.)
- **A guard-then-`$apply`-closure mechanism must prove the closure is NOT invoked on rejection.** Flip a `$applied = true` flag inside the test's closure and assert it stayed `false`, alongside the row/audit/event-unchanged assertions. Otherwise "the transaction rolled back" is proven but "the Action's own re-checks never ran" is not — and that is exactly the property the per-entity Actions rely on when they put a dedup / N ≥ 2 / constituent-state check inside the closure.
- **When two writers share an audit trail that a THIRD party reads by string-matching, the derivation must be one shared function, not a copy.** `ApprovalGovernance` suffix-matches `catalog.<segment>.<verb>`; `LifecycleTransition` and `CatalogContentEdit` both produce it. `CatalogAuditEnvelope::action()` makes divergence structurally impossible instead of merely unlikely.
- **"This query was skipped" is a real, cheap assertion — `DB::listen` + a needle only that query emits, pinned in LOCK-STEP with a positive.** A conditional query (a dedup re-check that runs only when the identity key moves) is otherwise UNOBSERVABLE from outside: the skipped path and the ran-but-found-nothing path return the same verdict. Capture `QueryExecuted::$sql` and match the join's table **alias** (`as "wine"` — both engines' grammars wrap an alias in double quotes; the unaliased read of the same table does not match). Then assert the SAME needle fires on the key-moving edit, so a broken needle reds instead of silently "proving" a skip. Precedent: `UpdateProductMasterIdentityTest::it('runs the dedup join only when the identity key moves')`.
- **A `TranslatableText` diffs by its i18n-keyed MAP, never by object identity.** `$old?->jsonSerialize() != $new?->jsonSerialize()` — loose `!=` because `==` on string-keyed arrays ignores key order, and two prose values carrying the same locale ⇒ text pairs are the same content whatever order they were built in. `null` is a legitimate value on either side (untranslated, or cleared) and must survive verbatim into the audit snapshot. Applies to `winery_story` (2.1) and `tasting_notes` (4.1 — whose no-op rule makes the same comparison LOAD-BEARING, not merely cosmetic).
- **A spec that says "every OTHER row" may still be unreachable-by-construction — write the guard, and say so.** The BR-Identity-1 edit dedup runs only when the key MOVED, and a moved key can never equal the row's own current key, so `whereKeyNot` excludes nothing today. It ships because the delta spec states the rule that way, and because an unconditional future caller (a surface that re-checks on every save) would self-collide without it. Mark the branch as defence-in-depth in the docblock rather than letting a reviewer read it as live logic — and don't let a test claim to exercise it.
- **An Action's own re-checks belong INSIDE `$apply`, never before `edit()` — even when the sibling Create action validates the same rule up front.** `CreateCompositeSku` runs the N ≥ 2 floor as pre-transaction input validation; `UpdateCompositeSkuComposition` cannot, because the mechanic's state guard and operator floor must WIN over it: a one-element edit of a `retired` entity has to report the `retired` state, not the count. Pin the precedence directly — submit a *deliberately invalid* payload against a `retired` entity and assert the state rejection (2.2). Every Action on the mechanic inherits this (3.1's unknown-CC-id, 4.1's no-op).
- **One invariant ⇒ one exception CLASS; one surface ⇒ one reason.** The edit path re-asserts the activation cascade (an `active` Composite must never come to reference a non-`active` constituent — the gate never runs again on an already-`active` entity), so it throws `ActivationCascadeViolation` — but under a SECOND factory + lang key, because the operator pressed *save composition*, not *activate*. A catcher learns the cascade broke whichever door it came through; the operator reads copy naming what they actually did. Never reuse a gate primitive whose copy names the wrong verb.
- **PHPStan max: `Collection::map()->all()` is `array<int,int>`, never `list<int>`** — Eloquent collection keys are not provably contiguous. Wrap in `array_values(...)`; do NOT relax the `@return list<int>`, because a `list` is exactly what an ORDERED, index-compared audit snapshot (constituent order, CC-id sets) needs. Bit the Action and its test helper alike (2.2); will bite 3.1's CC-id sets and 6.3's multi-select the same way.
- **`sync()` on a relation carrying `orderByPivot` is safe** (`newPivotQuery()` does not inherit the relation's ORDER BY): it detaches, attaches and rewrites the pivot column on survivors in ONE call. Follow it with `unsetRelation(...)` so the returned model does not serve the stale cache the `before` read materialised. And a pure REORDER is a real content change — `!==` on two `list<int>` compares element-wise by index, the correct diff for an ordered set.
- **PHPStan max rejects a `void` call as a `match` arm** ("Result of function … is used"). Bind the arms to `function (): void {}` closures and invoke the result (`$applyHistory();`) — also keeps `actingAs()` re-binding inside the arm where it belongs.
- **Assert an append-only trail as ONE ordered list of action strings, not as N per-verb counts.** `expect(auditActions($entity))->toBe(['…submitted', '…rejected', '…identity_updated', …])` pins order, per-verb counts AND the absence of any unexpected extra row in a single expectation — the count-per-verb style silently tolerates a spurious row of a verb nobody counted. Order is the point on an append-only trail: it is what makes "latest wins" observable. Precedent: `IdentityEditRearmsReviewTest::rearmAuditActions()` (2.3).
- **Under PHPStan max, hydrate models rather than `pluck()` a scalar column you will type.** `pluck('action')` yields `mixed`, and `(string) $mixed` is `cast.string` — forbidden, and a cast added only to silence the analyser is the wrong fix. `->get()->map(fn (AuditRecord $r): string => $r->action)->all()` reads the column through its `@property string` declaration, so the `list<string>` return type is *earned*. `pluck()` stays right where the value is immediately narrowed (`is_string(…)` / `is_numeric(…)`, as `ApprovalGovernance` does) or compared loosely (`->toEqual()` on an uncast bigint `actor_id`). Still wrap in `array_values()` per the `list` pattern above.
- **A derived "latest wins" predicate needs a test where TWO arming causes coexist.** A trail holding a `.rejected` *and* a later `.identity_updated` is the only shape that discriminates latest-wins from any-cause-pending: both implementations block, both pass every count assertion, and they differ ONLY in the localized reason the operator reads. Assert the discriminating token (`edited` vs `un-remediated`), never merely that *an* exception was thrown. Generalises to any audit-derived condition with more than one arming verb (2.3).

## [2026-07-08 15:42] — 1.3 CatalogContentEdit mechanic

- **What was implemented.** The shared content-edit mechanism (design D1/D2/D3; product-catalog — *In-Place Versioned Identity Edits*; Module 0 PRD BR-Audit-1 / DEC-073) — the second catalog audit writer, which task 1.2's verb filter made survivable.

That is the seam tasks 2.x/3.x/4.x plug into: their own re-checks (identity dedup, N ≥ 2, constituent state) and related-row writes (wine attributes, join tables) live inside the closure, inside this transaction.
