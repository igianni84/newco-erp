# Design — parties-holds

## Context

`parties-compliance` shipped the KYC and sanctions **state** on Customer/Producer and left two named seams in the shipped `party-registry` spec: *"The `kyc` Hold coupling (auto-place on KYC `pending`, auto-lift on `verified`) and the unified Hold registry SHALL remain deferred to `parties-holds`."* This change builds that registry and wires the one coupling. It is **additive** — one new table (`parties_holds`), new enums/Actions/events/exception/read-contract, and **two MODIFIED Actions** (`RequireKyc`/`RecordKycVerified` gain the `kyc`-Hold place/lift). It introduces **no new dependency** and steps through **no open ADR gate** (events deliver inline; no queue/object-storage/payment-provider).

The platform substrate this builds on (verified in `app/Platform/Events` + `app/Modules/Parties`):

- `DomainEventRecorder::record(string $name, string $module, ActorRole $actorRole, ?int $actorId, string $entityType, string $entityId, array $payload, ?string $correlationId = null, ?int $causationId = null): DomainEvent` — appends inside the caller's open transaction (throws at `transactionLevel() === 0`); inline delivery post-commit (no `queued` consumer → no ADR gate).
- `ActorContext::role(): ActorRole` / `actorId(): ?int` — the actor seam (`ActorRole::System` until principals wire in; an authenticated operator → `ActorRole::NewcoOps`).
- `Module::Parties->value === 'parties'`. `ActorRole` is a backed enum with an existing cast (used by `domain_events.actor_role`).
- The migration idiom (`2026_06_15_000005_create_parties_customers_table.php`): a value-set column is `string` + the backed-enum cast on both engines, **plus a PostgreSQL-only `CHECK`** whose accepted set derives from `Enum::cases()` (SQLite skips the `CHECK`; the cast carries the floor).
- The transition-Action idiom (`RecordCustomerScreening` is the closest mirror — it injects `DomainEventRecorder` + `ActorContext`): `DB::transaction` → `lockForUpdate` re-read → guard → write → `record(...)`. Models stay persistence-only (`$guarded = []`); the **Action is the sole writer**.
- The enum-with-first-predicate idiom (`KycStatus::clears()`): the predicate is the first method, file docblock cites design L-tag + spec §.
- i18n: only `lang/en/parties.php` exists; English is the authored baseline + final fallback, the other five locales fall back **per key** (the locale-parity test requires only `welcome.php` per locale — DEC-127). New copy goes in `lang/en/parties.php` only.
- Boundary law: every reference is within Module K (`ModuleBoundariesTest`); the one cross-module surface is the **read contract** (an interface + a PII-free DTO — never the `Hold` model).

## Goals / Non-Goals

**Goals:**
- The unified `parties_holds` registry — six types (`admin/kyc/payment/fraud/compliance/credit`), three scopes (`customer/account/profile`), `active|lifted` lifecycle, placement + lift audit metadata; Module K is registry-of-record; trigger-agnostic with a manual-placement path.
- `PlaceHold` / `LiftHold` Actions; the **per-type lift discipline** (auto-managed `kyc`/`payment` reject operator lift; `admin`/`fraud`/`compliance`/`credit` lift freely; already-lifted rejected).
- The `kyc`-Hold coupling — auto-place on `RequireKyc` (`pending`), auto-lift on `RecordKycVerified` (`verified`), leave-in-place on `RecordKycRejected`.
- `CustomerHoldPlaced` / `CustomerHoldLifted` events (verbatim § 15.1), PII-free, transactional.
- The uniform read-API tuple `(sanctions_status, active-Hold-list)` with Customer→Profile cascade — exposed **ready** as a contract + DTO + bound implementation.
- Every DB-touching task verified on PostgreSQL 17.

**Non-Goals:**
- **Downstream enforcement** — every DEC-181 transaction-initiation surface that *consumes* the read-API (Module S/C/E) and the by-position gate/decorator. Module K is Hold-blind by design.
- **Automatic Hold triggers** for `payment`/`fraud`/`compliance`/`credit`, and the finance-driven subtypes `STORAGE_PAYMENT_FAILED` / `CHARGEBACK_REVIEW` (Module E events) — the registry is trigger-agnostic; those consumers are Module-E seams.
- **Demand-side status FSMs** (the Hold→`suspended` coupling, AC-K-FSM-9) → `parties-membership-lifecycle`.
- **Club Credit** + the issuance-asymmetry (AC-K-J-16a) → `club-credit`. **GDPR erasure × Hold** (AC-K-J-9a) → the anonymisation change. **Hold expiry** job (§ 4.8.1, under-specified) → deferred automation.
- The **Filament Hold console** → `parties-operator-console`.

## Decisions

### L1 — One additive `parties_holds` table; polymorphic scope by (`scope_type`, `scope_id`), no DB FK
The Hold scope spans three tables (Customer/Account/Profile), so the scope is modelled **polymorphically** as `scope_type` (a `HoldScope` cast) + `scope_id` (`unsignedBigInteger`), with a composite index `(scope_type, scope_id, status)` for the read-API and **no** database foreign key (a single column cannot FK three tables). Within-module referential integrity is the Action's job (it resolves the scoped entity before placing). Columns: `id`, `hold_type` (string + `HoldType` cast), `scope_type` (string + `HoldScope` cast), `scope_id`, `status` (string + `HoldStatus` cast, default `active`), `reason` (nullable string), `placed_actor_role` (string + `ActorRole` cast), `placed_actor_id` (nullable), `lift_reason` (nullable string), `lifted_actor_role` (nullable string + `ActorRole` cast), `lifted_actor_id` (nullable), `lifted_at` (nullable `timestampTz`), `timestamps()` (created_at = placement moment). PG-only `CHECK`s for `hold_type`/`scope_type`/`status` derive from `Enum::cases()`. _Alternative rejected:_ three nullable FK columns (`customer_id`/`account_id`/`profile_id`) + a one-of CHECK — more columns, more constraints, no real gain over the indexed polymorphic pair for a within-module registry.

### L2 — `HoldType::autoLiftable()` is the first predicate; the lift discipline lives on the type
The per-type lift discipline (ADR `2026-06-18-hold-lift-discipline-per-type.md`; DEC-160; AC-K-FSM-11) is encoded as `HoldType::autoLiftable(): bool` → true for `Kyc`/`Payment` only, mirroring `KycStatus::clears()`. `LiftHold` (the operator path) rejects `$hold->hold_type->autoLiftable()` with `IllegalHoldLift::autoManaged($type)`; the `kyc` auto-lift is a **separate within-module path** invoked by `RecordKycVerified` (it lifts the system-managed Hold the operator path forbids). _Alternative rejected:_ a `LiftHold` flag (`auto: bool`) that bypasses the guard — muddies the operator action; the system auto-lift is a distinct caller (`RecordKycVerified`), so it gets its own clean lift path.

### L3 — Actions are the sole writers; place/lift mirror the screening Action
`PlaceHold` and `LiftHold` follow the `RecordCustomerScreening` shape (inject `DomainEventRecorder` + `ActorContext`): one `DB::transaction`, `LiftHold` re-reads `lockForUpdate` and guards `status`+type, both `record(...)` their event in the same transaction (the recorder's open-transaction guard makes write+emit atomic). The `Hold` model stays persistence-only. `PlaceHold` is the **manual operator path** and is **reused by `RequireKyc`** for the auto `kyc` Hold (one Action calling another — the `RetireProducer → SunsetClub` precedent). `RequireKyc`'s shipped docblock already anticipated this: *"that future Hold coupling will be within-module Action orchestration (this action calling the Hold place)."*

### L4 — Events: verbatim § 15.1 names, `entity_type = 'Hold'`, scope in the PII-free payload
The PRD § 15.1 names only `CustomerHoldPlaced` / `CustomerHoldLifted` — there is **no** `ProfileHold*` / `AccountHold*`. The zero-invention reading of AC-K-FSM-10's *"or Profile/Account analogs"* is to record these two verbatim names for **all** scopes, with the scope carried in the payload. The event subject is the **Hold** (`ENTITY_TYPE = 'Hold'`, `entityId = (string) $hold->id`); the payload is `{hold_id, hold_type, scope_type, scope_id, reason}` (placed) / `{hold_id, hold_type, scope_type, scope_id, lift_reason}` (lifted). Each event class is the single source of truth for `NAME` / `ENTITY_TYPE` / `payload(Hold)` (the `CustomerOnboardingScreeningPassed` shape). _Alternative rejected:_ inventing `ProfileHoldPlaced`/`AccountHoldPlaced` — would coin event names the PRD does not name (the codebase's "invent no event" discipline). _Alternative rejected:_ a scope-agnostic `HoldPlaced` — not the § 15.1 name.

### L5 — `reason` is a controlled business-reason; system-placed Holds carry `null` (PII-free)
The Hold `reason` is an operator-supplied **business** reason (the same class of data as the `:state` tokens and id references the codebase already treats as non-PII), so it is safe in the event payload (AC-K-EVT-2 wants it there). **System-placed Holds carry `reason = null`** — the type *is* the reason (the auto `kyc` Hold has no free text), which also keeps the i18n invariant clean (no hardcoded reason string anywhere; the operator console renders a type-derived label when `reason` is null). Operators are responsible for keeping PII out of the reason (UI-enforced later), exactly as for any audit reason. _Risk noted below._

### L6 — The read-API is a contract + DTO + bound implementation; cascade resolves at read
`PartyComplianceStatusReader` is an **interface** under `app/Modules/Parties/Contracts/`, returning a `ComplianceStatus` **DTO** (`?SanctionsStatus $sanctionsStatus`, `list<HoldType> $activeHoldTypes`, `isClear(): bool` = sanctions `Passed` ∧ no active Hold). It exposes `forCustomer(int $customerId)` and `forProfile(int $profileId)`. **Cascade resolves at read**, not by writing duplicate rows: `forProfile` returns the Profile's own active Holds **∪** the parent Customer's active Holds (BR-K-Hold-3), and the sanctions status is the parent Customer's; `forCustomer` returns Customer-scope active Holds. A concrete implementation (`DatabaseComplianceStatusReader`) is bound in `PartiesServiceProvider`. The DTO carries **`HoldType`s, never the `Hold` Eloquent model** — the no-model-leak boundary law. _Alternative rejected:_ returning `Hold` models — leaks an Eloquent model across the module boundary. _Alternative rejected:_ building the by-position enforcement gate/decorator now — there are no surfaces to gate (Module S/C/E are stubs); the contract is the seam.

### L7 — The `kyc`-Hold coupling is the one MODIFIED behaviour; flip the shipped "no Hold" test
`RequireKyc` and `RecordKycVerified` gain the Hold orchestration (each now injects the recorder/actor). `RequireKyc` after `→ pending` places a Customer-scope `kyc` Hold (reason null). `RecordKycVerified` after `→ verified` lifts the Customer's active `kyc` Hold(s) (the system path, recording `CustomerHoldLifted`). `RecordKycRejected` is **unchanged** (the Hold remains — § 9.1). The shipped `CustomerKycLifecycleTest` scenario *"drives require → verify without placing any kyc Hold"* and its `where('name','like','%Hold%')->count() === 0` assertions are **flipped** to assert `CustomerHoldPlaced` on require and `CustomerHoldLifted` on verify (this is the one behaviour change, handled exactly as `parties-compliance` flipped `ActivateProducer`'s ungated test). The `kyc_status` writes themselves are untouched.

### L8 — Operator/system actor via the `ActorContext` seam
`PlaceHold`/`LiftHold` resolve the actor from `ActorContext` exactly as the spine + screening Actions do. The auto `kyc` Hold place/lift (driven by `RequireKyc`/`RecordKycVerified`) records whatever actor is in context (an operator setting `kyc_required`, or `System` for unattended work) — no new actor concept; the console that drives these is `parties-operator-console`.

## Risks / Trade-offs

- **`reason` free-text in the immutable event log (L5).** A free-text field is the classic PII-leak risk against invariant 4 / the event-substrate ADR. Mitigated: **system holds carry `reason = null`** (the one path exercised end-to-end here is the `kyc` auto-hold, which is null-reason); operator reasons are business text by contract; the event payload is otherwise structural (ids + enums). If review prefers, `reason` can be dropped from the payload (it stays on the entity) — flagged for the gate. The conservative default shipped is null-reason for the system path.
- **Polymorphic scope without a DB FK (L1).** No referential-integrity guard at the DB for `scope_id`. Mitigated: `PlaceHold` resolves the scoped entity within the transaction before placing; a test asserts placing against a missing scope is rejected. This is the standard trade-off for a 3-way polymorphic reference.
- **Flipping a shipped, tested behaviour (L7).** The `CustomerKycLifecycleTest` "no Hold" assertions are **updated, not duplicated**; the `parties-compliance` archive's progress notes stay historical. A reviewer expecting "KYC records no event" must read it precisely: KYC records no *KYC* event, but the coupled Hold now records `CustomerHoldPlaced`/`CustomerHoldLifted`.
- **The read-API has no consumer at launch (L6).** `PartyComplianceStatusReader` is exposed but called by nothing (Module S/C/E are stubs). This is the seam, not dead code — it is the exact contract those surfaces will call by-position (DEC-181). A test drives it directly to pin cascade + the tuple.
- **`CLAUDE.md` Invariant #7 is over-broad (cross-cutting).** "Holds are never auto-lifted" is true for only four of six types. The build follows ADR `2026-06-18-hold-lift-discipline-per-type.md`; the `CLAUDE.md` reword is the human's (Protected file) — surfaced at the gate.
- **The asymmetric cascade (Customer→Profile only).** The PRD defines Customer→Profile cascade and Profile isolation; it is **silent on Account-scope cascade**. The read-API implements exactly the specified cascade (Profile reads its parent Customer's Holds); Account-scope cascade is left out (no spec basis) and noted in `CONTEXT.md`.
- **`payment` is `autoLiftable` but has no trigger here.** The classification ships (so the `LiftHold` guard is correct for it), but the `payment` auto-lift signal is Module E (`StoragePaymentSucceeded`) — deferred. A `payment` Hold placed manually in this slice can therefore only be lifted once that trigger lands; no `payment` Hold is placed by this slice, so this is a documented seam, not a live gap.

## Open Questions

None blocking. The slice boundary (Hold registry now; downstream enforcement / demand-side status / console deferred) is pre-ratified by the `parties-compliance` proposal's boundary table. The one item to surface at the gate is non-blocking: root `CLAUDE.md` Invariant #7 wording (ADR `2026-06-18-hold-lift-discipline-per-type.md` records the spec-faithful per-type discipline; whether to reword the protected `CLAUDE.md` line is the human's call).
