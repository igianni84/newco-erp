(consolidated reusable patterns — read first each iteration; seeded at authoring from the archived `2026-06-17-parties-compliance/progress.md` — read that file's full pattern library too, the Parties idioms carry over verbatim)

- **Carryover from parties-compliance (apply as-is).** Enum house style (string-backed, PascalCase cases, persisted-token values, the predicate is the first method, docblock cites design L + spec §; test pins verbatim order-sensitive case→value maps + `toHaveCount` + a `from()` `ValueError`). Quality runners: full suite `php -d memory_limit=512M vendor/bin/pest` (plain `php artisan test` OOMs at 128M); `vendor/bin/phpstan analyse --no-progress` (level max, 0 errors); `vendor/bin/pint` / `--test`; `openspec validate parties-holds --strict` is a standing per-task gate. `timestampTz` casts `immutable_datetime`; assign `CarbonImmutable::now()` in tests, never `now()`. PG-only-CHECK test is per-file-safe (inline the savepoint; branch on `DB::getDriverName()` — pgsql asserts `toContain('<constraint>')`, sqlite asserts the cast is the floor; never a vacuous skip). A compliance change's PG17 close runs `tests/Feature/Modules/Parties` + `tests/Architecture` together.
- **The migration idiom (this change's is NON-nullable CHECK — SHIPPED in 1.2).** `2026_06_18_000001_create_parties_holds_table.php` is a `Schema::create` (a NEW table — contrast parties-compliance's `Schema::table` additive columns), so the PG-only CHECK is the plain `CHECK (col IN (<Enum::cases()>))` of `2026_06_15_000005_create_parties_customers_table.php` (NOT the nullable `col IS NULL OR …` form — `hold_type`/`scope_type`/`status` are NOT NULL). The three CHECKs are emitted in ONE `foreach` over a `/** @var array<string, list<BackedEnum>> */ $valueSets = ['hold_type'=>HoldType::cases(), …]` map, with the constraint name derived from the column (`parties_holds_{$column}_check`) — DRY and PHPStan-max clean (the `@var` carries the heterogeneous-enum value type). `BackedEnum` is unqualified (global) in the no-namespace migration file; Pint keeps it. `scope_id` is a plain `unsignedBigInteger` with NO DB FK (polymorphic 3-way scope — design L1); the integrity is the Action's (`PlaceHold` resolves the scoped entity in-transaction). Composite index named explicitly `parties_holds_scope_status_index` on `(scope_type, scope_id, status)`. **PG17-proven** (catalog inspection: 3 named CHECKs with verbatim value-sets + the composite btree index; `migrate:fresh` + `migrate:rollback` both clean).
- **Pint `{@see \FQCN}` forward-ref trap (re-confirmed 1.2; the `knowledge/laravel/rules.md` rule).** A model/file docblock that references a class created in a LATER task (here the `Hold` model citing `PlaceHold`/`LiftHold`/`CustomerHoldPlaced`/`CustomerHoldLifted`, built in 3.1/3.2/2.1) must use **plain backticks**, NOT `{@see \FQCN}` — Pint's `fully_qualified_strict_types` auto-imports the FQCN into a `use`, and a `use` of a not-yet-existing class reds PHPStan. Concrete EXISTING classes (`HoldType`, `ActorRole`) keep `{@see}` fine. Eyeball the post-Pint `use` block on any rich-docblock file.
- **Evented-Action idiom (PlaceHold/LiftHold — design L3).** Both EMIT, so both inject `DomainEventRecorder` + `ActorContext` (the `RecordCustomerScreening` shape — contrast the eventless KYC Actions which inject nothing *today*; this change adds the recorder to `RequireKyc`/`RecordKycVerified` for the coupling). `record(NAME, Module::Parties->value, $actor->role(), $actor->actorId(), ENTITY_TYPE, (string) $hold->id, payload)` runs INSIDE the Action's `DB::transaction` (the recorder's open-transaction guard makes write+emit atomic). `LiftHold` re-reads `lockForUpdate` then guards `status` + `autoLiftable()`. Capture ONE `CarbonImmutable::now()` for `lifted_at`.
- **Lift discipline lives on the type (design L2; ADR 2026-06-18).** `HoldType::autoLiftable()` true for `Kyc`/`Payment` only (the first predicate — mirror `KycStatus::clears()`). `LiftHold` (operator path) rejects `autoLiftable()` types via `IllegalHoldLift::autoManaged($type)`; the `kyc` auto-lift is a SEPARATE within-module path in `RecordKycVerified` (it lifts the system-managed Hold the operator path forbids). Operator-lift of `kyc`/`payment` is rejected; `admin`/`fraud`/`compliance`/`credit` lift freely; already-`lifted` → `IllegalHoldLift::notActive`.
- **Event class idiom (design L4).** `final class` (no base/interface), untyped `const NAME` (verbatim § 15.1 — `CustomerHoldPlaced`/`CustomerHoldLifted`), `const ENTITY_TYPE = 'Hold'` (the Hold is the subject; the scope is in the payload), static `payload(Hold): array` PII-free. Read nullable enum `@property` with bare `?->value` (never combine `?->` with `?? fallback` — trips `nullsafe.neverNull`). Assert `NAME` byte-for-byte via the concrete class ref, never a `$class::NAME` on a string var. The two names cover ALL scopes (the § 15.1 catalog names no Profile/Account variant — design L4).
- **`reason` PII discipline (design L5).** `reason`/`lift_reason` are controlled business strings; **system-placed Holds carry `reason = null`** (the auto `kyc` Hold — the type is the reason; keeps i18n clean, no hardcoded reason string). The event payload carries `reason` (null for system holds). If review prefers, `reason` can drop from the payload (stays on the entity) — flagged at the gate.
- **KYC coupling is the one MODIFIED behaviour (design L7).** `RequireKyc` (after `→ pending`) calls `PlaceHold(Kyc, Customer, customerId, null)`; `RecordKycVerified` (after `→ verified`) lifts the Customer's active `kyc` Hold(s) recording `CustomerHoldLifted`; `RecordKycRejected` UNCHANGED (Hold remains — § 9.1). The shipped `CustomerKycLifecycleTest` `where('name','like','%Hold%')->count() === 0` assertions FLIP to assert the coupling (handled exactly as parties-compliance flipped `ActivateProducer`'s ungated test). Update both action docblocks (the "NO Hold is placed … parties-holds owns it" notes describe the coupling now).
- **Read contract = interface + DTO + bound impl (design L6).** `app/Modules/Parties/Contracts/{PartyComplianceStatusReader (interface), ComplianceStatus (readonly DTO)}` + `app/Modules/Parties/Reads/DatabaseComplianceStatusReader` bound in `PartiesServiceProvider`. The DTO carries `?SanctionsStatus` + `list<HoldType>` (NEVER the `Hold` model — boundary law) + `isClear()`. Cascade resolves at READ (`forProfile` ∪ parent-Customer holds; `forCustomer` = customer-scope holds), not by writing duplicate rows. This is the one surface other modules import — but they don't yet (Module S/C/E are stubs); a test drives it directly.
- **Scope-guard whitelist GROWS (carryover).** `SupplyLifecycleChainTest`'s exact-set "only these non-Create transition Actions exist" `toEqualCanonicalizing([...])` is the canonical home — adding `PlaceHold`/`LiftHold` (and the recorder dep on `RequireKyc`/`RecordKycVerified` is not a new Action) means **add `PlaceHold`/`LiftHold` to that whitelist or the suite goes RED**. The guard's intent (no demand-side **status** transition Action: no `ActivateCustomer`/`SuspendAccount`/…) is preserved — placing a Hold writes no status.
