- **Pint `{@see \FQCN}` forward-ref trap (re-confirmed 1.2; the `knowledge/laravel/rules.md` rule).** A model/file docblock that references a class created in a LATER task (here the `Hold` model citing `PlaceHold`/`LiftHold`/`CustomerHoldPlaced`/`CustomerHoldLifted`, built in 3.1/3.2/2.1) must use **plain backticks**, NOT `{@see \FQCN}` — Pint's `fully_qualified_strict_types` auto-imports the FQCN into a `use`, and a `use` of a not-yet-existing class reds PHPStan. Concrete EXISTING classes (`HoldType`, `ActorRole`) keep `{@see}` fine. Eyeball the post-Pint `use` block on any rich-docblock file.
- **Evented-Action idiom (PlaceHold SHIPPED 3.1 / LiftHold SHIPPED 3.2 — design L3).** Both EMIT, so both inject `DomainEventRecorder` + `ActorContext` (the `RecordCustomerScreening` shape — contrast the eventless KYC Actions which inject nothing *today*; this change adds the recorder to `RequireKyc`/`RecordKycVerified` for the coupling). `record(NAME, Module::Parties->value, $actorRole, $actorId, ENTITY_TYPE, (string) $hold->id, payload)` runs INSIDE the Action's `DB::transaction` (the recorder's open-transaction guard makes write+emit atomic). **Resolve the actor ONCE per Action** (`$actorRole = $this->actor->role(); $actorId = $this->actor->actorId();`) into locals — PlaceHold stamps them onto BOTH the `placed_actor_*` columns AND the event envelope, so a single resolution keeps the row and its event from ever disagreeing (the seam resolves lazily per call, so two reads *could* differ under a `runAs` boundary). PlaceHold sets `status = HoldStatus::Active` EXPLICITLY on create (so the returned model reads `active` in-memory, not relying on the column default) and passes no causation/correlation → ROOT event. `LiftHold` re-reads `lockForUpdate` then guards `status` + `autoLiftable()`. Capture ONE `CarbonImmutable::now()` for `lifted_at`.
- **Lift discipline lives on the type (design L2; ADR 2026-06-18).** `HoldType::autoLiftable()` true for `Kyc`/`Payment` only (the first predicate — mirror `KycStatus::clears()`). `LiftHold` (operator path) rejects `autoLiftable()` types via `IllegalHoldLift::autoManaged($type)`; the `kyc` auto-lift is a SEPARATE within-module path in `RecordKycVerified` (it lifts the system-managed Hold the operator path forbids). Operator-lift of `kyc`/`payment` is rejected; `admin`/`fraud`/`compliance`/`credit` lift freely; already-`lifted` → `IllegalHoldLift::notActive`.
- **Event class idiom (design L4 — SHIPPED in 2.1).** `final class` (no base/interface), untyped `const NAME` (verbatim § 15.1 — `CustomerHoldPlaced`/`CustomerHoldLifted`), `const ENTITY_TYPE = 'Hold'` (the Hold is the subject; the scope is in the payload), static `payload(Hold): array` PII-free. **`->value` vs `?->value` is decided by the model's `@property` nullability, NOT a blanket rule:** `Hold::hold_type`/`scope_type`/`status` are NON-nullable (`@property HoldType $hold_type`) → plain `->value` (a `?->value` here trips PHPStan-max `nullsafe.neverNull`); a NULLABLE enum prop (`Customer::sanctions_status`, `@property ...|null`) → bare `?->value` (and never combine `?->` with a `?? fallback`). `reason`/`lift_reason` are nullable STRINGS, not enums → emit the value directly, no `->value`. Assert `NAME` byte-for-byte via the concrete class ref, never a `$class::NAME` on a string var. The two names cover ALL scopes (the § 15.1 catalog names no Profile/Account variant — design L4). No central event-name registry exists and no arch test globs the Events dir — a new event class is purely additive; the recorder wiring is pinned at the EMIT site (3.x/4.1).
- **`reason` PII discipline (design L5).** `reason`/`lift_reason` are controlled business strings; **system-placed Holds carry `reason = null`** (the auto `kyc` Hold — the type is the reason; keeps i18n clean, no hardcoded reason string).
