## Codebase Patterns
(consolidated reusable patterns тАФ read first each iteration)

- **Platform substrate code lives under `App\Platform`** (design D1). Enums sit in `App\Platform\Events` (`app/Platform/Events/`). PSR-4 `App\` тЖТ `app/` already covers it тАФ no composer change.
- **Enum convention** (mirrors `App\Modules\Module`): string-backed, PascalCase case name тЖТ snake_case backing value; a `Module`-density class docblock citing the design/spec source. Enum tests live in `tests/Unit/Platform/`, assert the full caseтЖТvalue map verbatim with `->toBe([...])` (order-sensitive), and add one `from('<bad>')` тЖТ `ValueError` failure case. Mirror `tests/Unit/Modules/ModuleTest.php`.
- **Boundary-law amendment for a NEW platform root** (the only legitimate edit to the hardcoded `$platformNamespaces` array): add the root string to `$platformNamespaces` in `tests/Architecture/ModuleBoundariesTest.php` with a design-citing comment, then **prove non-vacuity with a red-proof** тАФ a temp class under the new root type-referencing a module symbol must turn the suite RED naming the exact pair, then removal returns it GREEN with a clean `git status`. This is mandatory (lessons.md 2026-06-12; design Risk "vacuously-green arch tests"). F1 3/3 adds `App\Platform\Money` etc. *under the same already-listed `App\Platform` root* тАФ no further amendment needed once `App\Platform` is in the array.
- **pest-plugin-arch RED failure message format**: `Expecting '<source>' not to use '<target>'.` тАФ useful to grep for when asserting a red-proof named the right pair.
- **Test runner emits JSON** (one line per run: `{"tool":"pest","result":...,"tests":N,...}`). A failure includes a `failures[]` array with `test`, `file`, `line`, `message`. Tail/grep accordingly; don't expect Pest's pretty TTY output.
- **Per-file `RefreshDatabase`**: the global Pest binding stays commented тАФ DB-touching test files opt in with `uses(RefreshDatabase::class)` per-file (tasks.md standing rule). 1.1 needed no DB so EnumsTest is a pure unit test.
- **Postgres-truthful migration idiom** (design D2): use the fluent column types as-is тАФ on the SQLite dev/test lane they fall back behavior-preservingly (`jsonb()`тЖТ`text`, `timestampTz()`тЖТdatetime `text`, `uuid()`тЖТ`varchar`; all identical under the Eloquent array/datetime casts). A constraint only PostgreSQL can express (e.g. an `actor_role` CHECK) goes in an `if (DB::getDriverName() === 'pgsql')` branch via `DB::statement(...)` **after** `Schema::create()`, with the SQLite fallback (NOT NULL + the enum cast) documented inline. **Derive the CHECK's enumerated values from the enum's `::cases()`** so the DB constraint can never drift from the PHP enum. The PG branch is unexercised on SQLite тАФ the `pgsql` CI lane (task 5.2 / first human push) is its proof, by design D8.
- **Portable schema assertions**: `Schema::hasColumns($t, [...])` тЖТ bool for column presence; `Schema::hasIndex($t, [col, col, ...], 'unique'|null)` matches on the **column array** (order-sensitive тАФ verified in `Builder::hasIndex`, compares `$value['columns'] === $index`), so you assert composite/unique indexes **without** knowing Laravel's generated index name. Pair structural checks with behavioral ones: a complete-row `DB::table()->insert()` for happy path; `->throws(QueryException::class)` for NOT NULL / unique violations (never engine-specific SQLSTATEs); a self-referencing FK is proven by inserting a child row whose FK column points at a parent's `id`. A row-builder `function fooRow(array $overrides = [])` keeps the complete-valid-row DRY while each test drops/changes exactly one field.
- **PHPStan level-max in test files** (two recurring traps): (1) plain helper functions need iterable-value types тАФ `@param array<string, mixed>` + `@return array<string, mixed>`; (2) never cast a DB-query result to a scalar тАФ `->first()` is `stdClass|null` (a prior `expect()->not->toBeNull()` does NOT narrow it for PHPStan) and `->value()` is `mixed`, so `(int) $row->col` trips `cast.int`. Assert DB values with Pest `->toEqual(...)` (loose ==, accepts `mixed` with no cast) instead of casting.
- **Partial index = raw `DB::statement('CREATE INDEX <name> ON <t> (<col>) WHERE <predicate>')`** (design D2). The installed Laravel Blueprint has NO fluent partial-index predicate тАФ `IndexDefinition` exposes only `algorithm/deferrable/initiallyImmediate/language/lock/nullsNotDistinct/online`, and both grammars' `compileIndex` emit a plain `create index тАж (cols)` with no `WHERE` (verified in vendor 2026-06-12). The raw `CREATE INDEX тАж WHERE` syntax is **identical and valid on BOTH SQLite and PostgreSQL**, so one statement serves both engines (unlike the PG-only `actor_role` CHECK). Give the index an **explicit stable name** and derive any literal in the predicate from the enum (`DeliveryStatus::Pending->value`) so it can't drift. **Assert existence by NAME**: `Schema::hasIndex($t, '<name>')` тАФ `Builder::hasIndex` matches `$value['name'] === $index || $value['columns'] === $index`, so a name string works and is portable (both engines' `getIndexes()` return the index by name). Caveat: `getIndexes()` surfaces name+columns but **NOT the partial predicate** on either engine, so the name-existence assertion is the portable proof; partiality itself is carried by the migration DDL (a malformed predicate aborts the migration тЖТ every test in the file errors, so green тЗТ valid partial DDL on the running engine). The pgsql lane (5.2) re-proves the PG side.
- **Cross-engine immutability triggers** (design D7) тАФ raw DDL via **`DB::statement`, NOT `DB::unprepared`**: Larastan types `Connection::unprepared()`'s `$query` as `literal-string` (an injection guard), so an interpolated clause (`"тАж ({$changed}) тАж"`, where `$changed` is `implode(array_map(...))`) trips `argument.type` at PHPStan max. `Connection::statement($query, $bindings=[])` is **not** literal-string-constrained (it carries a bindings param) тАФ exactly why the table migrations (000001/000002 `actor_role` CHECK, 000003 partial index) already use it for interpolated driver-branched DDL. Match them: one statement per call, **no trailing semicolon** (a single command per call dodges PG's "cannot insert multiple commands into a prepared statement"; the body's own `;` are safe тАФ inside SQLite `BEGIN тАж END` or PG `$$тАж$$`). Verified (prepare+execute installs a SQLite `CREATE TRIGGER` with internal semicolons and it fires). Engine dialects: **SQLite** `CREATE TRIGGER тАж BEFORE UPDATE/DELETE тАж BEGIN SELECT RAISE(ABORT, 'тАжimmutableтАж'); END` (single-event triggers тЖТ two for an append-only table; `RAISE(ABORT)` reverts only the offending statement, not the wrapping txn, so a test can read the row back unchanged after catching); **PostgreSQL** a plpgsql `RETURNS trigger` function (`RAISE EXCEPTION 'тАжimmutableтАж'`; `\$\$` is the PHP-escaped `$$` dollar-quote) fired by one combined `BEFORE UPDATE OR DELETE тАж EXECUTE FUNCTION`. A **partial-immutability** table (allow only some columns to change тАФ e.g. audit `before`/`after` for GDPR redaction) uses **ONE authoritative structural-column list** (every other column) driving both branches: SQLite `WHEN (OLD.c IS NOT NEW.c OR тАж)` (null-safe `IS NOT`), PG `IF (OLD.c IS DISTINCT FROM NEW.c OR тАж) THEN RAISE; END IF; RETURN NEW`. Tests assert **behaviour only** тАФ `QueryException` whose message `toContain('immutable')` + row unchanged/exists afterward тАФ **never SQLSTATEs**, so one test file proves both lanes; they ARE the red-paths (no separate red-proof). `event_deliveries` gets NO trigger (mutable by design) тАФ pin that with a positive "update succeeds" test so the trigger scope can't silently over-extend.
