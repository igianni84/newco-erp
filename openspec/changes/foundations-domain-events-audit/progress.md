# Progress — foundations-domain-events-audit

## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Platform substrate code lives under `App\Platform`** (design D1). Enums sit in `App\Platform\Events` (`app/Platform/Events/`). PSR-4 `App\` → `app/` already covers it — no composer change.
- **Enum convention** (mirrors `App\Modules\Module`): string-backed, PascalCase case name → snake_case backing value; a `Module`-density class docblock citing the design/spec source. Enum tests live in `tests/Unit/Platform/`, assert the full case→value map verbatim with `->toBe([...])` (order-sensitive), and add one `from('<bad>')` → `ValueError` failure case. Mirror `tests/Unit/Modules/ModuleTest.php`.
- **Boundary-law amendment for a NEW platform root** (the only legitimate edit to the hardcoded `$platformNamespaces` array): add the root string to `$platformNamespaces` in `tests/Architecture/ModuleBoundariesTest.php` with a design-citing comment, then **prove non-vacuity with a red-proof** — a temp class under the new root type-referencing a module symbol must turn the suite RED naming the exact pair, then removal returns it GREEN with a clean `git status`. This is mandatory (lessons.md 2026-06-12; design Risk "vacuously-green arch tests"). F1 3/3 adds `App\Platform\Money` etc. *under the same already-listed `App\Platform` root* — no further amendment needed once `App\Platform` is in the array.
- **pest-plugin-arch RED failure message format**: `Expecting '<source>' not to use '<target>'.` — useful to grep for when asserting a red-proof named the right pair.
- **Test runner emits JSON** (one line per run: `{"tool":"pest","result":...,"tests":N,...}`). A failure includes a `failures[]` array with `test`, `file`, `line`, `message`. Tail/grep accordingly; don't expect Pest's pretty TTY output.
- **Per-file `RefreshDatabase`**: the global Pest binding stays commented — DB-touching test files opt in with `uses(RefreshDatabase::class)` per-file (tasks.md standing rule). 1.1 needed no DB so EnumsTest is a pure unit test.
- **Postgres-truthful migration idiom** (design D2): use the fluent column types as-is — on the SQLite dev/test lane they fall back behavior-preservingly (`jsonb()`→`text`, `timestampTz()`→datetime `text`, `uuid()`→`varchar`; all identical under the Eloquent array/datetime casts). A constraint only PostgreSQL can express (e.g. an `actor_role` CHECK) goes in an `if (DB::getDriverName() === 'pgsql')` branch via `DB::statement(...)` **after** `Schema::create()`, with the SQLite fallback (NOT NULL + the enum cast) documented inline. **Derive the CHECK's enumerated values from the enum's `::cases()`** so the DB constraint can never drift from the PHP enum. The PG branch is unexercised on SQLite — the `pgsql` CI lane (task 5.2 / first human push) is its proof, by design D8.
- **Portable schema assertions**: `Schema::hasColumns($t, [...])` → bool for column presence; `Schema::hasIndex($t, [col, col, ...], 'unique'|null)` matches on the **column array** (order-sensitive — verified in `Builder::hasIndex`, compares `$value['columns'] === $index`), so you assert composite/unique indexes **without** knowing Laravel's generated index name. Pair structural checks with behavioral ones: a complete-row `DB::table()->insert()` for happy path; `->throws(QueryException::class)` for NOT NULL / unique violations (never engine-specific SQLSTATEs); a self-referencing FK is proven by inserting a child row whose FK column points at a parent's `id`. A row-builder `function fooRow(array $overrides = [])` keeps the complete-valid-row DRY while each test drops/changes exactly one field.
- **PHPStan level-max in test files** (two recurring traps): (1) plain helper functions need iterable-value types — `@param array<string, mixed>` + `@return array<string, mixed>`; (2) never cast a DB-query result to a scalar — `->first()` is `stdClass|null` (a prior `expect()->not->toBeNull()` does NOT narrow it for PHPStan) and `->value()` is `mixed`, so `(int) $row->col` trips `cast.int`. Assert DB values with Pest `->toEqual(...)` (loose ==, accepts `mixed` with no cast) instead of casting.
- **Partial index = raw `DB::statement('CREATE INDEX <name> ON <t> (<col>) WHERE <predicate>')`** (design D2). The installed Laravel Blueprint has NO fluent partial-index predicate — `IndexDefinition` exposes only `algorithm/deferrable/initiallyImmediate/language/lock/nullsNotDistinct/online`, and both grammars' `compileIndex` emit a plain `create index … (cols)` with no `WHERE` (verified in vendor 2026-06-12). The raw `CREATE INDEX … WHERE` syntax is **identical and valid on BOTH SQLite and PostgreSQL**, so one statement serves both engines (unlike the PG-only `actor_role` CHECK). Give the index an **explicit stable name** and derive any literal in the predicate from the enum (`DeliveryStatus::Pending->value`) so it can't drift. **Assert existence by NAME**: `Schema::hasIndex($t, '<name>')` — `Builder::hasIndex` matches `$value['name'] === $index || $value['columns'] === $index`, so a name string works and is portable (both engines' `getIndexes()` return the index by name). Caveat: `getIndexes()` surfaces name+columns but **NOT the partial predicate** on either engine, so the name-existence assertion is the portable proof; partiality itself is carried by the migration DDL (a malformed predicate aborts the migration → every test in the file errors, so green ⇒ valid partial DDL on the running engine). The pgsql lane (5.2) re-proves the PG side.

## [2026-06-12 16:45] — 2.3 Migration `create_event_deliveries_table`
- **What:** The third substrate table (design D2) — the per-consumer delivery ledger that makes domain-event delivery durable and at-least-once. One mutable row per (domain_event × consumer): lifecycle `pending → done | failed`, attempt count, exponential-backoff clock. R4 is resolved structurally (independent rows → independent retries). Unlike the two append-only tables this one is deliberately MUTABLE and carries timestamps.
- **Files changed:**
  - `database/migrations/2026_06_12_000003_create_event_deliveries_table.php` — new. `id` PK · `domain_event_id` foreignId→`domain_events.id` (NOT NULL) · `consumer` string (= the consumer class FQCN, design D4) · `status` string NOT NULL default `DeliveryStatus::Pending->value` · `attempts` unsignedSmallInteger default 0 · `available_at` timestampTz nullable (NULL = due now) · `last_error` text nullable · `timestamps()`. Unique `(domain_event_id, consumer)`. **Partial index** `event_deliveries_pending_index ON (available_at) WHERE status = 'pending'` via raw `DB::statement` (Blueprint has no fluent predicate — verified in vendor), the `'pending'` literal derived from the enum.
  - `tests/Feature/Platform/EventDeliveriesSchemaTest.php` — new, `uses(RefreshDatabase::class)`. 9 tests / 10 assertions: full columns (`hasColumns`), the unique pair (`hasIndex … 'unique'`), the partial index by NAME (`hasIndex($t, 'event_deliveries_pending_index')`), defaults (status=pending, attempts=0), happy-path full row, duplicate-pair → `QueryException`, the pair allows same-consumer/diff-event & diff-consumer/same-event (non-vacuity), FK orphan → `QueryException`, missing `consumer` → `QueryException`. Two helpers: `seedDomainEvent(): int` (FK parent via `insertGetId`, PHPDoc `@return int` so no cast) and `eventDeliveryRow(int, array): array`.
- **Quality loop:** green — format ✅ · EventDeliveriesSchemaTest 9/9 ✅ · full suite **88/88 (352 assertions)**, +9 vs the 79 baseline ✅ · phpstan level max 0 errors ✅ · pint --test ✅ · `openspec validate --strict` ✅. `git status`: only the two new files (no protected paths); `git diff main -- composer.json composer.lock` empty.
- **Vendor-verified before writing** (standing rule): `IndexDefinition` has no `where()`/partial-predicate method; SQLite & Postgres `compileIndex` emit plain `create index … (cols)`; `Builder::hasIndex` matches a NAME string OR a columns array (`$value['name'] === $index || $value['columns'] === $index`); `getIndexes()` returns name+columns+unique+primary (PG adds type) but **never the partial predicate**; `insertGetId` PHPDoc `@return int`; `Blueprint::timestamps()` makes `created_at`/`updated_at` **nullable**; `ForeignIdColumnDefinition::constrained()` defaults the referenced column to `'id'`.
- **Extra confidence check (not a committed test):** ran a throwaway `php -r` against `sqlite::memory:` creating the same partial index and read `sqlite_master.sql` — the stored DDL keeps the `WHERE status = '…'` predicate, confirming SQLite materializes a genuinely PARTIAL index (not a plain one). Documented because `getIndexes()` can't surface this.
- **Learnings for future iterations:**
  - 2.4 (immutability triggers) now has all three tables to guard — but note the trigger task targets `domain_events` (UPDATE+DELETE) and `audit_records` (DELETE + structural-UPDATE) ONLY; `event_deliveries` is deliberately MUTABLE (status/attempts churn during retries) and gets NO immutability trigger. Don't accidentally extend the trigger to it.
  - The FK `domain_event_id`→`domain_events.id` is enforced on SQLite `:memory:` (foreign_keys ON), so 3.4's recorder must insert the parent event BEFORE its delivery rows inside the same transaction (it does — the event is inserted, then `->id` feeds the fan-out rows).
  - Partial-index idiom is now in Codebase Patterns above (raw DDL, name-based portable assertion). It's the second "PG-truthful realization where SQLite needs a documented path" — but unlike the CHECK (PG-only), the partial index runs on BOTH engines.
