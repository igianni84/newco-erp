## Codebase Patterns
(consolidated reusable patterns ‚Äî read first each iteration)

- **Platform substrate code lives under `App\Platform`** (design D1). Enums sit in `App\Platform\Events` (`app/Platform/Events/`). PSR-4 `App\` ‚Üí `app/` already covers it ‚Äî no composer change.
- **Enum convention** (mirrors `App\Modules\Module`): string-backed, PascalCase case name ‚Üí snake_case backing value; a `Module`-density class docblock citing the design/spec source. Enum tests live in `tests/Unit/Platform/`, assert the full case‚Üívalue map verbatim with `->toBe([...])` (order-sensitive), and add one `from('<bad>')` ‚Üí `ValueError` failure case. Mirror `tests/Unit/Modules/ModuleTest.php`.
- **Boundary-law amendment for a NEW platform root** (the only legitimate edit to the hardcoded `$platformNamespaces` array): add the root string to `$platformNamespaces` in `tests/Architecture/ModuleBoundariesTest.php` with a design-citing comment, then **prove non-vacuity with a red-proof** ‚Äî a temp class under the new root type-referencing a module symbol must turn the suite RED naming the exact pair, then removal returns it GREEN with a clean `git status`. This is mandatory (lessons.md 2026-06-12; design Risk "vacuously-green arch tests"). F1 3/3 adds `App\Platform\Money` etc. *under the same already-listed `App\Platform` root* ‚Äî no further amendment needed once `App\Platform` is in the array.
- **pest-plugin-arch RED failure message format**: `Expecting '<source>' not to use '<target>'.` ‚Äî useful to grep for when asserting a red-proof named the right pair.
- **Test runner emits JSON** (one line per run: `{"tool":"pest","result":...,"tests":N,...}`). A failure includes a `failures[]` array with `test`, `file`, `line`, `message`. Tail/grep accordingly; don't expect Pest's pretty TTY output.
- **Per-file `RefreshDatabase`**: the global Pest binding stays commented ‚Äî DB-touching test files opt in with `uses(RefreshDatabase::class)` per-file (tasks.md standing rule). 1.1 needed no DB so EnumsTest is a pure unit test.
- **Postgres-truthful migration idiom** (design D2): use the fluent column types as-is ‚Äî on the SQLite dev/test lane they fall back behavior-preservingly (`jsonb()`‚Üí`text`, `timestampTz()`‚Üídatetime `text`, `uuid()`‚Üí`varchar`; all identical under the Eloquent array/datetime casts). A constraint only PostgreSQL can express (e.g. an `actor_role` CHECK) goes in an `if (DB::getDriverName() === 'pgsql')` branch via `DB::statement(...)` **after** `Schema::create()`, with the SQLite fallback (NOT NULL + the enum cast) documented inline. **Derive the CHECK's enumerated values from the enum's `::cases()`** so the DB constraint can never drift from the PHP enum. The PG branch is unexercised on SQLite ‚Äî the `pgsql` CI lane (task 5.2 / first human push) is its proof, by design D8.
- **Portable schema assertions**: `Schema::hasColumns($t, [...])` ‚Üí bool for column presence; `Schema::hasIndex($t, [col, col, ...], 'unique'|null)` matches on the **column array** (order-sensitive ‚Äî verified in `Builder::hasIndex`, compares `$value['columns'] === $index`), so you assert composite/unique indexes **without** knowing Laravel's generated index name. Pair structural checks with behavioral ones: a complete-row `DB::table()->insert()` for happy path; `->throws(QueryException::class)` for NOT NULL / unique violations (never engine-specific SQLSTATEs); a self-referencing FK is proven by inserting a child row whose FK column points at a parent's `id`. A row-builder `function fooRow(array $overrides = [])` keeps the complete-valid-row DRY while each test drops/changes exactly one field.
- **PHPStan level-max in test files** (two recurring traps): (1) plain helper functions need iterable-value types ‚Äî `@param array<string, mixed>` + `@return array<string, mixed>`; (2) never cast a DB-query result to a scalar ‚Äî `->first()` is `stdClass|null` (a prior `expect()->not->toBeNull()` does NOT narrow it for PHPStan) and `->value()` is `mixed`, so `(int) $row->col` trips `cast.int`. Assert DB values with Pest `->toEqual(...)` (loose ==, accepts `mixed` with no cast) instead of casting.
- **Partial index = raw `DB::statement('CREATE INDEX <name> ON <t> (<col>) WHERE <predicate>')`** (design D2). The installed Laravel Blueprint has NO fluent partial-index predicate ‚Äî `IndexDefinition` exposes only `algorithm/deferrable/initiallyImmediate/language/lock/nullsNotDistinct/online`, and both grammars' `compileIndex` emit a plain `create index ‚Ä¶ (cols)` with no `WHERE` (verified in vendor 2026-06-12). The raw `CREATE INDEX ‚Ä¶ WHERE` syntax is **identical and valid on BOTH SQLite and PostgreSQL**, so one statement serves both engines (unlike the PG-only `actor_role` CHECK). Give the index an **explicit stable name** and derive any literal in the predicate from the enum (`DeliveryStatus::Pending->value`) so it can't drift. **Assert existence by NAME**: `Schema::hasIndex($t, '<name>')` ‚Äî `Builder::hasIndex` matches `$value['name'] === $index || $value['columns'] === $index`, so a name string works and is portable (both engines' `getIndexes()` return the index by name). Caveat: `getIndexes()` surfaces name+columns but **NOT the partial predicate** on either engine, so the name-existence assertion is the portable proof; partiality itself is carried by the migration DDL (a malformed predicate aborts the migration ‚Üí every test in the file errors, so green ‚áí valid partial DDL on the running engine). The pgsql lane (5.2) re-proves the PG side.
- **Cross-engine immutability triggers** (design D7) ‚Äî raw DDL via **`DB::statement`, NOT `DB::unprepared`**: Larastan types `Connection::unprepared()`'s `$query` as `literal-string` (an injection guard), so an interpolated clause (`"‚Ä¶ ({$changed}) ‚Ä¶"`, where `$changed` is `implode(array_map(...))`) trips `argument.type` at PHPStan max. `Connection::statement($query, $bindings=[])` is **not** literal-string-constrained (it carries a bindings param) ‚Äî exactly why the table migrations (000001/000002 `actor_role` CHECK, 000003 partial index) already use it for interpolated driver-branched DDL. Match them: one statement per call, **no trailing semicolon** (a single command per call dodges PG's "cannot insert multiple commands into a prepared statement"; the body's own `;` are safe ‚Äî inside SQLite `BEGIN ‚Ä¶ END` or PG `$$‚Ä¶$$`). Verified (prepare+execute installs a SQLite `CREATE TRIGGER` with internal semicolons and it fires). Engine dialects: **SQLite** `CREATE TRIGGER ‚Ä¶ BEFORE UPDATE/DELETE ‚Ä¶ BEGIN SELECT RAISE(ABORT, '‚Ä¶immutable‚Ä¶'); END` (single-event triggers ‚Üí two for an append-only table; `RAISE(ABORT)` reverts only the offending statement, not the wrapping txn, so a test can read the row back unchanged after catching); **PostgreSQL** a plpgsql `RETURNS trigger` function (`RAISE EXCEPTION '‚Ä¶immutable‚Ä¶'`; `\$\$` is the PHP-escaped `$$` dollar-quote) fired by one combined `BEFORE UPDATE OR DELETE ‚Ä¶ EXECUTE FUNCTION`. A **partial-immutability** table (allow only some columns to change ‚Äî e.g. audit `before`/`after` for GDPR redaction) uses **ONE authoritative structural-column list** (every other column) driving both branches: SQLite `WHEN (OLD.c IS NOT NEW.c OR ‚Ä¶)` (null-safe `IS NOT`), PG `IF (OLD.c IS DISTINCT FROM NEW.c OR ‚Ä¶) THEN RAISE; END IF; RETURN NEW`. Tests assert **behaviour only** ‚Äî `QueryException` whose message `toContain('immutable')` + row unchanged/exists afterward ‚Äî **never SQLSTATEs**, so one test file proves both lanes; they ARE the red-paths (no separate red-proof). `event_deliveries` gets NO trigger (mutable by design) ‚Äî pin that with a positive "update succeeds" test so the trigger scope can't silently over-extend.
- **Platform Eloquent models** (design D1/D2): `$table` explicit, `protected $guarded = []` (the recorders are the only writers ‚Äî no request-driven mass assignment on a substrate model), `casts()` method (Laravel 11+ style, `@return array<string, string>`). The two append-only tables (`domain_events`/`audit_records`) set `public $timestamps = false` ‚Äî the `occurred_at` envelope clock is the only timestamp and the migrations have NO created/updated columns, so leaving timestamps on would make `create()` write non-existent columns (it's a correctness requirement, not a preference); the mutable `event_deliveries` KEEPS timestamps. Cast vocabulary: enum column ‚Üí the enum FQCN (`ActorRole::class`/`DeliveryStatus::class`) = the application-side value-set floor on BOTH engines ‚Äî reading an out-of-enum value throws `ValueError` via `BackedEnum::from()`, **uncaught by Eloquent** (`HasAttributes::getEnumCaseFromValue` ‚Üí `::from()`), which IS the failure-case test; `jsonb` ‚Üí `'array'`; `timestampTz` ‚Üí `'immutable_datetime'` (‚Üí `Carbon\CarbonImmutable`). **PHPStan level-max model-property access**: Larastan scans `database/migrations` by default (`databaseMigrationsPath: []` ‚Üí `database_path('migrations')`) and its `SchemaAggregator` understands `jsonb`/`uuid`/`foreignid`/`unsigned*integer`/`id`/`string`/`text`/`timestamps` but **NOT `timestamptz`** (singular), so `occurred_at`/`available_at` are NOT schema-inferred ‚Äî add explicit `@property` annotations (Larastan maps `immutable_datetime`‚Üí`CarbonImmutable`, annotate to match so there's no drift). `Eloquent\Model::__get` means an un-inferred property never trips `property.notFound`, but annotate anyway so `expect($x->prop)->toBeInstanceOf(...)` is a typed check, not `mixed`. **Pint's `fully_qualified_strict_types` fixer manages docblock types + imports together**: write `@property \Carbon\CarbonImmutable`, Pint adds the `use` and shortens to `CarbonImmutable` (and `no_unused_imports` keeps a docblock-only import) ‚Äî don't fight it. An uncast column reads back as the raw DB type (`audit_records.occurred_at` is left uncast per the task's explicit cast list ‚Üí reads as `string`, annotate it so).
- **Pest helper functions are GLOBAL across all test files** (one shared namespace) ‚Üí prefix per-file helpers to avoid a redeclaration fatal: `domainEventRow` (schema test), `immutabilityDomainEventRow` (immutability), `modelsDomainEventAttributes` (models) all build a valid `domain_events` row but must NOT share a name.
- **Provider-seam singleton** (3.2, design D4): a shared registry/service is a plain class with instance state, bound `$this->app->singleton(Foo::class)` in `AppServiceProvider::register()` (bindings go in `register()`, runs before any provider `boot()`). The container singleton is what makes module-provider `boot()` registrations and the consuming path see ONE instance. `singleton($class)` with no closure is valid (resolves the concrete via reflection; works when the ctor has no args).
- **Unit tests have NO Laravel app** ‚Äî `tests/Pest.php` binds `Tests\TestCase` (which boots the app) only `->in('Feature')`; `tests/Unit/**` is plain `PHPUnit\Framework\TestCase`, so `app()`/`$this->app` are unavailable there. Split accordingly: pure-logic assertions ‚Üí Unit (new up the SUT directly); anything that resolves from the container (singleton identity, binding persistence) ‚Üí a Feature test. DB-touching Feature tests still opt into `uses(RefreshDatabase::class)` per-file; a container-only Feature test needs no DB trait.
- **`is_subclass_of($class, Interface::class)` is a PHPStan type-guard**: after a negated guard `if (! is_subclass_of($x, Foo::class)) { throw; }`, PHPStan level-max narrows the `string $x` to `class-string<Foo>`, so it can be stored into a `list<class-string<Foo>>` with no cast (verified 2026-06-12). It accepts an anonymous-class name (`$anon::class`) and returns `false` (no fatal) for a missing class ‚Äî so ONE guard covers both "doesn't implement the contract" and "class doesn't exist" failure cases. Type the validated param `string` (not `class-string<Foo>`) so a negative test can pass a non-conforming class without a static-analysis error.
- **A single-case enum is a compile-time gate, not a runtime branch**: with `enum DeliveryMode { case Inline; }` you CANNOT write `if ($mode !== DeliveryMode::Inline) throw ‚Ä¶` ‚Äî PHPStan max flags it always-false / the throw unreachable. Pin the gate behaviorally instead: a test asserting `DeliveryMode::cases() === [DeliveryMode::Inline]` with a comment naming exactly what flips (the registry gains a reject-branch; the test flips to assert rejection) when the queue-driver ADR (F4‚ÄìF6) adds `Queued`. Carry the deferred mode as a defaulted param (`DeliveryMode $mode = DeliveryMode::Inline`) for forward-compatible call sites; PHPStan does not flag an unused method param.
- **Platform substrate speaks `string` module-ids, never the `Module` enum** (boundary law, 3.3 ‚Äî also `knowledge/architecture/rules.md`): a platform API carrying a module identity types it `string` (the `Module::X->value`, or `'platform'` for the platform pseudo-module), NOT `App\Modules\Module` ‚Äî importing `Module` under `App\Platform\**` trips the `it_forbids_platform_code_from_depending_on_any_module` arch test (proven RED this iter: `Expecting 'App\Platform' not to use 'App\Modules'.`). Module emitters pass `Module::X->value` (the registry stays their typed anchor); same shape as `event_deliveries.consumer` (string FQCN) and the plain-`string` `*.module` columns. **Binds 3.4's `DomainEventRecorder` (`string $module`, NOT `Module|string`)** and every F2+ emitter. Design D3 sketched `Module|string` but labels it "realization, loop may refine"; D1 (the canonical boundary) outranks that detail. Tests MAY import `Module` (the arch suite scans only `App\Platform` *source*, not `Tests\`).
- **A `DB::transactionLevel() === 0` guard is testable only under `DatabaseMigrations`** (3.3): `RefreshDatabase` wraps each test in a transaction (level ‚â• 1) so the guard's negative path is unreachable (design D3: "satisfied trivially" under the wrapper). `DatabaseTruncation` is out ‚Äî its per-table DELETE hits the append-only immutability triggers. `DatabaseMigrations` (migrate:fresh = DDL, which the triggers don't guard) leaves tests at level 0 ‚Üí wrap write tests in explicit `DB::transaction()` (the hint's instruction becomes load-bearing, not cosmetic), call the guard test bare. Bonus: under DatabaseMigrations `DB::transaction()` actually COMMITS observably (and a rollback discards) ‚Äî which **3.4's atomicity tests (commit-persists / rollback-discards-all-three) REQUIRE**; RefreshDatabase can't surface a real commit. Per-file `uses(DatabaseMigrations::class)`; the global Pest binding stays commented.
- **Shared substrate guard exception** `App\Platform\Events\NotInTransactionException` (3.3): ONE dedicated exception for the no-dual-write guard, used by BOTH recorders (3.4's `DomainEventRecorder` natively; 3.3's `AuditRecorder` by import ‚Äî `App\Platform\Audit` already imports `App\Platform\Events\ActorRole`, so the direction is established). Static factory `forRecording(string $what)` for a precise per-caller message (`'an audit record'` / `'a domain event'`); extends `RuntimeException`. Lives in `Events/` (the substrate core namespace owning the recorder/contract/registry).
- **Recorder fan-out + the constructor-injected singleton registry** (3.4, design D3/D4): a platform service that needs the `ConsumerRegistry` declares `public function __construct(private readonly ConsumerRegistry $registry) {}` ‚Äî the container auto-wires the constructor and resolves the AppServiceProvider-bound SINGLETON, so the injected instance IS the one tests/module-providers `register()` into; the service itself needs NO explicit binding (`app(DomainEventRecorder::class)` reflects the ctor and injects the singleton). The fan-out shape: `foreach ($this->registry->consumersFor($name) as $consumer) { EventDelivery::create([...status => DeliveryStatus::Pending, attempts => 0]) }`, inside the caller's transaction, AFTER `DomainEvent::create()` (so `$event->id` exists for the FK). The recorder NEVER opens its own transaction (it appends to the caller's) and NEVER calls `DB::afterCommit` ‚Äî delivery EXECUTION + the post-commit hook are task 4.1; 3.4 only enqueues the `pending` ledger rows. Enum-cast columns take the enum INSTANCE on write (`DeliveryStatus::Pending`, `ActorRole::X`), not `->value`. `correlation_id` default DIFFERS between the two recorders: audit ‚Üí a fresh independent `Str::uuid7()`; the event recorder ‚Üí the event's OWN `event_id` (root event is its own correlation root, design D3) ‚Äî capture `$eventId` once and reuse it for both `event_id` and the `?? $eventId` correlation default.
- **Atomicity tests need DatabaseMigrations + the `cache` table as the "state change"** (3.4): proving the recorder rides the caller's transaction requires a REAL commit/rollback, which only happens at `transactionLevel 0` (DatabaseMigrations, not RefreshDatabase ‚Äî see the guard pattern above). The canonical co-committed state change is a `cache`-table write (design D9: a platform table, idempotent, no domain pollution): `DB::table('cache')->insert(['key' => '‚Ä¶', 'value' => '‚Ä¶', 'expiration' => 9999999999])` (its three columns). Commit test: wrap state-write + `record()` in `DB::transaction()`, then assert the cache row + the `domain_events` row + the `pending` `event_deliveries` rows all exist. Rollback test: `expect(fn () => DB::transaction(fn () => { state-write; record(); throw new RuntimeException(...); }))->toThrow(RuntimeException::class)` then assert ALL THREE tables are empty. Reused by 5.1's E2E pipeline + demo command.
- **Two distinct `new class` statements ‚áí two distinct anonymous-class FQCNs** (3.4): a PHP anonymous-class declaration is compiled once per SYNTACTIC occurrence ‚Äî executing the same `new class {...}` statement twice yields two instances of the SAME class name, but two SEPARATE `new class {...}` statements yield two DIFFERENT `::class` names. A fan-out/multi-consumer test that needs N *distinct* registered consumers (so the unique `(domain_event_id, consumer)` pair gives N rows) must write N separate `new class implements DomainEventConsumer {...}` blocks (helper `registerTwoFakeConsumers` does exactly this and returns the two FQCNs in registration order). The container/registry singleton is rebuilt per test (fresh `createApplication()`), so re-registering the same FQCNs across tests never leaks. 4.1 reuses this for its throwing-vs-succeeding consumer pair.
- **Two PHPStan-max test-typing traps** (3.3): (a) spreading an `array<string, mixed>` as named args (`record(...$args)`) degrades EVERY parameter to `mixed` (`expects string, mixed given`) ‚Äî replace with a **fully-typed helper function** with defaulted params (enum and array-literal defaults are valid) returning the model; Larastan infers `DB::transaction(fn () => helper())` and `app(Foo::class)` return types so `$record->id` stays typed. (b) `$this->travelTo()`/`$this->freezeTime()` are unresolvable inside a Pest closure (`$this` resolves to `Pest\PendingCalls\TestCall`, not the TestCase) ‚Üí use the static `CarbonImmutable::setTestNow($frozen)` + an `afterEach(fn () => CarbonImmutable::setTestNow())` reset (Carbon and CarbonImmutable share the global mock). (c) Don't `{@see}` a not-yet-existent class ‚Äî Pint's `fully_qualified_strict_types` auto-imports docblock FQCNs, yielding a `use` of a missing class; use prose for forward references (e.g. "the domain-event recorder (task 3.4)").
- **The D5 `afterCommit`-under-test landmine, RESOLVED (4.1, verified in vendor)**: the base `DatabaseTransactionsManager::addCallback` attaches the callback to the current pending transaction (or runs it immediately at level 0); `afterCommitCallbacksShouldBeExecuted($level)` returns `$level === 0`, so callbacks fire when the OUTERMOST `DB::transaction()` commits ‚Äî exactly production. The testing override (`Illuminate\Foundation\Testing\DatabaseTransactionsManager`, used by `RefreshDatabase`) returns `$level === 1` and `callbackApplicableTransactions()` skips the wrapper, so the hook ALSO fires under RefreshDatabase (at the inner commit). **So the "afterCommit never fires under the wrapper" landmine is NOT present in this framework** ‚Äî no D5 fallback needed. Still choose **DatabaseMigrations** for delivery tests: only there are commit/rollback OBSERVABLE (level 0), which the atomicity / exactly-once assertions require. `DatabaseTransactionRecord::executeCallbacks()` is a plain FIFO `foreach`, so callbacks fire in registration order.
- **Per-record `afterCommit` hook = the realization of "one hook per transaction" (4.1, design D5)**: each `record()` registers `DB::afterCommit(fn () => $executor->deliver([$event->id]))`. Multiple records in ONE transaction attach their callbacks to the SAME `DatabaseTransactionRecord` and fire FIFO in record (= id) order ‚Äî so causal delivery order holds ‚Äî and a rollback discards them ALL (Laravel never runs uncommitted after-commit callbacks ‚Üí no delivery on rollback, automatically). This stateless per-record form was chosen over a transaction-scoped id-buffer + single hook: functionally identical at launch volumes, but with zero buffer state to clear on commit/rollback and no nested-transaction edge cases. (Design D3/D5 say "one hook per transaction"; that is the behavioral contract ‚Äî deliver post-commit, in causal order, never on rollback ‚Äî which per-record satisfies. If a high-fan-out emitter ever makes the per-record query count matter, batching is a drop-in behind the same contract.)
- **Reading typed config at PHPStan level max (4.1)**: `config('x', 5)` is typed `mixed`, so `(int) config(...)` trips `cast.int` at level max (do NOT cast mixed). Use **`Config::integer($key, $default)`** (`Illuminate\Support\Facades\Config`, Laravel 11+) ‚Äî returns a real `int` (throws `InvalidArgumentException` if the resolved value isn't an int). Baking the default into its second arg makes a service self-sufficient BEFORE its `config/*.php` exists (`Config::get` returns the default when the key is unresolved; the later file just makes the numbers explicit + env-overridable). Siblings: `Config::string/boolean/float/array`.
- **Container-resolve a consumer by FQCN + `instanceof` narrowing (4.1, design D4)**: `$this->container->make($stringFqcn)` returns `mixed` (a plain `string`, not `class-string`, defeats the generic), so guard with `if (! $x instanceof Contract) { throw; }` ‚Äî PHPStan-narrows to the contract AND is the runtime guard for a renamed/removed consumer class. Inject `Illuminate\Contracts\Container\Container`. **Empirically the container resolves ANONYMOUS classes too** (`make($anon::class)` builds + returns it) ‚Äî so a fan-out's anonymous fake-consumer rows actually execute under the inline hook (they don't silently no-op by failing to resolve).
- **Execution-after-rollback model resync (4.1)**: when a consumer handler and its `done` flip share one `DB::transaction()` and the handler throws, the transaction rolls back; `$delivery->refresh()` in the catch resyncs the model to its committed (pre-attempt) state before the SEPARATE failure write, so `attempts` counts from the persisted value (not an in-memory mutation a rolled-back `update()` left behind, nor a stale read after a commit-time failure). In the common path the throw precedes the `update()` line so the model is already clean ‚Äî `refresh()` is the belt-and-suspenders that also covers the rare commit-time failure.
- **Named test-double consumers under `Tests\Support\` (4.1)**: PSR-4 `Tests\ => tests/` autoloads `tests/Support/Platform/*.php` (namespace `Tests\Support\Platform`) with NO composer dump and NO Pest collection (`tests/Pest.php` binds `TestCase` only `->in('Feature')`, and they have no `it()`); the container resolves them by FQCN at delivery time (named, not anonymous, = how a real module consumer resolves). They do DB-work-only (write `cache` markers for observable effects) and append to a `public static array $handled` sink for invocation count/order ‚Äî **reset the static in `beforeEach`** (the DB trait resets the DB, never PHP statics).
- **When a downstream behavior lands, re-point a prior task's tests that incidentally relied on its absence (4.1‚Üí3.4)**: 3.4's two fan-out tests asserted `pending`/`attempts 0` AFTER a committed `record()`; 4.1's `afterCommit` hook now delivers post-commit (the anonymous fake consumers resolve ‚Üí rows go `done`). Fix = assert the recorder's REAL in-transaction contract INSIDE the `DB::transaction()` closure (immune to the post-commit hook), and relax the atomicity test from a `where('status','pending')->count()` to a status-agnostic `EventDelivery::count()` (atomicity is co-persistence, not delivery status). The tests get tighter, not weaker ‚Äî they now pin exactly what each task owns.
- **Laravel 13 scheduling + explicit console-command registration (4.2, design D1/D6, all vendor-verified)**: a sub-minute schedule goes in `routes/console.php` via the `Illuminate\Support\Facades\Schedule` facade ‚Äî `Schedule::command('events:sweep')->everyThirtySeconds()->withoutOverlapping()` (`everyThirtySeconds` ‚Üí `repeatEvery(30)`, sub-minute is first-class). A CLASS-based command living OUTSIDE `app/Console/Commands` (here `App\Platform\Events\SweepCommand`) is NOT auto-discovered (`Kernel::shouldDiscoverCommands()` only loads `app/Console/Commands`, and only when `get_class($this) === Foundation\Console\Kernel` ‚Äî true since this app has no custom `app/Console/Kernel.php`) ‚Äî register it explicitly with `->withCommands([SweepCommand::class])` in `bootstrap/app.php` (the `ApplicationBuilder` partitions class-string args to `$kernel->addCommands()`). `bootstrap/` is NOT in the phpstan paths (`app`/`database`/`routes`/`tests` are) so that file isn't type-checked, but `routes/console.php` IS ‚Äî the facade chain types clean (`command()` returns `Event`, the frequency/overlap methods return `$this`).
- **Introspecting the schedule in a Feature test (4.2)**: `app(\Illuminate\Contracts\Console\Kernel::class)->bootstrap()` forces the console kernel to `require routes/console.php` (via `discoverCommands()`'s `commandRoutePaths` loop), registering the entries on the `Schedule` singleton (bound shared in `FoundationServiceProvider`, so the facade in the routes file and `app(Schedule::class)` are the SAME instance). Then `collect(app(Schedule::class)->events())->sole(fn (Event $e): bool => is_string($e->command) && str_contains($e->command, 'events:sweep'))` ‚Äî `events()` is typed `Event[]` so `collect()` is `Collection<int,Event>` and `sole(callable): Event` (non-null, **throws `ItemNotFoundException` if absent/duplicated ‚Üí finding it IS the existence proof**). Assert the public introspectable props `$e->repeatSeconds` (30) and `$e->withoutOverlapping` (bool). Guard the mixed `$e->command` with `is_string()` (NOT `(string)` cast ‚Äî mixed‚Üístring casts trip PHPStan max like `(int)` does).
- **`Artisan::call('cmd')` returns a clean `int` exit code; the Pest `artisan()` helper returns `PendingCommand|int` (4.2)**: `Pest\Laravel\artisan()` is PHPDoc'd `@return PendingCommand|int`, so chaining `->assertExitCode()` trips PHPStan max on the `int` arm ‚Äî and `$this->artisan()` is unusable in a Pest closure (`$this` is `TestCall`, like `$this->travelTo`). When a test only needs execution + exit code (no output assertions), use `Illuminate\Support\Facades\Artisan::call('cmd')` ‚Üí `int`, asserted `expect(Artisan::call('cmd'))->toBe(0)` ‚Äî one line that runs the command (bootstrapping the console kernel + loading routes/console.php) and pins success. (Keep the Pest `artisan()`/`expectsOutputToContain` for 5.1's demo-command OUTPUT assertions.)
- **PHPStan-clean time assertions on a nullable Carbon cast (4.2)**: `event_deliveries.available_at` is `?CarbonImmutable`, so a direct `$d->available_at->equalTo(...)` trips "method on nullable" at level max.
