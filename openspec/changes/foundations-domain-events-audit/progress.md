## Codebase Patterns
(consolidated reusable patterns — read first each iteration)

- **Platform substrate code lives under `App\Platform`** (design D1). Enums sit in `App\Platform\Events` (`app/Platform/Events/`). PSR-4 `App\` → `app/` already covers it — no composer change.
- **Enum convention** (mirrors `App\Modules\Module`): string-backed, PascalCase case name → snake_case backing value; a `Module`-density class docblock citing the design/spec source. Enum tests live in `tests/Unit/Platform/`, assert the full case→value map verbatim with `->toBe([...])` (order-sensitive), and add one `from('<bad>')` → `ValueError` failure case. Mirror `tests/Unit/Modules/ModuleTest.php`.
- **Boundary-law amendment for a NEW platform root** (the only legitimate edit to the hardcoded `$platformNamespaces` array): add the root string to `$platformNamespaces` in `tests/Architecture/ModuleBoundariesTest.php` with a design-citing comment, then **prove non-vacuity with a red-proof** — a temp class under the new root type-referencing a module symbol must turn the suite RED naming the exact pair, then removal returns it GREEN with a clean `git status`. This is mandatory (lessons.md 2026-06-12; design Risk "vacuously-green arch tests"). F1 3/3 adds `App\Platform\Money` etc. *under the same already-listed `App\Platform` root* — no further amendment needed once `App\Platform` is in the array.
- **pest-plugin-arch RED failure message format**: `Expecting '<source>' not to use '<target>'.` — useful to grep for when asserting a red-proof named the right pair.
- **Test runner emits JSON** (one line per run: `{"tool":"pest","result":...,"tests":N,...}`). A failure includes a `failures[]` array with `test`, `file`, `line`, `message`. Tail/grep accordingly; don't expect Pest's pretty TTY output.
- **Per-file `RefreshDatabase`**: the global Pest binding stays commented — DB-touching test files opt in with `uses(RefreshDatabase::class)` per-file (tasks.md standing rule). 1.1 needed no DB so EnumsTest is a pure unit test.
- **Postgres-truthful migration idiom** (design D2): use the fluent column types as-is — on the SQLite dev/test lane they fall back behavior-preservingly (`jsonb()`→`text`, `timestampTz()`→datetime `text`, `uuid()`→`varchar`; all identical under the Eloquent array/datetime casts). A constraint only PostgreSQL can express (e.g. an `actor_role` CHECK) goes in an `if (DB::getDriverName() === 'pgsql')` branch via `DB::statement(...)` **after** `Schema::create()`, with the SQLite fallback (NOT NULL + the enum cast) documented inline. **Derive the CHECK's enumerated values from the enum's `::cases()`** so the DB constraint can never drift from the PHP enum. The PG branch is unexercised on SQLite — the `pgsql` CI lane (task 5.2 / first human push) is its proof, by design D8.
- **Portable schema assertions**: `Schema::hasColumns($t, [...])` → bool for column presence; `Schema::hasIndex($t, [col, col, ...], 'unique'|null)` matches on the **column array** (order-sensitive — verified in `Builder::hasIndex`, compares `$value['columns'] === $index`), so you assert composite/unique indexes **without** knowing Laravel's generated index name. Pair structural checks with behavioral ones: a complete-row `DB::table()->insert()` for happy path; `->throws(QueryException::class)` for NOT NULL / unique violations (never engine-specific SQLSTATEs); a self-referencing FK is proven by inserting a child row whose FK column points at a parent's `id`. A row-builder `function fooRow(array $overrides = [])` keeps the complete-valid-row DRY while each test drops/changes exactly one field.
- **PHPStan level-max in test files** (two recurring traps): (1) plain helper functions need iterable-value types — `@param array<string, mixed>` + `@return array<string, mixed>`; (2) never cast a DB-query result to a scalar — `->first()` is `stdClass|null` (a prior `expect()->not->toBeNull()` does NOT narrow it for PHPStan) and `->value()` is `mixed`, so `(int) $row->col` trips `cast.int`. Assert DB values with Pest `->toEqual(...)` (loose ==, accepts `mixed` with no cast) instead of casting.
- **Partial index = raw `DB::statement('CREATE INDEX <name> ON <t> (<col>) WHERE <predicate>')`** (design D2). The installed Laravel Blueprint has NO fluent partial-index predicate — `IndexDefinition` exposes only `algorithm/deferrable/initiallyImmediate/language/lock/nullsNotDistinct/online`, and both grammars' `compileIndex` emit a plain `create index … (cols)` with no `WHERE` (verified in vendor 2026-06-12). The raw `CREATE INDEX … WHERE` syntax is **identical and valid on BOTH SQLite and PostgreSQL**, so one statement serves both engines (unlike the PG-only `actor_role` CHECK). Give the index an **explicit stable name** and derive any literal in the predicate from the enum (`DeliveryStatus::Pending->value`) so it can't drift. **Assert existence by NAME**: `Schema::hasIndex($t, '<name>')` — `Builder::hasIndex` matches `$value['name'] === $index || $value['columns'] === $index`, so a name string works and is portable (both engines' `getIndexes()` return the index by name). Caveat: `getIndexes()` surfaces name+columns but **NOT the partial predicate** on either engine, so the name-existence assertion is the portable proof; partiality itself is carried by the migration DDL (a malformed predicate aborts the migration → every test in the file errors, so green ⇒ valid partial DDL on the running engine). The pgsql lane (5.2) re-proves the PG side.
- **Cross-engine immutability triggers** (design D7) — raw DDL via **`DB::statement`, NOT `DB::unprepared`**: Larastan types `Connection::unprepared()`'s `$query` as `literal-string` (an injection guard), so an interpolated clause (`"… ({$changed}) …"`, where `$changed` is `implode(array_map(...))`) trips `argument.type` at PHPStan max. `Connection::statement($query, $bindings=[])` is **not** literal-string-constrained (it carries a bindings param) — exactly why the table migrations (000001/000002 `actor_role` CHECK, 000003 partial index) already use it for interpolated driver-branched DDL. Match them: one statement per call, **no trailing semicolon** (a single command per call dodges PG's "cannot insert multiple commands into a prepared statement"; the body's own `;` are safe — inside SQLite `BEGIN … END` or PG `$$…$$`). Verified (prepare+execute installs a SQLite `CREATE TRIGGER` with internal semicolons and it fires). Engine dialects: **SQLite** `CREATE TRIGGER … BEFORE UPDATE/DELETE … BEGIN SELECT RAISE(ABORT, '…immutable…'); END` (single-event triggers → two for an append-only table; `RAISE(ABORT)` reverts only the offending statement, not the wrapping txn, so a test can read the row back unchanged after catching); **PostgreSQL** a plpgsql `RETURNS trigger` function (`RAISE EXCEPTION '…immutable…'`; `\$\$` is the PHP-escaped `$$` dollar-quote) fired by one combined `BEFORE UPDATE OR DELETE … EXECUTE FUNCTION`. A **partial-immutability** table (allow only some columns to change — e.g. audit `before`/`after` for GDPR redaction) uses **ONE authoritative structural-column list** (every other column) driving both branches: SQLite `WHEN (OLD.c IS NOT NEW.c OR …)` (null-safe `IS NOT`), PG `IF (OLD.c IS DISTINCT FROM NEW.c OR …) THEN RAISE; END IF; RETURN NEW`. Tests assert **behaviour only** — `QueryException` whose message `toContain('immutable')` + row unchanged/exists afterward — **never SQLSTATEs**, so one test file proves both lanes; they ARE the red-paths (no separate red-proof). `event_deliveries` gets NO trigger (mutable by design) — pin that with a positive "update succeeds" test so the trigger scope can't silently over-extend.
- **Platform Eloquent models** (design D1/D2): `$table` explicit, `protected $guarded = []` (the recorders are the only writers — no request-driven mass assignment on a substrate model), `casts()` method (Laravel 11+ style, `@return array<string, string>`). The two append-only tables (`domain_events`/`audit_records`) set `public $timestamps = false` — the `occurred_at` envelope clock is the only timestamp and the migrations have NO created/updated columns, so leaving timestamps on would make `create()` write non-existent columns (it's a correctness requirement, not a preference); the mutable `event_deliveries` KEEPS timestamps. Cast vocabulary: enum column → the enum FQCN (`ActorRole::class`/`DeliveryStatus::class`) = the application-side value-set floor on BOTH engines — reading an out-of-enum value throws `ValueError` via `BackedEnum::from()`, **uncaught by Eloquent** (`HasAttributes::getEnumCaseFromValue` → `::from()`), which IS the failure-case test; `jsonb` → `'array'`; `timestampTz` → `'immutable_datetime'` (→ `Carbon\CarbonImmutable`). **PHPStan level-max model-property access**: Larastan scans `database/migrations` by default (`databaseMigrationsPath: []` → `database_path('migrations')`) and its `SchemaAggregator` understands `jsonb`/`uuid`/`foreignid`/`unsigned*integer`/`id`/`string`/`text`/`timestamps` but **NOT `timestamptz`** (singular), so `occurred_at`/`available_at` are NOT schema-inferred — add explicit `@property` annotations (Larastan maps `immutable_datetime`→`CarbonImmutable`, annotate to match so there's no drift). `Eloquent\Model::__get` means an un-inferred property never trips `property.notFound`, but annotate anyway so `expect($x->prop)->toBeInstanceOf(...)` is a typed check, not `mixed`. **Pint's `fully_qualified_strict_types` fixer manages docblock types + imports together**: write `@property \Carbon\CarbonImmutable`, Pint adds the `use` and shortens to `CarbonImmutable` (and `no_unused_imports` keeps a docblock-only import) — don't fight it. An uncast column reads back as the raw DB type (`audit_records.occurred_at` is left uncast per the task's explicit cast list → reads as `string`, annotate it so).
- **Pest helper functions are GLOBAL across all test files** (one shared namespace) → prefix per-file helpers to avoid a redeclaration fatal: `domainEventRow` (schema test), `immutabilityDomainEventRow` (immutability), `modelsDomainEventAttributes` (models) all build a valid `domain_events` row but must NOT share a name.

SQLite: 4 `CREATE TRIGGER … RAISE(ABORT, '…immutable…')` (domain_events update+delete, audit delete, audit structural-update with a `WHEN (… IS NOT …)` guard). PostgreSQL: 2 plpgsql `RETURNS trigger` functions (`RAISE EXCEPTION '…immutable…'`) each fired by one `BEFORE UPDATE OR DELETE … EXECUTE FUNCTION`; the audit function checks `TG_OP='DELETE'` then `IS DISTINCT FROM` over the structural list and `RETURN NEW` for allowed redaction.   - `tests/Feature/Platform/ImmutabilityTest.php` — new, `uses(RefreshDatabase::class)`. 6 tests / 14 assertions: the five delta-spec scenarios (domain_events UPDATE & DELETE rejected; audit structural UPDATE rejected; audit before/after-only UPDATE allowed + structural columns untouched; audit DELETE rejected) + an `event_deliveries`-stays-mutable guard. Two row builders (`immutabilityDomainEventRow`/`immutabilityAuditRow`, names chosen to not collide with the sibling test helpers).
- **Quality loop:** green — format ✅ · ImmutabilityTest 6/6 ✅ · full suite **94/94 (366 assertions)**, +6 vs the 88 baseline ✅ · phpstan level max 0 errors ✅ · pint --test ✅ · `openspec validate --strict` ✅. `git status`: only the two new files (no protected paths); `git diff main -- composer.json composer.lock` empty.
- **Realization decision — `DB::statement` not `DB::unprepared`** (design D7/tasks.md 2.4 both name `unprepared`): `unprepared()`'s `$query` is Larastan-typed `literal-string`, so the interpolated audit structural clause (`{$changed}` from `implode`/`array_map`) trips PHPStan `argument.type` at level max. `DB::statement` carries no such constraint — and is already the codebase's established idiom for driver-branched interpolated DDL (000001/000002 CHECK, 000003 partial index). This is a realization detail (the ADR doesn't mandate a facade method), not an ADR re-decision — preserves driver-branching, both branches, the one authoritative list, and behaviour parity. No escalation. Documented in the migration docblock + Codebase Patterns above.
- **Verified before writing** (standing rule): throwaway `php -r` against `sqlite::memory:` proved (a) `RAISE(ABORT,'…immutable…')` surfaces the token in the PDOException message; (b) `IS NOT` is null-safe (NULL→value structural change caught); (c) a before/after-only UPDATE passes the `WHEN` guard untouched; (d) prepare+execute (= `DB::statement`) installs a `CREATE TRIGGER` with internal semicolons and it fires.
- **Learnings for future iterations:**
  - The schema group (2.x) is now COMPLETE — three tables + the immutability triggers. Next is 3.1 (Eloquent models). The triggers mean a model test that tries to `update()`/`delete()` a `domain_events`/`audit_records` row WILL throw — models for those two tables are effectively insert-only; only `event_deliveries` (mutable) round-trips update in tests.
