# Proposal — parties-membership-activation

## Why

This is the **demand-side activation slice of Module K (Parties)** — the first slice of the demand-side membership backend the three prior Parties slices (parties-core / parties-producer-lifecycle / parties-compliance / parties-holds) deferred. Phase 2 is backend-green but **not** complete: the demand-side status FSMs are unbuilt seams in the shipped `party-registry` spec, and the Admin Panel UI over them is the immediate next step (a separate slice). This change closes the **activation** subset of those seams.

- It implements **the one retained producer write** the Build Workplan names — *"the one retained producer write — membership approve/decline (L-PP / K-Q4)"* + *"originating-club one-shot immutability"* (`Build_Workplan_v0.3-MVP.md` § Phase 2, line 104). The producer-facing HTTP portal is deferred; the write ships as an operator/console-invocable Action (admin-parity, DEC-083).
- It closes three of the seams the shipped *Birth States Recorded, Lifecycle Transitions Deferred* requirement holds open: Customer `pending → active` (§ 4.1 / AC-K-J-1), Profile `Applied → Approved | Rejected → Active` (§ 4.2.1 / AC-K-FSM-2), and the **Originating-Club one-shot lock** (§ 6.1 / AC-K-J-4) — the `originating_club_id` no-mutation seam parties-core left.
- This is slice 1 of the change parties-holds pre-named **`parties-membership-lifecycle`**. It ships the **activation** subset only; the remaining demand-side lifecycle (`Active → Suspended | Lapsed | Cancelled | Inactive`, the **Hold→`suspended`** coupling, the **Hero Package Capacity Invariant**, customer **segments**) continues in follow-on slices (see the boundary table).

It steps through **no open ADR gate**: domain events deliver **inline** at launch (`decisions/2026-06-12-event-substrate-and-audit-store.md`), so `CustomerActivated` / `ProfileActivated` / `OriginatingClubLocked` record like the spine `*Created` events — no `queued` consumer. No object storage, no payment provider, no frontend (the producer portal is operator-console parity), no new dependency is touched. Every transition follows the established Module K lifecycle-Action idiom (`ActivateProducer` is the verbatim template).

## What Changes

- **Profile membership approve/decline (T1 — the producer write, audit-only)** (PRD § 4.2.1 / § 3.1; AC-K-FSM-2) — two operator/console Actions: `ApproveProfile` (`Applied → Approved`) and `DeclineProfile` (`Applied → Rejected`, terminal). Both are **from-state guarded** against a transaction-locked re-read (a localized `IllegalProfileTransition` on an out-of-state call). Per § 15.2 the event catalog **names no `ProfileApproved` / `ProfileRejected`** — so, exactly like the shipped KYC transitions (`RecordKycVerified` records no KYC event — the precedent), approve/decline are **audit-only**; the approve path's *only* domain event is the conditional `OriginatingClubLocked`.
- **Originating-Club one-shot lock (an in-tx side-effect of `ApproveProfile`)** (PRD § 6 / § 6.1; AC-K-J-4, AC-K-BR-OC-1/2/3, AC-K-EVT-10) — on the Customer's **first-ever** Profile approval across any Club, `ApproveProfile` (re-reading the Customer under a transaction lock) sets `Customer.originating_club_id` to the approving Club and records `OriginatingClubLocked`. **Gated on the link being currently unset** (idempotent — a second Club's approval does not re-fire or re-set), **immutable** thereafter (no admin-override surface), and **may stay unset** indefinitely for Discovery-only Customers (DEC-040). It is **not** a standalone Action (`LockOriginatingClub` / `SetOriginatingClub` stay forbidden — the codebase's own arch guards name them so).
- **Profile activation (T2)** (PRD § 4.2.1; AC-K-FSM-2, AC-K-EVT-5) — `ActivateProfile` (`Approved → Active`) records `ProfileActivated`, from-state guarded. In production the trigger is Module E's `MembershipFeePaid` (or free-club activation, no fee); Module E does not exist, so the **`MembershipFeePaid` listener is a deferred Module-E seam** — the Action is the within-module writer, invoked directly (the free-club / operator path) and in tests. No mock event class is fabricated (zero-invention — § 15.2: *"Module K consumes Module E's `MembershipFeePaid`"*).
- **Customer onboarding activation (T3)** (PRD § 4.1 / § 7.1; AC-K-J-1, AC-K-BR-Identity-3, AC-K-FSM-1, AC-K-EVT-1) — `ActivateCustomer` (`pending → active`) records `CustomerActivated`, from-state guarded, behind the **composite onboarding gate**: `email_verified_at` set ∧ `tc_accepted_at` set ∧ `privacy_accepted_at` set ∧ `sanctions_status = passed` ∧ KYC **cleared** if `kyc_required` (the four hard gates — AC-K-J-1 *"only after all four gates clear"* — plus the KYC-cleared rider, AC-K-BR-Identity-3). It is an **explicit gated Action**: a sanctions verdict does **not** auto-activate the Customer (AC-K-BR-Customer-1 *"suspension is explicit — not automatically driven"*; § 9.4 KYC/sanctions/status FSM separation). A gate-unmet call is rejected with a localized `IllegalCustomerTransition`.
- **Three demand-side activation events** (verbatim PRD § 15.1 / § 15.2 / § 15.6; AC-K-EVT-1/5/10) — `CustomerActivated` (`entity_type = Customer`), `ProfileActivated` (`entity_type = Profile`), `OriginatingClubLocked` (`entity_type = Customer`), each recorded transactionally through `DomainEventRecorder`, module `parties`, actor from `ActorContext`, with a **PII-free** payload (`OriginatingClubLocked` carries `customer_id`, `club_id`, `profile_id`, `locked_at` — § 6.1 verbatim).
- **One additive migration** — three nullable onboarding-acceptance timestamps on `parties_customers` (`email_verified_at`, `tc_accepted_at`, `privacy_accepted_at`; the gate inputs — § 4.1 *"tracked at Customer level with timestamps"*, DEC-071 additive pattern). No change to `parties_profiles` (the `ProfileState` domain + the partial-unique index already exist; the index already excludes the terminal `rejected` state, so re-application after a decline works with no index migration). New `IllegalProfileTransition` / `IllegalCustomerTransition` exceptions + localized copy. New `ProfileActivated` / `CustomerActivated` / `OriginatingClubLocked` event classes.
- **Guard-test realignment (two archived-change protected tests)** — `SupplyLifecycleChainTest` (the exact-set transition whitelist + the event-non-existence list) and `ComplianceIndependenceTest` (the forbidden-Action list + the `originating_club_id` write-scan) currently assert the **absence** of precisely the Actions/events this slice ships. They are narrowed to what **remains** deferred (e.g. `SuspendCustomer` / `ProfileApproved` / `CustomerSegmentChanged` stay forbidden/absent); the now-shipped names are removed and the OC write-scan admits `ApproveProfile`'s conditional write.
- **Docs** — extend `CONTEXT.md` with the resolved demand-side activation terms (approve/decline = audit-only; the OC one-shot lock; `ProfileActivated` on fee-paid/free-club; the `CustomerActivated` composite gate) and a Parties activation contract note documenting the three event payloads + the deferred seams below.

## Capabilities

### New Capabilities

_None._ This change extends the existing `party-registry` capability with the demand-side activation transitions; it introduces no new capability (the Module-K-is-one-capability pattern the prior slices established).

### Modified Capabilities

- `party-registry`: gains the demand-side activation FSMs (Customer / Profile) and the Originating-Club lock. **ADDED** — *Profile Membership Approval* (`ApproveProfile` / `DeclineProfile` + the OC one-shot lock), *Profile Activation* (`ActivateProfile` → `ProfileActivated`), *Customer Onboarding Activation* (`ActivateCustomer` + the composite gate → `CustomerActivated`), *Demand-Side Activation Events* (`CustomerActivated` / `ProfileActivated` / `OriginatingClubLocked`). **MODIFIED** — *Customer Identity* (the `originating_club_id` no-mutation seam closes — the one-shot lock is its mutation surface), *Birth States Recorded, Lifecycle Transitions Deferred* (narrowed — Customer/Profile **activation** + the OC lock are now implemented, while suspension / lapse / cancellation, the Hold→`suspended` coupling, the Hero Package invariant, WaitingList and segments remain deferred seams).

## Impact

- **New code** — `app/Modules/Parties/Actions/{ApproveProfile,DeclineProfile,ActivateProfile,ActivateCustomer}.php`; `app/Modules/Parties/Events/{ProfileActivated,CustomerActivated,OriginatingClubLocked}.php`; `app/Modules/Parties/Exceptions/{IllegalProfileTransition,IllegalCustomerTransition}.php` + new `profile`/`customer` lifecycle keys in `lang/en/parties.php`; one additive migration `database/migrations/2026_06_18_000002_add_onboarding_acceptance_to_parties_customers.php`; tests under `tests/Feature/Modules/Parties/` + `tests/Unit/Modules/Parties/`.
- **Modified code** — `app/Modules/Parties/Models/Customer.php` gains the three nullable acceptance-timestamp casts + `@property`; the two archived-change guard tests (`SupplyLifecycleChainTest`, `ComplianceIndependenceTest`) are narrowed to the still-deferred set. No change to `parties_profiles` or any other shipped Action.
- **One migration, no new dependency** — `git diff main -- composer.json composer.lock` stays empty; the only schema change is the **new additive** columns on `parties_customers` (nullable, no backfill, reversible `down()`); no change to `parties_profiles` (the `ProfileState` domain + partial-unique index pre-exist).
- **Reuses, does not modify** — `DomainEventRecorder` (open-transaction guard + `DomainEvent` return), `ActorContext` / `ActorRole`, the `Module` enum (`Module::Parties->value`), `ProfileState` / `CustomerStatus` / `KycStatus` (`clears()`) / `SanctionsStatus` (`Passed`), the spine Models, and the additive-migration idiom (`2026_06_17_000001`).
- **Boundary & arch** — every reference is within Module K (`ModuleBoundariesTest`); no cross-module DB access (Module A capacity + Module E `MembershipFeePaid` are deferred seams, not reads). The OC lock is a cross-**entity** write within one Action (Profile + Customer, both Module K) — the same in-module pattern as `RetireProducer` (Producer + Club) and `RecordKycVerified` (Customer + Hold).
- **DB engines** — the acceptance columns, the four transitions, and the three events all touch the database, so every DB-touching task is verified on **PostgreSQL 17** before close, in addition to SQLite (`knowledge/testing/rules.md`).
- **Deliberate traceability gaps** — each deferred concern above maps to a named future change. The §13 capacity gate (AC-K-J-13 / AC-K-XM-18), WaitingList, the `MembershipFeePaid` listener, Club Credit, Hold→suspension, segments, and the operator/producer surfaces are recorded as named seams; the Actions they will drive / consume ship ready.
